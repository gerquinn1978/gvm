GVM v2.2.0

What's changed in the plugin since v1.0.0 — four new skills, methodology hardening, and a finished mock-budget runtime across Python, TypeScript, and Go.

Cumulative release notes covering v2.0.0 → v2.2.0. If you're upgrading from v1.0.0, this is everything you'll see different. If you're already on a v2.x release, the v2.2.0 delta is at the bottom under v2.2.0 — the point release.

Four new skills

Four skills exist on the v2.x track that did not exist at v1.0.0. Each one is invoked by its slash command.

/gvm-impact-map

A discovery skill that asks the right questions before requirements gathering: what change in the world are we trying to cause, and who needs to do something differently for that change to happen? Produces an impact-map.md with four linked tables — Goals, Actors, Impacts, Deliverables. Downstream /gvm-requirements refuses to finalise any requirement that doesn't trace back to a leaf Deliverable.

/gvm-walking-skeleton

Wires a thin end-to-end execution thread through every external integration boundary — HTTP APIs, databases, cloud SDKs, file systems, subprocess, email — before feature work begins. Surfaces integration failures at hour 1 instead of month 6. Produces a runnable skeleton plus a boundaries.md registry that downstream skills gate against.

/gvm-explore-test

Time-boxed exploratory testing against built code. You write a short YAML charter (mission, tour, scope, duration); the skill runs the timer, captures defects through structured prompts, and produces a session report that /gvm-test reads when checking the verdict. Severity is your call — the skill never auto-classifies.

/gvm-analysis

Privacy-preserving exploratory data analysis. Point it at a CSV, TSV, xlsx, parquet, or JSON file and it produces a real findings report — per-column statistics with bootstrap confidence intervals, outlier detection (IQR + MAD always; IsolationForest + LOF for n ≥ 1000), time-series cadence and gap detection, driver decomposition (Decompose / Validate / Run-everything modes), privacy-safe duplicate summaries, deterministic comprehension questions, and rendered SVG charts. Raw rows never enter Claude's context — Claude orchestrates a Python engine that reads the file and returns aggregates only.

Methodology hardening

Eight rules came in across the v2.x track that change what a chunk is allowed to do. The rules apply to chunks authored on or after 2026-04-28; older chunks are exempt as "v2.0.x carry-over" so existing work continues to ship.

  • Outside-in tests come first (TDD-1). For any chunk that has a user-facing surface, the acceptance test that exercises that surface is the first deliverable in the chunk's TDD plan — written before any unit test. Pure internal helpers are exempt with a documented rationale.
  • One mock at the external boundary (TDD-2). Each test mocks at most one collaborator, and only when that collaborator sits at the network or process edge. Mocking an internal class is a violation. Eight named external-boundary categories are recognised by default; per-project extensions go in .ebt-boundaries. Targets listed in .cross-chunk-seams escalate from Important to Critical.
  • Realistic-fixture variants for known-edge domains (TDD-3). If your chunk touches data analysis, parsing, or security validation, a realistic-fixture variant ships alongside the synthetic happy-path fixture, or the chunk records a documented exemption. Synthetic fixtures encode the engineer's mental model; realistic fixtures expose the gap between the model and the world.
  • Wiring matrix gates the implementation guide. Every implementation guide carries a four-column wiring matrix: entry point, consumed modules, wiring chunk, and the chunk that demanded the producer. Empty cells refuse the impl-guide write; the matrix is the audit trail proving every built module has a path to the running product and a consumer that demanded it.
  • Phase-completion wiring audit. After every phase finishes, the build mechanically greps each entry point for the modules the matrix says it consumes. The inverse audit catches the dual failure mode — a module built but never written into the matrix.
  • Chunk-level acceptance smoke gate (Hard Gate 8). After tests pass and the wiring audit runs, but before the handover is written, the chunk runs a smoke command that exercises its user-facing surface end-to-end. Non-zero exit code blocks the handover.
  • You own finding triage (shared rule 28). Every Critical or Important finding from a review skill is presented to you before any disposition is recorded. The skill may recommend "fix / defer / dismiss" but must not decide. The forbidden patterns are spelled out and structurally tested.
  • Diff-budget tracking (NFR-2). Each chunk records a Diff budget delta; cumulative SKILL.md additions over 500 lines surface a prompt asking whether the methodology is bloating. Surface-and-track, not refusal.

Cost-optimised model selection

The pipeline now picks the right model for each step instead of using one model everywhere. Opus handles deep reasoning (architecture, diagnosis, requirements). Sonnet handles guided work (code generation, structured extraction). Haiku handles HTML rendering. Selective loading of expert references means smaller projects don't pay for context they won't use.

Better review reports

  • Defect-class panels. All review skills now use orthogonal defect-class panels (References, Contracts, Logic, Naming for code; Argument, Factual Accuracy, Prose, Scoring for documents). Each panel scans for one class of defect across every artefact — published research shows this catches 41% more unique defects than expertise-based panels at lower cost per defect.
  • Capture-recapture. After the first round, panels estimate remaining defect population from their overlap counts. Whether to run a second round becomes a calculation, not a guess.
  • Two passes per panel. Each panel makes a systematic scan and then a cross-reference scan, mitigating the well-documented effect where reviewers stop looking after the first defect they find.
  • "What Prevents a Higher Score" section. Every review report now ends with a one-sentence-per-dimension explanation of the gap and whether closing it is worth the cost.

v2.2.0 — the point release

v2.2.0 closes the last open known issue from the v2.1.x series: mock-budget detection was Python-only at v2.1.1, with TS and Go declared as forward-compat stubs. v2.2.0 wires both stacks for real.

  • TypeScript mock-budget. The detector now recognises the vitest, jest, testdouble, and sinon mock APIs — vi.mock, vi.spyOn, jest.mock, jest.spyOn, td.replace, sinon.stub, sinon.replace. The default external-boundary allowlist treats node:* builtins, major HTTP clients (axios, undici, got, ky, node-fetch), DB drivers (pg, mongodb, redis), and AI/cloud SDKs as the network edge. Internal mocks emit Important; matches in .cross-chunk-seams escalate to Critical. The wrapper-as-SUT exemption applies when the test-file stem equals the mocked target's basename.
  • Go mock-budget. Adapted to Go's idiom: the detector counts distinct test-double instances per func TestXxx(t *testing.T) body. Three orthogonal signals — types ending in Mock|Fake|Stub|Spy, gomock.NewController invocations, and types embedding testify's mock.Mock (recognised by the embedded library type, not by struct name). The default allowlist covers net/http, os, io, database/sql, context, plus AWS / Google / Anthropic / OpenAI / Redis / MongoDB / Postgres SDK package paths.
  • Per-project seam allowlists now apply on TS and Go. Previously the per-stack readers silently dropped the seam_allowlist parameter — they're now wired into the per-stack detector. Two STUBS.md entries close out as a result.

Compatibility

Non-breaking across v1.0.0 → v2.2.0. Existing projects continue to ship under the carry-over exemption (chunks authored before 2026-04-28 are not retroactively gated by the v2.1.0 hardening rules). The TS/Go mock-budget detection makes some test files emit findings that previously emitted nothing — those findings reflect protocol-drift defects the runtime previously couldn't see, not regressions in the test code.

Known limitations

  • TS detection does not handle multi-line vi.mock( declarations where the module string is on a continuation line (text-line heuristic). Full coverage would require an embedded TS parser, which the v2.x track has chosen not to take a dependency on.
  • Go test-function body extraction uses brace-depth counting that does not parse strings or comments. A } inside a string literal could misalign function bodies.
  • Go conservative recall: a struct used as a fake but named without *Mock|*Fake|*Stub|*Spy and without testify's mock.Mock embed will not be detected. Practitioner workaround: rename or extend .ebt-boundaries.

How the release was built and verified

Every release in the v2.x track was driven through the GVM pipeline itself — the plugin dogfoods its own methodology. Each chunk passed strict TDD with outside-in acceptance tests, the chunk-acceptance smoke gate, the wiring-matrix audit, and the converged independent review loop before being committed. Every release passed code review, test verification, and doc review under the three-verdict taxonomy (Ship-ready / Demo-ready / Not shippable). The build artefacts and review reports live outside this repository — what's checked in is the result.
