/gvm-tech-spec

A command for Claude Code that turns requirements and test cases into a suite of focused technical specifications, guided by named experts and calibrated to the project’s risk profile.

Overview

The /gvm-tech-spec command acts as a software architect. It reads the output of /gvm-requirements and /gvm-test-cases, assesses project complexity, then generates a suite of focused specification documents through structured conversation with user approval at every stage.

Unlike a monolithic spec, /gvm-tech-spec adapts the number and depth of documents to your project. A simple app might produce 3 specs; a complex agentic system might produce 8–10. Each document covers a distinct architectural concern.

For every spec it produces two files, always kept in sync:

  • *.html — a professionally designed document with collapsible ADR blocks and layered reading via <details> toggles
  • *.md — a Claude-friendly markdown version (flat, everything visible) that serves as the primary input for /gvm-build

All files are saved to your project's specs/ directory.

The skill captures what the system's architecture is and why each decision was made. It does not contain implementation code — that belongs to /gvm-build.

Prerequisites

/gvm-tech-spec requires two input documents from earlier pipeline stages:

  • requirements/requirements.md — produced by /gvm-requirements
  • test-cases/test-cases.md — produced by /gvm-test-cases

If either file is missing, the skill will tell you which upstream command to run first.

Using an existing spec or architecture document

If you have an existing technical specification, architecture document, or design document from another tool or process, provide it as input. The skill uses it as a seed — extracting architecture decisions, component structures, technology choices, and integration patterns that map to GVM spec structure.

The external spec accelerates the assessment and informs architecture decisions, but the full expert-guided flow still runs. Before finalising, an independent verification agent compares the GVM output against your original document, checking completeness, accuracy, hallucination, and consistency. Discrepancies are presented to you for resolution.

Pipeline order

Always run the commands in sequence: /gvm-requirements then /gvm-test-cases then /gvm-tech-spec. Each command builds on the output of the previous one.

The Six Phases

The skill proceeds through six phases, each with a user approval gate. You confirm or request revisions before the skill moves on.

Table 1 — Process phases
PhaseWhat happensYour role
1 Assessment Reads requirements and test cases in full. Identifies Architecturally Significant Requirements (ASRs), assesses complexity, and proposes a spec decomposition — which specs to write and why. Approve or adjust the proposed spec list.
2 Cross-Cutting Spec Always written first. Presents 2–3 tech stack options with trade-offs (ADR format). After your choice, covers project structure, authentication, error handling, data model overview, API and development conventions. Activates Tier 3 stack specialists. Choose the tech stack. Approve the cross-cutting spec.
3 Domain Specs Writes one domain spec at a time. Activates Tier 2a domain specialists as needed. Each spec covers purpose, ASRs, design decisions (ADRs), component design, integration points, error handling, and testing strategy. Review and approve each domain spec before the next one begins.
4 Architecture Overview Synthesises all domain specs into a high-level overview using C4 diagrams. Applies Brooks' conceptual integrity lens — does the system cohere as if designed by one mind? Flags inconsistencies. Review the overview. Flag anything that doesn't cohere.
5 Implementation Guide Decomposes the build into phases and chunks (P1-C01, P1-C02, etc.). Maps the dependency network, identifies parallelism opportunities, and defines context-loading rules for /gvm-build. The final phase must include an integration wiring chunk or product startup verification. Approve build ordering and chunk boundaries.
6 Coverage Audit Traces every requirement ID to at least one spec section, and every spec section back to at least one requirement. Flags gaps in both directions — uncovered requirements and spec sections that don't trace to requirements. When specs derive from an external document, a Phase 6b source verification runs automatically: the skill checks that the spec faithfully represents the source and flags any discrepancies. Resolve any coverage gaps.
The architecture overview (Phase 4) is written after the domain specs because it synthesises from them. It does not duplicate their content — it summarises key decisions and checks for cross-domain consistency.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load All Three Expert Tiers. Tier 1 (architecture) at session start, Tier 2a (domain) before writing any domain spec, Tier 3 (stack) after the tech stack decision — do not write spec content without loading the relevant tier first.
  2. Write Paired HTML + MD for Every Spec. Every spec file must exist in both formats before moving to the next phase — the MD version is what /gvm-build consumes.
  3. Implementation Guide is Mandatory. specs/implementation-guide.md must be written as the final spec — /gvm-build cannot proceed without it.
  4. User Approval Per Spec. Each spec must be presented to the user for approval before writing the next one.
  5. HTML Before Approval. For every spec file, write the MD first, then immediately write the HTML — present for approval only after both files exist.
  6. Wiring matrix mandatory. The implementation guide must contain a “Wiring matrix” section (entry point → consumed modules → wiring chunk). The skill refuses to write the implementation guide if any module-building chunk has no consumer chunk. Required for the downstream /gvm-build Hard Gates 6 and 7.

Phase 1: Assessment

The skill reads both input documents in full and performs three analyses:

Identifying ASRs

Architecturally Significant Requirements (Keeling) are requirements that force design decisions. Examples:

  • A "15-minute time budget" forces asynchronous processing
  • "Real-time progress" favours WebSocket or Server-Sent Events
  • "User accounts with roles" forces an authentication architecture
  • Quality attribute implications — performance, security, resilience

Complexity assessment

The skill evaluates how many distinct implementation boundaries exist, what C4 containers the system would have, and where risk is concentrated — novel, complex, or uncertain areas get deeper treatment (Fairbanks).

Decomposition proposal

Based on the assessment, the skill proposes a list of specs to write. For a simple project this might be just a cross-cutting spec and one domain spec. For a complex system it could be 6–8 domain specs plus the cross-cutting, overview, and implementation guide.

You control the scope

The proposed decomposition is a starting point. You can merge specs that feel too granular, split ones that feel too broad, or remove domains you don't need specified.

Adaptive Decomposition

The skill follows George Fairbanks' risk-driven architecture principle: invest specification effort where risk is highest, and keep things light where patterns are well-understood.

Table 2 — Decomposition scales with risk
Risk levelTypical specsDepth
Low (CRUD, well-understood patterns)1–2Light — pattern names and variations only
Medium (multi-component, some novelty)3–4Moderate — ADRs for key decisions, component sketches
High (distributed, novel domain, many integrations)5–8Thorough — full component design, sequence diagrams, error handling
Fairbanks' Just Enough Software Architecture argues against both under-specification (cowboy coding) and over-specification (Big Design Up Front). The sweet spot is risk-driven: thorough where it matters, light where it doesn't.

Three-Tier Expert Model

The skill draws on a structured panel of experts organised into three tiers. Tier 1 is always active. Tiers 2 and 3 activate conditionally, and the skill announces each activation so you know which perspectives are guiding the current spec.

The architecture panel is loaded from the architecture-specialists.md reference file. The experts govern the overall process, decomposition, and cross-cutting decisions. The panel can be updated by editing the reference file.

Tier 3 — Stack Specialists (Post-Decision)

Tier 3 is loaded selectively from per-stack files in references/stack/ and activates in Phase 2 after you choose the tech stack. The roster is organised by technology area — React specialists, Python specialists, database specialists, and so on. Only specialists relevant to the chosen stack are loaded.

The roster is intentionally extensible. New stacks and specialists can be added by editing the reference file.

Architecture Decision Records

Every significant design choice is captured as an Architecture Decision Record (ADR), following Michael Keeling's format from Design It!. ADRs appear throughout the specs as collapsible blocks in the HTML version.

Each ADR contains four sections:

  • Context — the forces at play and the requirements driving the decision
  • Options Considered — 2–3 viable approaches with trade-offs
  • Decision — the chosen option and the reasoning
  • Consequences — what follows from the decision, including any new constraints
ADR-001: Use WebSocket for real-time progress updates
Status: Accepted

Context:
  PL-9 requires real-time progress view.
  PL-14 requires agent transparency.

Options Considered:
  1. Polling — simple but latency and load trade-off
  2. Server-Sent Events (SSE) — unidirectional, simpler
  3. WebSocket — bidirectional, supports user interjection

Decision:
  WebSocket. Bidirectional communication required for
  PL-15 (agent questions) and PL-16 (user interjection).

Consequences:
  Need WebSocket server infrastructure. Connection
  management for long-running sessions (~20 min).
ADR format follows Keeling (Design It!, Ch. 13) and Nygard's original lightweight format. The key insight: capture why, not just what. Six months later, the ADR tells you what alternatives were considered and why they were rejected.
When ADRs are written

Technology choices, architectural patterns, data model decisions, integration approaches, security decisions. The tech stack decision in Phase 2 is always the first ADR. Frontend specs include additional ADRs for UI component library, styling, form management, and other tooling.

Frontend and UI specs

When a domain spec covers a frontend or client-side application, it must also include ADRs for tooling decisions. The goal: a developer reading the spec should know exactly which packages to install and which patterns to follow for every UI concern.

  • UI component library — headless (Radix, Headless UI) vs opinionated (MUI, Chakra) vs copy-paste (shadcn/ui)
  • Styling approach — utility-first (Tailwind), CSS Modules, CSS-in-JS
  • Form management — library (React Hook Form, Formik) with schema validation (Zod, Yup)
  • Notification/toast system — established library (Sonner, react-hot-toast) vs custom
  • Icons — consistent icon set matching the UI library
  • Design patterns — multi-step forms, optimistic UI, skeleton loading, confirmation dialogs
No wheel reinvention

Frontend tooling decisions are architectural decisions. They are captured in the spec as ADRs, not deferred to the build phase. Every library choice is made here so that /gvm-build can proceed without ambiguity.

The Implementation Guide

The implementation guide (Phase 5) bridges /gvm-tech-spec and /gvm-build. It decomposes the entire build into phases and chunks using a structured ID system.

Phase-Chunk ID system

Every unit of work gets a unique ID: P{phase}-C{chunk}. For example, P1-C01 is Phase 1, Chunk 1. These IDs provide unambiguous references across the implementation guide, /gvm-build sessions, and parallel agent dispatching.

Phase numbering — collision detection

Before assigning phase numbers, /gvm-tech-spec scans build/handovers/ and any prior specs/*implementation-guide*.md for in-use P{N} numbers. The new guide starts at max(in_use) + 1. This prevents collisions in build/handovers/ and build/prompts/ when a project produces multiple implementation guides over its lifetime — for example, an improvement guide written after the original build completed. Cross-skill phase references (e.g., "/gvm-requirements Phase 5") are not shifted — they refer to other skills' fixed phase numbers, not this guide's build phases.

Dependency network

The guide maps dependencies between chunks as a network (not a hierarchy), represented in three complementary formats:

  • Dependency matrix — a table showing what each chunk depends on, enables, and can run in parallel with
  • Two-track ASCII diagram — backend and frontend tracks side-by-side with cross-links
  • Critical path — the longest sequential chain that determines minimum build time

Test co-location

Every chunk includes its tests. There are no separate "write tests" chunks. Unit tests are created alongside the code they test — never as a separate phase.

Parallel work

Chunks that share no mutable state can run in parallel. The guide marks these explicitly. The best parallelism opportunities are typically domain agents (same structure, different data sources) and independent frontend features.

Chunks are sized for a single Claude context window. Each chunk should be implementable by loading: the implementation guide + the cross-cutting spec + one domain spec section. This keeps /gvm-build sessions focused.

Chunk sizing principles

Chunk boundaries follow the INVEST criteria (Cohn): each chunk should be Independent, Negotiable, Valuable, Estimable, Small, and Testable. A chunk that cannot be evaluated on all six axes is a candidate for splitting or merging.

Chunks cut vertically through all layers — data, service, API, and UI — for a single feature or capability. Slicing horizontally (all data models first, then all services) produces chunks that deliver no testable value until the final layer is built. Vertical slices keep every chunk shippable.

MVP-first ordering (MVP-1, where MVP = Minimum Viable Product). The first user-facing chunk in the implementation guide must constitute the smallest end-to-end shippable slice — exercising one user-visible behaviour through every layer it touches against the walking skeleton's wired boundaries. Subsequent chunks add slices, not layers. A guide whose first user-facing chunk is a layer fragment ("data model only", "API endpoint with no caller", "UI shell with no backing data") is refused at impl-guide write time and again at /gvm-build read time (Hard Gate 9). The pattern this prevents: Phase 1 is "all auth and data model", Phase 2 is "all services", Phase 3 is the first user-visible behaviour — vertical slicing within chunks but layer-first thinking across the sequence. The walking skeleton handles boundary wiring; everything after should compound user-visible value, not infrastructure depth.

Four named project shapes claim exemption with the literal in-guide form MVP-1 exempted: <category> — rationale: <why>: library (no user-facing surface — SDK or daemon consumed by other systems), refactor (stable interface, implementation-only change), performance-driven (the architectural constraint is the product; first slice must exercise it rather than a token user-visible behaviour), and fully-specified (regulatory/compliance rebuild where user feedback adds no information). An exemption with no rationale is treated as no exemption (Fagan: silent skips are not skips).

The refusal rule gates the shape of the first user-facing chunk. A non-gated companion paragraph in the skill prose directs practitioners to optimise chunk sequencing for the earliest runnable MVP — position is not gated because that would force exemption inflation on legitimate infrastructure-first cases (significant data migration, auth bootstrapping, regulatory setup). The judgment call is which infrastructure is genuinely required by the MVP versus speculative scaffolding for later phases.

Horizontal architecture, vertical sequence. MVP-1 governs the delivery sequence; the cross-cutting spec (Phase 2) and the architecture overview (Phase 4) govern the patterns every slice is built against. Slice 5 uses the same data-access conventions, error-handling approach, logging discipline, and naming rules as slice 1 — Brooks's conceptual integrity. The first slice exercises one user behaviour through the architecture the spec already defined; subsequent slices inherit those patterns rather than reinvent them. The walking skeleton establishes the boundary contracts; the cross-cutting spec establishes everything between the boundaries. MVP-1 does not change either — it changes the order in which user-visible value reaches the practitioner. A reader who interprets "vertical slice first" as licence to invent slice-local architecture is misreading the rule: if slice N's data-access pattern diverges from slice 1's, the divergence is a code-review finding regardless of whether each slice individually shipped a green smoke gate.

When a requirement varies by data type (e.g., different asset classes, different message formats), split by data variation rather than by operation. Each variation becomes its own chunk, sharing a single code path. Kleppmann's data-intensive application principles apply: model the variation in data first; the operations follow.

Parallel chunks must share nothing at runtime — no shared files, no shared mutable state. Interface contracts (agreed request/response shapes) serve as the synchronisation point between parallel chunks; they are defined before parallel work begins and held to until integration.

When a chunk involves significant uncertainty (novel algorithm, unfamiliar library, unclear performance characteristics), designate it a spike chunk. A spike has a time-box and a defined question to answer, not a deliverable to ship. The spike's output is a decision, which feeds a follow-on implementation chunk.

The final phase must always include an integration closure chunk: wiring all deferred seams, verifying the assembled system starts cleanly, and confirming that end-to-end smoke tests pass. Integration problems caught here are far cheaper than those discovered after hand-off.

Stale Detection

If specs already exist when you run /gvm-tech-spec, the skill compares file timestamps. If requirements are newer than the existing specs, it will:

  • Notify you that requirements have changed since specs were last generated
  • Identify which specs are impacted by the changed requirements (by matching requirement IDs)
  • Offer selective regeneration (only affected specs) or full regeneration
  • Preserve specs for unchanged domains

Context & Session Management

Spec generation is a long-running task. The skill proactively manages context to maintain output quality throughout the process:

  • After every 3 domain specs, the skill recommends /compact to free context
  • Progress is saved to a handoff memory file after every spec completion — cheap insurance against context loss
  • Sessions can be resumed — the skill reads the handoff file and skips completed phases
  • Context loading is selective — each phase loads only what it needs, not the full document set
Long projects

For complex projects with many domain specs, you may run /gvm-tech-spec across multiple sessions. The handoff memory ensures no work is lost and the skill picks up exactly where it left off.

Where This Fits

/gvm-tech-spec is the third step in the skill pipeline. It consumes requirements and test cases, and produces specs that drive implementation: 

[/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec
  → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review
  → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Optional phases shown in [ ].

The requirements document captures what the system must do. The test cases define how success is measured. The tech spec captures how the system is designed and why each architectural decision was made. The build command implements it.

Specification writing often reveals missing requirements or untested scenarios. When this happens, update the upstream documents before continuing. The feedback loop is the methodology working as designed.

The coverage audit in Phase 6 may reveal requirement gaps. When it does, update requirements.md and test-cases.md before finalising specs. This upstream feedback is expected and valuable.

Tips

  • Engage with tech stack decisions. Phase 2 presents 2–3 viable options with trade-offs. This is the most consequential decision in the process — take your time.
  • Challenge the ADRs. If a technology decision doesn't feel right, say so. The skill will reconsider with your input and update the record.
  • Trust the decomposition but question it. The proposed spec list is based on complexity analysis, but you know your project best. Merge or split specs as needed.
  • Push back on over-specification. If a domain feels over-detailed for a well-understood pattern, say so. The skill respects Fairbanks' "just enough" principle.
  • Open the HTML files as you go. The specs use collapsible sections and Tufte-style design. Seeing the formatted output helps you evaluate completeness.
  • Watch for expert activations. The skill announces when Tier 2a or Tier 3 specialists activate. This transparency helps you understand which perspectives are shaping each spec.
  • Review the implementation guide carefully. This is what /gvm-build will follow. The chunk ordering and dependencies determine the build sequence.
  • Don't skip the coverage audit. Phase 6 catches requirements that slipped through and spec sections that don't trace to any requirement.

Key Rules

The skill follows these principles consistently:

  1. Requirements drive decomposition — spec structure emerges from requirements complexity, not a fixed template
  2. Cross-cutting first, always — every domain spec references it; it must exist before domains are written
  3. One domain spec at a time — keeps context focused; user approves each before the next
  4. ADR-style decisions — every significant choice captures context, options, and rationale
  5. Depth proportional to risk — thorough for novel/risky, light for well-understood
  6. Every write touches both files — HTML and Markdown always in sync
  7. Specs reference requirements by ID — full traceability; coverage audit catches gaps
  8. Specs reference test cases — testing strategy links to relevant TC-IDs
  9. No implementation code — specs describe what and why, not exact code
  10. Announce expert activations — transparent about which specialists guide each spec
  11. Diagrams wherever established — ERDs, sequence diagrams, state machines, C4 containers. Not optional.
  12. Frontend specs include library ADRs — a developer should know which packages to install
  13. Mandatory API boundary contracts — every endpoint crossing a boundary (backend→frontend, service→service) must specify the exact response shape with a concrete JSON example. Prose descriptions of “returns user data” are not sufficient. Both sides of every boundary build from the same literal contract.
