/gvm-init

The entry point for greenfield projects — calibrates the expert roster across all tiers before any pipeline work begins.

Overview

/gvm-init is the fast entry point for greenfield projects. It asks one question — your business domain — then automatically scores experts, initialises the activation log, and writes a calibration report. The entire flow takes under a minute for the common case. Run it once at the start of a greenfield project, before /gvm-requirements.

For existing codebases, use /gvm-site-survey instead. It diagnoses the codebase and selects experts based on what it finds.

Application Brief (Optional)

Before expert calibration, /gvm-init offers to create an Application Brief — a structured one-page project justification. The brief answers: what problem does this solve, for whom, and is it worth doing?

When to use: recommended for organisations that gate project starts, end-user computing scenarios where you want approval before resources are committed, or when the project’s justification is unclear. Skip it when the project is already approved and the business case is obvious.

The brief feeds into downstream skills: the problem statement seeds /gvm-requirements, the architecture direction constrains /gvm-tech-spec, and the constraints become requirements constraints.

When to Use

  • Specialised domains (financial services, healthcare, safety-critical, regulated) — init loads industry domain experts before requirements start, ensuring the elicitation is grounded in the right frameworks from the first question. It may also surface the need to replace default Tier 1 process experts.
  • Re-calibrating experts after significant changes to your project’s domain
  • Adding industry domain experts for a new business area
When you can skip init

For general-purpose software with no specific industry domain, every pipeline skill automatically loads shared rules, discovers missing experts, and scores unscored ones on the fly. You can go straight to /gvm-requirements. Init adds a calibration report and an explicit domain declaration, but no functionality that downstream skills don’t already handle.

Pre-check

If an existing expert setup is detected (activation log entries, scored experts, custom industry files), init will warn you before overwriting. You can start fresh, alter specific steps, or cancel.

The Init Flow

Init is fast: one question for the common case, then automatic setup. The flow has three phases.

Table 1 — Init flow
PhaseWhat happens
Pre-check Detects prior initialisation. If found, asks whether to start fresh, alter, or cancel. If no prior setup, proceeds automatically.
Business domain The one question you answer. Select your business domain (financial services, healthcare, technology, general-purpose, or other). Init uses this to select industry domain files and, for specialised domains only, may ask a follow-up about adjusting Tier 1 process experts.
Automatic setup Bootstraps the GVM home directory, initialises the activation log, and writes the calibration report. Expert scoring happens during roster assembly (Step 2). No further interaction required.
Stack and component domains are handled later

Technology stack specialists activate during /gvm-tech-spec when the stack is chosen. Component-level domains emerge during /gvm-requirements and are handled by expert discovery at that point. Init does not ask about either — the information is not available yet.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Expert References. Read architecture-specialists.md and the relevant industry domain file before assessing expert coverage.
  2. Score Unscored Experts. If any loaded experts lack scores, score them automatically — do not leave experts unscored.
  3. Expert Roster Table is Mandatory. The output must include an Expert Roster Summary table — downstream skills consume it to assemble their expert panels.
  4. HTML Before Approval. Write the init MD file first, then immediately write the HTML — both must exist before presenting the init report for user review.

Output

Init produces paired HTML and Markdown files: 

init/init-001.html    init/init-001.md
init/init-002.html    init/init-002.md
...

Each run produces a new numbered pair. The report has six sections:

  1. Executive Summary — project name, date, business domain
  2. Process Experts (Tier 1) — defaults confirmed or replacements created, with rationale
  3. Industry Domain (Tier 2b) — files selected or created, coverage assessment, any newly discovered experts with scores
  4. Expert Roster Summary — single table of all active experts: name, work, tier, classification, reference file, status
  5. Activation Log — confirmation that the log was initialised or already existed
  6. Next Steps — direction to run /gvm-requirements, with a note that stack specialists and component domains will be configured as they emerge during the pipeline
The MD version is the primary input for downstream skills. The expert roster summary tells /gvm-requirements, /gvm-tech-spec, and other skills which experts are active for this project.

Pipeline Position

GVM v2.2 pipeline [/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Init is the precondition for greenfield projects. After completing init, proceed to /gvm-requirements to begin gathering requirements with the calibrated expert panel.
