/gvm-test

Verifies a completed build works as a product — the acceptance gate between “code exists” and “product works.”

Overview

/gvm-test runs after all build chunks are complete. It is not a test suite — it verifies that the product works end-to-end as a user would experience it. It catches the integration gaps that per-chunk TDD misses: unresolved seams between chunks, services that fail to start, UI flows that hit placeholders, and requirements that were stubbed rather than implemented.

Per-chunk tests verify that each piece works in isolation. /gvm-test verifies that the pieces work together.

When to Use

  • After all build chunks are complete and /gvm-code-review findings have been addressed.
  • After fixing code review findings, to confirm fixes did not break integration.
  • When you need to verify that the product works end-to-end as a user would experience it.

What It Does

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Verify Code Review Exists. In full mode, stop if no code review file exists or if unresolved criticals remain. In light mode, note the absence and proceed.
  2. Run Actual Commands. Every verification step (startup, health checks, smoke tests) must execute real commands — assumed success is not verification.
  3. Write HTML Output. The test report must produce test/test-{NNN}.html. Verification reports are HTML-only — findings are carried forward in reviews/calibration.md.
  4. Write HTML Before Presenting Results. The HTML must exist before presenting results.

Eight steps (seven mandatory, one optional), in order:

Table 1 — Verification steps
StepWhat it checks
1. Prerequisite checkVerifies all upstream artefacts exist: implementation guide, build handovers, test cases, requirements, code review report, and git repository.
2. Integration seam auditGreps for NotImplementedError, TODO, “deferred to integration”. Classifies each as blocking or non-blocking. Blocking seams are fixed before proceeding.
3. Startup verificationStarts infrastructure — docker compose (or equivalent), database migrations, backend server, frontend dev server. Verifies health endpoints respond before proceeding.
4. Acceptance test verificationReads test-cases/test-cases.md and verifies every test case against the built code. Classifies each as PASS, FAIL, BLOCKED, or STUB.
5. Mutation testing (optional)Offered after acceptance tests pass. Runs mutation testing on critical-path files only (MUST-priority requirements, security-sensitive code). Reports mutation score and flags surviving mutants as test gaps. Uses stack-specific tools from stack-tooling.md.
6. Smoke runBeyond test cases — verifies user experience. Starts infrastructure, hits health endpoints, walks through pages and interactive elements.
7. Stub vs implemented auditClassifies every requirement as IMPLEMENTED, PARTIAL, STUB, or NOT IMPLEMENTED. Mock providers and placeholder data are labelled as stubs.
8. Declare completeIf all checks pass, the build is complete. If stubs remain, the audit table is presented and the user decides what ships.
Mutation testing is optional because it can be slow on large codebases. It targets only critical-path code — MUST-priority requirements and security-sensitive logic — not the entire codebase.
The user decides what ships

If stubs remain, the audit table is presented and the user decides: wire real implementations, or ship with documented limitations. The build is only “complete” when the user explicitly accepts the current state.

Output

Produces an HTML-only report (per shared rule 13 exception — review/verification reports do not produce paired MD). Findings are carried forward in reviews/calibration.md: 

test/test-001.html
test/test-002.html
...

The report contains up to eight sections: executive summary, integration seam audit, code review results, acceptance test results, mutation testing results (if run), smoke run results, requirement coverage audit, and verdict.

Verdict Vocabulary

Every /gvm-test run ends in exactly one of three verdicts. The taxonomy is a finite state, not a sentence the skill assembles — the skill emits whichever of the three values the evaluator returns.

VerdictMeaningGating criteria
Ship-ready The build is fit for production. Real chains exercised, no unresolved code-review criticals, no expired stubs, no blocking integration seams, no Critical-severity exploratory defects in non-stub paths. VV-2 criteria all pass (and the OQ-5 manual-gate override has not downgraded the verdict — see callout below).
Demo-ready The build is fit to show, but not to release. Stubs may remain (registered, in-date), some integration paths may use sandbox endpoints, or the walking-skeleton CI gate was satisfied manually rather than via real CI confirmation. VV-2 fails on at least one bar; VV-3 criteria all pass.
Not shippable The build does not yet work as a product. Blocking seams are open, MUST-priority test cases fail, expired stubs are present, or critical code-review findings are unresolved. Any VV-4 criterion triggers a blocking condition; OR VV-2 fails on at least one bar AND VV-3 also fails (fall-through).
Retired vocabulary

The earlier two-tier verdict that used a partial-credit middle option has been retired across the plugin. The three-verdict taxonomy replaces it. A release-gate verifier flags the retired phrase if it reappears anywhere in the plugin tree.

OQ-5 manual-gate override

When the walking-skeleton CI gate was satisfied manually (no real CI confirmation), VV-2 may all pass yet the verdict is downgraded to Demo-ready. The skill prompts the practitioner; if the prompt is dismissed (non-interactive run), the default is Demo-ready. The override applies only on the VV-2-pass branch — once VV-3 or VV-4 are in play, the regular decision table governs.

Because the verdict is a fixed label, the skill cannot embed nuance in it. Practitioners who want to communicate "ships with caveats" record the caveats in the report's executive summary or the requirement coverage table — not by softening the verdict.

Pipeline Position

Greenfield [/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Optional phases shown in [brackets].

/gvm-test sits after code review. The build produced code, /gvm-code-review reviewed it for quality, and /gvm-test verifies the reviewed code works as a product. Fix code quality issues before investing in integration testing and smoke runs.

After /gvm-test completes, run /gvm-doc-write in pipeline docs mode to generate user-facing documentation (README, user guides, API docs) that reflects the verified build. Then run /gvm-doc-review to audit the documentation quality.

/gvm-test checks that /gvm-code-review has been run before starting. If no code review report exists, it tells you to run one first. Light mode is offered automatically when the skill detects a small build scope — the skill asks which mode to use via a question prompt. In light mode, the code-review prerequisite is waived and integration seam audit and stub audit are skipped if the change is localised.

Troubleshooting

Common issues and recovery (Nielsen H9)
ProblemCauseFix
Infrastructure fails to startMissing dependencies, port conflicts, environment not configuredCheck the README setup instructions. Run the startup command manually to see the error. Fix the environment before re-running /gvm-test.
Acceptance tests cannot executeTest runner not configured, or test-cases.md references features not yet builtCheck that the test framework is installed (step from /gvm-build P1-C01). For unbuilt features, /gvm-test classifies them as NOT IMPLEMENTED — this is expected.
Code review not found/gvm-code-review has not been run yetRun /gvm-code-review first. /gvm-test requires it as a prerequisite.
All tests pass but stubs remainTests are passing against mock/stub implementationsThis is the stub audit’s job — it flags stubs honestly. The user decides whether to ship with stubs or wire real implementations.
