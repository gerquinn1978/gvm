/gvm-doc-write

Expert-grounded document creation and revision — presentations, strategy documents, newsletters, training materials, status reports, whitepapers, and any prose document. Operates standalone or as the documentation phase after /gvm-test in the pipeline.

Overview

The /gvm-doc-write command creates and revises documents grounded in named experts. Where the pipeline skills (/gvm-requirements → /gvm-build) produce software artefacts, this skill produces and maintains documents that communicate about, support, or complement the software.

Two modes:

  • Create — write a new document from scratch, grounded in relevant experts
  • Revise — fix issues in an existing document, using the experts who identified the problems to guide the fixes
Pipeline artefacts (requirements, test cases, specs) are created by their respective pipeline skills. /gvm-doc-write handles standalone documents at any point, and also serves as the pipeline documentation phase after /gvm-test for user-facing docs like README and user guides.

Each document type has a specific expert roster drawn from the design system reference files. The experts are not decorative — they determine the structure, principles, and quality criteria applied to every document. A presentation built by presentation design experts reads differently from one built by copying bullet points into slides.

The skill handles any prose document that needs expert-grounded quality — whitepapers, origin stories, blog posts, and other written content, not just the typed categories below. The document types table provides structured guidance for common formats; documents outside these types still get writing experts, domain experts, and the full quality process.

Where This Fits

This skill operates in two modes: standalone for documents created at any point, and pipeline docs as the documentation phase after /gvm-test.

Standalone (any point) /gvm-doc-write → presentations, strategy, newsletters, training, status reports, whitepapers

In pipeline docs mode, the skill runs after /gvm-test completes and generates user-facing documentation (README, user guides, API docs) grounded in the verified build and pipeline artefacts. The output is then audited by /gvm-doc-review.

In standalone mode, the skill creates or revises documents at any point — presentations, strategy documents, whitepapers, newsletters, and any prose document. If the document concerns a project with pipeline artefacts, the skill reads those for context. Consistency with pipeline artefacts is maintained even though the document itself is not a pipeline artefact.

Quality gate

After creating a document with /gvm-doc-write, run /gvm-doc-review to audit it against expert criteria. The doc-review skill auto-detects presentations, strategy documents, and other types produced by doc-write.

Getting Started

Run /gvm-doc-write and describe what you need. The skill detects the document type from your request:

Create mode

/gvm-doc-write create a presentation on our Q2 architecture changes
/gvm-doc-write write a strategy document for the platform migration
/gvm-doc-write draft a newsletter for the engineering team
/gvm-doc-write build training materials for the new onboarding flow
/gvm-doc-write write a status report for the steering committee
/gvm-doc-write write a whitepaper on our attribution methodology

Revise mode

Triggered when the request references an existing document and an issue to fix: 

/gvm-doc-write fix the critical issues from the whitepaper review
/gvm-doc-write fix the structure of section 3 in the whitepaper
/gvm-doc-write improve the origin story

In revise mode, the skill reads the existing document, identifies the issues (from review findings, an explicit description, or its own assessment), loads the experts who diagnosed those issues, and applies their frameworks to fix them. The principle: experts who find should fix.

If the document type is ambiguous, the skill asks. You can also run it with no arguments and answer the prompts: 

/gvm-doc-write

The skill then asks for the document type, subject, audience, purpose, and scope before loading experts and drafting the structure.

The audience question matters. Minto's Pyramid Principle says the reader's existing knowledge determines whether to lead with the answer (deductive) or build to it (inductive). A technical audience gets the conclusion first. A sceptical audience may need the evidence first.

Presentations

Self-contained HTML slide decks using the slide template from the design system. The template provides CSS-based slide separation, keyboard navigation, a slide counter, and presenter notes.

Structure is governed by complementary expert concerns:

  • Content structure — story-first, three-act structure, one idea per slide, headlines as complete sentences (not topic labels)
  • Narrative arc — sparkline between "what is" and "what could be," audience as hero, contrast creates meaning
  • Visual design — simplicity, signal-to-noise ratio, full-bleed images, restraint over decoration
  • Data slides — data-ink ratio, small multiples, tables for precision, charts for trends
  • Document structure — lead with the answer, pyramid structure, MECE grouping
One idea per slide is non-negotiable. If a slide has two ideas, it becomes two slides. Headlines are sentences that state the slide's message, not topic labels like "Q2 Results."

Strategy Documents

Structure follows the kernel from Rumelt:

  1. Diagnosis — what is going on? What is the critical challenge (the crux)?
  2. Guiding policy — the approach to dealing with the challenge. Not goals — an actual approach.
  3. Coherent actions — coordinated, mutually reinforcing steps that implement the policy.

If the strategy needs operationalisation, the five cascading choices layer in: winning aspiration, where to play, how to win, capabilities required, management systems.

The skill checks for bad strategy patterns: fluff (buzzwords masquerading as insight), failure to face the challenge, goals disguised as strategy, and incoherent actions pulling in different directions. These are flagged explicitly.

Newsletters

  • Open with a hook that earns the next sentence
  • One core message per newsletter — do not try to cover everything
  • Utility: the reader should be able to do something with this information
  • Specific over generic — concrete examples, real numbers, named things
  • Close with a clear call to action or takeaway
  • Keep it scannable: short paragraphs, subheadings, bold key phrases

Training Materials

  • Start with learning objectives — what will the learner be able to do?
  • Identify the gap type: knowledge, skill, motivation, or environment. Training only fixes the first two.
  • Each module: concept → example → practice → assessment
  • Apply multimedia principles: words + relevant images, spatial contiguity, no extraneous content
  • Include practice activities — not just information. The learner must do something with the knowledge.
  • Provide job aids (checklists, quick-reference cards) where reference is better than memorisation

Status Reports

  • Lead with the headline: what is the most important thing the reader needs to know?
  • Situation-complication-resolution for the overall status
  • Traffic light (red/amber/green) for at-a-glance status where appropriate
  • Risks and decisions needed — prominently positioned, not buried
  • Progress against milestones — specific and countable
  • Next steps with owners and dates

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Expert References Before Writing. Read writing-reference.md, relevant per-domain files from references/domain/, and tufte-html-reference.md before writing any document content.
  2. Detect Mode Correctly. If an existing document is provided or referenced, this is Revise mode — read the document and the review findings before writing. Do not create a new document when the user asked to fix an existing one.
  3. Incremental Writing for Large Documents. If the document will exceed approximately 300 lines of HTML, write it section by section, not as a single monolithic write.
  4. User Approval of Expert Panel. Present the selected experts to the user before writing — the user must know which experts are shaping the document.
  5. HTML Before Approval. Write HTML for every major section before presenting it for approval — the user reads the HTML to evaluate the document.

Expert Roster

All expert definitions live in the design system reference files. No expert names are hardcoded in the skill. The roster files are the single source of truth.

Minto (Pyramid Principle) is the cross-cutting foundation. It applies to every document type: lead with the answer, group by MECE categories, support with evidence. The type-specific experts layer on top.

The writing reference loads for all document types. Every piece of prose — whether a slide headline, a strategy paragraph, or a newsletter hook — is subject to the same quality standard: Doumont for structure, Zinsser for economy, Orwell for clarity, Williams for grace.

Beyond the type-specific experts, the skill assesses the document’s content and topic for additional expert needs. A document with data charts loads Tufte and Few. A whitepaper about cognitive biases loads the relevant domain experts. A document covering a specific industry loads the matching industry reference file. If no expert covers a domain encountered in the content, the skill flags it per the Expert Discovery and Persistence Process, discovers and scores candidates, and persists them to the roster for future use.

How It Works — Create Mode

The skill follows nine steps from invocation to finished document:

Table 6 — Document creation process
StepWhat happens
1 Determine type Detects the document type from your request, context clues, or asks if ambiguous. Supports the five typed categories plus any other prose document.
2 Gather context Asks for subject, audience, purpose, scope, and type-specific details (slide count, delivery context, decision being informed)
3 Load experts Reads writing-reference.md (always), tufte-html-reference.md (always), per-domain files from references/domain/ selectively based on document type (document structure + type-specific sections), branding file if present. Assesses document content for additional expert needs — data presentation, diagrams, industry domains, security, or any uncovered domain. Presents the selected expert panel for your awareness before writing begins.
4 Structure Applies type-specific structural principles (pyramid, kernel, sparkline) and presents the proposed structure for your approval
5 Ask about MD For non-presentation documents, asks whether you want a Markdown version alongside the HTML. Skipped for presentations (HTML-only always).
6 Write Drafts the document applying all loaded expert principles and branding
7 Self-review Reviews the draft against loaded expert principles and fixes violations. Includes a factual claims check — every factual claim is verified against source material. Uncertain claims are flagged to you; unverifiable claims are removed. Max 3 passes.
8 Write output Saves to the type-specific directory; scans for existing files to avoid overwriting
9 Next steps Offers: revise, create another document, run /gvm-doc-review, or done
Step 4 is where structure approval happens. The skill proposes a document outline and waits for your confirmation before writing. Structure determines quality more than prose does.
Structure before writing

The skill proposes the document structure and gets your approval before writing content. This is deliberate. A well-structured document with average prose beats a beautifully written document with a confused structure.

HTML written before review

The skill writes the HTML document before presenting it for your review. Both the HTML and Markdown (if requested) are current at every approval checkpoint — you are always reviewing the actual output, not a draft proxy.

How It Works — Revise Mode

The revise process edits an existing document rather than writing from scratch. It preserves what works and changes only what the findings require.

Table 7 — Document revision process
StepWhat happens
1 Read the document Reads the existing document in full and identifies its type and subject matter
2 Identify issues From review findings (if a review file is referenced), from your explicit description, or from a quick expert assessment. Issues are prioritised: critical → important → minor.
3 Load experts Writing reference (always) plus the experts who diagnosed each issue. If Orwell found the problem, Orwell’s rules guide the fix. Also loads content-relevant experts (data, industry, diagrams) as needed.
4 Fix Works through issues in priority order. Applies the finding expert’s framework to determine the fix. Edits the document directly — no rewriting of sections that are not broken.
5 Self-review Re-reads changed sections against the expert principles that motivated the fix. Checks for new problems introduced by the changes. Includes a factual claims check on changed sections. Max 2 passes.
6 Present changes Summarises what was changed and which expert principles guided each fix. Offers: continue with remaining findings, run /gvm-doc-review, or done.
Experts who find should fix

In revise mode, the expert who identified a problem defines what “right” looks like. If a /gvm-doc-review finding cites Weinberg, Weinberg’s framework guides the fix — not a generic rewrite.

Relationship to /gvm-doc-review

/gvm-doc-write creates and revises documents. /gvm-doc-review audits them. The two skills are designed to work together:

  1. Create a document with /gvm-doc-write (create mode)
  2. Run /gvm-doc-review to score it against expert criteria
  3. Fix issues with /gvm-doc-write (revise mode) — reference the review findings and the skill loads the diagnosing experts to guide the fixes
  4. Re-review to confirm

The doc-write skill includes a self-review step that catches many issues before output, including a factual claims check that flags unverifiable assertions. But the doc-review skill applies a broader audit with formal scoring and severity classification. The self-review is a first pass; the doc-review is the quality gate.

After completing a document, the skill offers to run /gvm-doc-review as a next step. This is optional but recommended for any document that will be shared externally.

Key Rules

Expert grounding is the value. Every document type has named experts whose principles guide the output. This is what distinguishes /gvm-doc-write from "just write me a document."

Minto is the cross-cutting foundation. Pyramid structure, lead with the answer, MECE grouping. Applies to every document type regardless of other experts loaded.

Writing reference always loaded. Doumont for structure, Zinsser for economy, Orwell for clarity, Williams for grace. These experts apply to all prose in all document types.

Branding is identity, not design. The branding file controls logo, header colours, and heading font. The design system controls body typography, layout, spacing, and data visualisation.

Presentations are HTML-only. No Markdown pair. The slide format does not have a meaningful plain-text representation. All other types produce HTML by default, with an optional MD pair on request.

Content drives expert loading. Beyond the document type, the skill assesses the content for additional expert needs — data charts load Tufte and Few, an industry domain loads the matching industry file, an uncovered domain triggers expert discovery.

Revise means edit, not rewrite. In revise mode, the skill preserves what works and changes only what the findings require. Sections that are not broken are not touched.

Factual claims are verified. During self-review, every factual claim (statistics, dates, attributions, named results) is checked against source material. Claims that cannot be verified are flagged to you or removed.

One idea per slide. For presentations, this is non-negotiable. If a slide has two ideas, it is two slides.

Structure before writing. The skill proposes and gets approval on the document structure before writing content. Structure determines quality more than prose.

Self-review against experts. The draft is reviewed against loaded expert principles before presenting to you. Experts who find issues fix them.

Dual mode: standalone and pipeline. Standalone documents (presentations, strategy, whitepapers) can be created at any point. Pipeline docs (README, user guides, API docs) run after /gvm-test as the documentation phase, followed by /gvm-doc-review. Both modes stay consistent with pipeline artefacts.

Tips

  • Be specific about your audience. "Executive leadership" and "the engineering team" produce very different documents. The audience determines structure, depth, and vocabulary.
  • State the purpose as an action. "The reader should approve the migration budget" is better than "inform about migration." Purpose drives every structural decision.
  • Review the proposed structure carefully. Step 4 is your highest-leverage intervention point. Changing the structure after writing wastes effort.
  • Use branding for external documents. If the document leaves your team, a branding file ensures consistent visual identity across all outputs.
  • Run /gvm-doc-review on important documents. The self-review catches common issues, but the formal review provides scoring and severity classification.
  • For presentations, specify the delivery context. Live keynote, async read-ahead, and executive briefing produce fundamentally different slide counts and densities.
  • Iterate on sections, not the whole document. After the first draft, use revise mode to fix specific sections rather than regenerating everything.
  • Reference the review findings file. When fixing issues from /gvm-doc-review, point the skill at the review output. It reads the findings with their source experts and uses those experts to guide the fixes.
  • Any prose document works. The typed categories (presentations, strategy, etc.) provide structured guidance for common formats. But the skill handles whitepapers, blog posts, origin stories, or any other document that benefits from expert-grounded quality.
Best practice

For any document that will be shared beyond your immediate team: create with /gvm-doc-write, audit with /gvm-doc-review, fix issues, re-review. Two passes catches what one misses.
