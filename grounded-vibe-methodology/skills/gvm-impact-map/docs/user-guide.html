/gvm-impact-map

Discovery skill — produces the four-level Goal → Actor → Impact → Deliverable artefact that downstream /gvm-requirements gates against for traceability.

Overview

/gvm-impact-map is the discovery skill that produces an impact-map.md artefact: four flat tables (Goals, Actors, Impacts, Deliverables) linked by parent-ID columns. The artefact becomes the single source of truth for downstream traceability — /gvm-requirements Phase 5 (the IM-4 gate) refuses to finalise any requirement that does not trace to a leaf Deliverable in this file.

Adzic’s Impact Mapping is the structural model. Cagan’s Inspired shapes the elicitation order: outcomes first, never feature lists.

Run /gvm-impact-map after /gvm-init and before /gvm-requirements. The flat-table form is diff-friendly and mechanically validatable — the validator enforces referential integrity across the four levels and refuses to save the file if any parent-ID is missing.

When to Use

  • New product or feature initiatives — before requirements are written, when the team needs to align on outcomes (not outputs) and on which actors’ behaviour must change.
  • Re-scoping a project — the impact map exposes whether the candidate deliverables still trace to a measurable goal, or whether they have drifted into output-thinking.
  • Strategic prioritisation — the four-level tree makes it visible which deliverables have the strongest line of sight to the business outcome and which are speculative.
When to skip

If the project is a small bugfix or a clearly bounded technical task with no behavioural-change hypothesis, an impact map is overkill. Go straight to /gvm-requirements. The downstream IM-4 traceability gate is enforced only when an impact-map.md exists; without one, requirements stand on their own.

How /gvm-requirements Consumes This

The IM-4 traceability gate in /gvm-requirements Phase 5 reads impact-map.md and refuses to finalise any requirement that does not name a leaf Deliverable as its source. ("IM-4" is an internal gate label — the fourth cross-skill check on the impact map.) This converts "why are we building this?" from a guideline into a structural check — if the trace cannot be drawn, either the requirement is unsupported or the impact map needs another Deliverable row.

The Six-Phase Flow

Table 2 — Phases of the elicitation
PhaseWhat happens
0. Bootstrap Pre-flightVerify ~/.claude/gvm/ exists; load architecture experts. Refuses to run if /gvm-init has not.
1. DiscoveryElicit one Goal at a time via three AskUserQuestion steps: outcome statement, metric + target, deadline. The Goal row is not drafted until both Metric and Target are non-empty.
2. ActorsFor the current Goal, elicit one to five Actors. Above five the skill warns (Adzic recommends ≤ 5 per Goal) but accepts.
3. ImpactsFor each Actor, elicit one to three behavioural changes with a + / - direction column.
4. DeliverablesFor each Impact, elicit one or more deliverables with a type enum (feature / content / process / tool) and an optional risks column.
5. ValidationForeign-key check, ambiguity scan, schema-version check. Errors are reported per row; the practitioner edits, the validator re-runs.
6. ApprovalWrite paired impact-map.md and impact-map.html; AskUserQuestion: “Approve?”
Outcome-first discipline

Cagan: a Goal is a measurable change in user or business behaviour, not a feature. “Increase weekly active users by 20% within 6 months” is a Goal. “Add a notifications panel” is a Deliverable looking for a Goal — the skill will reject it at the ambiguity scan.

Artefact Schema

Four flat tables, each row keyed by an explicit ID, each non-Goal table carrying a parent-ID foreign key. The schema is versioned in YAML frontmatter (schema_version: 1).

Table 3 — Row shape per level
TableID prefixRequired columns
GoalsG-NStatement, Metric, Target, Deadline (ISO-8601 or empty)
ActorsA-NGoal-ID, Name, Description
ImpactsI-NActor-ID, Behavioural change, Direction (+ / -)
DeliverablesD-NImpact-ID, Title, Type (feature/content/process/tool), optional risks

Goal Ambiguity Scan

An aspirational verb on its own (“improve”, “enhance”, “optimise”, “leverage”) is not a measurable outcome. The ambiguity scan reads the Goal’s combined Statement + Metric + Target text against an extensible denylist; a hit passes only if a numeric quantity sits on the same line. The denylist lives in references/ambiguity-indicators.md and can be extended for project-specific vocabulary.

Loops, not refusals. An ambiguity hit does not crash the skill — it presents the offending word(s) and loops back to the outcome-statement question. The Goal row is not saved until the scan returns clean.

Hard Gates

Non-negotiable steps. If any are skipped, the impact-map output is invalid.

  1. Four-level tree validated before save. Every Actor’s Goal-ID points at a row in Goals; every Impact’s Actor-ID points at a row in Actors; every Deliverable’s Impact-ID points at a row in Impacts. A foreign-key violation refuses the save.
  2. Goal passes ambiguity scan. The Goal statement is checked against an extensible denylist of aspirational verbs ("improve", "enhance", "leverage"…). Hits pass only if a numeric quantity sits on the same line.
  3. Every level explicit (no implicit parents). No row may carry a blank parent-ID column. The validator refuses to fabricate parents.
  4. Paired MD + HTML before approval. Both impact-map.md and impact-map.html exist before the approval prompt fires.

Output

Paired files at the project root: 

impact-map.md
impact-map.html

The MD version is what the validator reads and what downstream skills consume. The HTML is the presentation copy — same content, Tufte CSS shell.

Pipeline Position

Greenfield /gvm-init → /gvm-impact-map → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → /gvm-design-review → /gvm-walking-skeleton → /gvm-build → /gvm-code-review → /gvm-test → /gvm-explore-test → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

/gvm-impact-map sits between /gvm-init (which calibrates the expert roster) and /gvm-requirements (which gates against the impact map for traceability). The skill is optional — small projects can skip it — but when present it raises the floor on outcome thinking across every downstream phase.
