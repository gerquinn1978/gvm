/gvm-status

Shows where you are in the GVM pipeline — what’s done, what’s next, and whether anything has gone stale.

Overview

/gvm-status is a read-only diagnostic command. It checks the file system for pipeline artefacts, detects staleness when upstream documents have changed after downstream outputs were generated, and summarises expert activation for the current project. It changes nothing — no files are created, modified, or deleted.

Grounded in Nielsen’s “visibility of system status” heuristic: the system should always keep users informed about what is going on, through appropriate feedback within reasonable time.

Run it at any point to orient yourself. It is particularly useful when resuming work after a break, or when you want to confirm that all upstream artefacts are still current before starting the next phase.

When to Use

  • Resuming a project after a gap — “where was I?”
  • Before starting a new pipeline phase — confirming prerequisites are in place
  • After editing upstream documents — checking whether downstream artefacts are now stale
  • Reviewing expert engagement — which experts have been loaded and cited across phases

What It Checks

/gvm-analysis is standalone and not part of pipeline tracking. It does not produce artefacts that status monitors.

Requirements rounds

Status scans requirements/ for numbered artefacts (requirements.md = round 1, requirements-002.md = round 2, etc.). It reports which round is current and checks whether corresponding downstream artefacts exist for each round — flagging, for example, when round 2 requirements exist but no matching test cases or specs have been generated.

Staleness detection

When upstream artefacts are newer than downstream outputs, the downstream work may no longer reflect the current state of the upstream document. Status flags these cases:

  • requirements.md newer than test-cases.md → test cases may need regeneration
  • requirements.md newer than spec files → specs may need updating
  • test-cases.md newer than spec files → specs may need updating
  • Spec files newer than build handovers → build may be against outdated specs
Staleness is a signal, not a verdict

A timestamp difference does not always mean regeneration is needed. A minor edit to a requirements document (fixing a typo, clarifying wording) may not affect downstream artefacts. Status flags the discrepancy; you decide whether to act on it.

Expert activation summary

Status reads the expert activation log (~/.claude/gvm/expert-activations.csv) and filters entries for the current project. The summary shows:

  • Total experts loaded and cited, grouped by pipeline phase
  • Tier breakdown (Tier 1 process / Tier 2a software domain / Tier 2b industry / discovered)
  • Classification breakdown (Canonical / Established / Recognised / Emerging / Provisional)
  • The five most-cited experts — the authorities most heavily relied on in your project’s output
If no activation log entries exist for this project, status reports that no GVM skills have been run yet and suggests /gvm-init or /gvm-site-survey as entry points.

Build progress

When the build phase is active (an implementation guide exists and handover files are being produced), status provides additional detail: how many chunks have been completed, which chunk is current, and what remains.

The output adapts to what actually exists. A freshly initialised project shows only the init phase with the full pipeline ahead. A project with no GVM artefacts shows nothing and points you to an entry command.

What to Do with the Results

Table 2 — Status signals and recommended actions
Status showsWhat it meansRecommended action
No GVM artefacts found The pipeline has not started in this directory Run /gvm-requirements (general-purpose) or /gvm-init (specialised domain) to begin
Pipeline stopped at a phase The next phase has not been run yet Run the next command in the pipeline — status shows which phase is next
Staleness detected An upstream document changed after the downstream output was generated If the change was substantive (not just a typo), re-run the downstream skill. It will detect the newer upstream file and offer to regenerate. If the change was cosmetic, no action needed.
Build in progress (X of Y chunks) Build is partially complete Run /gvm-build to continue from the next unfinished chunk. It reads handover files to determine where to resume.
Zero experts cited Expert reference files may not be loading Check that ~/.claude/skills/gvm-design-system/references/shared-rules.md exists. If missing, reinstall the plugin.
Low expert cite rate Experts are loaded but not being used in output This may be normal for early phases. If it persists across multiple phases, run /gvm-experts to review utilisation data and check whether the roster matches the project’s actual domain.
Many discovered experts The pipeline found gaps in the default roster and filled them No action needed — discovery is working as intended. Run /gvm-experts if you want to review the discovered experts’ scores.

No Output Files

Unlike every other GVM skill, /gvm-status does not produce HTML or Markdown documents. Its output is conversational only — displayed in the terminal and then gone. It is a diagnostic tool, not a document-producing phase.

Pipeline Position

Available at any point /gvm-status — usable before, during, or after any pipeline phase

Status sits outside the pipeline. It does not have prerequisites, does not produce artefacts consumed by other phases, and does not activate or cite experts. It reads the state that other skills have created and reports on it.
