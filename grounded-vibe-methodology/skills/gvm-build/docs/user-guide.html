/gvm-build

Execute technical specifications as working code — one chunk at a time, with strict TDD, automated code review, and handover files for session resilience.

What’s new in v2.1.0

The methodology-hardening release adds five user-visible build gates. See methodology-changelog.md for full rationale and validation method. Identifier prefixes (NFR, ADR, TDD, REVIEW, GATE) decode in the changelog.

  • Hard Gate 8 — chunk-acceptance smoke gate. After tests pass and wiring audits run, every chunk must execute a literal smoke command exercising its user-facing surface and assert structural contracts on the output (HTML elements present, JSON keys present). Refusal blocks the handover write.
  • Hard Gate 8 — carry-over and helper exemptions. Chunks built before 2026-04-28 are exempt as “v2.0.x carry-over”. Pure internal helpers may claim exemption with a written rationale (see ADR-MH-04 in the changelog).
  • TDD-1 outside-in ordering. For user-facing chunks, the outside-in acceptance test is the first deliverable — written before any unit test. Both Red and Green commit SHAs are recorded in the handover. Pure-internal-helper chunks may record TDD-1 exempted: <reason> in the handover (blank reason is treated as no exemption).
  • TDD-2 mock budget — severity escalation. Mock-budget violations escalate from Important to Critical when the target appears in the project’s opt-in .cross-chunk-seams file. Internal Python class mocks in collaboration-contract tests are flagged by the EBT lint.
  • TDD-3 realistic-fixture catalogue. TDD-3 defines a six-domain catalogue of realistic-fixture starter shapes (data-analysis, web/UI, API, parsing, security validation, concurrency). Panel C in /gvm-code-review currently emits Important findings in three of the six (data-analysis, parsing, security validation); practitioners ship variants in the others by judgement. Override per chunk via realistic-fixture-not-applicable: <rationale> in the handover.
  • NFR-2 diff-budget tracking. Each handover records a Diff budget delta integer (net SKILL.md LOC added). The build summary aggregates per-chunk values; you are prompted when the cumulative total crosses 500 LOC.

Overview

/gvm-build reads the implementation guide produced by /gvm-tech-spec and executes it as code — one chunk at a time. Each chunk produces two artifacts:

  • Pre-build prompt (build/prompts/P{X}-C{XX}.md) — a self-contained work order generated before execution. Everything a fresh agent needs to build the chunk.
  • Post-build handover (build/handovers/P{X}-C{XX}.md) — a record of what was actually built. Files created, tests passing, deviations from spec, lessons learned.
Prompts and handovers are immutable records. If a chunk needs to be redone, a v2 is created alongside the original — nothing is overwritten.

Prerequisites

Before running /gvm-build, you need:

  1. specs/implementation-guide.md — produced by /gvm-tech-spec
  2. specs/*.md — domain spec files referenced by the implementation guide
  3. A git repository (for commits, branches, and worktrees)

If the implementation guide doesn’t exist, /gvm-build will tell you to run /gvm-tech-spec first.

Getting Started

Starting a Build

Run /gvm-build in a project that has specs/implementation-guide.md. The skill detects what’s been built (from handover files) and offers your options:

  • Build next chunk — the first unbuilt chunk on the critical path
  • Build specific chunk — pick any pending chunk
  • Build all of Phase N — run all chunks in a phase sequentially
  • Parallel dispatch — when multiple chunks have no shared dependencies

Resuming

Stop any time. Handover files persist your progress. Next session, run /gvm-build again — it picks up where you left off. Works across /compact, session restarts, even different machines (if handovers are committed to git).

This is the key design insight: prompts and handovers make the build process session-independent. Context can be lost; progress cannot.

The Build Process

After 3, strongly recommends. After 4 without compacting, makes it mandatory — quality degrades before you notice it. Automatic. Accept compact at 2–3 chunks; mandatory at 4.
2 Present Options Offers: next chunk, specific chunk, full phase, or parallel dispatch. Choose what to build.

Then reads chunk description, spec sections, framework docs, conventions, dependency handovers. Assembles a self-contained prompt. Approve before execution. Decide on spec changes if flagged.

5b Execute — after all deliverables Dependency verification (verify every new import resolves) → lint → credential scanning (detects hardcoded secrets, API keys, tokens — hard block if found) → static analysis (security scanning and complexity checks, tools selected per stack from stack-tooling.md) → independent chunk review (spawned agent) → chunk-level tests (exit code + pass count verified) → integration seam check (verifies interfaces between this chunk and its dependencies actually connect). 6 Write Handover Records files created, tests passing, deviations, surfaced requirements, upstream fixes, downstream notes. Review the handover.
7 Phase Completion After all chunks in a phase: runs the full phase test suite across all chunks. Confirm integration passes.
8 Build Summary 8a. Between phases (optional): offers a build summary to capture progress before compacting. 8b. After the final phase (mandatory): writes paired HTML + MD to build/build-summary.html and .md covering what was built, what was not, open questions, surfaced requirements, and expert panel activation summary. Review summary. Between phases, accept or skip.

Test results are never trusted from interpretation of stdout — exit codes and parsed counts are the source of truth.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Bootstrap GVM Home Directory. Verify ~/.claude/gvm/ exists and is initialised before writing any output.
  2. Load Grounding Material Before Generating Any Prompt. Load both framework documentation (language docs, API references — for the “Framework Reference” section) and expert principles (architecture, domain, stack specialists — for the “Review Criteria” section). Construction uses framework docs; the review loop uses expert principles.
  3. Write the Prompt File. Every chunk must have a prompt file at build/prompts/P{X}-C{XX}.md written before execution begins.
  4. Code Review Loop. After writing code, review against the Review Criteria (expert principles and build checks). For fast unit tests: test first, then review working code. For slower chunk/phase tests: review first — catches API and design errors faster. Loop until clean; stop if issue count is not decreasing after 5 iterations.
  5. Write the Handover File. Every completed chunk must have a handover file at build/handovers/P{X}-C{XX}.md written after all tests pass.
  6. Test First, Always. Write the failing test before writing the implementation code.
  7. Wiring matrix required. Refuse to start a phase if the implementation guide lacks a wiring matrix section. Without (entry point → consumed modules → wiring chunk) rows, the build cannot verify what each phase delivered.
  8. Wiring verification at phase completion. For every (entry_point, module) row in the wiring matrix, mechanically grep the entry-point file for both an import statement and a call site of the named module. Acceptance is by code, not by chunk-handover claim. A missing import or call site refuses the phase. The inverse audit (_module_audit.py) catches modules built but never written into the matrix.
  9. Chunk-acceptance smoke gate (v2.1.0, NFR-1 / GATE-1). After tests pass, run a literal smoke command exercising the chunk’s user-facing surface and assert structural contracts on the output. Non-zero exit refuses the handover write. Carry-over and ADR-MH-04 internal-helper exemptions are recorded explicitly in the handover; silent skip is a violation.
  10. MVP-first ordering (v2.2.0, MVP-1 read-side mirror; MVP = Minimum Viable Product). When opening an implementation guide, verify the first user-facing chunk delivers a runnable end-to-end slice (acceptance test against walking-skeleton boundaries) OR carries the literal MVP-1 exempted: <category> — rationale: <why> marker with one of the four named categories: library, refactor, performance-driven, fully-specified. Refusal blocks chunk-prompt generation when neither holds. Companion gate to the write-side rule in /gvm-tech-spec Phase 5; the two compose so an impl guide that bypasses one is caught by the other.

Parallel Dispatch

When the dependency matrix shows chunks that can run simultaneously (e.g., 4 independent domain agents), /gvm-build offers to dispatch them to parallel subagents. Each gets its own git worktree, executes its prompt independently, and writes its own handover.

After all parallel chunks complete, you choose a merge strategy: sequential merge, combined PR, or manual.

Parallel dispatch only works when chunks have no shared dependencies. The implementation guide’s dependency matrix determines which chunks qualify.

Construction and Review Grounding

The build prompt separates two types of grounding, applied at different points:

Table 4 — Grounding separation
Prompt SectionKnowledge TypeUsed DuringContent
Framework ReferenceProceduralCode generation (writing code)Language docs, framework API sections, library method signatures
Review CriteriaDeclarativeCode review loop (evaluating code)Expert principles from architecture/domain/stack specialists + build checks

Expert principles are quality criteria for the review loop — they define what good code looks like. Framework documentation is for construction — it defines how to use specific APIs correctly. Preliminary evidence suggests expert grounding is demonstrably effective during review but has undemonstrated effectiveness during code generation (see whitepaper Section 9.1).

Review criteria are loaded from the expert reference files per chunk type:

The review loop is iterative: review working code, fix issues, re-test, re-review. Loop until clean — stop only if issue count is not decreasing after 5 iterations.

Finding quality gate

During the code review loop, findings must pass the consumer impact test: “An end user would [concrete outcome].” Findings that interrupt the build flow cost the most to address — polish findings during a TDD cycle waste time without improving quality. The loop is clean when all remaining observations are polish. See review-reference.md for the full quality gate.

Build Check Promotion

Build checks are recurring review findings that have been promoted to build-time checklists. They represent patterns that reviewers have flagged repeatedly — problems worth preventing at generation time rather than catching again in review.

How It Works

During prompt generation (Step 4), the skill loads reviews/build-checks.md if it exists. Relevant checks are included in chunk prompts under a “Known Patterns to Avoid” section, giving the implementation agent advance warning of pitfalls that have recurred in this project. The code review loop (Hard Gate 3) also checks against build checks during its review passes.

Where They Come From

Build checks are promoted automatically by /gvm-code-review, /gvm-doc-review, and /gvm-design-review when findings recur 3 or more consecutive rounds or regress after being resolved. The review skills write promoted checks to reviews/build-checks.md.

Tiers

Each build check is classified at promotion time. Tier 1 checks represent general engineering practice (DRY violations, resource lifecycle, single source of truth) and are never retired — they apply to any project regardless of stack. Tier 2 checks are tied to a specific library, SDK, or domain convention. When a tier 2 check has not triggered in 3 consecutive rounds, you are asked whether to retire it. The system flags candidates; you decide.

Working with STUBS.md

When honesty-triad is active on a project — that is, when .gvm-track2-adopted is present at the project root — /gvm-build runs the HS-1 gate immediately before writing each chunk handover. The gate inspects the chunk’s “Files Created” and “Files Modified” list. Any path under a stubs/ directory must be registered in STUBS.md; otherwise the handover is refused.

The intent is plain: a stub is a placeholder for real behaviour. Without a registry, placeholders accumulate silently and ship as features. The HS-1 gate makes that impossible — an unregistered stub blocks the chunk until the practitioner records the reason, the real-provider plan, an owner, and an expiry date.

The registry shape

Every entry in STUBS.md carries four fields, per cross-cutting ADR-004:

  • Reason — why a stub is acceptable here (e.g. “real provider unavailable in dev environment”).
  • Real-provider plan — the concrete path back to a real implementation (which API, which library, which chunk will wire it).
  • Owner — the person or team accountable for resolving the stub.
  • Expiry — an ISO-8601 date by which the stub must be replaced. Expired entries fail the release-gate verifier.
Missing STUBS.md is not a bypass

If STUBS.md doesn’t exist, the gate treats it as an empty registry. Any stub-namespaced file in the chunk surfaces as unregistered — not silently accepted. This is deliberate: the absence of a registry must not become a loophole.

Resolving a refused handover

When the HS-1 gate refuses a handover, you have two options:

  1. Register the stub. Add an entry to STUBS.md with the four fields above, then re-run the handover step. This is the right move when the placeholder is genuinely needed for now.
  2. Replace the stub. Move the file out of the stubs/ namespace by wiring real behaviour. This is the right move when the placeholder shouldn’t exist at all.

What the gate doesn’t do: silently let the chunk pass. The handover file is the audit trail; if its file list claims a stub, the registry must explain why.

Pre-existing legacy stubs

The HS-1 gate only sees files the current chunk touches. Stubs that pre-date the honesty-triad rollout are caught by a separate retroactive audit run by /gvm-explore-test (the HS-6 hook). /gvm-build does not run a retroactive sweep itself — it enforces the chunk-level gate, and exploratory testing owns the retroactive sweep.

Key Rules

1. Spec is the contract. Code follows the spec. Deviations recorded in handover with rationale.
2. Strict TDD. Failing test first. Red → Green → Refactor.
3. Review working code, not speculative code. For fast unit tests: test first, review after. For slow chunk/phase tests: review first — API and design errors are caught faster by review. Expert principles are review criteria (declarative quality), not construction framing. Framework docs ground construction; experts ground review.
4. Lint before commit. Tools selected per stack from stack-tooling.md. Unknown stacks trigger automatic tool discovery.
5. Never auto-rollback. On failure: present the error, offer options.
6. Never overwrite artifacts. Prompts and handovers are immutable. Re-doing a chunk creates a v2.
7. New requirements bubble up, not sideways — and they promote, never park. Discovered during build? Record in the handover’s “Surfaced Requirements” field as the staging ground. Then on triage, AskUserQuestion presents three promotion options (per shared rule 27): append as acceptance criterion to an existing requirement, add a new requirement entry, or run /gvm-requirements in “New round” mode. Dismiss the question and the finding is marked “deferred — awaiting triage” and re-prompted at the next checkpoint. The destination is always requirements/requirements.md. Surfaced requirements MUST NEVER be parked in STUBS.md — that file is for code-level placeholders only.
8. Backward fixes go forward. Need to change prior code? Do it in the current chunk, record in “Upstream Fixes”, re-run prior tests.

Simple Build Check

When the implementation guide has a single chunk with no dependencies, /gvm-build detects this and takes a shorter path. The dependency matrix and parallel dispatch steps are skipped entirely. The output is just two artifacts:

  • A single pre-build prompt (build/prompts/P1-C01.md)
  • A post-build handover once execution completes
When this applies

Simple projects — a single script, a standalone tool, a self-contained API with no cross-service dependencies — are common. The simple build path removes overhead that adds no value when there is only one chunk to build.

API Boundary Contracts

Build prompts for chunks that produce or consume an API endpoint now include the exact response shape from the spec. Both backend and frontend chunks reference the same contract, copied verbatim from the domain spec’s component design section:

  • Backend chunks include the response shape they must produce
  • Frontend chunks include the response shape they consume

This means the two sides of every API boundary are built against an identical contract, not independent interpretations of the same spec prose.

Where the contracts come from

The spec (produced by /gvm-tech-spec) must define the exact response shape with a concrete JSON example for every endpoint crossing a service boundary. If the spec lacks this, /gvm-build will flag it before generating the prompt rather than proceeding on ambiguous ground.
