/gvm-test-cases

A command for Claude Code that generates structured test cases from a requirements document, acting as a quality gate before technical specification and implementation.

Overview

The /gvm-test-cases command reads the requirements document produced by /gvm-requirements and generates a comprehensive test suite. It first assesses the quality of your requirements (catching issues early), then generates test cases domain by domain using expert-driven techniques.

It produces five files:

  • requirements-health-report.html — requirements health report (Tufte/Few design) — open this to review findings before deciding whether to fix requirements
  • requirements-health-report.md — health report in markdown
  • requirements-health-report-decisions.md — persisted decisions on requirements issues
  • test-cases.html — professionally designed document (Tufte/Few)
  • test-cases.md — Claude-friendly markdown for downstream commands

All files are saved to your project's test-cases/ directory.

This command serves as a quality gate. The health report catches requirement issues that /gvm-requirements might have missed — untestable requirements, inconsistencies, and gaps.

Prerequisites

You must have run /gvm-requirements first. The command expects to find requirements/requirements.md in your project directory. If the file doesn't exist, it will tell you to run /gvm-requirements.

Using Existing Test Cases

If you have existing test cases from another tool or process, provide them as input alongside your GVM requirements. The skill uses them as seed input — extracting what maps to GVM test case structure (Given/When/Then, traceability, priorities) and running the normal generation flow to fill gaps and apply expert techniques.

Before finalising, an independent verification agent compares the GVM output against your original test document, checking completeness, accuracy, hallucination, and traceability consistency. Discrepancies are presented to you for resolution.

External test cases accelerate the process but are not accepted as-is. They go through the same health report, technique selection, and coverage audit as generated tests.

The Health Report

Before generating any test cases, the skill reads your entire requirements document and produces a health report. This catches:

  • Untestable requirements — vague language, missing acceptance criteria, with suggestions to fix
  • Inconsistencies — requirements that contradict each other
  • Weak requirements — implementation details, duplicates, overly broad statements
  • Missing coverage — gaps where you'd expect a requirement but there isn't one
The health report applies risk-based assessment and testability criteria from the skill's reference files. The focus is "can this requirement be verified?" and "what's most likely to fail?"

For each issue, you choose:

  • Fix first — go back to /gvm-requirements and update the document
  • Proceed — issue noted as a caveat, tests generated anyway
  • Acknowledge — saved permanently so it won't be raised again on future runs
Persistent decisions

Acknowledged issues are saved to requirements-health-report-decisions.md. On subsequent runs, these issues are skipped unless the underlying requirement has been modified.

Test Case Generation

Automatic first pass

For each domain in your requirements, the skill automatically generates test cases covering the happy path and obvious negative scenarios. It selects the right test design technique based on what each requirement describes:

Table 1 — Technique selection
Requirement patternTechnique applied
Conditional logic ("if X then Y")Decision table testing
Workflows, multi-step processesState transition testing
User actions with expected outcomesUse case testing
Data inputs with ranges or limitsEquivalence class + boundary value
Multiple interacting parametersPairwise testing
Reversible, idempotent, or ordering operationsProperty-based testing [PROPERTY]
Calculations with algebraic propertiesProperty-based testing [PROPERTY]
Accepts user input, renders content, handles filesSecurity testing [SECURITY]
Exposes API endpoints, manages auth/sessionsSecurity testing [SECURITY]
Simple declarative ("system must X")Single Given/When/Then scenario
Technique selection follows a decision framework loaded from the skill's reference files: "this type of requirement calls for this type of test." The specific experts and techniques can be updated by editing the reference files.
Industry domain specialists (Tier 2b)

When the application’s business domain has an industry domain reference file (e.g., market-risk, credit-risk), those experts are loaded during test generation. They inform domain-specific edge cases, regulatory scenarios, boundary conditions, and failure modes that software testing experts alone cannot surface.

Selective deepening

After presenting each domain's tests, the skill decides whether to offer deeper analysis. It auto-skips for simple, low-priority requirements and offers deepening when it detects complexity:

  • Data inputs with ranges or boundaries
  • Multiple conditions interacting
  • Must or Should priority requirements
  • First pass generated 4+ test cases

When deepening, rigorous boundary value analysis and equivalence partitioning techniques are applied from the skill's expert reference files. Complex scenarios get traditional-format detail (preconditions, steps, expected results, test data) alongside the Given/When/Then format.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Expert References at Session Start. Read references/test-design-techniques.md and references/test-case-checklist.md before generating any test cases.
  2. Health Report Before Generation. Run the requirements health check before generating test cases — do not skip it.
  3. Write Paired HTML + MD. Both test-cases/test-cases.html and test-cases/test-cases.md must be written.
  4. Traceability Matrix is Mandatory. The output must include a traceability matrix mapping every requirement ID to its test case IDs.
  5. HTML Before Approval. Write the MD first, then immediately write the HTML — both must exist before presenting test cases for user review.

Test Case Format

Every test case uses BDD's Given/When/Then format with requirement traceability:

TC-RE-1-01: Candidate areas match budget constraint
  Given a user with a maximum purchase price of $500,000
  And seeking single family homes
  When the system identifies candidate areas
  Then 3-5 geographic areas are returned
  And each area has properties within the budget
  [Requirement: RE-1] [Priority: MUST]

Complex scenarios add traditional detail beneath the Given/When/Then:

TC-RE-2-03: Property listing price at budget boundary

  Given a user with a maximum purchase price of $500,000
  When the system searches for properties
  Then only properties at or below $500,000 are included

  Preconditions:
    - Candidate area identified (TC-RE-1-01 passes)
  Steps:
    1. Set maximum purchase price to $500,000
    2. Query properties in candidate area
  Expected Results:
    - $500,000 property: included
    - $499,999 property: included
    - $500,001 property: excluded
  [Requirement: RE-2] [Priority: MUST]
The BDD format comes from Dan North's Behaviour-Driven Development. Concrete values (not "a valid budget") come from Gojko Adzic's Specification by Example.

Requirements with definable invariants also generate property-based test specifications:

TC-RE-3-04: Currency conversion round-trip preserves value [PROPERTY]
  Property: For all amounts (0.01 to 999999.99) and currency pairs,
    convert(convert(amount, A, B), B, A) is within 0.01 of amount
  Counterexample strategy: very small amounts, very large amounts,
    currencies with extreme exchange rates
  [Requirement: RE-3] [Priority: MUST]
Property-based tests assert invariants rather than checking specific input/output pairs. The build phase implements these using the stack's property testing library (Hypothesis for Python, fast-check for TypeScript, etc.).

Requirements involving user input, APIs, authentication, or file handling generate security test cases:

TC-RE-5-03: Search query resists SQL injection [SECURITY]
  Attack vector: A03:2021 Injection
  Given a search form accepting user text
  When the input contains "'; DROP TABLE users; --"
  Then the query is parameterised and the input is treated as literal text
  And no SQL error is returned to the client
  [Requirement: RE-5] [Priority: MUST]
Security tests are mandatory for Must/Should requirements with web or API exposure. They target OWASP categories where AI-generated code has demonstrated, measurable failure rates — injection, XSS, path traversal, credential exposure.

EBT-1

Evidence-Based Testing (EBT) is the per-test-case contract enforced during Phase 4. EBT-1 is the example-based emission rule: every MUST-priority requirement produces at least one [EXAMPLE]-tagged test in the three-element shape below. EBT-1, the tag taxonomy below, and the Phase 4 contract / collaboration lint together define what a well-formed test case looks like. 

TC-RE-X-NN: <descriptive name> [EXAMPLE]
Input: <concrete value>
Given <precondition>
When <action with input>
Then the output MUST contain: <token | phrase | structural element>
And the output MUST NOT contain: <counterexample>
[Requirement: RE-X] [Priority: MUST]
[Trace: G-J → A-K → I-M → D-N → RE-X → TC-RE-X-NN]

The Input:, MUST contain, and MUST NOT contain lines are all mandatory. Phase 4's EBT-1 audit checks each one verbatim and fails the requirement closed if any is absent — the audit is structural, not semantic, so the literal line shapes matter.

Table 3 — Test-case tag taxonomy
TagWhen emittedWhat it asserts
[EXAMPLE] Every MUST requirement (mandatory). One concrete Given/When/Then with the three-element Input / MUST contain / MUST NOT contain shape.
[PROPERTY] Conditional — only when the property-detection heuristic returns a non-empty category match (e.g., idempotence, round-trip, ordering). An invariant that holds across the input space. The audit does not fault the absence of [PROPERTY] for non-property requirements.
[CONTRACT] Tests that exercise the real production class downward from the DI root, with mocks only at external boundaries. End-to-end behaviour through real internal collaborators. Internal-class mocks are a Metz violation.
[COLLABORATION] Tests that verify a caller invokes its collaborator with expected arguments. Interaction-level assertion (mocks and stubs are allowed).
[SECURITY] MUST/SHOULD requirements with web or API exposure (user input, API endpoints, auth, file handling). OWASP-aligned attack-surface coverage from the project's security expert templates.
Phase 4 contract / collaboration lint. The lint refuses two structural violations regardless of which test file they appear in:
  • Rainsberger violation — patching HTTP transport (requests, httpx, axios, etc.) inside the consumer's test. Replace with a contract test against a real fake server, or retag as [COLLABORATION] if the boundary is genuinely internal.
  • Metz violation — internal-class mocks inside a [CONTRACT]-tagged test. Replace with the real production class, or extend the project's .ebt-boundaries allowlist if the dependency is legitimately external.
Trace lines are mandatory. Every emitted test gets a [Trace: ...] line. When the requirement carries an [impact-deliverable: ...] tag, the trace resolves to G-J → A-K → I-M → D-N → RE-X → TC-RE-X-NN. Otherwise the literal line [Trace: not-yet-traced] is emitted; Phase 4 surfaces the count to the practitioner but does not hard-fail transitional projects.

The Five Phases

Table 4 — Process phases
PhaseWhat happensYour role
1 Health Report Reads all requirements. Identifies untestable, inconsistent, weak, and missing requirements. Filters previously acknowledged issues. Fix, proceed, or acknowledge each issue.
2 Auto-Generate For each domain: selects techniques, generates happy path + obvious negatives, writes to both files. Review test summaries per domain.
3 Deepening Offers full boundary analysis for complex/high-priority requirements. Auto-skips for simple ones. Accept or skip deepening offers.
4 Coverage Audit Builds traceability matrix. Flags uncovered requirements and orphan tests. Confirm coverage is acceptable.
5 Finalize Writes traceability matrix, test summary. Final sync check. Approve completed document.

The health report creates a natural feedback loop. Issues found during test case generation often lead to requirements updates, which strengthen both documents before any code is written.

Running Again

What happens when you run /gvm-test-cases again depends on the state of the project:

  • First run. No test-cases.md exists. The skill generates the full test suite from scratch, runs the health report, and writes test-cases/test-cases.md.
  • Stale detection. If requirements.md is newer than test-cases.md (by file timestamp), the skill detects the staleness, re-runs the health report against the updated requirements, and offers to regenerate only the affected domains. Previously acknowledged health report decisions remain valid unless the underlying requirement changed.
  • New requirements round. If requirements-002.md exists (a second requirements round), the skill creates test-cases-002.md. Test case IDs in the new file use round-prefixed identifiers (e.g., TC-R2-VS-1) to keep the two sets unambiguous across the pipeline.
