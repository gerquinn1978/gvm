/gvm-design-review

Expert-grounded pre-build design validation — parallel expert panels check whether your design correctly covers every requirement before a line of code is written.

What’s new in v2.1.0

v2.1.0 adds one user-visible change to this skill. Identifier prefixes (ADR, “shared rule N”) decode in methodology-changelog.md.

  • Hard Gate 6 — user owns finding triage (shared rule 28). Every Critical or Important finding emitted by panels is presented to you before any disposition is recorded. The skill may recommend “fix / defer / dismiss” per finding but must not decide. Forbidden patterns: recording emitted findings as “filtered under strict criterion” or “deferred to v{N}.{M}.{P} hardening” without your input; presenting a verdict before you have seen the findings. Zero-deferred runs are exempt from the summary-count-with-hidden-deferrals rule.

Overview

The /gvm-design-review command answers one question: does the design actually implement the requirements? It does this by running parallel expert panels — one per design domain — each checking coverage and best practice through its specific lens.

This skill reviews design correctness, not document quality and not implementation quality. For document quality, use /gvm-doc-review. For implementation quality, use /gvm-code-review.

The review produces a requirements coverage matrix: every requirement ID mapped to a design coverage status (Covered / Partial / Missing). This is the central output. Per-panel findings, cross-domain themes, and — for full reviews — 0–10 domain scores sit beneath it.

The skill works in two modes. For new projects it reads from the specs/ directory produced by /gvm-tech-spec. For existing codebases it can be triggered by /gvm-site-survey, in which case it reads code artefacts directly: schema files for data models, component trees for UI structure, route definitions for API contracts.

When to Use

The primary trigger is completing /gvm-tech-spec: you have requirements, you have specs, and you want to verify the specs cover everything before you start building. A design gap caught here costs minutes; the same gap caught mid-build costs hours.

  • After /gvm-tech-spec, before /gvm-build — the primary use. Validate that the spec designs are complete and coherent before committing to implementation.
  • After /gvm-site-survey — for existing codebases. The site survey identifies design concerns; this skill investigates them with a full expert review.
  • After updating specs — if requirements change after an initial design review, re-run to check that the spec updates are consistent.
  • Before a major refactor — use the site survey path to establish a requirements-coverage baseline before touching the code.
The cheapest quality gate

Fixing a data model gap in specs takes minutes. Fixing the same gap after the schema is live and populated takes migration scripts, downtime, and re-testing. Run the design review before build, not after.

Getting Started

Run without arguments to auto-detect all design domains from your specs/ directory: 

/gvm-design-review

The skill detects which domains are present in your specs, asks which ones you want to review, and asks for depth. You can also be specific: 

/gvm-design-review data model          # Review data model only
/gvm-design-review UI                  # Review UI/UX design only
/gvm-design-review API contracts       # Review API contracts only

Domain Selection

After scanning your specs, the skill presents the detected design domains and asks which ones to review. You are not forced to review all four — a backend-only project skips UI/UX; a static site with no external integrations skips API contracts.

Table 2 — Design domains and detection signals
DomainActivates when specs contain
Data ModelERDs, schema definitions, entity descriptions, database design sections, migration files
UI/UX and User DocsWireframes, screen layouts, component hierarchies, user flow diagrams, frontend design sections
API / Integration ContractsAPI endpoint definitions, service contracts, integration patterns, multi-service architecture
PerformanceActivates when specs contain performance requirements, or when the design involves database queries, API calls, or data-heavy UI screens. Reviews query patterns, caching strategy, pagination, and load handling.
If a domain is not detected, its panel does not run. The skill never invents work. If your project has no frontend, no UI/UX panel is dispatched regardless of depth mode.

If you invoke the skill with a specific domain argument, the selection step is skipped. If invoked from /gvm-site-survey, the referred domain is pre-selected and defaults to full review; other domains default to quick scan.

Depth Modes

After domain selection, the skill asks for depth. Both modes use expert panels. The difference is scope, not whether experts participate.

Quick scan is not a lightweight review

Quick scan still dispatches expert panels and still checks every requirement against the design. It is "quick" because it does not produce scored deep-dives. If you have time, run a full review — the coverage matrix is identical either way, but the full review surfaces issues the quick scan would not catch.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Dispatch Panels in Parallel. All activated panels must be dispatched as concurrent subagents in a single message — do not review panels sequentially.
  2. Load Expert References Before Dispatch. Read architecture-specialists.md and the relevant per-domain files from references/domain/ before writing dispatch prompts — each prompt must include loaded expert excerpts, not generic descriptions.
  3. Write HTML Output. The review must produce design-review/design-review-{NNN}.html. Review reports are HTML-only — findings are carried forward in reviews/calibration.md, not in a paired MD file.
  4. Update Calibration. After presenting findings, create or update reviews/calibration.md.
  5. Verdict. Every review includes a verdict: Build from this (ready), Build with caveats (conditionally ready), or Do not build (not ready). The verdict appears in a prominent box at the top of the report.

Expert Panels

Expert definitions are loaded from the GVM design system reference files at runtime. The skill reads specs/cross-cutting.md to determine which experts were declared for this project, then activates the panels that match.

Table 4 — Expert panel sources
Reference fileTierWhen active
architecture-specialists.mdTier 1Always
Per-domain files from references/domain/Tier 2aLoaded selectively based on which design domains are present: data modelling, UX/interaction, integration, user documentation
Per-stack files from references/stack/Tier 3Loaded selectively based on the technology stack declared in the specs
writing-reference.md—Loaded by the UI/UX panel for user documentation assessment
The same experts who guided the specification phase review the design. Consistency from spec through design review through build is intentional: the experts who find gaps should also guide the fixes (shared rule 3).

Each defect-class panel is dispatched as a parallel subagent. All panels run simultaneously: a four-panel full review takes the same wall-clock time as a single-panel review.

The Four Defect-Class Panels

Panels are partitioned by defect class, not by domain. Each panel's H3 below describes the defect class it scans for. The tables under each panel illustrate concrete domain-specific concerns the panel may surface during a review — data-model checks under Coverage, UI/UX checks under Contracts, API/integration checks under Structural Soundness, performance checks under Implementability. The tables are domain checklists the panel applies; they are not the panel's definition. The defect-class mandate is the contract.

Panel A — Requirements Coverage

Checks whether every requirement is traceable to a design element. Missing coverage is a Critical finding — requirements that have no corresponding design artefact will become missing features at build time.

Table 5 — Data model checks
CheckWhat it looks for
Entity coverageEvery noun/concept in requirements that implies persistent data has a table or entity. Missing entities are Critical findings.
Relationship correctnessCardinalities (1:1, 1:many, many:many) match requirement language. "A user can have multiple addresses" must be 1:many, not a JSON blob.
Attribute completenessEvery data field implied by requirements has a column. "Track when the order was placed, shipped, and delivered" needs three timestamps.
NormalisationSchema is normalised appropriately for the access patterns. Over-normalisation slows reads; under-normalisation creates update anomalies.
Index strategyQuery patterns implied by requirements have supporting indexes. "Search by location" implies a spatial index; "sort by most recent" implies a timestamp index.
Data lifecycleRequirements implying retention, archival, or deletion have corresponding schema support: soft deletes, archive tables, or TTL.

Panel B — Interface Contracts

Checks whether API contracts, data schemas, and component interfaces are internally consistent and aligned across boundaries. Contract mismatches — backend response shape vs frontend expectation, schema type vs API type — are bugs that survive to runtime.

Table 6 — UI/UX checks
CheckWhat it looks for
Journey coverageEvery user journey in requirements has a corresponding screen flow. Journeys with no UI representation are Critical findings.
Interaction completenessEvery action the user needs (create, edit, delete, search, filter, sort) has a UI affordance. Missing affordances are missing requirements implementations.
Navigation depthCommon tasks take fewer clicks than rare tasks. The panel counts clicks for the top five user journeys and flags any common journey exceeding three clicks.
ConsistencySame action, same gesture, same location across all screens. A "save" button that moves between screens is flagged.
Error statesEvery input has a defined error state. Every destructive action has a confirmation. Every long-running operation has a progress indicator.
Data display qualityChart types match the data. Data-ink ratio is high. Numbers are right-aligned in tables. No decorative elements that carry no information.
AccessibilityKeyboard navigation path defined. Colour contrast meets WCAG AA (4.5:1 for text). Focus management for modals. Screen reader labels for interactive elements.
User documentationHelp content is task-oriented, not feature-oriented. Microcopy is specific and actionable. Error messages tell the user what went wrong and what to do (“Email address is required” not “Validation error”).

Panel C — Structural Soundness

Checks whether the design is structurally sound — normalisation, layering, separation of concerns, and performance characteristics. Structural defects create systemic problems that are expensive to fix after build.

Table 7 — API/integration checks
CheckWhat it looks for
Endpoint coverageEvery operation implied by requirements has a corresponding API endpoint. Missing endpoints are Critical findings.
Contract alignmentBackend response shape matches what the frontend spec consumes. Field names, types, nesting, and optional/required markers must agree exactly.
Error contractEvery endpoint has defined error responses with a consistent shape. The frontend knows what a 400, 404, and 500 look like from this API.
PaginationList endpoints that could return unbounded results have a pagination contract. Unbounded list responses are a design defect.
Auth/authorisationRole-based access requirements are reflected in the endpoint design. The contract includes the auth mechanism and which roles are required where.
IdempotencyCreate and update operations that could be retried (network failures, double-clicks) are designed to be idempotent, not accidentally so.

Panel D — Implementability

Checks whether the design can actually be built as specified — ambiguous requirements, contradictory constraints, missing edge cases, and feasibility gaps. Implementability defects surface during build as blocked work and rework.

Table 8 — Performance checks
CheckWhat it looks for
N+1 query patternsScreens or endpoints displaying lists with related data should not require one query per item. 50 orders with line items should not mean 51 queries.
Unbounded queriesAny query that returns all rows without a LIMIT or pagination. “Show all transactions” with no ceiling is a performance time bomb.
Missing indexesQuery patterns implied by requirements (search by name, filter by date, sort by status) have appropriate indexes in the schema design.
Caching strategyFrequently-read, infrequently-written data has a caching approach. Missing cache strategy for hot-path data (sessions, frequently accessed records) is an Important finding.
Response payload sizeAPI endpoints serving the frontend return appropriately scoped responses. An endpoint returning 50 fields when the frontend needs 3 is a concern.
Sync vs asyncPotentially slow operations (file processing, email, report generation, third-party APIs) are designed as asynchronous with status polling or notification.
Frontend performanceData-heavy screens have pagination or virtualisation. Data loads on demand, not all upfront. Bundle size is reasonable for the library choices.

Panel D also runs a third pass — MVP-1 (Minimum Viable Product first ordering) compliance — that checks whether the design admits a thin end-to-end vertical slice delivering user-visible value as the first chunk, or whether it forces a horizontal build order (all data layer first, then services, then UI) that defers user-visible behaviour to the final phase. Designs that structurally prevent MVP-1 are flagged so they can be reshaped before /gvm-tech-spec Phase 5 refuses to sequence them. Exemption is permitted for the four named project shapes (library, refactor, performance-driven, fully-specified). Horizontal architectural consistency across slices is required and is NOT a violation; the rule is about the build sequence, not the design shape.

Review Process

Table 9 — Review process steps
StepWhat happens
1 Detect domains Reads specs/*.md and identifies which design domains are present. If invoked from /gvm-site-survey, reads code artefacts instead.
2 Select domains Presents detected domains as a checklist. User confirms which to review. Skipped if a specific domain was passed as an argument or if invoked from site survey.
3 Select depth User chooses quick scan or full review. Site survey referrals default to full review on the referred domain, quick scan on others.
4 Load context Reads requirements, test cases (if present), relevant spec sections, calibration data (if present), and expert excerpts from the GVM design system reference files.
5 Dispatch panels Launches one subagent per selected domain, all in parallel. Each subagent receives its expert identities, requirements, spec sections, domain-specific checklist, and depth mode.
6 Synthesise Collects findings from all panels. Deduplicates cross-panel findings. Cross-references: a data model gap that also causes an API contract gap is elevated in severity.
7 Write report Outputs HTML to design-review/design-review-NNN.html (review reports are HTML-only). Scans for existing files and increments — never overwrites.
8 Present consolidated review Presents the requirements coverage matrix, per-panel findings with severity, cross-panel themes, API boundary contract audit, and domain scores (full review only).
9 Offer next steps Offers: fix design gaps in specs, proceed to build, run /gvm-doc-review on updated specs, or re-run site survey.
10 Update calibration Creates or updates reviews/calibration.md with score history, dimension benchmarks, anchor examples, recurring findings, and build check promotion. Mandatory after every review regardless of which next step the user selects.
Build check promotion

During calibration update, findings that have been flagged in 3 or more consecutive rounds (systemic) or that regressed after being resolved are promoted to reviews/build-checks.md. This file is loaded by /gvm-build during prompt generation, injecting “Known Patterns to Avoid” directly into chunk prompts. The result is a feedback loop: review findings become build checks, which prevent the same findings from recurring. Tier 1 checks (general practice) are permanent. Tier 2 checks (project-specific) are flagged for retirement after 3 quiet rounds, but you decide whether to retire them.

Parallel is the point

All defect-class panels run as concurrent subagents dispatched in a single message. A four-panel full review takes the same wall-clock time as a single-panel review.

Site Survey Integration

When /gvm-site-survey identifies design concerns in an existing codebase, it can recommend /gvm-design-review with specific focus areas. This is the second entry point alongside the post-spec path.

When invoked with a site survey referral:

  • The skill reads the site survey report for the specific concerns the survey identified.
  • It defaults to full review on the referred domain and quick scan on others.
  • For codebases without formal specs, it reads code artefacts directly: schema files, component trees, route definitions.
  • Findings are explicitly linked to the survey: “The site survey flagged X. This review confirms/contradicts the concern.”
The site survey path does not require specs/ to exist. The design review adapts its source material to what is available.
Existing codebases without specs

The design review is not only for greenfield projects. If you have a codebase without formal specs, run /gvm-site-survey first. The survey will identify which design domains have concerns, and the design review will investigate them against whatever requirements documentation you have.

Severity Levels

Every finding is classified into one of four levels. The standard for classification is the requirements — not abstract best practice. A design that violates an expert principle but correctly implements the requirements is at most a Suggestion, not Critical.

CRITICAL Must fix before building

A requirement has no design representation at all. You cannot build it from this design.

RE-8 (order history) has no schema entity
The requirements specify that users can view past orders with status, date, and line items. The schema has no orders table, no order_items table, and no order_status type. This requirement cannot be built from the current data model.
Data Model panel → specs/spec-data.md — covers RE-8
Backend GET /users/{id} returns full_name; frontend expects firstName + lastName
The backend spec defines a single full_name string field. The frontend spec destructures firstName and lastName separately. This is a runtime integration failure. Agree on a contract before building both sides.
API panel → specs/spec-api.md §4 vs specs/spec-frontend.md §7
Finding quality gate

Every finding must pass the consumer impact test: “A developer implementing this design would [concrete outcome].” Design findings propagate to implementation — a wrong schema is harder to fix after code is built on it — so the threshold is strict. But defence-in-depth findings are theoretical when the defended-against scenario is already prevented by a stronger mechanism. State the precondition for failure: if it starts with “if a developer…” it is real; if “if an attacker gains write access to internal files…” it is theoretical.

IMPORTANT Should fix before building

The design partially covers a requirement but with gaps. Buildable, but the result will be incomplete.

Search by location (RE-14) has no spatial index in schema
The requirements specify location-based search. The schema has latitude and longitude columns but no spatial index or PostGIS extension. Queries will be full table scans. Add a spatial index or use a geography column type.
Data Model panel → specs/spec-data.md §Locations table

MINOR Covers requirement but violates best practice

The design covers the requirement but an expert's framework identifies a better approach. Functional but suboptimal.

Pagination contract uses page/pageSize; cursor-based preferred for ordered lists
The requirements do not specify a pagination strategy. The current design uses offset pagination, which produces incorrect results if rows are inserted between page requests. Cursor-based pagination eliminates this class of bug. This is a design preference, not a requirements violation.
API panel → specs/spec-api.md §Listings endpoint

SUGGESTION Alternative to consider

An approach that would improve the design, not a gap.

Soft deletes on users table would simplify audit log queries
The requirements include an audit trail (RE-19). Adding a deleted_at timestamp rather than hard-deleting users would let audit queries join directly to the users table without needing an archive table. The requirements do not require this approach — it is an efficiency suggestion.
Data Model panel → specs/spec-data.md §Users table

Output

Each review produces a new numbered HTML report. Review reports are HTML-only (per shared rule 13 exception) — no paired Markdown is produced. Findings are carried forward in reviews/calibration.md, not a paired MD file. Existing files are never overwritten: 

design-review/design-review-001.html
design-review/design-review-002.html
...

The HTML report follows the same Tufte/Few design system used across all GVM pipeline documents. Structure:

  1. Executive summary — domains reviewed, overall coverage assessment, critical gaps at a glance
  2. Requirements coverage matrix — every requirement ID mapped to its design coverage status
  3. Data Model findings (if reviewed) — entity coverage, relationship correctness, normalisation assessment
  4. UI/UX findings (if reviewed) — journey coverage, navigation audit, accessibility, user documentation
  5. API Contract findings (if reviewed) — endpoint coverage, contract alignment, error contract
  6. Cross-domain themes (full review) — issues that span multiple panels, with elevated severity
  7. API boundary contract audit (full review) — explicit comparison of backend response shapes vs frontend expectations
  8. Domain scores (full review) — 0–10 per domain with rubric anchors and concrete deductions

The coverage matrix is mandatory in every review. It appears immediately after the executive summary and gives the user the fastest possible answer to “what is missing?”

Where This Fits

The design review sits between spec and build. It has one standard position in the pipeline and one triggered entry point from site survey: 

[/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec
  → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review
  → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy
         ↑
   you are here

/gvm-site-survey → /gvm-design-review        (existing codebases)

Optional phases shown in [ ].

After the design review, the expected next step is either fixing design gaps in the specs (then optionally re-running /gvm-doc-review on the updated specs) or proceeding to build with acknowledged gaps recorded.

/gvm-doc-review checks document quality — are the specs well-written, traceable, and internally consistent?

/gvm-design-review checks design correctness — do the specs actually cover all the requirements?

They are complementary, not redundant. Run both before build.
Relationship to /gvm-doc-review

Both skills review specs, but from different angles. /gvm-doc-review asks: “is this spec document high quality?” /gvm-design-review asks: “does this design cover every requirement?” A spec can score 9/10 on document quality while missing three requirements. Run both.

Dual Review

From round 3 onwards, design review automatically runs a second set of panels in parallel — blind panels with the same experts, requirements, and criteria but no calibration data. Round 3 is the threshold because round 1 establishes the first scoring anchor, round 2 establishes expectations, and by round 3 anchoring bias (Tversky & Kahneman, 1974) has had enough history to create systematic blind spots. A reconciliation step merges only genuinely new findings and regressions into the final report. This corrects for calibration blind spots that develop as the reviewer builds expectations from project history.

This is the same mechanism used by /gvm-doc-review and /gvm-code-review — all three review skills share it via shared rule 16.

Blind Review Findings

The final report includes a “Blind Review Findings” section listing issues that only the blind panel caught — the calibration reviewer’s blind spots. No action is needed to trigger dual review; it activates automatically once the calibration file has sufficient history (2+ score rows from different dates).

Tips

  • API contract alignment is the most critical check. Backend/frontend contract mismatches are the bugs most likely to survive to production. They are invisible until runtime and expensive to fix after both sides are built. The design review catches them in specs.
  • Fix Critical findings before writing code. A Missing entry in the coverage matrix means the current design cannot produce the feature. There is no point building until the design gap is resolved.
  • Cross-domain themes matter more than per-domain findings. When multiple panels flag the same issue independently, it is almost certainly real. Elevated findings in the synthesis section should be resolved first.
  • Quick scan is enough for a sanity check. If you completed /gvm-tech-spec and want a fast coverage check before starting build, quick scan answers the essential question: is everything represented in the design?
  • Use full review as a formal quality gate. For projects where a design defect would be costly — schema migrations on live data, third-party API integrations — full review with scores is the appropriate standard before committing to build.
  • The requirements are the standard. If an expert flags a design choice that violates best practice but the requirements do not require the better approach, it is a Suggestion, not Critical. You are not obliged to build the most elegant possible solution.
  • Re-run after updating specs. If design review findings lead to spec changes, re-run to confirm the updates are consistent and did not introduce new gaps.

Troubleshooting

Common issues and recovery
ProblemCauseFix
Requirements file not found requirements/requirements.md does not exist Run /gvm-requirements first. The design review has nothing to validate against without requirements.
No design domains detected Specs exist but contain no design content (schema, UI, or API sections) Check that your specs use the expected section keywords. Alternatively, specify the domain directly: /gvm-design-review data model.
No specs directory found /gvm-tech-spec has not been run, and not invoked from site survey Run /gvm-tech-spec first, or use the site survey path if working with an existing codebase.
Coverage matrix is mostly “Partial” Specs exist but design sections do not explicitly reference requirement IDs Add requirement ID references to spec sections. The panel can only confirm coverage when the traceability is explicit.
API panel finds no contracts to review Specs describe an API but without formal endpoint definitions Add a structured endpoint section to your spec with method, path, request shape, response shape, and error codes. The skill cannot audit contracts it cannot read.
Site survey path finds no schema Codebase uses an ORM or code-first approach with no explicit schema files Point the skill at the model/entity class files instead. The skill reads code artefacts, not just SQL schema files.
Full review takes too long Large spec set with many domains Use specific domain selection to run one panel at a time, or run quick scan first to identify the highest-priority domain for a full review.
