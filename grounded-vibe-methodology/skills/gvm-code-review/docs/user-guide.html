Expert-grounded implementation code review that dispatches parallel specialist panels as subagents, each reviewing through a different lens from the project’s declared experts.

What’s new in v2.2.0

v2.2.0 closes the last v2.1.x issue: TS and Go projects now get mock-budget detection from Panel B, matching Python. Each language carries its own external-boundary allowlist (a per-language list of imports treated as "the network edge", so mocking them does not count against the budget):

  • TypeScript. The detector recognises the vitest, jest, testdouble, and sinon mock APIs — specifically vi.mock, vi.spyOn, jest.mock, jest.spyOn, td.replace, sinon.stub, and sinon.replace. The _TS_STACK_DEFAULTS allowlist treats the following as external boundaries: node:* builtins, major HTTP clients (axios, undici, got, ky, node-fetch), DB drivers (pg, mongodb, redis), and AI/cloud SDKs.
  • Go. The detector counts distinct test-double instances per func TestXxx(t *testing.T) body using three signals: types ending in Mock|Fake|Stub|Spy, gomock.NewController invocations, and types embedding testify’s mock.Mock (recognised by the embedded library type, not by struct name). The _GO_STACK_DEFAULTS allowlist treats net/http, os, io, database/sql, context, plus AWS/Google/Anthropic/OpenAI/Redis/MongoDB/Postgres SDK package paths as external boundaries.
  • Per-project extension. Each language allowlist is extensible via the project-level .ebt-boundaries file at the repo root — one pattern per line. Internal mocks emit Important; matches in .cross-chunk-seams escalate to Critical (same severity rule as Python).
  • Two STUBS.md retirements. Per-project seam allowlists now actually apply on TS and Go — the _lint_typescript.seam_allowlist and _lint_go.seam_allowlist parameters previously read but had no effect. STUBS.md retires both entries.

See methodology-changelog.md § v2.2.0 for the audit trail (ADR-TG-01..03, ADR-MH-02 wrapper-as-SUT parity).

What’s new in v2.1.0

The methodology-hardening release adds three user-visible review behaviours. See methodology-changelog.md for full rationale. Identifier prefixes (REVIEW, TDD, ADR, “shared rule N”) decode in the changelog.

  • Hard Gate 6 — user owns finding triage (shared rule 28). Every Critical or Important finding emitted by panels is presented to you before any disposition is recorded. The skill may recommend “fix / defer / dismiss” per finding but must not decide. Forbidden patterns: recording emitted findings as “filtered under strict criterion” or “deferred to v{N}.{M}.{P} hardening” without your input; presenting a verdict before you have seen the findings (the canonical example is “Verdict: Merge with caveats”). A “fix all” authorisation does not extend to new Critical findings discovered during the fix — those re-enter triage. Zero-deferred runs are exempt from the summary-count-with-hidden-deferrals rule.
  • Panel B — EBT contract / collaboration lint (REVIEW-1). When the changed file set includes test files, Panel B runs the EBT contract lint against every test (regardless of [CONTRACT] or [COLLABORATION] tag — tags name the test’s intent: a [CONTRACT] test exercises real production classes from the DI root downward; [COLLABORATION] verifies a caller invokes its collaborator with expected arguments). Three violation kinds:
    • Rainsberger pattern — HTTP transport (requests, httpx, axios) patched in the consumer’s test instead of using a real fake server.
    • Metz pattern — an internal Python class mocked inside a [CONTRACT]-tagged test, which silently bypasses the seam the test claims to verify.
    • Mock-budget violation — more than one mock in a test, or a mock target that is internal rather than at the external boundary (TDD-2 mandates one mock at the network/process edge). v2.1.0 ships Python-only enforcement; severity escalates from Important to Critical when the target appears in the project’s opt-in .cross-chunk-seams file. (Extended to TypeScript and Go in v2.2.0 — see “What’s new in v2.2.0” above.)
  • Panel C — realistic-fixture mandate (REVIEW-2 / TDD-3). When a chunk’s diff touches data-analysis, parsing, or security validation, and its tests cover only synthetic happy-path fixtures, Panel C emits an Important finding naming the missing variant. Practitioners can override with realistic-fixture-not-applicable: <rationale> in the chunk handover.

Overview

The /gvm-code-review command reviews implementation code by dispatching parallel expert subagents. Each subagent reviews through a specific lens — architecture, domain, or stack — grounded in the named specialists from the project’s tech specs. Multiple agents run simultaneously: four expert reviews in the wall-clock time of one.

This skill reviews code, not documents. For reviewing requirements, test cases, and specs, use /gvm-doc-review.

The key differentiator is that reviews are grounded in the same experts who guided the specification and build phases, loaded from the project's reference files. An AI grounded in specific React expertise catches different issues than one applying generic “React best practices.”

When to Use

  • After each build phase completes, to catch issues before they compound in later phases.
  • After fixing code review findings from a prior round, to confirm the fixes are clean.
  • Before running /gvm-test, to ensure code quality before investing in integration testing.

Getting Started

/gvm-code-review accepts several input modes: 

/gvm-code-review                    # Auto-detect from latest build phase
/gvm-code-review abc123..def456     # Review a git range
/gvm-code-review P5                 # Review all of build phase 5
/gvm-code-review P5-C03             # Review a specific chunk
/gvm-code-review src/api/           # Review a directory

If invoked without arguments after a /gvm-build phase completes, the skill auto-detects the phase from build/handovers/ and reviews it.

No setup required

The skill reads expert definitions from the tech-spec reference files and project-specific design principles from your specs. You do not need to configure panels manually.

Expert Panels

Expert definitions are loaded from reference files at runtime, not hardcoded. The skill selects which panels to activate based on what changed. Experts can be updated, swapped, or extended by editing the reference files.

Table 1 — Expert panel sources
Reference fileTierWhen Active
architecture-specialists.mdTier 1Always
Per-domain files from references/domain/Tier 2aLoaded selectively based on the domain of changed code
Per-stack files from references/stack/Tier 3Loaded selectively based on the language/framework of changed files
The project’s own specs/ directory declares which experts were activated during specification. The code review uses the same experts, ensuring consistency from spec through implementation.

Each panel is dispatched as a parallel subagent. Panel prompts explicitly state “do NOT comment on [other panels’ concerns]” to prevent overlap and keep each review focused on its area of expertise.

Expert discovery (Rule 10)

If the code under review uses technologies or patterns not covered by any existing reference file expert, the skill identifies authoritative experts with published works and documents them with citations. This ensures coverage even for niche or emerging technologies.

Panel E

Panel E — Stub Detection — runs alongside panels A–D whenever the project has adopted honesty-triad (i.e. .gvm-track2-adopted is present at the project root). It is dispatched as the same kind of parallel subagent as the other panels, but its scanning mandate is mechanical rather than principle-driven: every flagged finding maps to a documented registry entry, an expiry date, or a namespace boundary.

Table 3 — Panel E violation types
Violation typeWhat Panel E flagsHow to resolve
unregistered Stub-shaped function or constant body present in code, with no matching entry in STUBS.md. Either replace the placeholder with a real implementation, or register it in STUBS.md with reason, real-provider plan, owner, and ISO-8601 expiry.
expired STUBS.md entry exists, but today’s date is past the entry’s expiry field. Implement the real provider before the next review, or amend the expiry with a fresh rationale that the team has signed off on.
namespace_violation Stub body lives outside a stubs/ directory (HS-5). Production code must not house stub bodies inline. Move the stub into the stubs/ namespace and import it from there.

SD-5 applies to any of the three violation types — not only expired. Whenever a Panel E finding (regardless of type) recurs across consecutive review rounds, severity is promoted (see the .stub-allowlist callout below for the round-by-round rule).

The Panel E prompt is assembled mechanically, embedding STUBS.md verbatim, the project’s .stub-allowlist verbatim, and the language-specific section of stub-detection.md. Drift between the heuristic file and Panel E’s behaviour is structurally prevented — the panel only matches against the embedded text, never against principles.

The .stub-allowlist

Intentional matches that look like stubs but aren’t (test fixtures, contract tests, examples) belong in .stub-allowlist. These are suppressed during synthesis. SD-5 promotes severity for any stub that recurs across consecutive review rounds — round 2 raises the finding to Critical, and round 3 emits a build-check promotion so the same stub cannot quietly survive another cycle.

First-run migration prompt

On the first run against a legacy calibration file (no schema_version frontmatter), /gvm-code-review detects the unmigrated calibration and prompts the practitioner to run python -m gvm_migrate_calibration reviews/calibration.md before the panel run begins. The migration stamps a machine-readable schema_version: 0 frontmatter marker so future versions can reason about the file format. Existing rows, evidence, and narrative are preserved verbatim — only frontmatter is added. The hook lives in SKILL.md and dispatches before any panel work.

Why a prompt rather than a silent migration

The migration touches reviews/calibration.md — a file the team has been adding to over multiple review rounds. The prompt makes the schema upgrade visible rather than rewriting the file under the practitioner’s feet. The migration is idempotent: running it again on an already-migrated file is a no-op.

Review Process

Table 2 — Review process steps
StepWhat happens
1 Resolve scope Determine git range, list changed files, categorise by type and domain
2 Select panels Read project specs to determine declared experts, select panels based on what changed
3 Load context Read expert sections from reference files, relevant spec excerpts, ADRs, and conventions
4 Dispatch Launch one subagent per panel — all in parallel. Each gets expert identities, filtered file list, and spec context.
5 Synthesise Collect findings, deduplicate, identify cross-panel themes, elevate severity for issues flagged by multiple panels
6 Present Consolidated review with summary table, cross-cutting themes, and findings by severity
7 Next steps Offer: fix critical issues, fix all issues, create fix plan, or continue to next build phase
8 Update calibration Creates or updates reviews/calibration.md with score history, per-dimension benchmarks, and anchor examples from this project. This accumulated data enables the dual review feature from round 3 onward. Also checks for findings eligible for promotion to build checks (see below).
Build check promotion

During calibration update, findings that have been flagged in 3 or more consecutive rounds (systemic) or that regressed after being resolved are promoted to reviews/build-checks.md. This file is loaded by /gvm-build during prompt generation, injecting “Known Patterns to Avoid” directly into chunk prompts. The result is a feedback loop: review findings become build checks, which prevent the same findings from recurring. Tier 1 checks (general practice) are permanent. Tier 2 checks (project-specific) are flagged for retirement after 3 quiet rounds, but you decide whether to retire them.

Parallel is the point

Panels run as concurrent subagents dispatched in a single message. This is the core value: multiple expert reviews in the time of one. A 4-panel review takes the same wall-clock time as a single review.

Finding quality gate

Every finding must pass the consumer impact test: “A developer would [concrete outcome].” If you cannot finish that sentence with a specific consequence — a bug, a security boundary violation, wrong data — the finding is polish. Code findings use a stricter threshold than document findings because errors in code reach end users. Each review round holds new findings to a higher bar: Round 1 catches structural problems; Round 3+ requires a concrete consumer consequence. See review-reference.md for the full quality gate.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Dispatch Panels in Parallel. All panels must be dispatched as concurrent subagents in a single message — sequential review defeats the purpose.
  2. Load Expert References Before Dispatch. Read the relevant expert sections from the reference files before writing dispatch prompts — each prompt must include loaded expert excerpts, not generic descriptions.
  3. Write HTML Output. The review must produce code-review/code-review-{NNN}.html. Review reports are HTML-only — findings are carried forward in reviews/calibration.md, not in a paired MD file.
  4. Update Calibration. After presenting findings, create or update reviews/calibration.md.
  5. Verdict. Every review includes a verdict: Merge (ready), Merge with caveats (conditionally ready), or Do not merge (not ready). The verdict appears in a prominent box at the top of the report.

Issue Classification

Findings use four severity levels. Issues flagged independently by two or more panels are automatically elevated one level.

Output

The consolidated review is written to an HTML-only file (per shared rule 13 exception — review reports do not produce paired MD). Findings are carried forward in reviews/calibration.md: 

code-review/code-review-001.html
code-review/code-review-002.html
...

Each run produces a new numbered HTML report. The document contains:

  1. Summary table — panels run, findings count per severity
  2. Cross-cutting themes — issues that span multiple panels
  3. Critical findings — with structured detail (see below)
  4. Important findings
  5. Minor + suggestions — grouped for efficiency
  6. Per-panel detail — full findings from each expert panel

Finding format

Every finding follows a structured format designed to be both readable and parseable by /gvm-build for fix work:

Table 4 — Finding fields
FieldDescription
ExpertWhich expert’s framework identified the issue
SeverityCritical / Important / Minor / Suggestion
File:LineExact file path and line number
IssueWhat is wrong, in one sentence
Spec ReferenceWhich ADR, spec section, or convention is violated (if applicable)

The MD version is the primary input for fix work. /gvm-build reads the Expert field to ground fixes in the same framework that identified the issue (shared rule 3: experts who find should fix).

Optional phases shown in [ ].

/gvm-code-review sits after /gvm-build in the pipeline. It is most valuable when run after each build phase completes, before moving to the next phase.

For the final build phase, the skill offers a full-codebase review rather than just the latest phase — catching cross-phase integration issues that per-phase reviews miss.

No self-review

Subagents get fresh context, preventing confirmation bias from the build session. The code is reviewed by agents that did not write it.

Dual Review

From round 3 onwards, code review automatically runs a second set of panels in parallel — blind panels with the same experts and criteria but no calibration data (no anchors, no resolved findings, no score history). Round 3 is the threshold because round 1 establishes the first scoring anchor, round 2 establishes expectations, and by round 3 anchoring bias (Tversky & Kahneman, 1974) has had enough history to create systematic blind spots — the point at which rater drift becomes detectable in assessment research. A reconciliation step merges only genuinely new findings and regressions into the final report. This corrects for calibration blind spots: areas that scored well stop getting scrutiny, and resolved issues are trusted without re-verification.

This is the same mechanism used by /gvm-doc-review and /gvm-design-review — all three review skills share it via shared rule 16.

Tips

  • Review after each build phase, not just at the end. Issues caught in phase 2 are cheaper to fix than issues caught after phase 5.
  • Fix critical issues before continuing. Critical findings represent security vulnerabilities or spec violations that compound if carried forward.
  • Cross-panel themes are the most important findings. If two independent expert panels flag the same issue, it is almost certainly real.
  • The spec is the standard. The review checks code against the project’s own specs and ADRs, not generic best practices. If the spec says SQLite, the review does not penalise you for not using PostgreSQL.
  • Use “Fix critical issues now” when offered. The skill can fix issues immediately while the review context is fresh.
  • Don’t ignore suggestions. Suggestions are not mandatory, but they often represent the named expert’s specific framework applied to your code — the kind of insight that makes GVM valuable.
