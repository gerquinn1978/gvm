/gvm-site-survey

A codebase diagnostic and health assessment that scores an existing system across multiple dimensions, selects grounding experts based on what it finds, and recommends where to enter the GVM pipeline. Can also be run at any point as a health check.

Overview

The /gvm-site-survey command is primarily an entry point for existing codebases. It has the AI explore the codebase, produce a scored health assessment, diagnose its architectural state, and assemble the right expert panel for whatever comes next.

The name comes from the construction industry, where a site survey evaluates conditions, structural integrity, and constraints before any work begins. The same principle applies here: understand the system before changing it.

Unlike other GVM skills, /gvm-site-survey does not start with a fixed expert panel. It uses two distinct sets of experts: diagnostic experts who conduct the survey (pattern recognition, legacy assessment, data layer, boundary assessment), and project experts who are assembled as the survey’s output for downstream pipeline phases. The project expert roster emerges from the codebase, not the other way around.

While it is primarily an entry point, the diagnostic machinery also makes it a general-purpose health check. Run it mid-build on a greenfield project to verify the implementation tracks the intended architecture, or after a remediation effort to confirm the scores have improved. Each run produces a new numbered report, preserving the history for before-and-after comparisons.

Non-destructive

The site survey is a static analysis. It reads code, git history, and configuration files. It does not execute the application, run tests, or modify any files.

Getting Started

Run /gvm-site-survey with no arguments. The skill will begin by asking scoping questions before exploring the codebase: 

/gvm-site-survey

The skill opens with Phase 0 — scoping — to establish boundaries before exploring:

  • What is the intended work? A feature addition, a remediation effort, a modernisation, an audit?
  • Is there a specific area of concern? You may already know where the problems are.
  • How large is the codebase? For very large codebases (100k+ lines), the survey focuses on agreed-upon modules rather than the entire system.
Scoping is critical for large codebases. A full survey of a million-line monorepo is not useful. Agree on boundaries first.

The Five Scenarios

The survey classifies the codebase into one of five scenarios. Each implies a different expert panel and a different entry point into the GVM pipeline.

Table 1 — Scenario overview
ScenarioDescriptionTypical Cause
COHERENTConsistent, well-executed designStrong original team, maintained discipline
OUTDATEDInternally consistent but era-boundBuilt 5+ years ago, not refactored since
FRACTUREDOriginal design drifted over timeMany developers, no architectural stewardship
MID-MIGRATIONIncomplete deliberate transitionTeam turnover or reprioritisation during rewrite
INCOHERENTNo consistent designDeadline pressure, no architectural guidance

COHERENT Continue it

The codebase follows a consistent, well-executed design.
New work should extend the existing architecture. The experts are selected to align with the patterns already in use.
Evidence: consistent patterns throughout, clear module boundaries, conventions followed, tests present and meaningful.

OUTDATED Modernise it

The design is internally consistent but reflects an earlier era of best practice.
Modernisation should be deliberate, not piecemeal. The experts are selected to guide the transition while respecting what works.
Evidence: callback-based async when the language now supports async/await, jQuery-era frontend patterns, monolithic structure when the scale now demands decomposition.

FRACTURED Repair it

The original design drifted through accumulated quick fixes and feature additions.
The experts provide the evaluative framework to identify where the design has fractured and bring it back into line.
Evidence: traces of an original coherent design but deviations accumulate. Multiple conventions coexist. Hotspot files modified by many contributors.

MID-MIGRATION Finish it (or reverse it)

A deliberate migration was started but not completed.
The task is to decide which direction to finish going. The experts are selected for the target architecture.
Evidence: two distinct patterns coexist with directional intent. Adapter layers, compatibility shims, or bridge code. Half REST / half GraphQL, partial TypeScript migration.

INCOHERENT Redesign it

The system was never well-designed in the first place.
The AI identifies what the codebase is; the grounded experts provide the framework for what it should be; the gap becomes an actionable remediation plan.
Evidence: no consistent pattern. Each module built in isolation with different conventions. No discernible architectural intent.
A large codebase may have different diagnoses for different areas — a well-designed core with an incoherent reporting module. The survey flags mixed cases and diagnoses each area separately.

Survey Process

The survey follows six phases. Each builds on the previous one.

Table 2 — Survey phases
PhaseWhat happens
0 Scope Establish survey boundaries with the user: intended work, areas of concern, codebase size
1 Reconnaissance Project structure, tech stack, dependencies, test infrastructure, git archaeology, size and scope, and any existing requirements, test cases, or architecture documents (used as seed input for downstream pipeline phases)
2 Architectural Mapping Pattern identification, boundary assessment, data layer diagnosis, legacy assessment, convention audit (diagnostic experts loaded from reference files)
3 Diagnosis Score the codebase across multiple health dimensions (1–5 scale). Classify into one of the five scenarios based on the score pattern. Present scores and diagnosis to user for confirmation.
4 Expert Selection Assemble the project-specific expert roster across all three tiers based on the diagnosis and tech stack. Includes a mandatory expert coverage assessment: which areas are covered by existing experts, which have gaps, and discovery of new experts for uncovered domains. Expert discovery delegates to the centralised process in the design system.
5 Route Recommendation Classify the intended work type (user-facing feature, technical remediation, targeted fix, or unclear scope). Recommend the appropriate GVM pipeline entry point based on both diagnosis and work type.
Git archaeology is first-class evidence

Commit history, contributor patterns, and file hotspots are diagnostic data, not secondary to code structure. The most frequently modified files often reveal the most problematic areas of the codebase.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Expert References at Session Start. Read architecture-specialists.md and relevant per-domain files from references/domain/ and per-stack files from references/stack/ before beginning the survey.
  2. Read Actual Code. The survey must read actual source files — every architectural claim in the report must be backed by a file that was read. No guessing from directory listings.
  3. Write Paired HTML + MD Output. Both files must be written.
  4. Expert Roster Table is Mandatory. The output must include a “Recommended Project Experts” table — downstream skills read it to assemble their expert panels.
  5. HTML Before Approval. Write the survey MD first, then immediately write the HTML — both must exist before presenting the survey for user review.

Phase 1: What it looks at

The reconnaissance phase gathers facts through systematic exploration:

Table 3 — Reconnaissance areas
AreaWhat it examines
Project structureDirectory layout, organisational pattern (by feature, layer, or domain), entry points
Tech stackLanguages, versions, frameworks, key dependencies from config files
Test infrastructureTest directories, runner configuration, approximate coverage, test patterns
Git historyRecent activity, commit patterns, contributor count, most frequently modified files (hotspots)
Size and scopeLine counts by language, number of modules/services, external API integrations

Phase 2: Diagnostic experts

Diagnostic experts are loaded from the skill's reference files and guide the deeper analysis. These are not the project’s permanent expert panel — they are the lens through which the survey reads the codebase.

Table 4 — Diagnostic expert roles (loaded from reference files)
Diagnostic roleWhat they look for
Pattern recognition / refactoringApplication and enterprise patterns in use, code smells, structural drift
Legacy code assessmentTestability, seams, coupling, change risk
Data layer diagnosisStorage patterns, consistency models, caching strategies, data flow
Boundary assessmentService boundaries, module structure, coupling between components
The diagnostic experts guide what to look for. The codebase determines which downstream experts (Tier 1, 2, and 3) get locked in for the project. Diagnostic experts can be swapped by editing the reference files.

Expert Selection

Based on the diagnosis, the survey assembles a three-tier expert roster for downstream GVM phases. The roster is tailored to both the diagnosed state and the identified tech stack.

Table 5 — Expert emphasis by scenario (experts loaded from reference files)
ScenarioTier 1 Emphasis
COHERENTADR capture for new decisions, risk-calibrated depth
OUTDATEDRefactoring guidance, boundary extraction, conceptual integrity during transition
FRACTUREDRefactoring guidance, legacy code techniques, quality attribute restoration
MID-MIGRATIONMigration patterns, strangler fig, seam identification
INCOHERENTEstablishing architecture, C4 for structure, conceptual integrity

Tier 2a (domain) and Tier 3 (stack) specialists are loaded selectively from per-domain files in references/domain/ and per-stack files in references/stack/, selected based on what the codebase does and what technologies it uses. The full roster is presented to the user for approval before being locked in.

Mandatory expert discovery

Every survey must present a coverage assessment: which technologies, patterns, and domains are covered by existing experts, and which have gaps. When gaps are found, the survey discovers and offers to persist new experts. This step is not conditional — it runs on every survey.

Route Recommendations

The survey recommends where to enter the GVM pipeline based on two factors: the diagnosis (what state the codebase is in) and the nature of the intended work (what you want to do).

Work types

Table 6 — Work type classification
Work TypeDescriptionRequirements needed?
User-facing featureNew functionality that changes what users experienceYes — always needs /gvm-requirements
Technical remediationRefactoring, pattern consolidation, modernisationNo — the survey itself defines the problem and target state. Go to /gvm-tech-spec.
Targeted fixSmall, well-understood change in a healthy codebaseNo — go to /gvm-build directly
Unclear scopeAreas of concern but severity or scope is uncertainRoute to /gvm-doc-review for deeper investigation first

For technical remediation, the survey report itself serves as the requirements document — it has defined the gap between what the codebase is and what it should be. Routing through /gvm-requirements would be ceremony without substance.

Existing documentation

The survey scans for existing requirements, test cases, specifications, and architecture documents in any format. If found, they are included in the route recommendation as seed input for the corresponding pipeline phase. For example, an existing PRD would be fed into /gvm-requirements as a rich seed — the skill extracts what it can, runs the normal elicitation flow to refine and fill gaps, then verifies the GVM output against the original for completeness and accuracy.

Existing documents accelerate the pipeline without skipping the quality process. Each downstream phase still runs its full expert-guided flow and independent verification.

The survey recommends, it does not mandate

The user always decides where to enter the pipeline. The route matrix is guidance, not a gate.

Output

The survey produces paired HTML and Markdown files: 

site-survey/site-survey-001.html    site-survey/site-survey-001.md
site-survey/site-survey-002.html    site-survey/site-survey-002.md
...

Each run produces a new numbered pair. Previous surveys are never overwritten, preserving the history for before-and-after comparisons.

The report follows the same Tufte/Few design system used by all GVM documents. It is structured in this order:

  1. Executive Summary — one-paragraph diagnosis, scenario classification, recommended entry point
  2. Codebase Profile — tech stack, size, structure, activity level, contributor count
  3. Architectural Map — patterns identified, module boundaries, data flow, conventions
  4. Health Scorecard — scored dimensions (Coherence, Currency, Testability, Modularity, Documentation, Dependency Health, Usability if applicable) as a 1–5 table
  5. Diagnosis — scenario classification with supporting evidence, showing how the score pattern maps to the scenario
  6. Risk Areas — hotspots, high-coupling zones, untested areas (prioritised as Critical / Important / Minor)
  7. Diagnostic Experts Used — which experts were loaded to conduct the survey and their roles (pattern recognition, legacy assessment, data layer, boundary assessment)
  8. Expert Coverage Assessment — which technologies, patterns, and domains are covered by existing experts, which have gaps, and what was done about each gap (expert discovered, scored, added — or gap acknowledged)
  9. Recommended Project Experts — the assembled three-tier panel for downstream pipeline phases, with a summary table: expert name, work, tier, classification, reference file, status (existing / newly added)
  10. Route Recommendation — recommended pipeline entry point based on both the diagnosis and the work type, with rationale
  11. Open Questions — anything the survey could not determine from code alone
The Markdown version is designed for Claude consumption in downstream commands. The recommended project experts section is the key input for /gvm-requirements, /gvm-tech-spec, and /gvm-build.

Where This Fits

/gvm-site-survey is the alternative entry point for existing codebases. It replaces /gvm-init as the entry point.

GVM v2.0 pipeline [/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Existing codebases /gvm-site-survey → /gvm-requirements | /gvm-tech-spec | /gvm-build (depends on diagnosis)

The survey output — particularly the expert roster and diagnosis — travels with the project through all subsequent pipeline phases. Downstream skills read the survey report to understand which experts are active and why.

Tips

  • Scope before you explore. For large codebases, agree on survey boundaries first. Surveying everything produces noise, not insight.
  • Your domain knowledge matters. The code tells the AI what the system is. You know why it is that way. When the survey presents its diagnosis, confirm or correct it.
  • Don’t skip the git archaeology. Commit history and contributor patterns often reveal more about a codebase’s health than the code itself. Hotspot files are where the problems live.
  • Mixed diagnoses are valid. A well-designed core with an incoherent reporting module is not unusual. The survey handles mixed cases explicitly.
  • The expert roster is a starting point. You can add or remove experts based on your knowledge of the project. The survey’s selections are evidence-based suggestions, not mandates.
  • Run it once, use it everywhere. The survey report persists and is consumed by all downstream commands. You do not need to re-survey for each pipeline phase.
  • Re-survey after major changes. If you complete a significant modernisation or remediation effort, a fresh survey confirms the codebase has moved to a healthier state.
Best practice

Run /gvm-site-survey before your first change to any existing codebase. The upfront investment of 10–15 minutes prevents weeks of working against the grain of an architecture you did not understand.
