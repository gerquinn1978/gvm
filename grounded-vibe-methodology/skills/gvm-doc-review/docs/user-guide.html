/gvm-doc-review

A cross-cutting quality audit for Claude Code that scores requirements, test cases, specs, and standalone documents against expert criteria, flags issues by severity, and checks consistency across your entire document set.

What’s new in v2.1.0

v2.1.0 adds one user-visible change to this skill. Identifier prefixes (ADR, “shared rule N”) decode in methodology-changelog.md.

  • Hard Gate 7 — user owns finding triage (shared rule 28). Every Critical or Important finding emitted by panels is presented to you before any disposition is recorded. The skill may recommend “fix / defer / dismiss” per finding but must not decide. Forbidden patterns: recording emitted findings as “filtered under strict criterion” or “deferred to v{N}.{M}.{P} hardening” without your input; presenting a verdict before you have seen the findings. Zero-deferred runs are exempt from the summary-count-with-hidden-deferrals rule.

Overview

The /gvm-doc-review command acts as a quality gate across your project documents. Point it at your project and it automatically detects which documents exist — requirements, test cases, technical specifications, user documentation, presentations, strategy documents, newsletters, and training materials — reviews each one against expert-sourced criteria, and checks that they agree with each other.

The output is a scored HTML report with every issue classified as Critical, Important, or Minor, so you know exactly what to fix and in what order.

Think of /gvm-doc-review as an automated peer review drawing on the same expert panels used to create each document type. It applies requirements expertise to requirements, test design expertise to test cases, and architecture expertise to specs. The specific experts are loaded from reference files.

Unlike the other pipeline commands, /gvm-doc-review is not a single stage — it is cross-cutting. You can run it at any point in the pipeline to audit whatever documents exist so far.

Getting Started

Run /gvm-doc-review with no arguments. The skill scans your project for documents in standard locations. When multiple document types are detected, a checklist is presented. Pipeline documents (requirements, test cases, specs) are pre-checked; standalone documents are not. You select which types to review: 

/gvm-doc-review

No setup required

The skill loads its own criteria reference and design system. You do not need to configure anything — just run the command and read the report.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Review Criteria Before Scoring. Read both references/review-criteria.md (local scoring rubrics) and review-reference.md (design system assessment methodology) before scoring any document.
  2. Multi-Select When Multiple Types Found. When multiple document types are detected, present a multi-select — the user decides which types to review.
  3. Write HTML Output. The review must produce doc-review/{document-name}-review-YYYY-MM-DD.html. Review reports are HTML-only — findings are carried forward in reviews/calibration.md, not in a paired MD file.
  4. Update Calibration. After presenting scores, create or update reviews/calibration.md.
  5. Write HTML Before Presenting Findings. The HTML must exist before presenting the review.
  6. Verdict. Every review includes a verdict using type-specific language. The exact phrasing depends on the document type — requirements say ‘Proceed’, presentations say ‘Present’, and so on. All map to one of three states: ready, conditionally ready, or not ready. The verdict appears in a prominent box at the top of the report.

Scoring System

Every document is scored 0 to 10 against ten type-specific criteria (one point each). These individual scores are then combined into an overall project score using fixed weights:

Table 2 — Score weights
Document TypeWeight
Requirements30%
Test Cases25%
Specs30%
Cross-document Consistency15%
If a document type does not exist yet, its weight is redistributed proportionally across the types that do exist. The review scores what is present, not what is absent — missing documents are flagged as issues instead.

Issue Classification

Every finding is classified into one of three tiers. The report groups issues by tier so the most urgent problems appear first.

CRITICAL Must fix before building

Critical issues block the pipeline. They represent gaps or contradictions that will cause real problems if carried forward into implementation.

MUST requirement RE-4 has no test coverage and no spec coverage
This requirement will be built without any way to verify correctness or any technical design. Add test cases and ensure at least one spec section addresses it.
Requirements → Test Cases, Requirements → Specs
Direct contradiction: specs say SQLite, requirements say PostgreSQL
The data storage spec (spec-data.md §3) specifies SQLite, but RE-12 requires PostgreSQL for concurrent access. Resolve the conflict before building.
specs/spec-data.md §3 vs requirements/requirements.md RE-12

IMPORTANT Should fix soon

Important issues weaken document quality but do not block progress outright. They represent missed coverage, ambiguity, or incomplete reasoning.

RE-7 uses vague language: "the system should be fast"
This requirement is not testable. Define a specific performance target — e.g., "response time under 200ms for the 95th percentile."
requirements/requirements.md RE-7
No boundary value tests for budget range (RE-3)
RE-3 specifies a budget of $100,000 to $2,000,000. Add tests for the boundaries: $99,999, $100,000, $2,000,000, $2,000,001.
test-cases/test-cases.md, RE-3 coverage

MINOR Nice to improve

Minor issues are polish items. They improve clarity, consistency, or professionalism but are safe to defer.

Test TC-12 uses mechanism-focused name: "Test RE-5 validation"
Rename to describe the behaviour — e.g., "Budget below minimum shows validation error." Behaviour-focused names are easier to scan in test reports.
test-cases/test-cases.md TC-12
Inconsistent vocabulary: "report" vs "summary" vs "output"
The requirements use "report," test cases say "summary," and the spec calls it "output." Pick one term and use it consistently.
Cross-document vocabulary check

Cross-Document Consistency

When multiple document types exist, the review checks that they tell a coherent story. This is where many real-world projects break down — each document is fine in isolation, but they contradict or ignore each other.

Cross-document consistency is weighted at 15% of the overall score. It has ten criteria of its own, scored like each document type.

Traceability

  • Requirements → Test Cases: Every requirement ID has at least one test case. Uncovered requirements are flagged.
  • Requirements → Specs: Every requirement ID appears in at least one spec section. Unspecified requirements are flagged.
  • Test Cases → Specs: Test cases reference spec sections where relevant. Tests that cannot be implemented from the specs are flagged.
  • Specs → Requirements: Every spec section traces to requirement IDs. Spec sections with no requirement basis are flagged as potential scope creep.

Vocabulary and contradictions

  • Vocabulary consistency: Are the same concepts named the same way across all documents? Different names for the same thing (e.g., "report" vs "summary") are flagged.
  • Priority consistency: Do test and spec depth match requirement priority? A MUST requirement with thin test coverage is flagged.
  • No contradictions: Do the documents agree on system behaviour? A requirement that says one thing while the spec says another is flagged as Critical.
  • Open questions tracked: Are open questions from requirements resolved as decisions in specs, or explicitly deferred?
  • Acknowledged gaps respected: If a health-report-decisions file exists, the review ensures those acknowledged decisions are not contradicted.

Output

The review produces a single HTML report saved to: 

doc-review/requirements-review-2026-04-03.html

The document name is derived from the reviewed file (e.g., requirements-review-2026-04-03.html for a requirements document). Subsequent reviews of the same document on the same date append -r2, -r3, etc. Reports are HTML-only: no downstream skill parses a Markdown version, and findings are carried forward in reviews/calibration.md rather than in the report itself.

The report follows the same Tufte/Few design system used by all pipeline documents. It is structured in this order:

  1. Score card — overall score and per-document breakdown at the top
  2. Critical issues — everything that blocks the pipeline
  3. Important issues — everything that weakens quality
  4. Minor issues — polish and consistency items
  5. Requirements review — criterion-by-criterion assessment with scores
  6. Test cases review — criterion-by-criterion assessment
  7. Specs review — criterion-by-criterion assessment
  8. Cross-document consistency — traceability matrix and vocabulary check
Each issue in the report is actionable: it cites the specific document, section, and ID where the problem lives, and suggests how to fix it. The review does not just say "this is bad" — it says what to do about it.
Date-stamped reports

Each review creates a new date-stamped file. This means you can track quality improvement over time by comparing successive reports.

Where This Fits

/gvm-doc-review is both cross-cutting (usable at any point) and has an explicit pipeline position after /gvm-doc-write in the documentation phase:

    → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Optional phases shown in [brackets].

Cross-cutting — /gvm-doc-review can run after any of these stages:
/gvm-requirements  →  /gvm-test-cases  →  /gvm-tech-spec  →  /gvm-design-review  →  /gvm-build
      ↑               ↑               ↑               ↑                ↑
      └───────────────────────── /gvm-doc-review ──────────────────────┘

Pipeline documentation phase:
/gvm-test  →  /gvm-doc-write (pipeline docs)  →  /gvm-doc-review  →  /gvm-deploy

The review adapts to whatever documents exist at the time you run it:

  • After /gvm-requirements — reviews requirements only. Scores completeness, quality attributes, and structure.
  • After /gvm-test-cases — reviews requirements and test cases. Adds traceability checks between them.
  • After /gvm-tech-spec — reviews all three document types plus full cross-document consistency.
  • Before /gvm-build — the most valuable checkpoint. Catches problems that would be expensive to fix in code.
  • After /gvm-doc-write (pipeline docs) — reviews user-facing documentation produced after /gvm-test. This is the final quality gate in the full pipeline.
The review is designed to prevent the most costly pattern in software: discovering requirements gaps during implementation. Fixing a requirement costs minutes at the requirements stage and hours during build.

Review Process

When you run /gvm-doc-review, the skill follows seven steps:

Table 8 — Review process steps
StepWhat happens
1 Detect Scans standard paths for all document types — pipeline documents (requirements, test cases, specs) and standalone documents (user docs, presentations, strategy, newsletters, training materials)
2 Load criteria Reads the review-criteria.md reference file (the scoring rubric). For standalone document types, also loads type-specific per-domain files from references/domain/ selectively based on the document’s subject matter.
3 Review Single calibrated review by default. When reviews/calibration.md contains 2 or more score history rows with different dates, dual review triggers automatically — a calibrated reviewer and a blind reviewer run in parallel, then findings are reconciled (see Dual Review below)
4 Cross-document checks Checks traceability, vocabulary, and contradictions across documents
5 Score Rates each document 0–10 and computes the weighted overall score
5b Source verification For externally-supplied documents only: if factual accuracy scored below 7, runs the source verification protocol against claims that scored poorly. Checks completeness, accuracy, and hallucination against the original source.
6 Write report Writes the HTML report to doc-review/{document-name}-review-YYYY-MM-DD.html. No paired Markdown file — review reports are not parsed by downstream skills.
7 Update calibration Creates or updates reviews/calibration.md with score history and anchor examples. When this file accumulates 2 or more score history rows with different dates, dual review triggers automatically on the next run. Also checks for findings eligible for promotion to build checks (see below).
Build check promotion

During calibration update, findings that have been flagged in 3 or more consecutive rounds (systemic) or that regressed after being resolved are promoted to reviews/build-checks.md. This file is loaded by /gvm-build during prompt generation, injecting “Known Patterns to Avoid” directly into chunk prompts. The result is a feedback loop: review findings become build checks, which prevent the same findings from recurring. Tier 1 checks (general practice) are permanent. Tier 2 checks (project-specific) are flagged for retirement after 3 quiet rounds, but you decide whether to retire them.

The entire process runs autonomously. The skill will only ask you a question if the review uncovers an ambiguity that genuinely requires your input — otherwise it produces the report and presents a summary.

Dual Review

When reviews/calibration.md contains 2 or more score history rows with different dates, dual review triggers automatically: a calibrated reviewer (with full project history) and a blind reviewer (with no calibration data, seeing the document fresh) run in parallel. A reconciliation step merges the results into one report.

This corrects for a systematic weakness in progressive calibration: the more the reviewer knows about the project, the more it expects certain patterns and the less it scrutinises areas that scored well previously. The blind reviewer has no such expectations. It catches:

  • New problem categories that prior rounds did not look for
  • Issues masked by worse adjacent problems that have since been fixed
  • Drift in areas that scored well and stopped getting scrutiny
  • Regressions — previously-resolved issues that have crept back
The blind reviewer also rediscovers resolved issues and flags things the calibrated reviewer correctly excluded. The reconciliation step filters this noise — the practitioner sees one clean report, not two.

In the final report, findings from the blind review are tagged so you can see where calibration had blind spots. Over time, the dual review metadata (how many findings came from each source) reveals whether calibration is developing systematic biases.

This mechanism applies to all three review skills — /gvm-doc-review, /gvm-code-review, and /gvm-design-review — via shared rule 16. Any review that uses progressive calibration automatically gets dual review correction once the calibration file has sufficient history.

Blind Review Findings

The final report includes a “Blind Review Findings” section listing issues that only the blind reviewer caught — the calibrated reviewer’s blind spots. No action is needed to trigger dual review; it activates automatically based on the calibration file.

Why not on the first review?

Until reviews/calibration.md contains at least 2 score history rows from different dates, the calibration file has insufficient history to meaningfully bias the reviewer. There are no established anchors to create blind spots, and no resolved findings that could mask regressions. The dual review adds cost without corresponding value until that threshold is reached.

Finding Quality Gate

Not every observation is a finding. A review system calibrated to keep discovering issues will always find more — reviewers can generate findings indefinitely by lowering the threshold for what counts. The finding quality gate defines what counts.

A finding is real if a consumer of this document would misunderstand, take wrong action, or lose trust because of it. A finding is polish if fixing it makes the author feel better but changes nothing for the consumer.

Every finding must pass three tests:

  1. Consumer impact. State the consequence: “A reader would…” If you cannot finish that sentence with a specific outcome (misunderstand the argument, cite a wrong source, implement the wrong thing), the finding is polish. Standalone documents (whitepapers, presentations) use the low-propagation threshold — the reader is the only consumer. Pipeline documents (requirements, test cases, specs) use a higher threshold because errors propagate downstream.
  2. Propagation. Does the finding cause a problem in another artefact? A wrong requirement ID propagates through test cases, specs, and code. A left-aligned number in a sources table propagates nowhere.
  3. Diminishing returns. Each review round holds new findings to a higher bar. Round 1 catches structural problems. Round 3 should not surface “redundant sentence” or “right-align this column.”

The stop condition: when all remaining observations fail the consumer impact test, the document is done. A review that produces only polish observations reports “No actionable findings” — a clean review is a valid outcome.

Tips

  • Run early and often. The review is most valuable when run after each pipeline stage, not just at the end. Problems caught early are cheap to fix.
  • Use it as a gate. Before moving from /gvm-requirements to /gvm-test-cases, run /gvm-doc-review and resolve any Critical issues. Before moving from /gvm-tech-spec to /gvm-build, run it again.
  • Fix Critical issues first. The tier system exists for a reason. Ignore Minor issues until all Critical and Important issues are resolved.
  • Track improvement over time. Each review is date-stamped. Compare reports to confirm that fixes actually landed and scores improved.
  • Don't chase a perfect 10. A score of 7–8 across all documents with no Critical issues is a strong, buildable project. Diminishing returns set in above 8.
  • Re-review after edits. If you fix issues manually (outside of the pipeline commands), run /gvm-doc-review again to confirm the fixes are correct and haven't introduced new problems.
  • Use requirements-health-report-decisions.md. If a review flags something you have intentionally decided to accept, document that decision in a health-report-decisions file. The review will respect acknowledged decisions on subsequent runs.
Best practice

Run /gvm-doc-review before every pipeline transition: after /gvm-requirements, after /gvm-test-cases, and after /gvm-tech-spec. Three reviews across a project lifecycle takes minutes and prevents hours of rework.

Light Mode

For targeted reviews — when only a single document has changed and prior reviews exist — the skill offers light mode. Light mode skips cross-document consistency checks and focuses solely on the criteria for the changed document.

Light mode is appropriate when:

  • You have made focused edits to one document (e.g., tightened requirements wording) and want to confirm the document-level score without re-running the full consistency audit
  • A prior full review already established baseline cross-document consistency and nothing else has changed
  • You are iterating quickly on a single document and want fast feedback between edits

Light mode is not appropriate when:

  • This is the first review of the project
  • Changes to one document may have introduced inconsistencies with others (e.g., requirement IDs changed, spec sections renamed)
  • You are reviewing before a pipeline transition — always run a full review at gates
How to invoke light mode

Pass the specific file path and prior review: /gvm-doc-review requirements/requirements.md. The skill detects that only one file is specified and that prior reviews exist, then offers light mode as an option. You confirm before it skips the consistency phase.
