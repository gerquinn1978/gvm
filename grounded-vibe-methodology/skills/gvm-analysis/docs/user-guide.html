/gvm-analysis

Privacy-preserving exploratory data analysis — Claude orchestrates a Python engine that reads your CSV, produces a grounded findings report, and never lets raw rows enter its context.

Overview

/gvm-analysis is the GVM data-analysis skill. Point it at a CSV (or TSV / xlsx / parquet / JSON) and it produces a structured findings.json plus a self-contained HTML hub. Four modes cover the main analytical patterns: Explore (descriptive only), Decompose (drivers when a target column is supplied), Validate (drivers plus baseline comparison), and Run-everything (every block fires).

Privacy boundary: the Python engine reads your input file. The orchestrating Claude reads findings.json, which carries aggregates only — never raw cell values. NFR-1 (the project’s first non-functional requirement — "raw rows never leave the engine") is enforced by an end-to-end sentinel-string privacy audit.

The skill is the engine + renderer behind the slash command. A direct-CLI invocation (python scripts/analyse.py ...) produces the same output as an LLM-orchestrated run, with one difference noted in the Comprehension Questions section below.

When to Use

  • First-pass exploratory analysis on any tabular data — understand the shape, find outliers, surface time-series patterns, identify drivers, before reaching for full statistical software.
  • Privacy-sensitive data — the engine never echoes cell values into output. The optional anonymisation pipeline (scripts/anonymise.py) tokenises high-cardinality columns before analysis, with a deterministic round-trip via scripts/de_anonymise.py.
  • When you need provenance — every report carries SHA-256 of the input, library versions, deterministic seed, and a methodology appendix. The same input always produces the same report.
  • Baseline comparison — Validate mode joins your input against a baseline file and runs driver decomposition to surface what changed between the two.

Invocation

Through the Claude skill, call the slash command: 

/gvm-analysis path/to/input.csv

Claude orchestrates mode selection, target-column choice (when applicable), and the comprehension-question synthesiser. The orchestrated path is what most users want.

Direct CLI invocation runs the same engine without an orchestrator: 

python3 scripts/analyse.py \
    --input path/to/input.csv \
    --output-dir /tmp/analysis \
    --mode run-everything \
    --target-column outcome

python3 scripts/render_report.py \
    --findings /tmp/analysis/findings.json \
    --out /tmp/analysis

The engine writes findings.json; the renderer writes report.html alongside. Both are self-contained — the report opens in any browser without external assets.

Comprehension Questions

Every report carries exactly three plain-language Q/A pairs. The questions help any reader (statistical background or not) understand the report's most important findings without consulting the analyst.

Two-path semantics

The engine produces deterministic baseline questions from the headline findings — structurally-real, reference real headline IDs, plain language. Direct-CLI invocations ship these.

Under SKILL.md orchestration with an LLM in the loop, the orchestration step overwrites the deterministic baseline with richer LLM-synthesised content. The LLM reads the headlines, writes more contextual questions, and patches them in via scripts/_patch_questions.py.

Either path produces three plain-language questions referencing real headlines. The orchestrated path is more contextually-rich; the deterministic path is reproducible without an LLM.

Privacy

NFR-1: raw cell values must not appear in any output artefact under --output-dir. The boundary is enforced end-to-end by tests/integration/test_privacy_audit.py, which drives the engine against a fixture whose every cell is a unique sentinel string and asserts no sentinel appears in findings.json, the rendered HTML, or any other file the run produces.

The duplicates module specifically returns aggregates only (group counts, row indices, similarity scores) — never the duplicate values themselves. The internal find_exact and find_near helpers continue to return value-bearing shapes for in-engine use, but the public summariser strips values at the JSON boundary.

Anonymisation

When the input contains identifying values, run them through the anonymisation pipeline before passing to the analysis engine: 

python3 scripts/anonymise.py \
    --input raw.csv \
    --output anonymised.csv \
    --mapping-out mapping.csv \
    --cols email,phone \
    --i-accept-the-risk

python3 scripts/analyse.py \
    --input anonymised.csv \
    --output-dir /tmp/analysis \
    --mode explore

The mapping file is the only place where original values live; keep it on a trusted filesystem. To reverse the tokenisation in a rendered report: 

python3 scripts/de_anonymise.py \
    --input /tmp/analysis/report.html \
    --mapping mapping.csv \
    --output report.deanonymised.html

AN-40 detection: if the engine notices an input whose values match the TOK_<column>_NNN token pattern in ≥ 80% of one or more columns, the provenance footer notes "input appears anonymised" and lists the matched columns.

Provenance

Every run records:

  • SHA-256 of every input file (and baseline file in Validate mode)
  • ISO-8601 UTC timestamp of the run
  • Resolved seed: explicit --seed value, or a deterministic 31-bit derivation from the input file's SHA-256 prefix when --seed is omitted (per ADR-202 — the decision record governing deterministic seed derivation)
  • Sub-seeds for every analytical module that uses RNG — outliers, drivers, forecasts, per-column bootstrap CIs — so reproducibility is column-isolated
  • Library versions captured via importlib.metadata
  • Preferences hash (when a prefs file is supplied)
  • AN-40 anonymisation flag and the columns that triggered it

Same input → same report. The deterministic seed default plus column-isolated sub-seeds mean two runs of the same input file produce byte-identical findings. Different inputs produce different seeds; reproducibility holds without coordinating seed values manually.

Pipeline Position

/gvm-analysis is a standalone GVM skill — it does not sit in the build-pipeline sequence. It produces a self-contained report from a CSV; nothing downstream consumes its output. Invoke it whenever you have data to analyse.

Three common situations to reach for it: before /gvm-requirements to understand the shape of the data the product will work with; after /gvm-build to validate that real outputs match the expectations the requirements set; or independently of any project when someone hands you a CSV and asks "what's in this?". The skill runs the same way in all three contexts.

Standalone /gvm-analysis <your-data.csv> → findings.json + report.html

The skill itself was built using the full GVM build pipeline (/gvm-impact-map → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → /gvm-design-review → /gvm-walking-skeleton → /gvm-build → /gvm-code-review → /gvm-test). The analysis work is the case study under the methodology, not part of the methodology pipeline.
