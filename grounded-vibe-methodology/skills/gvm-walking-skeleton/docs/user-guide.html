/gvm-walking-skeleton

Discovery + scaffolding skill — wires a thin end-to-end execution thread through every external boundary the project touches, surfacing integration failure modes at hour 1 rather than month 6.

Overview

/gvm-walking-skeleton produces two artefacts: a runnable walking-skeleton/ directory with a single test that exercises every external boundary (HTTP outbound, database, cloud SDK, filesystem, subprocess, email), and a project-root boundaries.md registry of those boundaries with their wiring status. The skeleton is an architectural spike, not production code — the integration thread runs end-to-end, but no feature behaviour is implemented.

Freeman & Pryce’s Growing Object-Oriented Software, Guided by Tests is the structural model. Cockburn names the discipline in Crystal Clear: keep it thin. Reinertsen names the failure mode the skeleton mechanically prevents — horizontal slicing defers integration risk to month 6.

Run /gvm-walking-skeleton after /gvm-tech-spec and before /gvm-build. The boundaries registry becomes the authority that downstream skills gate against: /gvm-build Hard Gate WS-5 (the fifth walking-skeleton cross-skill check) refuses chunks if the skeleton is red in CI; /gvm-test VV-4(d) (the verdict-evaluator’s fourth criterion, branch d — sandbox divergence) audits sandbox-divergence notes against the same registry.

When to Use

  • Greenfield projects with multiple external integrations — before any feature work begins, when auth, CORS, rate limits, and connection strings would otherwise surface during integration weeks later.
  • Projects switching providers (e.g. moving from one payment processor to another) — the skeleton verifies the new provider’s contract end-to-end before consuming chunks rely on it.
  • When the boundary set is non-trivial — three or more external dependencies of mixed kind (DB + HTTP + cloud SDK) is the threshold above which an explicit boundaries registry repays its cost.
When to skip

If the project has zero or one external boundary (a single database, no third-party APIs), the registry's cost exceeds its value. The skill is opt-in: WS-5 only fires when boundaries.md exists, so the gate is inactive on projects that never ran the skill. Any project that touches multiple external boundaries should still run it — the inactive gate is not a free pass.

The Eight-Phase Flow

Table 2 — Phases
PhaseWhat happens
0. Bootstrap Pre-flightVerify ~/.claude/gvm/ exists; load architecture experts and the walking-skeleton domain spec.
1. Prereq checkRefuse to run if specs/cross-cutting.md or requirements/requirements.md are missing. If walking-skeleton/ already exists, confirm overwrite.
2. DiscoverySource-scan via _discovery.scan() using boundary-discovery heuristics. Each candidate becomes a draft row; the practitioner confirms each via AskUserQuestion. Free-text capture for boundaries the scan missed.
3. Provider selectionPer boundary: provider, endpoints, cost, SLA. Mark wired / wired_sandbox / deferred_stub. Write the row immediately so the skill is resumable if interrupted.
4. Skeleton generationGenerate walking-skeleton/test_skeleton.{py,ts,go}, per-boundary client wrappers (real auth wired), and stubs under walking-skeleton/stubs/ for deferred boundaries. Single-flow lint refuses skeletons with more than three tests.
5. CI integrationDetect CI provider via filesystem markers; offer to add a walking-skeleton job. Falls back to Makefile + RUNBOOK if no CI is detected.
6. First runPractitioner executes the skeleton locally. Pass/fail recorded in walking-skeleton/.first-run.log. The point of WS-3: a red first run is the signal to fix auth / CORS / rate limits NOW, before feature work begins.
7. ApprovalSkeleton summary via AskUserQuestion. On approval the skill exits; /gvm-build can proceed (subject to the WS-5 red-skeleton CI status check).
First-run failure is the feature, not a bug

If the skeleton fails on the first execution, the skill has done its job: integration risk surfaced at hour 1 instead of month 6. The practitioner debugs auth / CORS / rate limits / connection strings; the skill does not advance to approval until the first run is green (or the practitioner explicitly accepts a red status with a rationale).

Boundaries Registry

The boundaries.md registry is a flat table at the project root. One row per external boundary; row schema versioned in YAML frontmatter.

Table 3 — Row shape
ColumnRequiredDescription

Hard Gates

Non-negotiable steps. If any are skipped, the walking-skeleton output is invalid.

  1. Every external dependency registered in boundaries.md. The source-scan in Phase 2 produces draft rows; the practitioner confirms each via AskUserQuestion. No undeclared boundary may be left unregistered when the skill exits.
  2. Every boundary either wired or deferred — no in-between. Each row carries real_call_status ∈ {wired, wired_sandbox, deferred_stub}. wired_sandbox requires a non-trivial divergence note; deferred_stub requires a corresponding STUBS.md entry with expiry.
  3. Mocked-boundary refusal at validation. A wired boundary must be exercised by a real (or sandbox) runtime call. The validator inspects the skeleton test file and refuses skeletons whose tests mock instead of calling.
  4. Paired MD + HTML before approval. boundaries.md and boundaries.html exist at project root before the approval prompt fires.
  5. CI hook installed on completion. The skill writes (or amends) a CI job, or falls back to a Makefile + RUNBOOK. The CI hook is what makes the WS-5 red-skeleton refusal in /gvm-build enforceable downstream.

Output

Two artefacts: 

boundaries.md
boundaries.html

  test_skeleton.{py,ts,go}

  stubs/  (for deferred boundaries)
  .first-run.log

Plus a CI job entry in your existing CI configuration (or a Makefile + RUNBOOK if no CI was detected).

How /gvm-build and /gvm-test Consume This

  • WS-5 red-skeleton refusal in /gvm-build. Before any chunk runs, /gvm-build queries CI status for the walking-skeleton job. If red, the chunk is refused with a pointer to fix the integration first. The honour-system fallback (manual confirmation when CI is unavailable) writes a sidecar that /gvm-test later audits.
  • VV-4(d) sandbox-divergence audit in /gvm-test. The verdict evaluator reads boundaries.md and refuses Ship-ready when wired_sandbox rows lack a divergence note or when manual WS-5 fallbacks were used without later CI confirmation.

The registry is the contract. Adding a new external boundary later means re-running the skill or updating the registry by hand and re-running the validator. The downstream gates read the registry, not the source.

Pipeline Position

Greenfield /gvm-init → /gvm-impact-map → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → /gvm-design-review → /gvm-walking-skeleton → /gvm-build → /gvm-code-review → /gvm-test → /gvm-explore-test → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Sits between /gvm-tech-spec (which names the architectural boundaries) and /gvm-build (which gates against the resulting registry). The boundaries registry is the authority for downstream skills; if the skill is skipped, the gates that read it become no-ops.

Relationship to MVP-1

The walking skeleton is the mechanical enabler of MVP-1 (Minimum Viable Product first ordering). MVP-1 is the rule that /gvm-tech-spec Phase 5 sequences chunks so the first user-visible flow lands first, then later chunks add slices; /gvm-build Hard Gate 9 mirrors the rule on the read side, and /gvm-design-review Panel D Pass 3 flags designs that structurally prevent it. This skill produces the wiring that the first slice runs on. Even projects claiming an MVP-1 exemption (library, refactor, performance-driven, fully-specified) may still need the skeleton if they touch external boundaries — the exemption is about chunk sequencing, not about whether boundaries need wiring.
