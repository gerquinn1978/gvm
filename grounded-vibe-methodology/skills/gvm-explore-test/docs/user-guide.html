/gvm-explore-test

Practitioner-driven exploratory testing — runs a timeboxed charter against built code, captures defects in Given/When/Then form, and produces an explore-NNN session report that /gvm-test consumes for VV-4(d).

Overview

/gvm-explore-test is the structural skill behind a human-driven exploratory testing session. It writes a charter (YAML frontmatter), validates it, times the session, captures defects through AskUserQuestion-driven prompts (severity, Given, When, Then, reproduction), and on debrief writes paired test/explore-NNN.md and test/explore-NNN.html files. The artefact is what /gvm-test reads when evaluating the VV-4(d) gate (Critical-blocks-Ship-ready).

Bach & Bolton’s Rapid Software Testing, Hendrickson’s Explore It!, Whittaker’s tour catalogue. The skill is structural — the charter, the GWT discipline, the artefact contract — not analytic substitution for the human tester.

Run /gvm-explore-test after /gvm-test verifies the build and before /gvm-doc-write produces release notes. The session is timeboxed (the charter declares a duration). Severity is the practitioner’s call — the skill never auto-classifies.

When to Use

  • Pre-release exploratory tour — after structured tests pass, when the team needs human-driven testing against the built product to surface emergent behaviour structured tests miss.
  • After a significant change to a critical user flow — an explicit charter against the changed flow catches integration surprises that unit and integration tests cannot frame.
  • To document the team’s testing intent — the charter is an artefact that says “we explored these tours, in this timebox, and this is what we found”. Useful for audits and post-incident reviews.
When to skip

If the change is small, scoped, and covered by structured tests, exploratory testing adds friction without proportional value. The skill is opt-in — /gvm-test VV-4(d) treats a missing charter as a no-op rather than a refusal.

The Five-Phase Flow

Table 2 — Phases
PhaseWhat happens
0. Bootstrap Pre-flightVerify ~/.claude/gvm/ exists; load architecture and exploratory-testing experts.
1. CharterWrite stub at test/explore-NNN.charter.yml; practitioner fills in mission, tour, scope, duration; validator runs. On error the skill refuses to start the session.
2. SessionPractitioner explores the product against the charter. Wall-clock timer enforces stop-at-time. The skill stays out of the way — no auto-narration of what the practitioner should test.
3. Defect intakeFor each finding, AskUserQuestion sequence: severity (Critical/Important/Minor/Observation), Given, When, Then, reproduction. Blank reproduction at ≥ Minor severity re-files as Observation.
4. DebriefAskUserQuestion: overall assessment of the session against the mission. Captured into the artefact.
5. Report writePaired MD + HTML at test/explore-NNN.{md,html}. Atomic write — either both exist or neither does. Every practitioner-supplied string HTML-escaped.
Severity is the practitioner’s call

The skill never up- or down-grades severity. The practitioner’s classification flows verbatim into /gvm-test VV-4(d) (the verdict-evaluator’s fourth criterion, branch d — explore-test artefacts), which uses Critical findings to refuse Ship-ready. If the practitioner says Important, the skill records Important — even if the LLM thinks the defect “sounds like Critical”. Authority lives with the human in the loop.

The Charter

Phase 1 of the flow above writes a charter stub at test/explore-NNN.charter.yml; the practitioner fills it in. The charter is YAML frontmatter; the validator checks every field by name before the session is allowed to start.

Table 3 — Charter fields
FieldRequiredDescription
session_idYesStable identifier (explore-NNN with NNN zero-padded to three digits).
schema_versionYesSchema version of the charter format. Tracks evolutions.
runnerYesPractitioner running the session.
missionYes (ET-2)The session’s purpose — what are we trying to learn about the product?
tourYes (ET-2)One of feature / data / money / interruption / configuration — Whittaker’s catalogue.
scopeYes (ET-2)Which part of the product is in/out of bounds for this session.
durationYes (ET-2)Timebox in minutes. The skill enforces stop-at-time.

“ET-2” identifies the four mandatory charter fields in the explore-test contract; the validator names each by its ET-2 identifier in error reports.

Hard Gates

Non-negotiable steps. If any are skipped, the session output is invalid.

  1. Charter validated before the session starts. The skill writes a stub at test/explore-NNN.charter.yml, the practitioner fills it in, and the validator checks the four ET-2 fields plus schema_version, session_id, and runner. On any error the skill refuses to start the session and reports the offending field by name.
  2. Paired MD + HTML before debrief finalised. Both test/explore-NNN.md and test/explore-NNN.html exist atomically before the skill announces the charter closed.
  3. Defect without reproduction → observation. Every entry of severity ≥ Minor requires a reproduction path. A blank reproduction at the prompt re-files the entry as an Observation (no severity, separate ID space) — not rejected, not silently dropped.
  4. Severity is the practitioner’s call. The skill never auto-assigns severity. The practitioner picks Critical / Important / Minor / Observation through AskUserQuestion. LLM second-guessing is forbidden.
  5. Every practitioner-supplied string HTML-escaped in the report. The HTML renderer uses html.escape() on every field before substitution. No template-engine raw-string escape hatches.

Defect & Observation Format

Every defect entry uses Given/When/Then to make reproduction unambiguous. Observations carry no severity and no reproduction requirement — they capture context, surprise, or incidental findings without forcing them into a defect frame.

Reproduction is the load-bearing field. A defect without a reproduction path can’t be fixed; it can’t even be confirmed. The skill enforces this by re-filing severity-bearing entries with blank reproduction as Observations — preserving the data without claiming it’s actionable.

How /gvm-test Consumes This

The VV-4(d) gate in /gvm-test step 8 (Evaluate Verdict) reads the latest test/explore-NNN.md charter via _explore_parser.load(...). Open Critical findings refuse Ship-ready; Important findings cap at Demo-ready; Minor and Observation are surfaced but do not block. A missing charter is treated as a no-op (the skill is opt-in).

Paired files per session, numbered:
