# Tufte/Few HTML Document — Core Design Reference

Shared HTML/CSS patterns for all document output. Apply **Tufte** (*The Visual Display of Quantitative Information*, *Envisioning Information*) and **Few** (*Show Me the Numbers*, *Information Dashboard Design*) principles — their core rules (data-ink ratio, chartjunk removal, tables over charts for precision, subtle zebra striping, right-aligned numbers) are well-documented and the model should apply them directly.

## Companion Files (load only if needed)

This file contains the CORE patterns used by every document. Load the companion files only when writing that specific output type:

| If writing… | Also load |
|---|---|
| Review report (doc-review, code-review, design-review, test) | `tufte-review-components.md` — verdict box, score card, issue blocks, criterion rows |
| Tech spec (ADRs, component disclosures) | `tufte-spec-components.md` — ADR and component-detail CSS |
| Presentation (slide deck) | `tufte-slide-template.md` — self-contained slide HTML/CSS/JS |

Presentations are self-contained — load `tufte-slide-template.md` instead of this file, not alongside.

## Document Structure

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{{Project Name}} — {{Document Type}}</title>
  <style>
    /* See CSS section below */
  </style>
</head>
<body>
  <nav class="toc" id="toc">
    <div class="toc-title">Contents</div>
    <!-- Auto-generated by script at bottom -->
  </nav>

  <main>
    <p class="date">{{Month Year}} — Living Document</p>
    <h1>{{Project Name}}</h1>
    <p class="subtitle">{{One-line description of the project and this document's purpose.}}</p>

    <!-- Sections follow -->
  </main>

  <script>/* TOC + scroll highlight script */</script>
</body>
</html>
```

## TOC Script

Place at the end of `<body>`, after all content:

```javascript
(function () {
  const toc = document.getElementById('toc');
  const headings = document.querySelectorAll('h2[id], h3[id]');
  const links = [];

  headings.forEach(function (h) {
    const a = document.createElement('a');
    a.href = '#' + h.id;
    a.textContent = h.textContent;
    if (h.tagName === 'H3') a.classList.add('toc-h3');
    toc.appendChild(a);
    links.push({ el: a, target: h });
  });

  function onScroll() {
    let current = links[0];
    for (let i = 0; i < links.length; i++) {
      if (links[i].target.getBoundingClientRect().top <= 80) current = links[i];
    }
    links.forEach(function (l) { l.el.classList.remove('active'); });
    if (current) {
      current.el.classList.add('active');
      const tocEl = document.getElementById('toc');
      const elTop = current.el.offsetTop - tocEl.offsetTop;
      const elBot = elTop + current.el.offsetHeight;
      if (elTop < tocEl.scrollTop || elBot > tocEl.scrollTop + tocEl.clientHeight) {
        tocEl.scrollTop = elTop - tocEl.clientHeight / 2;
      }
    }
  }

  window.addEventListener('scroll', onScroll, { passive: true });
  onScroll();
})();
```

## Sidenote Usage

Sidenotes appear beside the content they annotate. Use them for caveats, source citations ("Per Wiegers, Ch. 16…"), cross-references to related requirements, or data integrity notes.

```html
<span class="sidenote">
  This constraint applies to all financial figures. See DI-1.
</span>
```

Place the `<span class="sidenote">` immediately after the paragraph or requirement block it annotates.

## Requirement Block Pattern

```html
<div class="req">
  <div class="req-id">{{DOMAIN}}-{{NUMBER}} <span class="priority {{must|should|could}}">{{PRIORITY}}</span></div>
  <p>{{Requirement text}}</p>
</div>
```

**Domain prefixes** — use short, consistent 2-3 character prefixes per domain (e.g., RE Real Estate, AU Authentication, PY Payments, DT Data). Adapt to the project.

## Table Patterns

**Requirements Index** (always the second-to-last section):

```html
<table>
  <caption>Table N — All requirements at a glance</caption>
  <thead>
    <tr><th>ID</th><th>Domain</th><th>Summary</th><th>Priority</th></tr>
  </thead>
  <tbody>
    <tr><td>RE-1</td><td>Real Estate</td><td>Summary text</td><td><span class="priority must">Must</span></td></tr>
  </tbody>
</table>
```

**Comparison Tables** — same structure, descriptive caption, right-align numeric columns using inline `style="text-align:right"` or a `.num` class.

## Section Pattern

Every major section uses an `h2` with an `id` for TOC linkage. Subsections use `h3` with `id`.

```html
<h2 id="purpose">Purpose &amp; Vision</h2>
<p>Introductory paragraph for the section.</p>

<h3 id="target-user">Target User</h3>
<p>Content…</p>
```

## Cross-Cutting Section Patterns

**Assumptions / Constraints / Out of Scope** — simple list sections:

```html
<h2 id="assumptions">Assumptions</h2>
<ul>
  <li><strong>Single user system</strong> — no multi-tenancy or role-based access required.</li>
</ul>
```

**Open Questions** — visually distinct to signal unresolved items (uses `.open-question` CSS above):

```html
<h2 id="open-questions">Open Questions</h2>
<div class="open-question">
  <p><strong>OQ-1:</strong> Should the export include historical data or only current snapshot?</p>
  <p class="oq-context">Raised during data domain discussion. Affects storage and performance requirements.</p>
</div>
```

**Priority Model** — MoSCoW definition table:

```html
<h2 id="priority-model">Priority Model</h2>
<table>
  <caption>Table N — MoSCoW priority definitions</caption>
  <thead><tr><th>Level</th><th>Meaning</th></tr></thead>
  <tbody>
    <tr><td><span class="priority must">Must</span></td><td>Non-negotiable. The product fails its purpose without this.</td></tr>
    <tr><td><span class="priority should">Should</span></td><td>Expected by users. Omit only with clear justification.</td></tr>
    <tr><td><span class="priority could">Could</span></td><td>Desirable enhancement. Include if effort is low.</td></tr>
    <tr><td><span class="priority wont">Won't</span></td><td>Explicitly deferred to a future version. Not in scope for this release.</td></tr>
  </tbody>
</table>
```

## Key Rules

1. Every `h2` and `h3` must have an `id` attribute (kebab-case) for TOC generation
2. Requirement IDs must be unique across the document
3. Every requirement must have exactly one MoSCoW priority
4. Sidenotes must be placed adjacent to their relevant content
5. Tables must have captions with sequential numbering
6. Use `&amp;` for ampersands in HTML text content
