What’s new in v2.2.0

v2.2.0 closes the last v2.1.x issue: TS and Go projects now get mock-budget detection from /gvm-code-review Panel B, matching Python.

  • TypeScript mock-budget detection. The detector now recognises the vitest, jest, testdouble, and sinon mock APIs — vi.mock, vi.spyOn, jest.mock, jest.spyOn, td.replace, sinon.stub, and sinon.replace — via _lint_typescript and _TS_STACK_DEFAULTS. The allowlist treats the following as external boundaries: node:* builtins, major HTTP clients (axios, undici, got, ky, node-fetch), DB drivers (pg, mongodb, redis), and AI/cloud SDKs. Internal mocks emit Important; matches in .cross-chunk-seams escalate to Critical. The wrapper-as-SUT exemption applies when the test-file stem equals the mocked target’s basename — matching ADR-MH-02.
  • Go mock-budget detection. Adapted to Go: the detector counts distinct test-double instances per func TestXxx(t *testing.T) body. Three signals (ADR-TG-02): types ending in Mock|Fake|Stub|Spy, gomock.NewController invocations, and types embedding testify’s mock.Mock (recognised by the embedded library type, not by struct name). _GO_STACK_DEFAULTS covers net/http, os, io, database/sql, context, plus AWS/Google/Anthropic/OpenAI/Redis/MongoDB/Postgres SDK package paths.
  • Per-project seam allowlists now apply on TS and Go. Previously the per-stack readers silently dropped the _lint_typescript.seam_allowlist and _lint_go.seam_allowlist parameters — they are now wired into the per-stack detector. STUBS.md retires both entries as a result.

See methodology-changelog.md for the audit trail.

What’s new in v2.1.0

v2.1.0 hardens the methodology: same pipeline, stricter gates. See methodology-changelog.md for the audit trail. Identifier prefixes (NFR, ADR, TDD, REVIEW, GATE) decode in the changelog.

  • Build — chunk-acceptance smoke gate (Hard Gate 8). Every chunk runs a literal smoke command exercising its user-facing surface and asserts structural contracts on the output before the handover is written. Chunks built before 2026-04-28 are exempt as “v2.0.x carry-over”; pure internal helpers may claim exemption with rationale (see ADR-MH-04 in the changelog).
  • Build — TDD-1 outside-in ordering. The acceptance test that exercises the user-facing surface must be the first deliverable for user-facing chunks — written before any unit test.
  • Build — TDD-2 mock-budget escalation. Mock-budget violations escalate from Important to Critical when the target appears in the project’s opt-in .cross-chunk-seams file.
  • Build — TDD-3 realistic-fixture catalogue. Six-domain catalogue of realistic-fixture starter shapes (data-analysis, web/UI, API, parsing, security validation, concurrency). Panel C in /gvm-code-review currently emits Important findings in three of the six.
  • Reviews — user owns finding triage (shared rule 28). Every Critical or Important finding from /gvm-code-review, /gvm-design-review, and /gvm-doc-review is presented to you before any disposition is recorded. The skill may recommend “fix / defer / dismiss” but must not decide. (Hard Gate 6 in code-review and design-review; Hard Gate 7 in doc-review.) Added after an incident where the skill unilaterally deferred 2 findings and filtered 10.
  • Reviews — EBT contract / collaboration lint (REVIEW-1). /gvm-code-review Panel B runs the EBT contract lint against every test file, flagging HTTP transport patched in consumers, internal-class mocks in collaboration-contract tests, and TDD-2 mock-budget violations. (Mock-budget detection covered Python at v2.1.0 and was extended to TypeScript and Go in v2.2.0 — see “What’s new in v2.2.0” above.)
  • Diff-budget tracking (NFR-2). Each chunk handover records a Diff budget delta integer; the build summary aggregates per-chunk values and prompts you when cumulative SKILL.md additions cross 500 LOC.
  • Methodology-changelog inclusion heuristic (NFR-3). /gvm-status §6 documents the heuristic for what gets a changelog entry — behaviour-changing methodology evolution requires an entry; typo and formatting edits do not.

Getting Started

AI coding assistants without spec discipline produce code that drifts from intent. Requirements stay vague, test cases miss edge cases, architecture decisions go unrecorded, and the team has no audit trail when something breaks. "Vibe-coded" output looks plausible but cannot be traced back to a stated need.

The Grounded Vibe Methodology (GVM) addresses that drift. Each pipeline phase activates named experts — Wiegers, Beizer, Clements, Fairbanks, Beck, and others — whose published frameworks guide the AI's work, instead of relying on undifferentiated training data. The result is more complete requirements, more thorough test cases, and better-documented architecture decisions, all traceable through the pipeline. Expert names and their contributions are detailed in Tier 1 Expert Sources below. This plugin is the reference implementation of GVM for Claude Code.

What you need

  • Claude Code with the GVM plugin installed

The Pipeline

Entry points (pick one) /gvm-init — greenfield, calibrate experts before anything else (recommended for specialised domains: financial services, healthcare, regulated)
/gvm-impact-map — greenfield, outcome-framing first (skip /gvm-init when general-purpose software needs no specialised calibration)
/gvm-site-survey — existing codebase (routes to /gvm-requirements, /gvm-tech-spec, or /gvm-build based on diagnosis)

Greenfield pipeline [/gvm-init →] [/gvm-impact-map →] /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → /gvm-design-review → /gvm-walking-skeleton → /gvm-build → /gvm-code-review → /gvm-test → /gvm-explore-test → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

Quality gate (any phase) /gvm-doc-review — cross-cutting audit, usable at any stage

Standalone documents (any point) /gvm-doc-write — create or revise presentations, whitepapers, and any prose document

Standalone analysis (any point) /gvm-analysis — privacy-preserving exploratory data analysis on any tabular file

Diagnostic (any point) /gvm-status — shows pipeline progress, staleness, and expert engagement

Square brackets [/gvm-init →] and [/gvm-impact-map →] mark optional steps. Both are recommended for greenfield projects but neither is a hard prerequisite for downstream skills. The shared bootstrap rule creates ~/.claude/gvm/ on first invocation of any skill, so a project that goes straight to /gvm-requirements still has a valid GVM home directory.

Each phase reads the output of the previous phase. The pipeline is sequential but not rigid — you can re-enter at any point if you already have the prerequisite artefacts.

Quick Start: Greenfield Project

Starting a new project from scratch. This walkthrough assumes the plugin is installed.

Step 1 — Calibrate experts (recommended, not required)

Run /gvm-init. It asks one question — your business domain — then selects industry domain experts, scores all unscored experts, and writes a calibration report.

When init matters

Init is recommended for specialised domains (financial services, healthcare, regulated industries) where industry experts should be loaded before requirements start. For general-purpose software, skip it and go straight to /gvm-requirements — the pipeline discovers and scores experts automatically.

On completion, init writes a calibration report to init/init-001.html and init/init-001.md with six sections: executive summary, Tier 1 decisions, Tier 2b coverage, expert roster summary table, activation log status, and next steps. Stack specialists and component-level domains are not configured here — they are handled by /gvm-tech-spec and /gvm-requirements respectively, where that information is available.

Step 2 — Gather requirements

Run /gvm-requirements with your project idea. Describe what you want to build. The skill runs an elicitation conversation grounded in Wiegers' methodology, then produces a structured requirements document with MoSCoW priorities.

Step 3 — Generate test cases

Run /gvm-test-cases. It reads your requirements document and generates acceptance tests. Review the output — the test cases become the contract that the build phase must satisfy.

Step 4 — Write the spec

Run /gvm-tech-spec. It reads requirements and test cases, then produces one or more specification documents depending on project complexity. The spec includes an implementation guide that the build phase consumes.

Step 5 — Design review

Run /gvm-design-review. It reads the spec and validates data models, UI/UX, and API contracts against requirements before any code is written. Fix findings before proceeding.

Step 6 — Build

Run /gvm-build. It reads the implementation guide from the spec and executes it chunk by chunk, writing code and tests for each chunk. Each chunk produces a handover file so you can resume across sessions.

Step 7 — Review and verify

Run /gvm-code-review for a full codebase review. Fix any findings. Then run /gvm-test to verify the build works as a product — integration seam audit, acceptance test verification, startup verification, smoke testing, and a stub vs implemented audit.

Step 8 — Documentation

Run /gvm-doc-write in pipeline docs mode to generate user-facing documentation (README, user guides, API docs) grounded in the verified build. Then run /gvm-doc-review to audit the documentation quality.

Optional quality gate

Run /gvm-doc-review after any document-producing phase (requirements, test cases, specs) to audit quality before proceeding. This is not mandatory but catches issues early. After /gvm-test, the documentation phase (/gvm-doc-write → /gvm-doc-review) is the final pipeline step.

Quick Start: Existing Codebase

Working with code that already exists. The site survey replaces /gvm-init as your entry point.

Step 1 — Run the survey

Run /gvm-site-survey from your project directory. The survey explores the codebase, scores it across multiple health dimensions (coherence, currency, testability, modularity, documentation, dependency health), and produces a diagnosis.

Step 2 — Review the diagnosis

The survey classifies the codebase into one of five diagnostic scenarios: Coherent, Coherent but Outdated, Fractured, Mid-migration, or Incoherent. Each diagnosis comes with a recommended pipeline entry point.

Step 3 — Follow the recommended route

The survey tells you where to enter the pipeline based on both the codebase state and the nature of your intended work. A new user-facing feature on a coherent codebase might route directly to /gvm-requirements. Technical remediation on a fractured codebase might route to /gvm-tech-spec. If the survey found existing documentation — requirements, test plans, architecture docs in any format — it will recommend feeding them as seed input to the corresponding pipeline phase, accelerating the process without skipping the GVM quality checks. Follow the recommendation and continue through the pipeline from that point.

The site survey selects experts based on what it finds in the codebase — the technology stack, architectural patterns, and problem areas. You do not need to run /gvm-init separately.

Example: A Project from Start to Finish

A concrete walkthrough of a greenfield project — a neighbourhood comparison tool — showing what you type, what GVM produces, and what happens at each phase.

1. Requirements

You: /gvm-requirements

I want to build a tool that helps people compare neighbourhoods
when they're thinking about moving. It should look at schools,
commute times, and housing prices, and produce a report they
can share with their partner.

GVM loads the requirements expert panel (Wiegers, Patton, Cooper, Christensen/Moesta) and begins an elicitation conversation. It asks one question at a time:

  • “Who is the primary user? A first-time buyer, someone relocating for work, a family upsizing?”
  • “When you say ‘compare neighbourhoods’, do you mean side-by-side comparison of specific areas, or discovery of areas that match criteria?”
  • “What data sources are available? Government APIs, estate agent listings, school league tables?”

After 10–15 questions, it writes the requirements document with MoSCoW priorities.

Output: requirements/requirements.html + requirements/requirements.md

2. Test Cases

You: /gvm-test-cases

GVM reads the requirements document, runs a health report (catching untestable or ambiguous requirements), then generates Given/When/Then test cases per domain. For a MUST requirement like “users can compare up to 3 neighbourhoods side by side”, it generates boundary tests (0 areas, 1 area, 3 areas, 4 areas) using Copeland’s boundary value analysis.

Output: test-cases/test-cases.html + test-cases/test-cases.md

3. Tech Spec

You: /gvm-tech-spec

GVM reads requirements and test cases, assesses complexity, and asks about the tech stack. For this project it might produce 4 specs: cross-cutting conventions, a data integration domain spec, a comparison engine domain spec, and an implementation guide with 8 build chunks across 3 phases. Stack specialists (e.g., Vanderkam for TypeScript, Winand for SQL) activate at this point.

Output: specs/cross-cutting.md, specs/data-integration.md, specs/comparison-engine.md, specs/implementation-guide.md (+ paired HTML)

4. Build

You: /gvm-build

GVM reads the implementation guide and presents the dependency graph. It builds one chunk at a time with strict TDD — failing test first, minimal code to pass, review against expert principles, commit. Each chunk produces a handover file recording what was built. After 2–3 chunks it recommends /compact to manage context. If chunks can run in parallel, it offers to dispatch them to separate git worktrees.

Output: Working code, build/prompts/P1-C01.md through P3-C02.md, build/handovers/ for each

5. Code Review

You: /gvm-code-review

GVM dispatches parallel defect-class panels as subagents — each scanning for a specific class of defect (references, contracts, logic, naming) rather than a broad area of expertise. This orthogonal partitioning, grounded in Basili’s Perspective-Based Reading research, catches more issues on the first pass because panels cannot miss the same defects. Findings come back with structured format: expert, severity, file:line, issue, fix. You fix the findings, re-run on changed files until clean.

Output: code-review/code-review-001.html (HTML-only — per shared rule 13 exception, review reports do not produce paired MD)

6. Build Verification

You: /gvm-test

GVM verifies the reviewed code works as a product: greps for integration seams (TODO, NotImplementedError), starts the infrastructure, runs every acceptance test case from test-cases.md against the built code, walks through the UI, and produces an honest stub vs implemented audit. The audit might show: 

Requirement Coverage: 12 IMPLEMENTED / 2 PARTIAL / 1 STUB / 0 NOT IMPLEMENTED
Test Cases: 45 PASS / 3 FAIL / 1 BLOCKED / 2 STUB

You decide whether to fix the remaining items or ship with documented limitations.

Output: test/test-001.html (HTML-only — per shared rule 13 exception, review/verification reports do not produce paired MD)

At any point

Run /gvm-status to see where you are. Run /gvm-doc-review to audit the quality of any document before proceeding to the next phase. After /gvm-test, run /gvm-doc-write for pipeline documentation, then /gvm-doc-review as the final quality gate.

How Expert Grounding Works

GVM uses a three-tier expert model. Each tier activates at different points in the pipeline:

Table 3 — Expert tiers
TierNameWhen activeExamples
Tier 1 Process experts Always — they govern the methodology for each phase Wiegers (requirements), Copeland (test design), Clements & Fairbanks (architecture)
Tier 2a Software-domain specialists When specific software domains are encountered (data modelling, UX, resilience, etc.) Cooper/Krug (frontend), Kleppmann (data systems), Nygard (resilience)
Tier 2b Industry-domain specialists When the application operates in a regulated or specialised industry Industry-specific experts (model risk, credit risk, operational risk, etc.)
Tier 3 Stack specialists After technology choices are made Ramalho (Python), Vanderkam (TypeScript), Winand (SQL)

Experts are defined in reference files stored within the plugin. When a pipeline phase activates, it loads the relevant reference files into context. The AI then applies those experts' frameworks rather than drawing from undifferentiated training data. Every pipeline artefact (requirements, test cases, specs) includes an Expert Panel table near the top showing which experts were active and what each governed — so you know which frameworks shaped the decisions.

Shared rules

Every GVM skill loads shared-rules.md at startup. The load-bearing rules are: expert activation and discovery, pipeline artefact contracts, hallucination prevention, dual review from round 3+, and expert panel attribution in output. Secondary rules govern when-not-to-ground guidance, what to surface vs keep invisible, automatic scoring, industry domain loading, and incremental writing for large outputs. The single shared file is what keeps every skill behaving the same way.

The roster is not fixed. You can swap experts during /gvm-init, add industry domain files for your specific field, and extend the stack specialist list as new technologies are adopted. The expert management guide covers scoring and roster management in detail.

Where Things Live

GVM stores files in two locations. Understanding the split avoids confusion about where to find things and what belongs where.

Table 4 — Project vs shared files
LocationWhat lives hereScope
Your project directory
requirements/, test-cases/, specs/, build/, code-review/, etc.		 All pipeline output — requirements documents, test cases, specs, build handovers, review reports, site survey reports One project. Lives in your repo. Committed to git. Visible to your team.
~/.claude/skills/gvm-*/ The plugin itself — skill definitions (SKILL.md), expert reference files, design templates, shared rules, scoring methodology, pipeline contracts All projects. Shared across every project that uses GVM on this machine.
~/.claude/gvm/ Shared runtime data — activation log (expert-activations.csv), discovered experts (discovered-experts.jsonl), rescore log (rescore-log.jsonl), plugin user guide (docs/plugin-guide.html) All projects. Shared across every project that uses GVM on this machine.
The activation log (~/.claude/gvm/expert-activations.csv) tracks entries per project using a project name column. /gvm-status filters it to show only your current project.

Why the split

Pipeline output is project-specific — your requirements are not another project’s requirements. It belongs in the project directory where it can be committed, reviewed, and shared with collaborators.

Expert references and skill definitions are shared resources — the same Wiegers methodology applies to every requirements phase, the same Tufte design system applies to every document. Duplicating these per project would create maintenance burden and version drift.

This is also why expert discovery persists to the central roster, not to project-local files. An expert discovered during one project may seem specific to that project at the time, but the same domain knowledge often surfaces again in a different context. If the expert is already scored and in the roster, the next project picks them up automatically. If they were stored locally, you would have to remember they exist and manually transfer them — or rediscover them from scratch.

What this means in practice

  • To see your project’s GVM output: look in your project directory — requirements/, test-cases/, specs/, build/, etc.
  • To customise experts: edit files in ~/.claude/skills/gvm-design-system/references/ or use /gvm-experts to score and manage them.
  • To see expert engagement for your project: run /gvm-status — it reads the shared activation log and filters to your project.
  • To share GVM output with your team: commit the project-level directories. The shared plugin files are not needed — each team member installs the plugin independently.

When NOT to Ground

Not every action needs expert grounding. Expert activation is for decisions — architectural choices, design trade-offs, technique selection, quality judgements. The following do not benefit from it:

  • Inline bug fixes
  • Simple renames and refactors
  • Formatting corrections
  • Dependency updates
  • Routine housekeeping

If the task does not involve a judgement call, just do it. The overhead of loading experts, citing frameworks, and tracking activations is only worth paying when the task involves a decision that benefits from authoritative grounding.

Troubleshooting

A /gvm-* command is not recognised

The plugin is not loaded. Run /gvm-status as a test. If that is not recognised either, verify that plugin.json is at the expected path and restart Claude Code.

A skill says a prerequisite file is missing

Pipeline skills check for upstream artefacts before starting. The error message tells you which upstream command to run. This is intentional — the pipeline is sequential and each phase depends on the previous one’s output.

A skill was interrupted mid-way through

  • Document-producing skills — check for partial output files. Re-run the skill; it will detect existing output and offer to continue or start fresh.
  • Build skill — designed for interruption. It reads build/handovers/ to determine progress and continues from the next unfinished chunk.
  • Init or site survey — re-run. Both detect prior state and offer options.

Downstream artefacts are stale after upstream changes

Run /gvm-status to check for staleness. If flagged, re-run the downstream skill — it will detect the newer upstream file and offer to regenerate.

A reference file is missing or corrupted

Reinstall the plugin to restore reference files to their default state. Custom industry domain files in references/industry/ are preserved.

Reference files are shared

Files in ~/.claude/skills/ are shared across all projects on this machine. If you corrupt or delete a reference file, every project is affected.

Experts are not being cited in output

Check that shared-rules.md and the expert reference files exist in ~/.claude/skills/gvm-design-system/references/. Run /gvm-status to check expert engagement.

Expert scores seem wrong

Run /gvm-experts and select “Rescore.” Scoring uses independent verification, so occasional disagreements are expected. Rescoring with a fresh assessment resolves persistent issues.

Customising the Expert Roster

Adding industry domain experts

Industry domain files live in references/industry/ within the design system skill. GVM ships with files for market risk, credit risk, model risk, and operational risk. To add a new domain:

Recommended: let /gvm-init or /gvm-site-survey do it. Both commands delegate to the centralised Expert Discovery and Persistence Process in the design system. In the init flow (step 2), describe your business domain. In a site survey, expert discovery runs automatically after diagnosis. Either way, the AI will identify authoritative experts from its training data, present them with scores and evidence, and persist approved experts to the reference files — all in the correct format.

Manual alternative: Create a new file at ~/.claude/skills/gvm-design-system/references/industry/{your-domain}.md, following the structure of the existing files. Make it verbose and self-contained. Then run /gvm-experts and select “Score experts in a reference file” to score the entries.

Whether created by /gvm-init or manually, industry files are append-only. Experts discovered during projects are added, never removed. The roster grows as your understanding of the domain deepens.

How scoring works

Every expert and work carries a classification: Canonical (4.5+ average), Established (3.5–4.4), Recognised (2.5–3.4), Emerging (1.5–2.4), or Provisional (below 1.5). Scores are assessed across five dimensions for experts (authority, publication, breadth, adoption, currency) and four for works (specificity, depth, currency, influence).

Scoring uses independent verification: a second agent assesses the same expert without seeing the first agent's scores. If the two assessments converge, the averaged scores are used. If they diverge, the user decides. Over time, utilisation data from the activation log provides feedback on whether classifications align with actual usage.

The expert management guide covers the full scoring methodology, utilisation feedback, and divergence criteria.
