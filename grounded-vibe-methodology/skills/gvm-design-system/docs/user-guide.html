Design System

The shared foundation for all GVM skills — design templates, expert reference files, shared rules, and pipeline contracts.

Overview

The design system is the central shared resource for the Grounded Vibe Methodology. It is a passive store — other skills load its files at runtime, but it is not invoked directly by users. It provides six categories of reference files:

  • Design templates — tufte-html-reference.md (core Tufte/Few patterns) plus three companion files for skill-specific components: tufte-review-components.md (verdict box, score card, issue blocks), tufte-spec-components.md (ADR and component-detail CSS), and tufte-slide-template.md (self-contained slide deck). Each companion is loaded only when that output type is being produced
  • Shared rules (shared-rules.md) — cross-cutting rules loaded by every skill at startup: expert activation logging, expert discovery, industry domain loading, when-not-to-ground guidance, automatic scoring, pipeline contracts, what to surface vs keep invisible, and hallucination prevention
  • Expert reference files — Tier 1 architecture specialists, Tier 2a per-domain specialist files (in references/domain/), Tier 3 per-stack specialist files (in references/stack/), and Tier 2b industry domain files
  • Expert scoring (expert-scoring.md) — the full scoring methodology loaded on demand by /gvm-experts
  • Pipeline contracts (pipeline-contracts.md) — structural contracts between pipeline phases: required sections, ID formats, and structural elements that downstream consumers depend on
  • Roster assembly (roster-assembly.md) — parameterised expert roster assembly process used by /gvm-init and /gvm-site-survey
For expert scoring and roster management, use /gvm-experts. That skill provides the user-facing interface; the design system provides the storage.

For Skill Authors

If you are writing or modifying a GVM skill, you need two things from the design system:

  1. At skill startup, load ~/.claude/skills/gvm-design-system/references/shared-rules.md. This gives you the cross-cutting rules (expert logging, discovery, scoring, when-not-to-ground). Every skill does this.
  2. Before the first document write, load the core files: ~/.claude/skills/gvm-design-system/references/tufte-html-reference.md (core HTML/CSS patterns) and ~/.claude/skills/gvm-design-system/references/writing-reference.md (prose quality standards). Then — only if applicable to your output type — load one of the Tufte companion files: tufte-review-components.md (review reports), tufte-spec-components.md (tech specs with ADRs), or tufte-slide-template.md (presentations — self-contained; load instead of the core file).
  3. Before any review or scoring activity, also load ~/.claude/skills/gvm-design-system/references/review-reference.md (assessment methodology: rubric design, reviewer calibration, measurement reliability).

Everything else in this guide is reference material for understanding what the design system contains and why.

Visual Design Philosophy

The HTML/CSS design templates are grounded in two information design experts:

Edward Tufte

  • Data-ink ratio — maximise the share of ink devoted to data. Every pixel should convey information.
  • Remove chartjunk — no decorative borders, gradients, drop shadows, or ornamental rules.
  • Small multiples — use the same visual format repeated for comparison.
  • Layered information — documents should work at multiple zoom levels.
  • Let whitespace do the structural work — space communicates grouping and hierarchy.

Stephen Few

  • Tables over charts when precision matters — if users need exact values, use a table.
  • Subtle zebra striping — alternate row backgrounds for tracking, not decoration.
  • Minimal gridlines — top and bottom rules only. No internal borders.
  • Left-align text, right-align numbers — follows natural reading patterns.
  • Caption tables descriptively — every table has a numbered caption.
The result is documents that look nothing like typical AI output. No rounded cards, no gradient headers, no icon grids. Just clean typography, generous whitespace, and information density where it matters.

Sidenotes

Sidenotes float in the right margin, providing context without interrupting the main text flow. They are the Tufte equivalent of footnotes, but visible alongside the text they annotate. 

<span class="sidenote">
  This appears in the right margin alongside the paragraph it annotates.
</span>

Tables

Tables follow Few’s principles: minimal gridlines (top and bottom rules only), zebra striping on even rows, left-aligned text, right-aligned numbers, and a descriptive italic caption.

Priority chips

Inline badges for MoSCoW priorities: MUST SHOULD COULD WON'T

Callout blocks use a left border with a label in small caps. Use them for important notes, best practices, or warnings that need visual emphasis.

The font stack begins with ET Book (Tufte’s preferred typeface) and falls back through Palatino to Georgia. All are serif faces suited to sustained reading.

The background is #fffff8 — a warm off-white that reduces eye strain compared to pure white. Body text is #111, not pure black, for slightly softer contrast.

Responsive behaviour

Below 1200px viewport width, the layout collapses to single-column:

  • The TOC becomes a static block at the top of the page
  • Sidenotes reflow inline below their associated paragraph
  • All content fills the full viewport width
The TOC script

Every document includes a JavaScript snippet at the bottom that auto-generates the TOC from h2 and h3 elements and highlights the current section on scroll. This is part of the design system — skills do not need to build TOC entries manually.

Build Check Promotion (Shared Rule 21)

Shared rule 21 defines a cross-cutting mechanism for self-improving builds. When review skills (/gvm-code-review, /gvm-doc-review, /gvm-design-review) find recurring issues (3+ consecutive rounds or regressions), they promote those findings to reviews/build-checks.md. This file is a pipeline artefact loaded by /gvm-build during prompt generation, injecting “Known Patterns to Avoid” into chunk prompts. Checks are tiered: tier 1 (general practice — DRY, resource lifecycle, single source of truth) are permanent and never retired. Tier 2 (project-specific — tied to a particular library or domain convention) are flagged for retirement after 3 quiet rounds, but you decide whether to retire them.

Expert Scoring & Management

Expert scoring, rescoring, discovery, and utilisation analysis are handled by /gvm-experts. The design system stores the reference files and scoring methodology; /gvm-experts provides the user-facing interface. See the Expert Management User Guide for details on scoring dimensions, classification bands, utilisation feedback, and divergence criteria.

Pipeline skills score unscored experts automatically when they load reference files (shared rule 9). User data — the activation log, discovered experts, and rescore history — is stored in ~/.claude/gvm/, separate from the plugin install path at ~/.claude/skills/gvm-design-system/references/. This means user data survives plugin updates. /gvm-status summarises expert engagement per project.

Tips

  • Load it once, use it everywhere. The design reference only needs to be read once per session. All subsequent HTML writes use the patterns from context.
  • Sidenotes are for context, not content. If the information is essential to understanding the document, it belongs in the main text. Sidenotes are for caveats, citations, cross-references, and secondary observations.
  • Caption every table. Use numbered captions (Table 1, Table 2) with descriptive text. This follows Few’s principle that tables should be self-explanatory.
  • Prefer tables over prose for structured data. If you are listing items with attributes, a table communicates more efficiently than a paragraph.
  • Use the priority chips. Every requirement should have a visible MoSCoW priority chip. This is the design system’s way of making priority scannable.
  • Let whitespace breathe. Tufte’s key insight is that whitespace is structural, not wasted. Do not fill every margin with content.
  • No decorative elements. No rounded corners on content blocks, no gradient headers, no icon grids, no shadows. Every visual element must convey information.
