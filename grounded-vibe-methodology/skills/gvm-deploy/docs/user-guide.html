/gvm-deploy

Release preparation — version bump, release notes, documentation check, changelog, and git tag.

What It Does

Prepares a release from your current pipeline state. It aggregates review verdicts, checks that your documentation matches what was actually built, increments the version, generates release notes, updates the changelog, and tags the commit.

It does not deploy anything. Deployment mechanisms — CI/CD pipelines, cloud providers, container orchestration — are organisation-specific and outside the pipeline's scope. This skill produces a release-ready tagged commit.

Pipeline Position

/gvm-deploy is the final pipeline phase: 

/gvm-build → /gvm-code-review → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

It runs after the build is verified, reviewed, tested, and documented. /gvm-explore-test is optional; run it between structured testing and documentation when charter-driven exploratory sessions are needed. /gvm-doc-review runs after documentation is written and before the final release gate.

When to Use

Run /gvm-deploy when the build is verified (Ship-ready or Demo-ready), code-reviewed, and documented. The skill aggregates verdicts, checks documentation accuracy against the build, bumps the version, generates release notes, and creates a git tag. It does NOT perform deployment — that is organisation-specific.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Quality Gate Check. If any review verdict is “not ready” (Do not merge, Do not build, Revise first, Do not publish), the release is blocked.
  2. Documentation Accuracy Check. README.md and user-facing documentation must match the current build. Stale documentation blocks the tag.
  3. Version and Tag. Every release must produce a git tag.

How to Use

/gvm-deploy

The Quality Gate

The skill reads the most recent verdict from each review type (code, doc, design). Verdicts come from the review reports in code-review/, doc-review/, and design-review/.

SituationWhat happens
All reviews “ready”Proceed to release
Any review “ready with caveats”Caveats presented — you decide whether to proceed
Any review “not ready”Release blocked. Fix the issue and re-run the review first.
No reviews existWarning. You can proceed but the release has no quality evidence.

Documentation Accuracy Check

This is not a full /gvm-doc-review. It is a targeted accuracy check: does the documentation describe what was actually built?

  • README.md — features match implemented requirements, setup instructions are current, configuration options are accurate.
  • User docs (if they exist in docs/) — navigation paths match the UI, API endpoints match the routes, no references to descoped features.

If inaccuracies are found, you choose: fix now (the skill applies the fixes), acknowledge and proceed, or abort the release.

Versioning

The skill proposes a semver version based on what changed:

  • Major — breaking changes or new MUST-priority requirements that change public interfaces
  • Minor — new features (new requirements implemented)
  • Patch — bug fixes, review finding fixes, documentation updates only

You always have final say on the version number. The first release defaults to v0.1.0.

Output

The skill produces:

  • RELEASE-NOTES.html — user-facing release notes (Tufte/Few design, overwritten each release). Describes what changed in terms users care about: new capabilities, improved workflows. No requirement IDs or review scores.
  • RELEASE-NOTES.md — developer-facing release notes (overwritten each release). Includes requirement IDs, review scores, and implementation details.
  • CHANGELOG.md — cumulative changelog (appended each release, follows Keep a Changelog format)
  • An annotated git tag (v{VERSION})
  • Documentation fixes (if any were applied in Step 3)

Extending for Your Environment

The skill produces a tagged commit with accurate documentation. Your CI/CD pipeline can take it from there. A common pattern: configure your CI to trigger on version tags (v*). When /gvm-deploy creates the tag and you push it, your pipeline runs automatically — build, test, deploy to staging, promote to production, whatever your organisation requires.

The skill is also extensible. If your release process needs additional checks (security scan results, compliance sign-off, performance benchmarks), add them as review skills or build checks. The release gate reads all review verdicts — any new review type that produces a verdict is automatically enforced.

What It Does NOT Do

  • Deploy to any environment
  • Trigger CI/CD pipelines
  • Generate runbooks or operational procedures
  • Manage environments (staging, production, feature flags)
  • Create GitHub/GitLab release objects
  • Roll back releases
  • Push to the remote repository

These are all organisation-specific concerns. The skill draws a clear line: it prepares the release artefacts, you handle the deployment.
