/gvm-requirements

A command for Claude Code that turns rough project ideas into structured, complete requirements documents through expert-driven conversation.

Overview

The /gvm-requirements command acts as a requirements analyst. You describe what you want to build — a paragraph, a few bullet points, or just a vague idea — and it drives a structured conversation to produce a complete requirements document.

It produces two files, always kept in sync:

  • requirements.html — a professionally designed document following Tufte and Few design principles
  • requirements.md — a Claude-friendly markdown version for use by downstream commands

Both files are saved to your project's requirements/ directory.

The skill is implementation-agnostic. It captures what the system must do, not how to build it. Architecture and decomposition belong to downstream commands like /gvm-tech-spec.

Getting Started

Starting from an Application Brief

If /gvm-init produced an Application Brief (init/brief-*.md), the skill auto-detects it and uses it as seed input. It then asks if you have any additional documents to include — meeting notes, competitor analysis, existing requirements from other tools. All seeds are loaded together for richer elicitation.

Starting with an idea

The most common way to use the command: type /gvm-requirements and provide your rough description in the same message or when prompted. 

/gvm-requirements

I want to build a tool that helps people compare neighbourhoods
when they're thinking about moving. It should look at schools,
commute times, and housing prices, and produce a report they
can share with their partner.

The skill will analyse your input, identify what's clear, what's ambiguous, and what's missing, then begin asking questions to fill the gaps.

Starting from scratch

If you don't have a description yet, just run /gvm-requirements with no input. The skill will start with broad discovery questions:

  • "What problem are you trying to solve?"
  • "Who is this for?"
  • "What would a highly successful solution look like?"
These are "context-free questions" from Gause & Weinberg's Exploring Requirements — designed to work regardless of domain and surface the most important information first.

Starting with an existing document

If you already have requirements from another tool or process — a Word document, Confluence page, Jira export, spreadsheet, or plain text — provide it as input. The skill treats the entire document as a rich seed: it extracts what maps to GVM structure, flags what needs clarification, and identifies gaps.

The external document accelerates the process but is not accepted as-is. It goes through the same expert-driven elicitation flow, and before finalising, an independent verification agent compares the GVM output against your original document. The agent checks four things:

  • Completeness — nothing from the source was dropped
  • Accuracy — the meaning of each requirement was preserved
  • Hallucination — nothing was invented that wasn’t in the source or discussed during elicitation
  • Priority alignment — MoSCoW priorities match the source where applicable

Any discrepancies are presented to you for resolution before the document is finalised.

The Conversation

The skill asks one question at a time. It prefers multiple-choice when the options are known, and open-ended questions when exploring new territory. You don't need to give perfect answers — the skill will circle back if something needs clarification.

Expert lenses

Behind the scenes, the skill draws on techniques from requirements experts defined in its reference files. The expert panel is loaded from reference files at runtime, not hardcoded into the skill. This means experts can be updated, swapped, or extended without changing the skill itself.

Adaptive depth

Not every project needs the same level of rigour. The skill starts with core concerns (purpose, user, functional requirements, priorities) and escalates when it detects signals that warrant deeper probing:

  • User accounts or roles → authentication, authorisation, privacy
  • Payments or pricing → financial constraints, regulatory, error recovery
  • Third-party APIs → integration requirements, failure handling
  • Personal or sensitive data → compliance, data retention, security
  • Multiple platforms → platform-specific constraints, responsive design

If the skill asks about something that isn't relevant, just say so. It will note your answer in the Assumptions section and move on.

Incremental Writing

After each major domain is explored, the skill writes that section to both files and asks you to confirm it. This means:

  • You can see progress as you go — open the HTML file at any time
  • Misunderstandings are caught early, not at the end
  • The context window stays manageable (no massive document generation at the end)
How revision works

If a section needs changes, the skill edits just that section — it doesn't rewrite the whole document. If you want to revisit an earlier section later, just say so.

Cross-cutting items — assumptions, open questions, and out-of-scope decisions — are logged to the document as they surface during conversation, not batched up for later.

Process Phases

Table 2 — Process phases

5 Finalise Confirms priorities, generates the requirements index, writes the executive summary, and does a final sync check. Also asks for the application name — you can provide one, request AI-generated suggestions based on the requirements content, or skip. The chosen name becomes the document and project title used by downstream commands. Review and approve the completed document.

Hard Gates

Non-negotiable steps. If any are skipped, the output is invalid.

  1. Load Shared Rules and Expert References at Session Start. Read shared-rules.md and references/elicitation-techniques.md before starting elicitation.
  2. Write Paired HTML + MD. Every requirements section must be written to both requirements/requirements.html and requirements/requirements.md.
  3. One Question at a Time in Free-Text Conversation. Do not dump a list of 10 questions — ask one question per message in open-ended exploration.
  4. User Approval Per Domain. Each domain section must be presented for confirmation before moving to the next domain.
  5. HTML Before Approval. Write the MD content first, then immediately write the HTML — both must be current before presenting the section for user approval.

IM-4: Finalisation Gates

Phase 5 (Finalise) runs two structural finalisation gates before the requirements document is locked. Both gates loop until clean — they will not let Phase 5 advance while errors remain.

The two gates are IM-4 (Impact Map traceability) and RA-3 (Risk Assessment completeness).

Table 3 — Phase 5 finalisation gates
GateSourceWhat it checks
IM-4 impact-map.md Every Must, Should, and Could requirement traces to at least one Impact Map ID (Impact / Actor / Deliverable). Only Won't requirements are skipped.
RA-3 risks/risk-assessment.md Each of the four risk sections (Value, Usability, Feasibility, Viability) is either properly populated or uses the rigid *accepted-unknown* four-line form: Rationale:, Validator:, and Review date: with a date that is today or later.

Both gates apply uniformly: IM-4 to every Must/Should/Could requirement, RA-3 to every one of the four risk sections. There is no per-section escape hatch other than the explicit *accepted-unknown* form for RA-3 or downgrading the requirement to Won't for IM-4.

When IM-4 fires: for each untraceable Must/Should/Could requirement, the skill presents three options — add a trace to an existing Impact Map ID, downgrade the requirement to Won't (out of scope this round), or edit impact-map.md to introduce a new Impact / Actor / Deliverable (typically by re-running /gvm-impact-map). The loop only exits when every Must/Should/Could has a trace.
When RA-3 fires: the validator returns errors keyed by section name and rule code (RA-1..RA-4). For each error the practitioner either populates the section properly or converts it to the *accepted-unknown* four-line form. The loop exits when the validator returns no errors.

Running Again

What happens when you run /gvm-requirements a second time depends on where the pipeline is and what you want to do.

Pipeline still in progress — append mode

If the pipeline is still in progress (test cases or specs not yet complete), the skill appends new domains to the existing document rather than replacing it. Existing requirement IDs are preserved; new requirements are added after them.

Pipeline complete — new scope

If the pipeline is complete and you want to add new scope, the skill creates requirements-002.md. Requirements in the second document use round-prefixed IDs (e.g., R2-VS-1) to distinguish them from the original set.

Updating existing requirements in-place

If you want to revise existing requirements rather than append new ones, the skill edits them directly. Removed requirements are marked with [REMOVED] rather than deleted, so downstream documents can identify what changed. After in-place updates, run /gvm-status to see which downstream artefacts are now stale.

Phase 5 (Finalise) runs after any in-place update to confirm priorities and regenerate the requirements index.

Output Documents

HTML document

The HTML version is a professionally designed document following Edward Tufte's and Stephen Few's information design principles:

  • Floating table of contents on the left
  • Sidenote annotations on the right — caveats, cross-references, sources
  • Requirement blocks with unique IDs and MoSCoW priority chips
  • Comparison tables with minimal gridlines and subtle zebra striping
  • Responsive layout that reflows on narrow screens

Markdown document

The markdown version mirrors the HTML exactly — same sections, same order, same requirement IDs. It's designed for consumption by downstream Claude commands:

  • /gvm-test-cases — generates test cases from the requirements
  • /gvm-tech-spec — decomposes requirements into technical specifications
  • /gvm-build — implements the project
The markdown file uses blockquotes for sidenotes and bold brackets for priorities — e.g., [MUST]. This keeps it readable in any markdown renderer while preserving all the information from the HTML version.

Priority Model

Every requirement gets a MoSCoW priority, confirmed with you before finalisation:

Table 5 — MoSCoW priority levels
LevelMeaning
MustNon-negotiable. The product fails its purpose without this.
ShouldExpected by users. Omit only with clear justification.
CouldDesirable enhancement. Include if effort is low.
Won'tExplicitly deferred to a future version. Not forgotten — tracked for later.
MoSCoW prioritisation as applied in Wiegers' Software Requirements, Chapter 16. Originally developed by Dai Clegg for DSDM. The priorities directly inform how /gvm-tech-spec decomposes the work — Must-haves are built first.

Where This Fits

/gvm-requirements is the first step in a skill chain. Each command produces an artifact that feeds the next:

GVM v2.0 pipeline [/gvm-init] → [/gvm-impact-map] → /gvm-requirements → /gvm-test-cases → /gvm-tech-spec → [/gvm-design-review] → [/gvm-walking-skeleton] → /gvm-build → /gvm-code-review → /gvm-test → [/gvm-explore-test] → /gvm-doc-write → /gvm-doc-review → /gvm-deploy

The requirements document is complete but implementation-agnostic. It captures what the system must do and how success is measured, but makes no decisions about architecture, technology, or task decomposition. Those decisions belong to downstream commands.

Feedback loop

The /gvm-test-cases command may surface ambiguities in the requirements. When this happens, update the requirements document before proceeding to /gvm-tech-spec.

Tips

  • Be honest about uncertainty. "I'm not sure yet" is a valid answer. The skill will log it as an open question rather than forcing a premature decision.
  • Push back on questions. If something feels irrelevant, say so. The skill respects de-escalation and records your reasoning.
  • Open the HTML as you go. The document is valid at every step. Seeing the formatted output can help you think about what's missing.
  • Revisit earlier sections. Just say "let's go back to the user journeys section" — the skill will edit in place rather than rewriting.
  • Don't worry about format. Focus on answering questions. The skill handles structure, IDs, priorities, and formatting.
