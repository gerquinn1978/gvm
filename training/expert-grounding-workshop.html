Expert Grounding Workshop

Why citing specific experts produces different output than asking for "best practices" — and how to apply this in your own work.

What you will be able to do

  1. Explain why a citation acts as a retrieval key into the model's training data
  2. Write a grounded prompt that activates specific expert knowledge
  3. Choose which experts to cite for a given task
  4. Recognise the difference between grounded and ungrounded output
  5. Understand where grounding works (review) and where it doesn't (construction)

The problem with "write good code"

When you tell an AI "write good code," it draws from everything in its training data — a broad, undifferentiated blend of Stack Overflow answers, blog posts, documentation, textbooks, and production code of varying quality. The output is plausible. It compiles. It might even work. But it converges on the average of everything the model has seen, not on any specific standard of quality.

When you tell it "apply Fowler's refactoring principles from Refactoring, 2nd edition," something different happens. The citation shifts the model toward a focused application of a specific body of published work. The output reflects Fowler's principles — extract method, replace conditional with polymorphism, remove feature envy — because the citation activates the model's knowledge of that specific work.

Ungrounded

"Write a well-structured dashboard for market risk data."

Grounded

"Apply Tufte's principles from The Visual Display of Quantitative Information and Few's from Show Me the Numbers."

The ungrounded prompt produces a dashboard with decorative gradients, pie charts, and generic layouts. The grounded prompt produces a dashboard with high data density, appropriate chart types, and no decoration. The difference is measurable — in a controlled experiment across three model families, the effect was statistically significant with large effect sizes (p < 0.001, all pairwise comparisons).

How citations work

The citation acts as a retrieval key. Large language models organise knowledge internally in ways that respond to specific names and titles. When you write "Tufte" in a prompt, the model does not search a database — it shifts its attention toward the regions of its parameter space that encode Tufte's principles. The more specific the citation (chapter, edition, specific principle), the more focused the activation.

This is not RAG (retrieval-augmented generation). Nothing is retrieved from an external source. The knowledge is already in the model's parameters from training. The citation tells the model which knowledge to prioritise.

The analogy

Telling a student to "do research" produces generic output. Telling them to "read Chapter 16 of Wiegers' Software Requirements and apply his elicitation framework" produces focused output. The student's library already contains the book. The citation tells them which shelf to go to.

Specificity is the mechanism

Not all citations are equally effective. The experiment found that:

  • Specific principles work. "Tufte's data-ink ratio" and "Few's bullet graphs instead of gauges" produce measurable improvement.
  • Book-level citations are weaker. "Jorion, Value at Risk" (cited at book level, no specific principles) produced no measurable effect on domain dimensions.
  • Chapter-level citations are strongest. "Wiegers' elicitation framework from Chapter 16 of Software Requirements" is more effective than "Wiegers' Software Requirements."

The rule: cite the specific principle, not just the authority. The model needs the retrieval key, not just the author's name.

Practice 1: Ground a prompt

Take this ungrounded prompt: "Review this Python code for quality issues."

Rewrite it with specific expert citations. Which experts would you cite? What specific principles would you include? Write your grounded version before reading on.

One possible answer: "Review this Python code applying Ramalho's idioms from Fluent Python (2nd ed.) — Protocol-based abstractions over inheritance, frozen dataclasses for value objects, MappingProxyType for shared state. Apply Beck's test quality principles from Test-Driven Development — one assertion per test, behaviour-focused test names. Apply Hunt & Thomas's DRY principle: flag duplicated logic across modules."

Where grounding works — and where it doesn't

This is a finding we discovered through testing, not one we predicted.

Expert grounding during design is demonstrably effective. An architect grounded in Fairbanks produces risk-driven decomposition — thorough where the project is risky, light where it isn't — instead of the uniform-depth spec that everything-equally-detailed produces. An architect grounded in Bass, Clements & Kazman writes quality-attribute scenarios with measurable response measures, not bullet-list non-functionals. A spec author grounded in Keeling produces ADRs that capture the decision, the alternatives rejected, and the reason — not just the chosen option. The output is structurally different from what the same model produces ungrounded.

Expert grounding during review is demonstrably effective. A reviewer grounded in Tufte and Few sees chartjunk that an ungrounded reviewer misses. A reviewer grounded in Fowler and Ramalho catches design patterns that an ungrounded reviewer accepts. The experimental evidence for review specifically is strong — reproduced across three model families.

Expert grounding during code construction is undemonstrated. Citing Fowler during code generation does not reliably produce Fowler-quality code. Code construction is largely procedural — what matters is the exact framework API, the correct method signature, the right import path. Citing an expert who wrote about quality does not substitute for documentation that tells the model how to call this specific library. This is the declarative/procedural distinction the next subsection unpacks.

The separation

Construction is grounded in framework documentation — procedural knowledge about how to use the tools. Design and review are grounded in expert principles — declarative knowledge about what quality looks like. The expert patterns you see in the code exist because the spec defined them (grounded design) and the review enforced them (grounded review), not because experts were cited during code generation.

This separation was discovered during the Assay case study. The Anthropic SDK streaming hallucination compiled and passed tests — but the code review, grounded in the actual SDK interface, caught it. See Building a Production Tool with GVM for the full story.

Choosing which experts to cite

GVM organises experts into tiers based on how they're used:

TierWhen loadedExamples
Tier 1 — ProcessEvery project, every phaseWiegers (requirements), Copeland (test design), Beck (TDD)
Tier 2a — ArchitectureWhen the domain matchesFowler (enterprise patterns), Kleppmann (data-intensive systems), Tufte (data visualisation)
Tier 2b — IndustryWhen the business domain matchesJorion (market risk), Hull (derivatives), HIPAA experts (healthcare)
Tier 3 — StackWhen the technology stack matchesRamalho (Python), Percival & Gregory (Python architecture), Hevery (Angular)

GVM loads the right experts automatically based on your project's domain and stack. You don't need to remember which experts to cite — the pipeline handles it. But understanding the tiers helps you know why certain experts appear in your review reports.

When no expert covers your domain

If your project involves a domain that the built-in roster doesn't cover, GVM flags it: "No authoritative specialist identified for [domain]." You can then run /gvm-experts or /gvm-init to discover and score relevant experts. Discovered experts are persisted and available to all future projects.

Practice 2: Identify the expert gap

You're building a mobile health app that tracks medication adherence. Which GVM tiers would you expect to be covered by the built-in roster? Which might have gaps?

Answer: Tier 1 (process) is always covered. Tier 2a (architecture) is covered for general patterns. Tier 3 (stack) is covered if you're using a supported framework. Tier 2b (industry) is the likely gap — healthcare/pharma domain expertise, medication safety regulations, HIPAA compliance. You would run /gvm-init to discover healthcare domain experts.

Ungrounded reviewers can't see grounded quality

The experiment's most surprising finding was not that grounded output is better. It was that ungrounded reviewers can't tell.

In the controlled experiment, dashboards produced with Tufte/Few grounding scored below ungrounded dashboards when evaluated by reviewers without the expert framework. The grounded output was measurably better by expert standards, but reviewers who lacked those standards rated it worse.

This has practical consequences. If you use an AI to review AI output without grounding the reviewer, the review is confidently approving code that a grounded reviewer would reject. You can't see what you don't have a framework to see.

Practice 3: See the difference

Ask Claude to generate two versions of a data table:

  1. "Create an HTML table showing quarterly revenue by region."
  2. "Create an HTML table showing quarterly revenue by region. Apply Few's principles from Show Me the Numbers: right-align numeric columns, use tabular-nums for digit alignment, top and bottom rules only (no vertical gridlines), subtle zebra striping for row tracking, left-align text. Apply Tufte: maximise data-ink ratio, no decorative borders or backgrounds."

Compare the two outputs. The differences are visible — but only if you know what to look for. That is the grounding effect.

Grounding outside the pipeline

Citation-based grounding isn't pipeline-bound. The same mechanism that activates Beck during code review activates Tukey for an exploratory data analysis, or Hull for a market risk discussion, or Doumont for a presentation. Anywhere a knowledge-dependent decision is made, citing the right expert focuses the output.

The clearest example is /gvm-analysis. It performs privacy-preserving exploratory data analysis: a Python engine reads your data and Claude orchestrates the analysis using grounded experts (McKinney for pandas idioms, Tukey for exploratory technique, Cleveland for visualisation, plus industry-domain experts where applicable) — without ever seeing the raw rows. The data stays in the engine; the model reasons with the right frameworks about what the engine reports.

This is the proof that grounding is about activating reasoning frameworks, not about giving the model access to data. The same logic applies to /gvm-doc-write standalone mode: presentations grounded in Tufte and Doumont, strategy documents grounded in Rumelt and Christensen, training materials grounded in Mayer and Bloom — the experts change with the document type, the activation mechanism stays the same.

Where citation grounding earns its place

A useful distinction: declarative knowledge (what good code looks like, what a sound argument is, what a clear visualisation shows) responds well to expert grounding — the citation activates the framework that defines "good." Procedural knowledge (how to call this specific API, what the exact field name is) responds better to direct framework documentation — the citation can't substitute for the actual interface specification.

GVM's pipeline implements the split. The build prompt loads framework documentation for construction (procedural). Expert principles get applied during design (where the question is "is this the right structure?") and during the code-review loop after construction (where the question is "does this meet quality standards?"). When you're deciding whether to add a citation, ask which kind of knowledge the task needs.

Key takeaways

  • Citations are retrieval keys — they activate specific knowledge the model already has.
  • Specificity matters — cite the principle, not just the author. Chapter-level beats book-level.
  • Grounding fits declarative tasks; framework documentation fits procedural tasks — expert principles define what quality looks like; API and library documentation define how to call the tools. GVM's pipeline splits the two: framework docs for construction, expert principles for design and review.
  • Ungrounded reviewers miss grounded quality — the reviewer needs the same framework as the producer to see the improvement.
  • Grounding is not pipeline-bound — /gvm-analysis, /gvm-doc-write standalone mode, and the standalone modes of the other skills all use the same activation mechanism for non-pipeline work.
  • GVM handles expert loading automatically — the pipeline activates the right experts at each phase. You benefit from the grounding without managing it manually.
