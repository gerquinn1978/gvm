Engineering Leader's Guide to GVM

Adopting the methodology — what it changes, what it costs, how to roll it out.

The adoption decision in one paragraph

GVM is a methodology, not a tool. Installing the Claude Code plugin takes two minutes; adopting GVM takes a bit of setup, some organisation, and Claude Code — the only supported environment today, with ports to other AI development frameworks treated as future work. The agent does most of the work — the team's part is choosing to let the review verdicts shape decisions rather than overriding them when the deadline presses. If your team already treats structured review as a quality gate, GVM gives it grounding and traceability. If they treat reviews as skippable, the methodology makes that visible — and that visibility is the thing the plugin cannot do for you.

The plugin is GVM's reference implementation. It structures AI-assisted development as a phased pipeline from requirements through release, activates named expert authorities at each phase, and accumulates a calibration history across review rounds. The pipeline is optional in parts; the review discipline is not. Leaders who install the plugin without establishing the review discipline will get artefacts but not quality gates.

The key question

Before any other evaluation: is your team currently reading AI-generated code carefully before merging it? If yes, GVM structures and amplifies that practice. If no, the problem isn't tooling — GVM adoption needs to start with that conversation.

What problem GVM solves

AI code generation produces plausible output. It compiles, the obvious tests pass, and the PR looks fine on a fast read. The failures show up in production: an edge case the model didn't consider, a security assumption that was wrong, an integration boundary that wasn't wired correctly. These failures are not random — they cluster in predictable defect classes.

GVM's core mechanism is forcing a structured read. The /gvm-code-review skill dispatches parallel expert panels — each panel scanning for a specific defect class (contracts and interfaces, logic and completeness, security, naming and spec compliance) — grounded in specific published works. The review produces scored findings and a verdict. The verdict is either acted on or explicitly deferred; it is never silently bypassed.

The whitepaper Grounding Produces Measurably Better Output reports 1,080 blind reviews across three AI models and three grounding conditions. Grounded output scores consistently higher under blind review than ungrounded output across all three models. The effect is not subtle. The full experimental design and per-dimension results are in that paper; this document does not restate those findings as if they were proven here.

The experiment uses 180 AI-generated market risk dashboards. Blind reviewers scored each without knowing its grounding condition. The paper reports dose-response results: more specific grounding produces incrementally better scores.

What changes for the team

The pipeline structures the work into phases

A full GVM project runs through up to 14 phases: impact mapping, requirements, test cases, tech spec, design review, walking skeleton, build, code review, test, exploratory testing, doc write, doc review, and release. Most teams use a subset. The phases are not bureaucracy — each one produces a traceable artefact that the next phase consumes. Requirements trace to impact-map deliverables. Test cases trace to requirements. Code review findings trace to the expert principles that identified them.

The structural change most likely to surprise the team: test cases come before architecture. If a requirement can't be expressed as a test, the requirement is ambiguous. /gvm-test-cases forces that question before any design work begins. Teams that normally write tests after code find this backwards; that is the point.

Reviews are scored and the calibration history persists

Each code review round produces scored findings and appends to reviews/calibration.md. By round three, the reviewer knows which modules have recurring findings and which have clean histories. Reviews get more precise over time — not because the AI gets smarter, but because the calibration file gives it project-specific context it would otherwise lack.

A finding that recurs across rounds is signal, not noise. GVM tracks recurring findings and can promote them to build checks — gates that run automatically on every subsequent build. This converts a discovered defect pattern into a prevention mechanism.

Findings are user-triaged, not silently filtered

Shared rule 28: the verdict is not recorded until the user has triaged every finding. The practitioner cannot mark a finding "not applicable" without supplying rationale. The leader's job is to audit that this is happening. The calibration file is the audit trail. If you see findings marked N/A with no rationale, the process has been bypassed.

The stopping rule prevents polish loops

GVM includes a stopping rule: once Critical findings are resolved and Importants are triaged, the build is done. Teams without a stopping rule tend to iterate indefinitely on minor issues — burning time on polish when the substantive problems are already fixed. The rule makes "good enough to merge" an explicit, auditable decision rather than a judgment call made under deadline pressure.

What it costs

GVM has a real cost in time — review rounds add work before a passing verdict, and calibration takes a few rounds to become useful. The cost is bounded and predictable, and observed adoption shows it sits in a sensible bracket: hours-not-weeks for the document tooling, one bounded project to prove out the full pipeline. In the long run it saves time: defects caught upstream of production are defects that do not turn into rework, hotfixes, or incidents. The framework's whole proposition is that prevention beats cure — that catching defects early is cheaper than fixing them in production. If you accept that premise, the upfront cost is part of the deal, not something to optimise away.

Review rounds take real time

Each /gvm-code-review round dispatches multiple parallel expert panels and waits for findings to be triaged before proceeding. For a substantive change, expect 1–2 review rounds before a passing verdict. The whitepaper reports 4 review rounds for a public-facing document under the doc-review gate; internal documents typically pass in 1–2. Code reviews tend to converge faster than document reviews once the calibration file has history.

This is not a bug. The rounds catch things. The cost of a review round is bounded; the cost of a production regression is not.

Practitioners need calibration before scores are trustworthy

The first two or three reviews a practitioner runs are not yet decision-grade. The scores reflect the review instrument, not the project's actual quality profile. Calibration training (see Review Calibration Training) teaches practitioners how to read scores, when to push back on a finding, and how to distinguish a signal from a rubric artefact.

Insist on at least three calibrated reviews before treating a score as a basis for go/no-go decisions. This is not optional. A team that skips calibration will have numbers without meaning.

The expert roster needs maintenance for niche domains

GVM ships with a default expert roster for general software development. For specialised domains — regulated industries, domain-specific languages, emerging frameworks — the default experts may not cover the relevant failure modes. The /gvm-init skill discovers and scores domain experts automatically, but the output needs a sanity check from someone who knows the domain. That someone is usually you. Budget one session for this if your team works in a niche domain; it compounds across every subsequent review.

What to measure

Measure things that tell you whether the quality gate is working. Do not measure things that tell you how busy the team is.

MetricWhat it tells youAction threshold
Calibrated vs blind score gap How much context the calibration file adds. A wide gap means the project has a specific quality profile the blind instrument misses. Gap > 1.5 points: review the calibration file for staleness or anchor drift.
Finding count by expert panel Where defects cluster. If security findings are consistently high while logic findings are low, you know where to invest. Any panel averaging > 2 Critical findings per round: schedule a targeted improvement session.
Review rounds to passing verdict How much rework each change requires. More than 3 rounds consistently suggests upstream problems (requirements or spec quality). Average > 3 rounds: look upstream at test cases and tech spec quality, not at the reviews themselves.
Rework rate after /gvm-code-review Whether review is catching things before merge or just generating work. High rework means the gate is working; very low rework on first-pass may mean the review is being bypassed. Zero rework on three consecutive reviews: audit the calibration file and findings for silent suppression.

Do not measure: reviewer count, lines of code reviewed, AI tokens spent, number of phases completed. These are activity metrics. GVM's value is not in activity; it is in finding defects before they reach production.

Adoption path

Three options, in order of how much of the methodology you turn on at once. All three are legitimate; the right one depends on whether you want to see the document tooling in isolation first, prove out the full pipeline on a single project, or layer the pipeline phases in one at a time.

Option 1 — Standalone first (recommended starting point)

Run /gvm-doc-review and /gvm-doc-write only. No pipeline, no SDLC (software development lifecycle) change. Practitioners typically pick these tools up in their first session — the document tooling is the lowest-friction part of the methodology, and observed adoption with colleagues has been a matter of hours, not weeks. The team feels the review discipline on documentation — a bounded, low-stakes context — before committing to it on code. The six-test gate in the doc-review skill is the same gate that applies to code review: the team learns what "finding triaged" means, what a verdict looks like, and what calibration adds, without changing how they write software.

Run this on internal documents long enough to observe the pattern: are findings catching real problems (not just style), is triage being applied rather than every finding silently fixed, is the calibration file accumulating useful data. When that becomes ordinary — for most practitioners after a handful of documents — the case for extending to code is self-evident. If it never becomes ordinary, that is useful signal too.

Option 2 — Toe-in-water (one team, one project, full pipeline)

Pick one team and one project where the cost of getting it wrong is bounded — a greenfield internal tool, not a customer-facing service. Run the full pipeline: requirements, test cases, tech spec, build, code review, test. Treat the project as a calibration exercise for the methodology, not a production delivery commitment.

One completed pipeline run — requirements through deploy — gives you full signal. The duration depends on the project, not the methodology; what matters is that the pipeline runs end to end at least once before you commit to it more broadly. The risk: if the team is under pressure to deliver, the pipeline becomes the first thing cut. Protect the project scope explicitly, or the signal will be contaminated.

Option 3 — Pipeline by phase

Introduce /gvm-requirements and /gvm-test-cases first. Let the team build confidence in the upstream artefacts before touching the build and review phases. Add /gvm-build once the team trusts that requirements and test cases are producing useful outputs; add /gvm-code-review once the builds are complete enough to review.

This is the most incremental path. Teams that layer phase-by-phase tend to develop sharper opinions about each skill because they see the value of each in isolation before the next one is introduced. The risk: momentum stalls between phases if there is no explicit push to keep moving.

Failure modes to watch for

These are not hypothetical. Each one has a signal in the artefacts if you know where to look.

Silent suppression

Practitioners mark findings as "not applicable" without supplying rationale. The calibration file is the audit trail. Open reviews/calibration.md and look at the N/A entries. If there is no rationale column, or the rationale entries are blank, the gate is not being enforced. Fix it at the process level, not by asking the team to try harder.

Batched verdicts

The verdict is recorded before the user has triaged all findings. This is a shared-rule-28 violation. It typically happens when a practitioner is under time pressure and wants to move to the next phase before completing triage. The calibration training document covers this explicitly; if you see it happening, route the practitioner back to that training rather than accepting the early verdict.

Skipping calibration

A team that goes straight to using review scores as decision inputs, without the calibration ramp, will miscalibrate. A score of 7.5 on a first-pass review of a new codebase means something different from a score of 7.5 on round four of a calibrated review. The numbers look the same; the information content is not. Three calibrated reviews before treating scores as decision-grade is a hard minimum, not a guideline.

Bypassing the pipeline for "small" changes

The team decides that a small change doesn't need the review gate. Then another small change. Then another. Small changes accumulate, and the gate erodes. Define the threshold up-front and hold it: for example, "any change touching production code paths goes through /gvm-code-review." The threshold can be wherever you want it; what matters is that it is written down and not renegotiated under pressure.

Watch for this

The most common failure mode is not any one of the above in isolation — it is silent suppression combined with deadline pressure combined with small-change bypass. Each is individually manageable; the combination unwinds the methodology quickly. The calibration file is your early warning system. Check it monthly.

What this is not

GVM is not a replacement for human code review. It amplifies it: the structured expert panels surface defect classes that humans miss under time pressure, and the calibration file gives reviewers better context. Human judgment on findings is required; GVM does not remove that judgment, it informs it.

GVM is not a guarantee of correctness. It raises the floor on AI-generated output. A passing verdict means the review found nothing it couldn't triage, not that no bugs exist. The experiment results in the whitepaper show a consistent quality improvement under grounding; they do not show perfection.

GVM is not a way to ship faster. The pipeline adds review overhead. What you get in return is fewer production regressions and a documentation byproduct that reflects what was actually built. If the current team is shipping fast with acceptable regression rates, the cost-benefit calculation looks different than for a team that is shipping fast with high rework rates.

Three decisions you have to make personally

These cannot be delegated to the team. Each one sets a constraint that the team will work within for months.

The triage call on Critical findings under deadline pressure

At some point, a release will have a Critical finding and a deadline. The question is not whether this will happen; it is whether you have a pre-committed answer when it does. Options: fix before release (the default); defer with an explicit ticket and a date; push the deadline. "Defer with rationale" is acceptable. "Ignore because we're busy" is not — not because of process purity, but because it signals to the team that the gate is optional when things get hard.

Write down your policy before the first deadline conflict. It is much easier to hold a written policy under pressure than to make the call cold.

The expert roster for your primary domain

The default roster covers general software development adequately. If your team works in finance, healthcare, embedded systems, or any domain with specific regulatory or technical constraints, the default experts will produce reviews that miss domain-specific failure modes. Run /gvm-init for your domain and review the scored output. The skill discovers candidate experts automatically; your job is to verify that the high-scoring experts are actually authoritative in your domain, not just plausible-sounding. One session, done once, compounds across every subsequent review.

The branding file

The plugin's /gvm-doc-write skill reads a branding configuration (logo, accent colour, heading font, team name) when generating documents. If you set this once for the team, every document the team generates shares a consistent visual identity. If you leave it unset, each practitioner either uses the default or sets their own — and the documents drift. This is a small decision with a disproportionate effect on the readability and professionalism of team output. Set it once at onboarding.

Practice exercises

Exercise 1 — Interpreting a calibration gap

Your team's most recent review round shows a calibrated score of 8.2 and a blind score of 7.0 for the same codebase. What does the 1.2-point gap mean? What would you do about it?

Answer key

The gap means the calibration file is adding meaningful context. The blind instrument — which has no project history — rates the codebase at 7.0. The calibrated instrument — which has project-specific anchors, score histories, and resolved-finding records — rates it at 8.2. The calibration file is doing its job.

A gap this size is healthy. It indicates that the project has a specific quality profile the generic rubric would underrate — likely because previous findings have been resolved and the calibration file reflects that improvement.

What to watch for: a gap that widens over time without corresponding fixes being recorded. That would suggest the calibration file is accumulating positive anchors without tracking real regressions — drift toward optimism. Check that the calibration file has recent entries in both directions: improvements and newly identified recurring findings.

What to do: nothing urgent. Continue the current review cadence. Review the calibration file quarterly to check that anchors still reflect the current codebase.

Exercise 2 — Deadline-pressed release with open findings

A release is scheduled for end of week. The code review verdict is "Merge with caveats" — 1 Critical finding and 3 Important findings remain open. The deadline is real: a customer commitment. Walk through the decision tree.

Answer key

Step 1: Understand the Critical finding. A Critical finding is not automatically a blocker — it depends on what it is. Is it a security vulnerability that could be exploited in production? A data-loss path? Or a design issue that has a safe workaround? Read the finding. Do not treat the severity label as a substitute for judgment.

Step 2: Apply your pre-committed policy. If you set a policy before this situation arose (see the decisions section above), apply it. If a Critical finding in a production code path is always a blocker, hold the release. If the policy allows deferral with explicit rationale and a fix date, use that path.

Step 3: Triage the Importants. Each Important finding needs a triage decision: fix now, defer with a ticket, or accept as-is with documented rationale. "We'll get to it" is not a triage decision. The calibration file records these decisions; that is your audit trail if something goes wrong later.

Step 4: Record the release decision in the calibration file. Whether you release or hold, record the decision and the rationale. A future reviewer will need to understand why these findings were open at release time.

The wrong answer: shipping without recording the open findings, or recording them as "not applicable" to clear the verdict. That is silent suppression under deadline pressure, which is the most common way the quality gate fails.

Takeaways

  • Installing the plugin is not adopting GVM. The plugin implements the methodology; the team's process is what enforces the quality gate.
  • Start with doc review if you're unsure. It gives the team review discipline experience at low cost before the pipeline touches production code.
  • Three calibrated reviews before the scores mean anything. Enforce this as a hard rule, not a guideline.
  • The calibration file is the audit trail. Check it monthly. If findings are being suppressed without rationale, the gate is failing silently.
  • Write down your Critical-finding policy before the first deadline conflict. It is much harder to hold a position you haven't pre-committed to.
