/gvm-analysis Training

Privacy-preserving exploratory data analysis — the model never sees your raw data.

What you will be able to do

  1. Explain the privacy contract that makes /gvm-analysis structurally different from every other GVM skill
  2. Trace the data flow: what enters the Python engine, what returns to the model
  3. Identify which analytical questions are addressable and which are not
  4. Run a first analysis session end-to-end
  5. Recognise when /gvm-analysis is the right tool vs when you need a different approach

Your data cannot leave the building

You have a dataset. It contains customer PII (personally identifiable information), medical records, financial positions, or commercially sensitive transactions. You need exploratory data analysis — distributions, quality checks, outlier detection, trend signals — but the data cannot be pasted into a chat window, uploaded to a web service, or included in a prompt.

Every other AI-assisted analysis workflow breaks here. The model needs to see the data to reason about it. Or so it seems.

/gvm-analysis is built on a different architecture. The model never sees your data. A Python engine runs locally on your machine, processes the file, and returns only aggregated, privacy-cleared results. The model reasons about those results — schemas, summary statistics, distribution shapes, quality flags — using grounded analytical frameworks. The raw rows stay in the engine throughout.

The privacy contract

Raw rows never enter the model's context window. The Python engine is the only actor authorised to read the input file. Claude reads only findings.json — an aggregated, privacy-cleared output produced by the engine. This is not a convention or a preference; it is a hard architectural boundary enforced at every step of the process flow.

What makes this structurally different

Every other GVM skill works by giving the model content to reason about: source code, requirements documents, test cases, review findings. The model reads the artefact; the model produces output about the artefact.

/gvm-analysis inverts this. The model never reads the artefact. Instead:

  • The Python engine reads the raw file. It computes distributions, detects outliers, runs data-quality checks, and optionally runs driver decomposition or time-series analysis.
  • The engine writes findings.json: an aggregated, privacy-cleared summary. No raw row values appear in this file.
  • The model reads findings.json only. It orchestrates the session, interprets the findings, and produces the report — but it has no path to the underlying data.

The consequence: you can run AI-assisted EDA on data you could not ordinarily share with an AI. The model contributes its analytical reasoning and grounded expert knowledge. The engine contributes numerical computation on data the model is not permitted to see.

The data flow

Here is what actually happens when you invoke /gvm-analysis:

  1. Model You invoke /gvm-analysis path/to/your/data.csv. Claude asks you to select a mode (Explore, Decompose, Validate, or run everything) and confirm preferences. No data file is read at this stage.
  2. Engine scripts/domain_detect_cli.py reads the file's header row only to identify column names and infer the business domain. It never reads row values.
  3. Model Claude receives the detected domain and loads the matching industry expert reference, or proceeds in general-purpose mode if no domain matches.
  4. Engine scripts/analyse.py runs the full analysis pass: data-quality checks, distribution profiling, outlier detection, and mode-specific analysis (driver decomposition for Decompose mode; differential against a baseline for Validate mode). All of this executes in the Python subprocess. The model receives no intermediate output.
  5. Engine The engine writes findings.json to the output directory. This file contains aggregated statistics, quality flags, and headline findings — no raw row values.
  6. Model Claude reads findings.json. This is the first and only time the model has any contact with information derived from your data.
  7. Model Claude generates three comprehension questions grounded in the headline findings and writes them back into findings.json via a patching script. Column names are permitted; raw values are not.
  8. Engine scripts/render_report.py reads findings.json and emits an HTML report hub.
  9. Model Claude announces the report path and summarises the three-to-ten headline findings. The session ends.
Blue steps run in the Python engine on your local machine. Amber steps run in the model. Your raw data touches only the blue steps.

The boundary holds across all branches: if the engine fails, Claude surfaces the error verbatim and stops — it does not open the input file to diagnose the failure. If you decline the forecast, Claude does not open the input file to provide an alternative. The boundary is unconditional.

What you can use it for

Any dataset where the constraint is "the model must not see raw rows" is a candidate for /gvm-analysis. Common cases:

  • Customer data — transaction histories, behavioural logs, CRM exports with names or account identifiers
  • Financial data — position files, P&L records, trade blotters with counterparty information
  • Healthcare / clinical data — patient cohort exports, trial datasets, anything under HIPAA or equivalent
  • HR and payroll data — compensation data, performance records
  • Internal operational data — datasets that are commercially sensitive even if not personally identifiable

The skill handles CSV, TSV, Excel (.xlsx), Parquet, and JSON inputs. For multi-sheet Excel files it prompts you to select which sheets to include.

Four analysis modes

All four modes run the same full statistical pass (data-quality checks, distribution profiling, outlier detection). The mode selects the report emphasis and any additional engine behaviour:

ModeAdditional engine workReport emphasis
ExploreNone beyond the full passTop data-quality issues and striking distributions
DecomposeDriver decomposition on a target column you nameWhat drives variance in the target column
ValidateFull pass on current + baseline file; differential between the twoDivergences between the two datasets
Not sure — run everythingFull pass + driver decomposition on top 5 numeric columns by varianceHub with all sections; navigate via the table of contents

What you cannot use it for

The privacy contract has a cost. It constrains the class of analysis questions the model can answer directly.

The model cannot:

  • Look at specific rows to understand what an outlier actually looks like
  • Pattern-match against raw string values (e.g., "find rows where the address field contains 'Ltd'")
  • Inspect individual records to understand a data-quality failure
  • Sample and display example rows for context
  • Reason about the semantic content of free-text fields

What the engine can compute — and what the model sees in findings.json — is aggregated information: distributions, quantiles, null rates, cardinality, outlier counts (not outlier values), correlation coefficients, trend direction and significance, driver importance scores. The model reasons about the shape of the data, not its contents.

The cost of the privacy contract

If your analytical question requires the model to inspect individual rows or match against raw field values, /gvm-analysis cannot answer it directly. The question must be reframed as an aggregation the engine can compute. Section 7 — Limitations — covers this in detail and gives examples of reframes that work.

Walkthrough: a first analysis session

Invoke the skill

Point /gvm-analysis at a file path. The skill accepts absolute or relative paths:

/gvm-analysis /data/customer_transactions_q1.csv

You can pass flags to override auto-detection:

/gvm-analysis /data/customer_transactions_q1.csv --domain financial_services

Select a mode

The skill's first action is to ask which mode you want. It will not run any analysis until you choose:

  • Explore — use this when you are new to the dataset and want a general profile
  • Decompose — use this when you have a specific outcome column and want to understand what drives it
  • Validate — use this when you have a new file and want to compare it against a known-good baseline
  • Not sure — run everything — use this for a thorough first pass on an unfamiliar dataset

Preferences

If this is your first run, the skill will offer to customise analytical preferences (headline count, outlier methods, bootstrap confidence, data-quality check toggles) or use shipped defaults. Your choices are saved to ~/.claude/gvm/analysis/preferences.yaml and reused on subsequent runs.

The engine profiles the data; the model sees the profile

After mode selection and preferences confirmation, the engine runs. Depending on dataset size, this takes a few seconds to a few minutes. The model receives no intermediate output during this phase — it waits for the engine's exit code and then reads findings.json.

What the model sees in findings.json: column names and inferred types, per-column null rates, cardinality, distribution statistics (min, max, mean, median, quartiles, skew), flagged data-quality issues, detected outlier counts, correlation matrix, and headline findings ranked by analytical significance. No row values appear anywhere in this structure.

The report

The renderer produces an HTML hub at ~/.claude/gvm/analysis/<run-id>/report.html. The hub contains:

  • A headline findings section (three to ten findings, ordered by significance)
  • Data-quality section with per-column diagnostics
  • Distribution section with summary statistics for every column
  • Mode-specific section (driver importance table for Decompose; delta section for Validate)
  • Forecast section, if you accepted the forecast offer
  • Three comprehension questions with answers, each anchored to a specific headline finding
  • A methodology appendix with provenance: engine version, mode, seed, sampling decisions, and expert references loaded

The HTML file is self-contained. You can open it in a browser, share it with colleagues, or archive it. It contains no raw data from your input file.

Expert grounding

/gvm-analysis uses the same citation-as-retrieval-key mechanism as the rest of the GVM pipeline. The experts the model activates when interpreting findings.json and composing the report are grounded in the detection results from the domain detection step.

The general-purpose framework draws on the EDA canon. The engine itself cites these experts in the methodology appendix it produces — the citations are not decoration, they are the published source for the method actually run on your data. The principal experts:

  • John W. Tukey, Exploratory Data Analysis (Addison-Wesley, 1977) — outlier detection via interquartile-range fences (Q1 − k·IQR, Q3 + k·IQR), rank-based correlation, partial rank correlation for confound control. Engine cites EDA Ch. 2 and Ch. 6 directly.
  • William S. Cleveland, Visualizing Data (Hobart Press, 1993) and Cleveland, McRae & Terpenning, Journal of Official Statistics (1990) — correlation methods for continuous near-normal data, STL (Seasonal-Trend decomposition using Loess).
  • Rob J. Hyndman & George Athanasopoulos, Forecasting: Principles and Practice (3rd ed., OTexts, 2021) — ARIMA (Auto-Regressive Integrated Moving Average), exponential smoothing, seasonality detection. Engine cites Ch. 5 and Ch. 9.
  • Hyndman, Koehler, Ord & Snyder, Forecasting with Exponential Smoothing: The State Space Approach (Springer, 2008) — the state-space framework for the exponential-smoothing forecast pass.

In domain-matched mode, industry-specific expert references are loaded alongside the general experts — for example, a credit-risk dataset triggers the credit_risk.md industry reference, which includes domain-appropriate analytical standards.

This skill was itself built using GVM. The expert citations above were activated during its construction (per the methodology described in the GVM whitepaper) and persisted into the engine's methodology.py module so that every report carries the provenance of the methods it ran. When you read the methodology appendix in your own report, you are reading the same citations the model used to choose those methods in the first place.

The comprehension questions the model writes are required to use plain language only — no statistical method names, no jargon. Column names are permitted. Raw values are not. This is enforced by the patching script: questions that contain jargon are rejected and regenerated.

Limitations

The model cannot inspect rows

This is the central trade-off. When a traditional analyst spots an outlier, they look at the row: what values surround it? Is the date plausible? Does the name field suggest a test record? The model has none of this. It knows that an outlier exists at a given percentile in a given column. It does not know what that row contains.

For many analytical questions, this is sufficient. "How many outliers are there? Are they in the upper or lower tail? Do they correlate with outliers in other columns?" are all answerable from aggregated statistics. The question "what do these outliers have in common?" requires reframing — see the practice exercise below.

Free-text column content is opaque

If your dataset has a free-text notes or comments field, the model sees its cardinality and null rate but not its values. It cannot scan for patterns in the text, classify sentiments, or extract entities. The column appears in the schema; its contents do not.

Sampling on large datasets

For datasets with 10,000 or more rows, the skill offers a sampled run (reproducible via a fixed seed) or a full-data run. The model sees the choice and the sample size; it does not see which rows were selected. If you accept sampling, the methodology appendix records the sample size and seed for reproducibility.

Summary of the cost

Traditional EDA with AI

Model sees raw rows. Can inspect any value. Can pattern-match text. Can explain specific outliers. Requires sharing data with the model.

/gvm-analysis

Model sees aggregated findings only. Cannot inspect rows. Can characterise distributions, quality issues, drivers, and trends. Data stays local.

Practice exercises

Exercise 1: Is /gvm-analysis the right tool?

For each scenario below, decide whether /gvm-analysis is appropriate or whether you need a different approach. Write your answers before expanding the key.

  1. A dataset of 50,000 insurance claims with policyholder names, claim amounts, and claim outcomes. You need to understand the distribution of claim amounts and identify columns with high null rates.
  2. A small dataset of 200 customer support tickets (free-text). You want the model to read the tickets, classify the issues, and surface recurring themes.
  3. A financial positions file with counterparty names and notional values. You need to compare this month's file against last month's to find new positions and changed notionals.
  4. A synthetic test dataset you generated yourself with no real PII. You want the model to look at a few example rows to verify your test data generator is producing plausible values.
Show answer key

1. Insurance claims — Yes. This is the target use case. The dataset is large and contains PII (policyholder names). The questions (distribution of claim amounts, null rates) are answerable from aggregated statistics. Use Explore or Decompose mode with claim outcome as the target column.

2. Free-text tickets — No. The analytical question requires the model to read the ticket text, which is raw row content. The privacy contract prevents this. /gvm-analysis can tell you how many tickets there are, the null rate on the text field, and its cardinality — but not what the tickets say. Use a different workflow where sharing the text with the model is permissible.

3. Financial positions for comparison — Yes. This is precisely what Validate mode is built for. Point the skill at the current file; it will ask for the baseline file. The differential section of the report will show added positions, removed positions, and changed values — all as aggregated statistics, without the model seeing counterparty names or notionals.

4. Synthetic test data you want to visually verify — No. If your goal is for the model to inspect specific rows and confirm they look plausible, that requires row-level access. Since your data is synthetic and you have no privacy constraint, use a standard workflow where you paste or share example rows directly. /gvm-analysis adds overhead and removes the row-level access you actually need here.

Exercise 2: Reframe a row-level question as an aggregation

You are analysing a dataset of customer transactions. The engine has flagged an outlier cluster in the transaction_amount column — roughly 300 rows with values above the 99th percentile. You want to understand what these outliers have in common.

Your first instinct is to ask: "Can you show me those 300 rows so we can see what they share?" That question requires row-level access the model does not have.

Reframe this as one or more questions the engine can answer from aggregated statistics. Write your reframes before expanding the key.

Show answer key

The goal is to characterise the outlier cluster without inspecting individual rows. Here are effective reframes:

  • Column co-occurrence: "For transactions where transaction_amount exceeds the 99th percentile, what is the distribution of transaction_type? Of merchant_category? Do high-amount transactions cluster in specific categories?" — The engine can cross-tabulate the outlier slice against categorical columns and return counts. No individual row is surfaced.
  • Temporal pattern: "Do the high-amount transactions concentrate in a particular date range or day of week?" — The engine can group by time bucket and return counts and mean amounts per bucket.
  • Null pattern: "Do the high-amount rows have higher null rates in other columns than the rest of the dataset?" — The engine computes null rates for the outlier slice vs the full dataset.
  • Bivariate correlation: "Is transaction_amount correlated with account_age_days? Does that correlation hold in the outlier slice?" — Correlation is a statistic; the engine returns a coefficient, not rows.

The pattern: every reframe converts "what do these rows look like?" into "what do these rows aggregate to?" The model reasons about the aggregate; you investigate the rows yourself if needed, using your own tooling on data that never left your environment.

Key takeaways

  • The privacy contract is structural, not configurable. Raw rows never enter the model's context window. This is enforced at every step, including error paths.
  • The model reasons about the shape of the data, not its contents. Distributions, quality flags, outlier counts, driver importance scores, trend direction — these are the inputs to the model's reasoning.
  • Four modes cover the main EDA use cases. Explore for general profiling, Decompose for driver analysis, Validate for dataset comparison, and run-everything for unfamiliar datasets.
  • Row-level questions must be reframed as aggregations. The engine can answer them; the model cannot answer them directly. The reframe is usually straightforward.
  • The output is self-contained HTML. The report hub includes findings, quality diagnostics, comprehension questions, and a methodology appendix. It contains no raw data from your input file and can be shared freely.
  • Domain detection loads appropriate expert grounding automatically. The model's analytical interpretation of findings.json is calibrated to your data's business domain where one can be detected.
