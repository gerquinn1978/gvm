Review Calibration Training

How GVM reviews get more precise over time — progressive calibration, dual review, build check promotion, and how to read and act on review reports.

What you will be able to do

  1. Explain how progressive calibration improves reviews across rounds
  2. Read a review report: interpret scores, findings, and verdicts
  3. Understand why dual review exists and what it catches
  4. Decide when to push back on a finding vs accept it
  5. Understand how recurring findings become build checks

The problem with one-shot reviews

Most AI code review tools treat every review as independent. The tenth review of a codebase discovers the same issues as the first. Nothing remembers what was resolved. Nothing tracks which dimensions are improving. Nothing accumulates project-specific knowledge.

The result is inconsistent reviews. An AI reviewer that scored your error handling 8/10 yesterday might score the same code 6/10 today — not because the code changed, but because the reviewer has no memory of yesterday's assessment. Deming called this measurement system variation: the gauge varies, so you can't tell whether changes in the score are real.

How progressive calibration works

GVM reviews accumulate knowledge in a calibration file (reviews/calibration.md). Each round adds to it. The file tracks:

  • Score history — per-dimension scores across rounds, so you can see trends
  • Anchor examples — the 2 best and 2 worst concrete examples per dimension from your project. These ground the reviewer in what "good" and "bad" actually look like in your codebase, not in the abstract.
  • Recurring findings — issues that appear in 2 or more consecutive rounds
  • Resolved findings — issues from prior rounds confirmed fixed

On round 1, the reviewer has only expert principles: "McConnell says functions should have 3 or fewer parameters." By round 3, the reviewer has project context: "McConnell says 3 or fewer parameters. In this project, the auth module averages 2.1 (good); the search module averages 4.8 (flagged in rounds 1 and 2, partially fixed in round 2)."

The feedback loop

Round 1: expert principles only → review → scores + anchors → create calibration.
Round 2: expert principles + calibration → review → append scores, update anchors → update calibration.
Round N: expert principles + calibration → review → append scores, curate anchors → update calibration.

Reading a review report

The score card

Every review report opens with an overall score and per-dimension scores. The score reflects quality relative to what the loaded experts would expect, on a 0–10 scale:

ScoreMeaning
9–10Publication quality. Could be used as a teaching example.
7–8Strong. Minor issues that don't affect usability.
5–6Adequate. Some issues that should be addressed.
3–4Weak. Significant gaps that will cause problems.
1–2Incomplete. Missing critical content.

The verdict

After the score card, the verdict tells you whether the artefact is ready for its next use. Three tiers:

Verdict
Merge

0 critical, 2 minor. All fixable without restructuring.

Verdict
Merge with caveats

0 critical, 5 important. Error handling in the retry module needs attention before production load.

Verdict
Do not merge

2 critical: budget race condition allows overspend; streaming path uses wrong API call pattern.

The verdict language changes by artefact type: Merge/Do not merge (code), Publish/Do not publish (documents), Build from this/Do not build (designs), Ship/Do not ship (user docs).

Findings

Findings are categorised into three tiers by severity:

Critical: Must fix. Blocks the pipeline. Missing requirements, security vulnerabilities, contradictory specs. Example: "Budget enforcement has a race condition — concurrent tasks can exceed the limit because record() and check_budget() are separate calls."
Important: Should fix soon. Weakens quality but doesn't block. Example: "Provider names defined in both factory.py and config.py — single source of truth violation."
Minor: Nice to improve. Polish items. Example: "Test names use mechanism format (test_config_loading) rather than behaviour format (test_invalid_yaml_raises_clear_error)."

Every finding cites a specific location (file and line) and the expert principle that identified it. This traceability is important: if you disagree with a finding, you can evaluate the expert's principle directly rather than arguing with an opaque judgement.

Practice 1: Classify a finding

A code review finds: "The AuditLogger class has __enter__ and __exit__ methods but callers use bare instantiation instead of with statements."

Is this critical, important, or minor? Why?

Answer: Important. The resource lifecycle is implemented but not enforced by callers. It won't crash the application (the logger works without with), but it means file handles may not be properly closed on errors — a real problem under load. Not critical because it doesn't cause data corruption or security issues.

Dual review: catching what calibration misses

Progressive calibration makes reviews more precise — but it also introduces bias. A reviewer that has seen 5 rounds of your codebase develops expectations. It knows what usually scores well and focuses less on those areas. It trusts resolved findings without re-verifying. Areas that scored high in prior rounds receive less scrutiny.

From round 3 onwards, GVM runs two reviewers in parallel. Why round 3? Round 1 establishes the first scoring anchor; round 2 establishes a pattern of expectations. By round 3, anchoring bias has had enough history to create systematic blind spots — the point at which rater drift becomes detectable in assessment research.

The two reviewers:

ReviewerWhat it receivesWhat it catches
CalibratedFull project history, resolved findings, anchor examples, recurring findingsNuanced issues that require project context to see
BlindSame criteria and expert references, but no calibration data — sees the project freshIssues that calibration bias causes the calibrated reviewer to overlook

After both reviewers finish, GVM reconciles their findings:

  • New — only the blind reviewer found it. A genuine blind spot. Merged into the report.
  • Confirmed — both reviewers found it independently. High confidence. Kept.
  • Regression — matches a previously resolved finding. The fix didn't stick. Merged.
  • Rediscovered — matches a finding already in the calibrated report. Discarded.
  • Noise — matches an acknowledged decision. Discarded.

You see one merged report, not two. The dual review is invisible except for a "Blind Review Findings" section listing what only the blind reviewer caught.

Real example from the Assay build

In round 7, the calibrated panel was ready to ship. The blind reviewer found one critical issue: an await on an async generator that should have been iterated directly. The calibrated panel had seen similar patterns resolve correctly in prior rounds and did not flag it. Without dual review, round 7 would have shipped with a critical bug.

Practice 2: Reconcile findings

A calibrated reviewer finds 3 issues: missing error handling in retry.py, test name format in test_state.py, and a style inconsistency in config.py.

A blind reviewer finds 4 issues: missing error handling in retry.py, a budget race condition in cost.py, test name format in test_state.py, and an unused import in signals.py.

Classify each blind finding: new, confirmed, regression, rediscovered, or noise. The resolved findings list shows "budget race condition fixed in round 4."

Answer:

  • Missing error handling in retry.py — confirmed (both found it)
  • Budget race condition in cost.py — regression (was resolved in round 4, now back)
  • Test name format in test_state.py — rediscovered (already in calibrated report)
  • Unused import in signals.py — new (only the blind reviewer caught it)

The merged report includes: the 3 calibrated findings + the budget race (regression) + the unused import (new blind finding). The rediscovered test name finding is not duplicated.

When to push back on findings

Not every finding requires action. The review provides the finding and the expert principle behind it. You evaluate whether the principle applies to your context.

Reasons to push back:

  • The finding is theoretical. "If an attacker gains write access to internal config files..." — if that's outside your threat model, acknowledge it and move on.
  • The expert principle doesn't apply. A review grounded in Fowler might flag a simple script for lacking design patterns. A 50-line utility doesn't need the Strategy pattern.
  • The fix introduces more risk than it removes. Refactoring a working module to satisfy a minor style finding can introduce regressions.

When you push back, the finding is recorded as "acknowledged" in the calibration file. Future reviews won't raise it again.

The six-test gate (standalone documents)

Code review and document review are different problems. For code, "would a consumer fail?" is a clean test — the consumer is a downstream skill or a build pipeline. For prose (whitepapers, strategy, training, blog posts), that test is too vague. /gvm-doc-review applies a six-test gate before emitting a Critical or Important finding on any standalone document. A finding has to pass at least one:

  1. Integrity — a reader who can verify the claim against a published source would find it wrong (misquote, wrong year, fabricated statistic).
  2. Misdirection (load-bearing) — a reader could form the wrong understanding of a load-bearing claim. Peripheral misdirection is Minor.
  3. Reproducibility — a researcher trying to replicate an empirical claim would lack a needed piece of information.
  4. Honesty of scope — the document claims more than its evidence supports.
  5. Audience hospitality — the document assumes reader knowledge inconsistent with its stated audience. Reader forms no understanding rather than wrong understanding.
  6. Cut-damage protection — if a finding recommends a cut, the cut must not damage orientation, evidence, scope-limits, or honesty disclaimers. Guard against over-trimming.

Findings that fail all six are still recorded — as Minor [POLISH]. They don't block publication. The gate exists because dual review will keep finding sentence-level items indefinitely; the gate filters those out at panel-emit time so synthesis isn't drowned in polish.

Build check promotion

Some findings keep coming back. Round after round, the review flags the same pattern. Individual fixes address individual instances, but the pattern persists because the root cause is in the process, not the code.

When a finding recurs in three or more consecutive rounds (or regresses after being resolved), GVM promotes it to a build check — a concrete instruction loaded into the code generation prompt under "Known Patterns to Avoid."

Two tiers of build checks

TierWhat it coversRetirement
Tier 1 (permanent)General engineering practice: DRY violations, resource lifecycle, single source of truth. Applies to any project.Never retired
Tier 2 (project-specific)Tied to a specific library, SDK, or domain convention. Example: "Anthropic streaming requires async with, not await."Flagged for retirement after 3 quiet rounds — you decide

The tier is assigned at promotion time. The test: would this check be useful on a completely different project with a different tech stack? If yes, tier 1. If it references a specific package or API, tier 2.

Practice 3: Classify build checks

Classify each of these as tier 1 or tier 2:

  1. "Provider names must be defined in exactly one place" (flagged 5 consecutive rounds)
  2. "Google Gemini adapter must use system_instruction parameter, not concatenate system prompt into messages" (regressed after fix)
  3. "Token summation must not be duplicated across phases" (flagged 4 consecutive rounds)
  4. "All pd.read_csv() calls must use explicit file paths, not glob patterns" (regressed after fix)

Answers:

  1. Tier 1 — single source of truth is a general principle (Hunt & Thomas)
  2. Tier 2 — specific to the Google Gemini SDK
  3. Tier 1 — DRY principle, applies to any project with multi-phase computation
  4. Tier 2 — specific to pandas API behaviour

Mechanical verification: the Pre-Publication Integrity Pass

Panels do judgement. They read, weigh, and decide whether something passes the gate. Judgement misses things that mechanical verification catches — and vice versa.

The Pre-Publication Integrity Pass is opt-in for high-stakes documents (public-facing whitepapers, signed-off strategy, anything where a credibility-killer at release would hurt). It is not a panel; it is five mechanical checks:

  1. Direct-quotation verification — every direct quotation in the document is fetched and compared against the actual cited source. Catches misquotes (wrong wording, dropped word, wrong year).
  2. Statistic verification — every specific statistic is fetched against its cited source. Numbers that don't match the source, or aren't traceable to a source at all, are surfaced.
  3. BC-3 extended audit — every body-named author appears in References; every References entry has body citations; no duplicates; companion-papers tracked. Scope includes table cells and pipeline-phase descriptions, not only inline body prose.
  4. Title/topic match — for every external paper cited, the cited title is compared against what the source paper actually says it is about. Catches the case where the title sounds right but the paper is about something else.
  5. Edition/date precision — for every cited book, the cited edition is verified to contain the cited content. Catches the case where the quote is from edition X but the citation says edition Y.

The pass is the complement to dual review, not a substitute. Dual review's blind panel catches what calibration misses through fresh judgement; the Integrity Pass catches what judgement misses through external verification. The two together bound the residual risk of a credibility-killer surviving to publication.

Run the pass after the regular round terminates and before any public release. If you make substantive prose edits afterwards, run it again.

The full cycle

Each review round makes the next one more precise:

  1. Round 1 — expert principles only. Finds structural problems, missing content, factual errors. Establishes baseline scores and first anchor examples.
  2. Round 2 — calibrated with round 1 data. Focuses on what round 1 couldn't see because major issues were masking them. Recurring findings begin to emerge.
  3. Round 3+ — dual review activates. Blind reviewer catches what calibration misses. Recurring findings eligible for build check promotion. The review system is now self-correcting.

The methodology gets better the longer you use it. Each round feeds the next. Build checks prevent findings from recurring. Calibration narrows the focus to what actually matters in your codebase. The blind reviewer prevents calibration from developing tunnel vision.

When to stop: the stopping rule

Without a stopping rule, dual review keeps surfacing sentence-level items forever. The methodology punishes its own users by continuing to find diminishing-return findings. The stopping rule converts "should we run another round?" from a feeling into a calculation.

A round is terminal — the document is publication-ready, further review is opt-in polish — when all four conditions hold:

  1. Zero Critical findings on the six-test gate.
  2. Zero Important findings on the four reader-impact tests (Integrity, Misdirection-load-bearing, Reproducibility, Honesty-of-scope). Audience-hospitality and cut-damage-protection findings need attention but don't by themselves block publication.
  3. Calibrated AND blind Panel D scores both meet the document's threshold. Defaults: ≥9.3 for a public-facing whitepaper that names the methodology, ≥9.0 for a standard whitepaper or strategy doc, ≥8.0 for a newsletter or status report. The user can override per document.
  4. The forcing function: no more than two consecutive verification-only rounds. After two, the next round must declare publication-ready or send the document for restructure. This prevents indefinite polish loops.

The design-decision-recurrence clause

Dual review by design doesn't show the blind panel prior fixes or the calibration story. Once a deliberate scoping or caveat has been added to the document, the blind panel will keep surfacing the same architectural concern as if new. Without correction, condition 3 above never triggers.

The clause: condition 3 is satisfied when calibrated meets the threshold AND every blind-only concern below the threshold is confirmed-recurring across two or more rounds AND is explicitly addressed in the document by caveat, scope-disclaimer, or design-decision documentation. The practitioner declares which specific blind concerns are design-decision-documented and where the documentation lives, in the calibration row's stopping-rule notes.

The clause exists because documents that have done the work of scoping their claims should not be punished by a stopping rule that requires the blind panel to forget them.

Key takeaways

  • Reviews improve across rounds — calibration accumulates project-specific knowledge
  • Dual review from round 3 — catches what calibration systematically misses
  • The six-test gate (standalone documents) — Integrity, Misdirection-load-bearing, Reproducibility, Honesty-of-scope, Audience-hospitality, Cut-damage-protection. Polish-only findings stay Minor.
  • The Pre-Publication Integrity Pass — mechanical verification (quotes, statistics, citations, titles, editions) that complements judgement-based panels for high-stakes documents
  • The stopping rule terminates the cycle — zero Critical, zero reader-impact Important, calibrated and blind both at threshold, max two consecutive verification-only rounds. The design-decision-recurrence clause prevents documents that have already absorbed honest scoping from being punished by recurring blind concerns.
  • Verdicts are clear decisions — ready, ready with caveats, or not ready; publication-ready when the stopping rule is satisfied
  • Findings cite expert principles — you can evaluate the principle, not just the judgement
  • Recurring findings become build checks — the process learns from its own history
  • Tier 1 checks are permanent — general practice that applies everywhere
  • Tier 2 checks require your approval to retire — you decide what stays
