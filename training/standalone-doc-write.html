Standalone /gvm-doc-write

Producing presentations, strategy, newsletters, training, and prose grounded in named experts — outside the pipeline.

What standalone mode is

/gvm-doc-write has two modes. Pipeline docs mode sits after /gvm-test and produces README, CHANGELOG, and user guides grounded in build artefacts. Standalone mode has no pipeline position. You can invoke it when you have no requirements file, no spec, no codebase — just a document you need to write and a reason to write it well.

You might be writing a strategy memo before a board meeting. A quarterly newsletter. A conference talk. A training module for a new joiner. A whitepaper you intend to publish. None of these involve software. All of them benefit from expert-grounded structure, argument, and prose.

What standalone mode gives you that "just write me a document" does not: a named expert panel is selected before writing starts, the structure is proposed and approved before content begins, and a self-review pass checks the draft against the same panel before output reaches disk. The experts are not decorative. They govern the shape of the document and the quality of the prose.

The key distinction

Pipeline docs mode produces documentation about software. Standalone mode produces any prose document that needs to be good — whether or not software is involved at all. The expert mechanism is the same. The document type determines which experts load.

The five typed categories

Five common document formats get structured guidance: the skill loads type-specific domain experts on top of the writing baseline. The table below shows what each type loads.

Type Domain experts loaded Structure principle
Presentations document-structure.md + presentation-design.md + data-presentation.md One idea per slide; headlines as sentences
Strategy document-structure.md + strategy.md Kernel: diagnosis → guiding policy → coherent actions
Newsletters document-structure.md + newsletter.md + writing reference Utility × inspiration × empathy; one reader, one message
Training materials document-structure.md + training.md Identify the gap; segment into learner-paced units
Status reports document-structure.md + writing reference Lead with the headline; situation-complication-resolution

All five types also load writing-reference.md (Doumont, Zinsser, Orwell, Williams) for prose quality, and the appropriate Tufte HTML template for visual design. Presentations use tufte-slide-template.md (self-contained, replaces the standard HTML reference). All other types use tufte-html-reference.md.

Model selection follows the cognitive demand of the task. Strategy documents and whitepapers use the primary model (deep diagnosis and argument construction need genuine reasoning). Presentations, newsletters, training, and status reports use Sonnet (structure is expert-guided; execution is more mechanical). HTML generation always dispatches a Haiku subagent immediately after the matching section is written.

The “any prose” branch

The five typed categories provide structured templates for common formats. But /gvm-doc-write handles any prose document that needs expert-grounded quality. Whitepapers, origin stories, blog posts, board memos, conference papers — all of these get the same expert loading, structure approval, and self-review pass. There is no rigid template, but the process is identical.

When you invoke the skill with "write a whitepaper on X," the skill detects no matching typed category, treats it as an "Other" document type, and proceeds through the same Create mode steps: gather context, load writing experts plus any domain experts relevant to the subject matter, propose structure, ask about an MD version, write, self-review, offer to run /gvm-doc-review.

Whitepapers and origin stories use the primary model (argument construction and narrative structure need judgement). Blog posts and internal memos may use Sonnet if the structure is relatively constrained — but this is model-selection guidance, not a ceiling on quality.

What “any prose” means in practice

A board memo on entering a new market loads Rumelt's kernel structure from strategy.md even though you didn't say "strategy document." A conference paper on a technical topic loads appropriate domain experts alongside the writing experts. An origin story for a methodology loads McKee's narrative arc from writing-reference.md. The typed categories are entry points, not hard walls. Content drives expert loading.

Walkthrough: writing a strategy document

You are preparing a strategy document for your team: whether to build an in-house data platform or continue using a third-party vendor. Here is what the Create mode process looks like step by step.

Step 1 — Invoke and choose Create

You type

/gvm-doc-write

The skill detects no existing document in context, so it enters Create mode. It asks: "What type of document?" You select Strategy document.

Step 2 — Answer the audience, purpose, and scope questions

The skill asks three questions before touching structure. These are not formalities — the answers govern every subsequent decision.

Skill asks

"Who is the audience for this document? What do they already know, and what do they need to decide?"

You answer

"The CTO and two senior engineers. They understand the current vendor's limitations. They need to decide whether to approve a six-month build project."

Skill asks

"What should the reader do or know after reading this? And what is the scope — high-level argument, or detailed cost-benefit analysis?"

You answer

"Approve or reject the build option. High-level argument with cost estimates — not a technical spec."

Minto's principle: the reader's existing knowledge and the decision they face determine structure. Because the audience already knows the problem, the document opens with the recommendation, not background. An audience new to the topic would need the evidence first.

Step 3 — Approve the expert panel (Hard Gate 4)

Before writing a word of content, the skill reads back the proposed expert panel. This is Hard Gate 4 — you must see and approve the panel. Do not skip this step.

Skill presents

"For a strategy document, I will load the following experts: Richard Rumelt (Good Strategy Bad Strategy, The Crux) for kernel structure, diagnosis, and bad-strategy detection; Roger Martin (Playing to Win) for the five cascading choices if we operationalise the strategy; Jean-Luc Doumont (Trees, Maps, and Theorems) for structure-by-message and answering ‘so what?’ at every level; William Zinsser (On Writing Well) for economy of prose; Barbara Minto (The Pyramid Principle) for deductive structure suited to your senior audience. Shall I proceed with this panel?"

Read the panel. If you are writing about a subject with no coverage in the roster — say, a document about AI safety governance — you may see "No authoritative specialist identified for [domain]" and the skill will propose discovered experts for your approval. Approve, request changes, or let the skill proceed.

Step 4 — Approve the structure

Skill proposes

"Proposed structure: (1) Recommendation — one sentence. (2) Diagnosis — what is the critical challenge with the current vendor? (3) Guiding policy — the approach we are choosing and why. (4) Coherent actions — four steps that implement the policy, each reinforcing the others. (5) Cost and timeline. (6) Risks and how the guiding policy addresses them. Does this structure work, or would you like to adjust it?"

The kernel structure (diagnosis → guiding policy → coherent actions) is Rumelt's. If any element is missing from your strategy document, /gvm-doc-review will flag it as Critical. Approve or request changes before the skill writes.

Step 5 — Decide on the MD pair

For strategy documents (and all non-presentation types), the skill asks whether you want a Markdown version alongside the HTML. The default is No. Say Yes only if you plan to edit the MD directly, use it in another tool, or share it as plain text. If in doubt, say No — the HTML is the document.

Presentations are HTML-only. No MD pair is produced and the question is not asked. The slide format has no meaningful plain-text representation.

Step 6 — Self-review pass; write to disk

The skill writes the document section by section, applying expert principles throughout. Before anything reaches disk, it runs a self-review pass against the loaded panel: checking structure, prose quality, and factual claims. Any claim it cannot verify from the provided context is flagged to you rather than asserted. After the review pass is clean, it writes the HTML (and MD if you said Yes).

Output location: strategy/{filename}.html. If a file with that name already exists, the skill increments to avoid overwriting.

Step 7 — Offer to run /gvm-doc-review

At the end of Create mode, the skill offers: review and revise specific sections, create another document, run /gvm-doc-review, or done. For a strategy document you intend to present to the CTO, run the review.

Walkthrough: revising after /gvm-doc-review

The review has run. You have a findings file at doc-review/strategy-build-vs-vendor-review-2026-05-02.html. Two Important findings. Here is what Revise mode looks like.

Step 1 — Load the findings

You type

"Fix the findings in the strategy document review."

The skill detects an existing document referenced with issues to fix. It enters Revise mode. It reads the document in full, then reads the review findings file and extracts each finding with its severity and the expert who raised it.

Step 2 — Load the experts who found the issues

Shared rule 3: experts who find should fix. The expert that identified the problem defines what "right" looks like — not a generic writing improvement pass.

Say the first finding reads: "[Rumelt] The guiding policy is a goal disguised as an approach. 'Become the leading data platform' states an aspiration, not a method for dealing with the challenge." The skill loads Rumelt's framework to guide the fix: a guiding policy must be an actual approach to the challenge, not a goal. The fix replaces the aspiration with a genuine approach.

The second finding: "[Zinsser] Section 4 contains three consecutive sentences built around nominalisations: 'the establishment of', 'the implementation of', 'the achievement of'. Each can be cut to a verb." The skill loads Zinsser and Williams to guide the rewrite of those sentences.

Step 3 — Edit, not rewrite

This is the distinction that matters most in Revise mode. The skill changes what the findings require. It does not restructure sections that are not broken, add new content the findings did not request, or "improve" prose the panel did not flag.

King's principle: kill your darlings — but only your darlings. The darlings the review panel flagged are candidates for cutting. The rest of the document is not.

After each fix, the skill notes what changed and which expert framework guided it. You see the reasoning, not just the edit.

Step 4 — Self-review; offer re-review

After all findings are addressed, the skill re-reads the changed sections against the panel's principles. It checks that fixes did not introduce new problems (Zinsser: rewriting is writing — every edit is a new opportunity to create clutter). It verifies the document still coheres after the changes.

It then offers: continue with remaining findings (if any), run /gvm-doc-review on the updated document to verify the fixes, or done. For Important findings, running the review again is the right call.

Termination rule

Once every finding from the step 2 list has been addressed, the "continue with remaining findings" option is removed. Revise mode does not loop back to re-assess the full document after all findings are resolved. That is the review skill's job.

Expert discovery for uncovered domains

The loaded roster covers writing, document structure, strategy, presentations, newsletters, and training. It does not cover everything. When your document touches a domain with no matching expert, the skill flags it: "No authoritative specialist identified for [domain]. Would you like me to find one?"

Say you are writing a whitepaper on cognitive biases in investment decisions. The writing experts and document structure experts load as normal. But cognitive psychology and behavioural economics have no entries in the existing roster. The skill surfaces the gap and, with your approval, runs the expert discovery process.

Discovery works as follows: the skill identifies candidates with published, peer-recognised work in the gap area (Kahneman and Tversky for cognitive biases; Thaler and Sunstein for behavioural economics in decision-making). It scores each candidate using the standard scoring methodology and presents the proposed classifications with evidence. Approved experts are persisted to the appropriate reference file and become available to all future skills — not just this document. The discovered expert is also recorded in ~/.claude/gvm/discovered-experts.jsonl.

Once approved, the discovered experts shape the document exactly as roster experts do: they are cited in the panel read-back, their principles govern the draft, and they appear in the self-review pass.

The six-test gate at review time

When /gvm-doc-review runs on a standalone document (any type outside the pipeline), findings earn Critical or Important status only if they pass at least one of six tests. Findings that pass none are recorded as Minor [POLISH] and do not block publication.

# Test What it catches
1 Integrity A verifiable claim is wrong, partially wrong, or unsupported. Misquoted source, wrong year, fabricated statistic.
2 Misdirection (load-bearing) A reader could come away with the wrong understanding of the thesis, a central argument, or a key result.
3 Reproducibility A researcher attempting to replicate an empirical claim lacks information they need. Applies to methodology and empirical papers.
4 Honesty of scope The document claims more than its evidence supports, or fails to flag a significant limitation where the unqualified claim would mislead.
5 Audience hospitality The document assumes reader knowledge inconsistent with the stated audience. Unflagged jargon, undefined acronyms, implicit specialist knowledge.
6 Cut-damage protection A recommended cut would damage orientation, evidence, scope limits, or honesty disclaimers. Guards against over-trimming.

Understanding this gate shapes how you use Revise mode. When a review finding is Critical or Important, it passed at least one of these tests — the finding has real reader impact. When a finding is Minor [POLISH], it is a prose preference, not a credibility risk. Address Critical and Important findings first. Polish findings are optional.

See also: the triage workshop for practice working through findings of mixed severity.

Pre-Publication Integrity Pass

The Integrity Pass is a separate, opt-in pass run after a regular terminal round. It exists for one purpose: catch credibility-killers that survived earlier panels.

When to invoke it:

  • Public-facing documents (whitepaper with your name on it, methodology document, published blog post): invoke when the regular review reaches ≥9.3.
  • Standard internal documents (strategy memo, training module): the regular terminal round at ≥9.0 is sufficient. The Integrity Pass is not required.

What the Integrity Pass scans for, and nothing else:

  1. Direct-quotation verification. Every quotation in the document is verified against its cited source via independent lookup. Misquotes are surfaced.
  2. Statistic verification. Every specific number is checked against its cited source. Numbers that do not match, or are untraceable to a source, are surfaced.
  3. Full citation audit. Every body-named author appears in References; every References entry has body citations; no duplicates.
  4. Title/topic match. For every external paper cited, the cited title is verified against what the source is actually about.
  5. Edition and date precision. For every cited book, the stated edition is verified to contain the cited content.

The pass produces a single-page report listing only Integrity findings. A document that passes with zero findings is credibility-killer-free against those specific checks. Invoke it with: "Run the integrity pass on this whitepaper."

What the Integrity Pass catches that panels miss

Standard review panels work at the argument level: is the structure sound, is the audience right, does the evidence support the claim? The Integrity Pass works at the source level: is this quote accurate to the letter, does this statistic match its source, does this citation point to the right paper? These are different failure modes. Panels catch the argument. The Integrity Pass catches the detail.

Practice exercises

Exercise 1 — Pick the right document type

You need to communicate the following to your team. For each, identify the document type /gvm-doc-write should create.

  1. You want to explain why the team is changing its approach to machine learning infrastructure, and get buy-in for the change.
  2. You are giving a 20-minute talk at an internal conference on lessons learned from the last year.
  3. You need to show the VP of Engineering where the project stands this week: what is done, what is at risk, what decisions are needed.
  4. You are onboarding new team members and want them to understand how to use the deployment pipeline correctly.
Answer key
  1. Strategy document. "Why we are changing approach and what the new approach is" maps directly to Rumelt's kernel: diagnosis (what's broken with the current approach), guiding policy (the new approach), coherent actions (what changes). A strategy document is also the right format for getting buy-in from a senior audience, because it leads with the recommendation.
  2. Presentation. A 20-minute live talk is a presentation. The skill will ask for approximate slide count and delivery context (live). At 1–2 slides per minute for a live talk, expect 20–40 slides.
  3. Status report. Weekly project status is a status report: lead with the headline, situation-complication-resolution, traffic-light signals, risks and decisions needed, next steps with owners and dates.
  4. Training material. Onboarding people to use a process correctly is a training problem. The skill will identify the gap type (knowledge gap — they don't know how the pipeline works) and structure the material as concept → example → practice → assessment.
Exercise 2 — Match the finding to the fixing expert

You receive a review finding for each document below. Identify which expert's framework should guide the fix.

  1. "The guiding policy section states 'we will be more agile and customer-focused.' This is a goal, not an approach. It does not tell the reader how the team will deal with the diagnosed challenge."
  2. "Section 3 opens with three consecutive passive constructions: 'it has been decided', 'the project was initiated', 'a review will be undertaken'. Subjects are buried."
  3. "Slide 7 carries three distinct messages. Each deserves its own slide."
  4. "The training module presents six concepts before any practice activity. Learners need to do something with new knowledge before the next concept loads."
Answer key
  1. Rumelt (Good Strategy Bad Strategy). The finding is a bad-strategy detection — goals masquerading as guiding policy. Rumelt's kernel defines what a guiding policy is: an approach to dealing with the diagnosed challenge, not an aspiration. His framework tells you what the fix must deliver.
  2. Joseph Williams (Style: Lessons in Clarity and Grace). Passive constructions that bury subjects are Williams's core concern. The fix: identify who is doing the action, move the actor to subject position, rewrite with a verb. Zinsser or Orwell both notice this pattern too, but Williams provides the most precise sentence-level framework for fixing it.
  3. Cliff Atkinson / presentation design experts (Beyond Bullet Points). One idea per slide is the non-negotiable presentation principle. The finding names a structural violation. The fix splits the slide. Any presentation design expert in the panel would govern this fix.
  4. Clark & Mayer (e-Learning and the Science of Instruction). The segmenting principle: break complex material into learner-paced segments and include practice before introducing new concepts. The fix restructures the module so practice follows each concept rather than queuing after all six.
Exercise 3 — Branding vs. design system

For each of the following, say whether it is controlled by branding/branding.md or by the design system (tufte-html-reference.md). Then say what happens if no branding file exists.

  1. The background colour of the title slide in a presentation.
  2. The line-height and measure of body text.
  3. The heading font family.
  4. The data-ink ratio principle applied to a chart in the strategy document.
  5. The accent colour used on links and emphasis borders.
Answer key
  1. Branding controls header and title-slide background colour. If no branding file exists, the design system default applies (typically a neutral or off-white background).
  2. Design system. Body typography — line-height, measure, spacing — is governed by Tufte/Few principles in tufte-html-reference.md. The branding file cannot override it.
  3. Branding controls the heading font family. The body font stays et-book or system serif regardless of the branding file.
  4. Design system. Data-ink ratio is a Tufte principle. Branding controls identity elements only; it does not govern how charts are designed or what visualisation principles apply.
  5. Branding controls the accent colour for links and emphasis. If no branding file exists, the design system default accent colour applies.

Common mistakes

Skipping the expert panel approval

Hard Gate 4 requires you to see and approve the expert panel before the skill writes. If you skip past the panel read-back, you lose the opportunity to catch a mismatch between the experts loaded and the document's actual needs. A whitepaper on machine learning inference might load general writing experts when it also needs performance engineering experts — but you only know that if you read the panel.

Skipping the panel does not speed up the document. It moves quality risk from a 30-second read to a full review round.

Treating Revise as Rewrite

Revise mode is targeted. It addresses the findings. It does not restructure what the findings did not flag, and it does not improve prose the panel accepted. When you say "fix the strategy document," the skill should change the guiding policy section and the three passive-voice sentences — and nothing else.

If you want a full rewrite, invoke Create mode on the existing content. But ask yourself first: did the review find structural problems, or prose problems? Structural problems (missing kernel element, wrong audience, no argument) warrant a rethink. Prose problems (three passive sentences, two nominalisations) are surgical fixes. Reach for the scalpel, not the sledgehammer.

Checking pipeline-doc boxes for a non-pipeline document

When the skill asks which documents to create, pipeline docs mode presents a checklist: README, CHANGELOG, user guide, API docs. These are only relevant in pipeline docs mode — invoked after /gvm-test with build artefacts present. If you invoke /gvm-doc-write standalone to write a strategy memo and somehow end up at the pipeline checklist, you have triggered the wrong mode. Exit and reinvoke standalone.

The signal: pipeline docs mode asks what build artefacts exist. Standalone Create mode asks what type of document you want to write. If the first question is about artefacts, you are in pipeline mode.

Requesting an MD pair when you do not need it

The default answer to "Would you like a Markdown version?" is No, and the default is correct for most standalone documents. Say Yes only if you have a concrete use for the MD: editing it in a plain-text tool, feeding it to another system, or sharing it where HTML does not render. An MD file that nobody reads is maintenance overhead — if you revise the HTML, you have to keep the MD in sync. When in doubt, HTML only.

Takeaways

  • Standalone mode has no pipeline position. You can invoke it any time, for any prose document, whether or not software is involved.
  • The five typed categories load expert panels automatically. Presentations, strategy, newsletters, training, and status reports each load the domain experts appropriate to the format. Any other prose document also gets expert loading — experts are selected from content, not just document type.
  • Hard Gate 4 is the value point you must not skip. The expert panel read-back is where you confirm the skill has loaded the right experts for your subject. Read it. Adjust if needed.
  • Structure before prose. The skill proposes and gets approval on structure before writing content. The structure determines quality more than word choice does.
  • Revise mode is surgical. Load the experts who found the issues. Fix what the findings require. Preserve what works. Do not rewrite what the panel accepted.
  • Expert discovery fills gaps. If your topic has no matching expert, the skill surfaces it and discovers one. Approved experts persist for all future work.
  • The six-test gate tells you what matters. Critical and Important findings at review time passed at least one of six reader-impact tests. Minor findings did not. Triage accordingly.
  • The Integrity Pass is for public-facing work. Invoke it after a terminal round at ≥9.3. It catches misquotes, wrong statistics, and citation errors that argument-level review misses.
