Getting Started with GVM

Install the plugin, run your first project through the pipeline, and understand what each phase produces.

What GVM is

The Grounded Vibe Methodology is a methodology for building well-engineered software with AI. The Claude Code plugin is its reference implementation, structuring AI-assisted development as a phased pipeline. Each phase activates named experts — published authorities whose cited works focus the AI's output. Instead of asking for "good code," you get code grounded in Fowler's refactoring principles or Ramalho's Python idioms. Instead of asking for "a good review," you get a review calibrated against Tufte's data-ink ratio or Beck's testing principles.

The pipeline runs from requirements through release. Each phase produces artefacts that the next phase consumes. Every artefact is traceable — you can follow a test case back to the requirement it verifies, and a code review finding back to the expert principle that identified it.

Install and verify

Installation

git clone <repository-url>
claude plugin add grounded_vibe_methodology/grounded-vibe-methodology

Verify it works

In Claude Code, run:

/gvm-status

If it responds with the plugin version and available skills, you're ready. If not, check that the plugin path is correct and that Claude Code has restarted since installation.

Several skills also work outside the pipeline as standalone tools. /gvm-doc-write creates any document type — presentations, strategy docs, newsletters, training materials, blog posts. /gvm-doc-review reviews any standalone document under the six-test gate. /gvm-analysis is privacy-preserving exploratory data analysis: it orchestrates a Python engine that reads your data; raw rows never enter the model's context. /gvm-site-survey assesses an existing codebase. /gvm-experts manages the expert roster. /gvm-impact-map, /gvm-walking-skeleton, and /gvm-explore-test can also run standalone outside a full pipeline.

Your first project

The walkthrough below covers the load-bearing phases. Optional/conditional phases (impact-map, walking-skeleton, exploratory testing, doc-write, doc-review) are described after the core walkthrough.

Step 0 (optional): Impact map

/gvm-impact-map

Before requirements, the impact map asks why. It produces four linked tables: goals (the business outcome), actors (who will achieve it), impacts (what behaviour change matters), deliverables (what gets built). Every requirement that follows must trace back to a deliverable. This catches scope drift early — if a requirement doesn't map to an impact-map deliverable, it's a new feature, not a clarification.

Optional but recommended for projects where outcomes are unclear or where multiple stakeholders disagree about what success means. Skip if you have a clear, agreed deliverable already.

Step 1: Define requirements

Start with a rough description of what you want to build. It can be a sentence, a paragraph, or a detailed brief.

/gvm-requirements

The skill asks clarifying questions about your project — who uses it, what problem it solves, what constraints exist. It then produces a structured requirements document with:

  • Requirements organised by domain, each with a unique ID (e.g., AU-1 for authentication requirement 1)
  • MoSCoW priorities: Must, Should, Could, Won't
  • An index table for quick reference

The output goes to requirements/requirements.html (for reading) and requirements/requirements.md (for downstream skills to consume).

Try it

Run /gvm-requirements with a simple project idea: "a CLI tool that converts CSV files to JSON." Follow the prompts. Open the HTML output in a browser. Notice the requirement IDs, priorities, and domain organisation.

Step 2: Generate test cases

/gvm-test-cases

This runs before the tech spec — deliberately. If a requirement can't be expressed as a test, the requirement is ambiguous. The test cases phase catches this before any architecture exists.

Each test case traces to a requirement ID, uses given/when/then format, and includes property-based invariants where appropriate. The output goes to test-cases/.

Step 3: Create the tech spec

/gvm-tech-spec

Reads your requirements and test cases, then produces architecture specs with:

  • Component decomposition proportional to risk (Fairbanks: risk-driven architecture)
  • Architecture Decision Records (ADRs) that capture what was decided, why, and what was rejected
  • An implementation guide that breaks the build into ordered chunks

Step 4: Design review

/gvm-design-review

Validates your designs against requirements before code is written. Dispatches parallel expert panels per design domain — data models, UI layouts, API contracts — and checks that the spec actually covers what the requirements describe. Produces a verdict: Build from this, Build with caveats, or Do not build.

This step is optional but recommended for complex projects. Catching a data model gap before implementation is cheaper than discovering it during code review.

Step 4.5 (recommended): Walking skeleton

/gvm-walking-skeleton

Cockburn's walking skeleton: a thin end-to-end execution thread that touches every external integration boundary — database, message bus, third-party APIs, file system, anything outside your code. Real calls or sandbox calls, not mocks. Produces boundaries.md, a registry the build phase reads to know which boundaries exist.

WS-5 is the load-bearing rule: /gvm-build refuses to run feature chunks until the walking skeleton is green. This catches integration problems while the cost of fixing them is one boundary, not a half-built feature set. If your project has any external integration at all, run this step.

Step 5: Build

/gvm-build

Reads the implementation guide and works through chunks in order. Each chunk follows a TDD loop: write the test, write the code, verify the test passes. Multiple chunks can be completed in a single session, and chunks that don't depend on each other can be built in parallel. When the context window fills up, the skill produces a handover document that gives the next session everything it needs to continue.

Session resilience

If your context fills up mid-build, start a new session and run /gvm-build again. It reads the handover from the last completed chunk and picks up where you left off. No work is lost.

Step 6: Code review

/gvm-code-review

Dispatches parallel expert panels to review your code. Each panel is grounded in specific published works. The review produces scored findings (critical, important, minor) and a verdict: Merge, Merge with caveats, or Do not merge.

Fix the findings, run /gvm-code-review again. The second review knows what the first review found — it tracks what was resolved and focuses on what changed. Reviews get more precise over time.

Step 7: Test and verify

/gvm-test

Runs after code review passes. Performs integration seam audits (verifying all components are wired together), acceptance test validation against the test cases from step 2, stub audits (checking that no test stubs remain in production code), and startup verification. This is the final verification that what was built matches what was specified.

Step 7.5: Exploratory testing

/gvm-explore-test

A timeboxed exploratory charter against the running product, grounded in Bach & Bolton's Rapid Software Testing and Hendrickson's Explore It!. Catches what automated tests cannot: workflows nobody specified, edge cases the test cases didn't anticipate, surprising user paths. Each defect found is logged in given/when/then form.

Recommended before doc-write so the documentation reflects what the product actually does, including any defects deferred for the next round.

Step 7.7: Documentation

/gvm-doc-write

Generates README, CHANGELOG, user guides, and API documentation grounded in build artefacts. The docs describe what was actually built, not what was planned. Also runs in standalone mode for non-pipeline documents (presentations, strategy, training, blog posts) — see the standalone-tool sidenote above.

Step 7.8: Doc review

/gvm-doc-review

Quality gate on the documentation. For pipeline docs (README, user guides), uses traceability and contract checks. For standalone documents (whitepapers, strategy), applies the six-test gate covered in the Review Calibration training. High-stakes documents can additionally run the Pre-Publication Integrity Pass — a mechanical scan that verifies every direct quotation, every statistic, every cited title against external sources.

Step 8: Release

/gvm-deploy

Checks all review verdicts, verifies documentation matches the build, proposes a version number, generates release notes and a changelog, and creates a git tag. If any review said "not ready," the release is blocked.
