Writing Requirements That Pass

Acceptance-criteria-first requirements that survive /gvm-test-cases Phase 4.

The contract between the two skills

/gvm-requirements produces a structured document: requirements grouped by domain, each with a unique ID, a MoSCoW priority (Must / Should / Could / Won't), and whatever acceptance criteria you gave it. /gvm-test-cases reads that document and turns each requirement into test cases — selecting the technique that fits the requirement shape. Most go to standard Given/When/Then example-based scenarios. Reversible, idempotent, or algebraic operations get tagged [PROPERTY] and use property-based testing (MacIver / Hypothesis-style generators) instead. Requirements that cross an external system boundary get tagged [CONTRACT] or [COLLABORATION] — the distinction the EBT lint enforces. (EBT here is External-Boundary-Test, not Example-Based-Test; example-based is the industry term for the Given/When/Then style above, contrasted with property-based.) Mutation testing comes later, in /gvm-test, applied to critical-path code only — it scores how well the test suite kills synthetic faults and surfaces gaps the example-based and property-based tests missed.

That second step is not automatic. Before generating any test case, /gvm-test-cases runs a requirements health report (Phase 1). It reads every requirement and flags those it cannot turn into a concrete, verifiable test. Common flags: vague language with no measurable outcome, missing boundaries, no stated failure behaviour, internal state claims with nothing observable from outside.

Flagged requirements stop the pipeline. You are asked to fix them or explicitly acknowledge the gap. Either way, the health report is written to disk — you cannot silently proceed past a vague requirement.

The health report is saved to test-cases/requirements-health-report.html and the corresponding .md. Acknowledged decisions are persisted to requirements-health-report-decisions.md so they are not re-surfaced on subsequent runs.

The most common early failure is arriving at Phase 4 of /gvm-test-cases with MUST-priority requirements that have no example-based test. Phase 4 runs the EBT-1 audit: it checks every MUST requirement for at least one test in the required shape (concrete input, a positive assertion about the output, and a counterexample the output must not contain). A requirement with vague acceptance criteria produces tests that cannot satisfy that shape. The pipeline blocks.

The fix is not in /gvm-test-cases. It is in the requirement itself, written before you ever run the test-cases skill. This training covers how to write requirements that clear Phase 4 on the first run.

Anatomy of a good requirement

A well-formed requirement gives /gvm-test-cases everything it needs to produce a concrete test: something to set up (the precondition), something to do (the action and its inputs), and something to verify (the measurable outcome). That maps directly onto Given/When/Then.

The traceability tag ([impact-deliverable: D-N]) connects the requirement back up to the impact map. If you ran /gvm-impact-map, this is set automatically. If you did not, the trace is left as not-yet-traced — not a hard failure, but a gap that Phase 4 will count and report.

The acceptance-criteria-first principle

A requirement is not done until you can state how you would verify it. If you cannot describe the test, the requirement is incomplete — not vague in a fixable-later sense, but structurally broken.

The practical discipline: before you confirm a requirement in /gvm-requirements, ask yourself whether you could write the Given/When/Then right now, with real values. If the answer is no, the requirement needs more work. Common blockers:

  • The action is described but the expected output is not quantified ("the system returns results quickly").
  • The happy path is described but error cases are absent.
  • The behaviour depends on an external system, but that system is not named.
  • A boundary condition exists (maximum, timeout, rate limit) but its value is not specified.
The whitepaper makes this structural argument in section 5.3: “test case generation is a quality gate on requirements, not a downstream activity. Before any architecture is designed, every requirement must be expressible as a concrete test with given/when/then conditions, boundary values, and expected outcomes.”

The EBT-1 audit in Phase 4 operationalises this principle mechanically. For every MUST requirement, it checks for an example test with three mandatory elements: a concrete input value on an Input: line, a MUST contain assertion, and a MUST NOT contain counterexample. A test that says “the response is successful” without naming a token or phrase the output must contain will fail the audit. That failure traces back to a requirement that did not specify a verifiable output.

Writing acceptance criteria first is not extra work. It is the moment where vagueness is cheapest to eliminate: before any architecture, any code, any specification. A fix at this stage is editing a sentence. The same fix discovered during build is a rework cycle.

Five common failure modes

1. Vague verbs

Words like “handle,” “support,” “manage,” and “gracefully” describe intent without describing behaviour. /gvm-test-cases flags these as untestable because there is no measurable outcome to verify.

Before
ER-3 (Must): The system shall handle errors from the payment gateway gracefully, so users are not left in an uncertain state.
After
ER-3 (Must): When the payment gateway returns a 5xx response, the system shall display the message "Payment could not be completed — please try again" and preserve the user's cart contents. The transaction shall not be recorded as pending or complete. The error response code from the gateway shall be logged with a correlation ID.

Acceptance criteria: (1) Cart contents unchanged after gateway 500. (2) No order record created. (3) Error log entry contains gateway response code and correlation ID.

The after version gives /gvm-test-cases enough to write distinct test cases: a happy path test for the normal flow, a negative test for the 5xx path, boundary tests for the cart preservation, and a log-content test. The before version produces a single vague test that no reviewer could call pass or fail.

2. Missing boundaries

Requirements that describe a behaviour without specifying its limits cannot be boundary-value tested. Every numerical, temporal, or volume constraint needs a value — not a placeholder.

Before
SR-2 (Must): The search API shall return results within an acceptable time and limit the number of results to prevent overload.
After
SR-2 (Must): The search API shall return a response within 2 seconds for queries returning up to 1,000 results. Results are capped at 1,000 per response; a next_cursor field is included when more results exist. Queries returning zero results shall respond within 500 ms.

Acceptance criteria: (1) Response time ≤ 2 s at 1,000 results. (2) Response time ≤ 500 ms at 0 results. (3) Result set exactly 1,000 when > 1,000 match. (4) next_cursor present when truncated, absent when not.

The after version produces four distinct boundary-value tests. Beizer's boundary analysis — which /gvm-test-cases applies during deepening — needs the exact limit values (1,000 and 1,001) to write useful tests. "Limit the number of results to prevent overload" is not a number the test can use.

3. Implementation leakage

A requirement that names a technology, library, or internal component is a specification fragment, not a requirement. It constrains the architecture in the requirements document, where it does not belong, and it often leaves the actual user-visible behaviour unstated.

Before
ST-1 (Must): User sessions shall be stored in Redis with a 30-minute TTL so that session data persists across requests.
After
ST-1 (Must): An authenticated user's session shall remain active for 30 minutes from their last request. After 30 minutes of inactivity, any request shall redirect the user to the login page. A user who makes a request within 29 minutes of their last action shall not be asked to log in again.

Acceptance criteria: (1) Request at 29 min succeeds without re-authentication. (2) Request at 31 min redirects to /login. (3) Each successful request resets the 30-minute timer.

The after version says nothing about Redis. The technology decision belongs in the tech spec. The requirement states what the user experiences: a 30-minute inactivity window, a clean redirect on expiry, and a rolling timer. Those three criteria produce three tests. The before version produces a test that checks Redis TTL configuration — which is an implementation test, not an acceptance test, and belongs in the build phase.

External constraints are a third category

Naming a vendor or external system is not always leakage. When the choice is an organisational or environmental constraint — procurement has signed the contract, compliance dictates the provider, the integration already exists and is out of scope to replace — the vendor is a given, not a decision. It is neither a requirement nor a spec choice; it is a constraint the team works within.

In that case, naming it in the requirement is correct: "shall send the notification via the SendGrid transactional email API" is fine when SendGrid is the mandated provider. It tells the test author what the external boundary actually is. Where possible, cite the constraint briefly — "(per the existing SendGrid contract)" or "(mandated by the procurement standard)" — so a future reader can tell the constraint from a preference.

The leakage test is: could the developer have chosen something else? If yes, it belongs in the spec. If no, it is a constraint and belongs wherever it makes the requirement testable.

4. Untestable observability

A requirement that claims something about internal system state, without identifying an externally observable signal, cannot be verified by an acceptance test. "The system shall be resilient" or "the circuit breaker shall open" are internal-state claims unless the requirement also names what a caller or user can observe.

Before
RS-4 (Must): The system shall be resilient to upstream service failures by implementing a circuit breaker that opens after repeated failures.
After
RS-4 (Must): When the upstream pricing service has returned errors on 5 consecutive requests, subsequent calls shall return a cached price (if available, up to 60 seconds old) without contacting the pricing service. If no cached price is available, the response shall include "price_status": "unavailable" and HTTP 200. The pricing service shall be retried after 30 seconds.

Acceptance criteria: (1) On 5th consecutive error, 6th request uses cached price without calling upstream. (2) With no cache, response body contains "price_status": "unavailable" with HTTP 200. (3) Pricing service receives a retry request after 30 s.

Every criterion in the after version is observable: the response body, the HTTP status code, and the timing of the retry call. The word “circuit breaker” does not appear — that is an implementation pattern, not an acceptance criterion. A test can verify all three criteria without knowing what the implementation looks like internally.

5. Hidden contracts

This failure mode is the one most likely to trip Phase 4 of /gvm-test-cases. A hidden contract is a requirement that implies an interaction with an external system but does not name that system. Phase 4 runs a contract/collaboration lint that classifies tests as [CONTRACT] (real external boundary, mocked at the boundary only) or [COLLABORATION] (internal call, stub allowed). When the requirement doesn't name the boundary, the test writer cannot make this classification — and the lint flags the violation.

Before
NT-2 (Must): When an order ships, the customer shall receive a notification.
After
NT-2 (Must): When an order transitions to “shipped” status, the system shall deliver a notification to the customer's registered email address via the SendGrid transactional email API. The notification shall include the order ID, the tracking number, and the carrier name. If the SendGrid API returns a non-2xx response, the event shall be queued for retry up to 3 times.

Acceptance criteria: (1) On order status = “shipped,” SendGrid API receives a call containing order ID, tracking number, and carrier name. (2) On SendGrid 5xx, the event is retried up to 3 times before being marked as failed. (3) Customer receives email at the registered address.
CONTRACT tests exercise a real production call all the way to the external boundary, with mocks only at that boundary (e.g. a fake HTTP server standing in for SendGrid). COLLABORATION tests verify that a caller invokes a collaborator with the right arguments — stubs and mocks are allowed inside the call chain. Mixing them is the violation the lint catches.

The after version names the external boundary (SendGrid), the protocol (transactional email API), the fields the call must contain, and the retry behaviour on failure. /gvm-test-cases can now emit a [CONTRACT]-tagged test that stands up a fake HTTP server at the SendGrid boundary and verifies the outbound call's content — without patching HTTP transport inside the consumer, which would be a Rainsberger violation.

Naming SendGrid here is fine because it is an environmental constraint, not a developer's preference (see the callout under Implementation leakage). If the provider were still an open choice, the requirement would say "the configured transactional email provider" and the spec would pick the vendor.

The rule of thumb: if your requirement implies a call leaves your system's boundary, name the service it calls. If you cannot name it yet, that is a design gap that belongs in open questions, not silently absorbed into the requirement text.

The MoSCoW discipline

MoSCoW is not a filing system. Each priority level changes how /gvm-test-cases behaves.

PriorityWhat it meansTest-cases behaviour
MustThe product does not ship without this.EBT-1 audit runs: every MUST needs an example test with concrete input, MUST contain, and MUST NOT contain assertions. Phase 4 blocks until satisfied or explicitly exempted.
ShouldHigh value; included unless there is a strong reason to omit.Full test generation. Deepening offered. Priority inherited by happy-path tests.
CouldNice to have; included if effort permits.Test generation proceeds. Deepening auto-skipped. Edge-case tests deprioritized one level.
Won'tExplicitly out of scope for this round.No test generation. The requirement is indexed but excluded from coverage calculations. Prevents scope creep by naming what is deferred.

Won't is the most misused priority. Many practitioners omit it entirely, treating the absence of a requirement as implicit exclusion. This creates two problems: the scope boundary is invisible to the team, and future requirements rounds have no record of what was deliberately deferred. A Won't requirement with a brief rationale ("deferred to round 2: depends on partner API not yet available") is a decision, documented.

The Must trap

Over-assigning Must is the other failure mode. If everything is Must, the priority carries no information and the EBT-1 audit becomes a tax on every requirement. Reserve Must for genuine ship-blockers. When in doubt between Must and Should, ask: "Would we delay the release for this?" If no, it is a Should.

Practice exercises

Each exercise presents a vague requirement as you might write it in an early session. Rewrite it so that /gvm-test-cases can generate concrete tests in Phase 1 and clear the EBT-1 audit in Phase 4. Open the answer key after you have attempted a rewrite.

Exercise 1

Exercise

Original requirement (Must): The API shall authenticate requests securely and reject unauthorised callers.

Identify the failure modes present. Rewrite the requirement with explicit acceptance criteria that would let /gvm-test-cases produce at least three distinct test cases, one of which names a specific HTTP response for the rejection case.

Answer key — Exercise 1

Failure modes present:

  • Vague verb — "authenticate securely" is not a behaviour.
  • Missing boundary — no token type, no header name, no rejection status code.
  • Hidden contract — if validation delegates to an external identity provider, that boundary is unnamed.

Rewritten requirement:

AU-2 (Must): Every request to the API shall include a valid Bearer token in the Authorization header. Tokens are validated against the Auth0 tenant at https://auth.example.com. A request with a missing or malformed token shall receive HTTP 401 with body {"error":"unauthorized"}. A request with a valid but expired token shall receive HTTP 401 with body {"error":"token_expired"}. A request with a valid, unexpired token shall proceed to the requested resource.

Acceptance criteria: (1) Valid unexpired token → HTTP 200 (or resource-appropriate status). (2) Missing Authorization header → HTTP 401, body {"error":"unauthorized"}. (3) Expired token → HTTP 401, body {"error":"token_expired"}. (4) Auth0 validation endpoint called with each request's token.

Why this clears Phase 4: Criterion 4 names Auth0 as the external boundary, enabling a [CONTRACT]-tagged test. Criteria 2 and 3 provide the MUST NOT contain counterexamples (e.g., the response must NOT contain "error":"token_expired" on a valid token). The EBT-1 audit can find a concrete input and both assertion types for each test.

Exercise 2

Exercise

Original requirement (Must): The file upload feature shall handle large files and provide feedback to the user.

This requirement has at least three failure modes. Name them. Then rewrite the requirement so that boundary-value testing is possible and the success/failure paths are each verifiable.

Answer key — Exercise 2

Failure modes present:

  • Missing boundary — "large files" is not a size. No maximum specified.
  • Vague verb — "handle" does not describe a behaviour. Rejected? Streamed? Queued?
  • Untestable observability — "provide feedback" is not observable without specifying what feedback, when, and through what channel.

Rewritten requirement:

UP-1 (Must): Users may upload files up to 50 MB. During upload, the UI shall display a progress indicator showing percentage complete, updated at least every 500 ms. On successful upload, the UI shall display the message "Upload complete" and the file shall be available for processing within 5 seconds. Files exceeding 50 MB shall be rejected before transmission begins, with the message "File too large — maximum 50 MB" displayed inline. Supported formats: PDF, PNG, JPEG. Files of other formats shall be rejected with "Unsupported file type."

Acceptance criteria: (1) 49.9 MB file uploads successfully; "Upload complete" shown. (2) 50.1 MB file rejected before upload begins; "File too large" message shown. (3) Progress indicator visible and updates during a 10+ MB upload. (4) File available for processing within 5 s of upload completion. (5) .docx file rejected; "Unsupported file type" shown.

Why this clears Phase 4: The 50 MB boundary enables Beizer boundary-value tests at exactly 50 MB, 50 MB - 1 byte, and 50 MB + 1 byte. Each acceptance criterion pairs a concrete input with a verifiable output, satisfying the MUST contain / MUST NOT contain shape for EBT-1.

Exercise 3

Exercise

Original requirement (Must): The system shall send order confirmation emails to customers after a successful purchase.

This requirement looks reasonable at first glance. Identify why it will still trip Phase 4's contract lint. Rewrite it to be lint-clean.

Answer key — Exercise 3

Why it trips the lint: The requirement implies an outbound call to an external email service, but does not name that service. When /gvm-test-cases generates a test, it cannot determine whether to tag it [CONTRACT] (real external boundary, mock only at the boundary) or [COLLABORATION] (internal call). The contract/collaboration lint in Phase 4 (ADR-503) flags tests where it cannot classify the boundary from the requirement. In practice, the generated test will likely patch HTTP transport inside the consumer code — a Rainsberger violation that the lint catches.

Rewritten requirement:

EM-1 (Must): When an order is recorded as paid (payment status = CAPTURED), the system shall send a transactional email via the Mailgun API to the customer's registered email address. The email shall contain the order ID, an itemised list of purchased products with quantities and prices, and the total amount charged. If the Mailgun API returns a non-2xx response, the event shall be queued and retried up to 3 times at 5-minute intervals before being marked as failed.

Acceptance criteria: (1) On payment status = CAPTURED, Mailgun API receives a call containing order ID, item list, and total. (2) Customer's registered email is the recipient address. (3) On Mailgun 500, the event is retried after 5 minutes; after 3 failures the event is marked failed and not retried. (4) No email is sent if payment status is not CAPTURED.

Why this clears the lint: Mailgun is named as the external boundary. /gvm-test-cases can emit a [CONTRACT]-tagged test that stands up a fake HTTP endpoint at the Mailgun boundary and verifies the outbound call's content, without patching requests or httpx inside the consumer — which is the Rainsberger violation the lint prevents. Criterion 4 enables a [COLLABORATION]-tagged test for the internal gate condition.

The Phase 4 lint: what it catches and how to avoid it

Phase 4 of /gvm-test-cases runs two lints against the generated test suite. Both trace back to how requirements describe external boundaries.

The Rainsberger violation occurs when a test patches HTTP transport directly inside the consumer's code (e.g., mocking requests.get, stubbing axios, or intercepting httpx at the call site). This happens because the requirement named a behaviour without naming the boundary — so the test writer had no real external endpoint to test against and substituted a transport patch instead. The fix is to name the external boundary in the requirement, so the test can stand up a fake server at that boundary rather than patching transport.

The Metz violation occurs when a test tagged [CONTRACT] (real external boundary) contains a mock for an internal production class. The two concepts are mixed: a contract test is supposed to exercise the real call chain all the way to the external boundary, mocking only at that boundary. An internal mock stops the chain short. This typically happens when the requirement conflates an internal collaboration with an external contract — naming a class or component rather than an external system.

At the requirements level, both violations have the same root cause: the requirement does not clearly distinguish between what happens inside your system and what happens at a real external boundary. The rule is straightforward:

  • If the behaviour involves calling a service, API, database, message bus, or file system outside your codebase, name it in the requirement.
  • Do not name internal classes, modules, or components in requirements. That is architecture, not behaviour.
  • When a requirement implies both (internal routing logic + external call), split the acceptance criteria: one for the internal gate condition, one for the external call's content.
Mock budget

The contract/collaboration lint enforces a mock budget at the architectural level: each test is allowed to mock either at the external boundary (CONTRACT) or at an internal seam (COLLABORATION), not both. Requirements that muddy this line produce tests that the lint cannot classify — and a test the lint cannot classify is a test that cannot be trusted to remain valid as the implementation evolves.

Takeaways

Requirements that clear /gvm-test-cases Phase 4 on the first run share four properties. Check each one before confirming a requirement:

  1. Concrete verbs with measurable outcomes. The requirement states what the system does and what an observer can verify. "The system displays error message X" is testable. "The system handles the error gracefully" is not.
  2. Explicit boundaries. Every numerical constraint has a value. Every time limit has a duration. Every volume cap has a number. Vague qualifiers ("large," "fast," "many") are placeholders, not requirements.
  3. Named external systems. If the behaviour involves an external service, API, or data store, the requirement names it. This is what enables [CONTRACT] tagging and prevents the Rainsberger and Metz violations in Phase 4.
  4. Both paths described. The happy path and the failure path are stated. A requirement that only describes the success case will produce one test. Phase 4 expects at least a positive and a negative assertion for every MUST requirement.

If you can write the Given/When/Then in your head before confirming the requirement, it will pass Phase 4. If you cannot, it will not.
