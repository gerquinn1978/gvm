Well-Engineered Software Needs an End-to-End Pipeline

Structure does work that staffing alone cannot — the structural decisions and durable artefacts that distinguish the GVM pipeline.

Conor Brennan & Gerard Quinn

April 2026

This paper in context. This paper argues that the pipeline structure matters independently of expert grounding. For the grounding mechanism and expert model, see The Grounded Vibe Methodology. For the experimental evidence, see Grounding Produces Measurably Better Output. For a full end-to-end build using the pipeline, see the case study Building a Production Tool with GVM.

The Argument in Brief

The Grounded Vibe Methodology is named for its citation-based expert activation — naming Wiegers for requirements, Beizer for testing, Fowler for architecture. That mechanism is described in the companion paper, The Grounded Vibe Methodology. This paper makes a different argument: the pipeline structure itself solves problems that the current generation of AI development methodologies does not address, regardless of whether the pipeline is staffed by named experts, generic personas, or no personas at all.

Building well-engineered software with AI assistance is not just a question of who or what staffs each phase — it is a question of what the pipeline itself does between and around the human and AI work. The structural decisions described below cover the gates, audits, the wiring matrix and boundary registry, persistence artefacts, calibration mechanisms, and governance rules that catch failure modes the spec, the tests, and the code cannot catch on their own.

None of these decisions requires expert grounding to function. They stand independently as structural commitments, each addressing a specific failure mode.

The Landscape: What Exists Today

Three approaches dominate AI-assisted software development in early 2026:

Spec-Driven Development (Thoughtworks, GitHub Spec Kit, AWS Kiro) places a specification at the centre of the process. Write a spec, generate a plan, break it into tasks, generate code. GitHub’s Spec Kit has over 92,000 stars and supports 30+ AI agent platforms (as of May 2026). AWS’s Kiro provides an agentic IDE built around spec-driven workflows. The approach works — teams adopting structured specs consistently report significant velocity improvements. But the lifecycle is linear — spec → plan → tasks → code — with no structured mechanism for review, change, or the second pass.

SDD tools: GitHub Spec Kit (open source, 30+ agent platforms), AWS Kiro (agentic IDE), Augment Code. The O’Reilly guide by Addy Osmani (“How to Write a Good Spec for AI Agents”) is the closest thing to a methodology document.

BMAD (Breakthrough Method for Agile AI-Driven Development) introduces role-based agents — PM, Architect, Developer, UX Designer, Scrum Master — that collaborate through structured artefacts: PRDs, architecture documents, user stories. It is the most complete open methodology available, with 12+ agent roles, quality gates between phases, ATDD (acceptance test-driven development) workflows, brownfield support for existing codebases, and a course-correction workflow for mid-sprint changes. BMAD recognises context loss as a problem and uses “hyper-detailed development stories” to manage it.

Vibe coding (Karpathy, 2025) remains the default for most developers: prompt, generate, iterate, ship. No formal structure. Works for small tasks. Breaks on anything that requires coordination across sessions, files, or concerns.

These approaches are strongest on the first pass: write the spec, generate the code, ship. BMAD has begun addressing iteration with its course-correction workflow and brownfield support. But software spends most of its life being changed, not being created. Requirements shift. Specs become stale. Code needs review against evolving standards. The second time through the pipeline — and the tenth — is where structural decisions about change management, review calibration, and session resilience matter most.

1. Test Cases Before Specs — Requirements Must Prove They Are Testable

Most methodologies generate test cases after implementation or, in BMAD’s case, before implementation code but after the architecture is designed. GVM generates them earlier — after requirements and before technical specification:

impact map → requirements → test cases → tech spec → design review → walking skeleton → build → code review → test → exploratory test → doc write → doc review → deploy

SDD: spec → plan → tasks → code. Testing happens after code exists. BMAD: PRD → architecture → stories → build, with ATDD generating acceptance tests before implementation. GVM’s distinction: test cases are generated before the technical spec, not just before code. This means requirements are validated for testability before any architecture is designed.

This is not an arbitrary ordering. Test case generation is a quality gate on requirements. Before any architecture is designed, the pipeline forces the question: can every requirement be expressed as a concrete test with given/when/then conditions, boundary values, and expected outcomes? Requirements that cannot be tested are identified and fixed before they propagate downstream.

The test-cases phase also produces a health report that identifies untestable, inconsistent, and contradictory requirements. These are resolved or explicitly acknowledged before the pipeline proceeds. By the time the tech spec phase begins, every requirement has been validated through the lens of testability. The spec inherits clean inputs.

This matters for AI-assisted development specifically because LLMs are compliant. Ask an LLM to implement a vague requirement and it will produce something — confidently, fluently, and wrong in ways that are hard to detect. The test-first gate catches vagueness at the cheapest point in the lifecycle: before any code exists.

An optional upstream gate moves the question one step further back. /gvm-impact-map (grounded in Adzic’s Impact Mapping and Cagan’s outcome-first framing) produces four linked tables — Goals, Actors, Impacts, Deliverables — before requirements are written. Each requirement must trace to a leaf Deliverable; the IM-4 gate refuses to finalise requirements that don’t. The discipline is the same as the test-first gate, applied one level higher: a requirement that has no observable change in the world is caught before it becomes a test case to be written.

2. Review Is a Pipeline Phase, Not a CI/CD Afterthought

In most AI development workflows, review means “run a linter” or “have Copilot comment on the PR.” The review happens after the code is written, often by the same model that wrote it. There is no structured rubric, no calibration, no mechanism for the review to improve over time.

GVM treats review as four distinct pipeline phases, each examining work from a different angle: document review (requirements, test cases, specs, and user documentation reviewed for quality and consistency), design review (data model, UI/UX, API contracts, and performance validated against requirements before build), code review (parallel defect-class panels scanning references, contracts, logic, and naming after build), and build verification (integration seams, smoke tests, stub audit, documentation check before release). All four share the same structural discipline:

  • Analytic rubrics with per-dimension scoring, not holistic impressions
  • Severity classification separated from defect finding — find everything first, classify second
  • Concrete deductions — every score states what the rubric requires, what was counted, and what prevents a higher score
  • Go-live criteria defined before scoring begins, not after
The review methodology draws on Fagan inspection (1976), Cronbach’s reliability theory, and Gilb’s quantified criteria from Competitive Engineering (2005). These are the mechanics of reliable measurement, applied to AI output evaluation.

Code review dispatches parallel defect-class panels as independent subagents, grounded in Perspective-Based Reading research (Laitenberger, Atkinson, Schlich & El Emam, “An experimental comparison of reading techniques for defect detection in UML design documents,” Journal of Systems and Software, 2000) — teams with non-overlapping scanning mandates detected 41% more unique defects than checklist-based teams at 58% lower cost per defect (the original study covered UML design-document inspection; the analogy to parallel code-review panels is structural, not a replicated finding). Each panel scans for a specific class of defect (references and dependencies, contracts and interfaces, logic and completeness, naming and specification compliance). Named experts are assigned to the panel whose defect class matches their primary framework. Panels run concurrently, do not see each other’s findings, apply a liberal detection threshold (Green & Swets’s Signal Detection Theory), and scan in two passes to counter satisfaction-of-search bias (Drew, Võ & Wolfe). The synthesis phase deduplicates, cross-references, uses capture-recapture estimation (Wohlin) to gauge residual defects, and elevates severity when multiple panels flag the same issue independently. This is not one reviewer scanning the code — it is structured inspection with orthogonal scanning mandates and measurable reliability.

Spec-driven and ATDD-style toolchains have begun to incorporate adversarial-review patterns — parallel scanning with non-overlapping mandates is the documented structural choice for defect detection, and the convergence reflects the same underlying research. Where GVM remains distinctive is in three additions to the panel pattern: capture-recapture estimation that converts “should we run another round?” into a calculation, calibration data that persists across rounds (§3), and the user-owned triage rule below.

The distinction from current practice is not sophistication; it is rigour. A review that produces “looks good, maybe clean up the naming” is not a quality gate. A review that says “8/10 on security (rubric: 9–10 requires input validation on all endpoints; 2 of 14 endpoints lack it; the gap is auth/callback and webhook/receive)” is.

2.1 The User Owns Finding Triage

Review skills can fail in a particular way. After dispatching panels and surfacing findings, the skill is tempted to triage them itself: silently filter borderlines, defer Important findings to a future hardening release, or bundle the verdict with the user’s first sight of the finding list. Each move is locally reasonable — the panel’s emit threshold becomes a post-synthesis filter; the skill is “saving the user time” — and globally a defect: the user never sees what was hidden, and the calibration record drifts away from what panels actually found.

The methodology now treats this as a structural defect class. Shared rule 28 (added after an incident in which a review skill silently deferred two Important findings and filtered ten Minor findings under cover of “R2+ strict criterion”) makes finding triage user-owned. Every Critical or Important finding is presented to the user before any disposition is recorded. The skill may recommend “fix / defer / dismiss” per finding but must not decide. Forbidden patterns are spelled out verbatim and pinned by structural tests: the gate cannot be paraphrased away. A “fix all” authorisation does not extend to new Critical findings discovered during the fix — those re-enter triage.

This is a small rule with a large effect on trust. The review report a practitioner reads is the same one the panels emitted. Anything the skill might have hidden, it cannot hide. The user’s authority over disposition is not advisory; it is the gate.

3. Reviews Improve Over Time — Progressive Calibration

The first review of a project has no project-specific context. It applies general principles to unfamiliar code. The second review should be better — it knows what scored well, what scored poorly, what the recurring issues are, and what “good” looks like in this specific codebase.

GVM accumulates this knowledge in a calibration file that persists across review rounds:

  • Score history — per-dimension scores across rounds, with trend direction
  • Anchor examples — the 2 best and 2 worst concrete examples from this project for each dimension, updated when better examples are found
  • Recurring findings — issues flagged in 2+ consecutive rounds, tracked until resolved
  • Dimension benchmarks — baseline (round 1), current, and trend per dimension

On round 3 of a project, the reviewer knows: “This project’s auth module scores 9 on security (good anchor). The search module scored 5 in rounds 1 and 2 (bad anchor, recurring). Input validation has been flagged twice.” The review is calibrated to the project, not just to general principles.

No other AI development methodology has this. Each SDD cycle starts fresh. Each BMAD sprint starts fresh. There is no institutional memory across iterations. GVM’s calibration file is that memory.

3.1 Calibration Creates Blind Spots — Dual Review Corrects Them

Calibration solves one problem and introduces another. A reviewer that knows the project’s history is more precise — it catches recurring issues, tracks regressions, and applies project-specific anchors. But that same history creates expectations. Areas that scored well in prior rounds receive less scrutiny. Resolved findings are trusted without re-verification. The reviewer sees what it has learned to look for and misses what it has not.

From round 3 onwards, GVM runs two reviewers in parallel: a calibrated reviewer (with full project history) and a blind reviewer (with the same criteria and expert references but no calibration data — no anchors, no resolved findings, no score history). The threshold is round 3 because round 1 establishes the first scoring anchor, round 2 establishes a pattern of expectations, and by round 3 anchoring bias (Tversky & Kahneman, 1974) has had enough history to create systematic blind spots — the point at which rater drift becomes detectable in assessment research. The blind reviewer sees the document fresh, as if encountering the project for the first time.

This applies Wang et al.’s self-consistency principle (Self-Consistency Improves Chain of Thought Reasoning in Language Models, ICLR, 2023) at the review level — two independent assessments with different information sets, then reconciliation that keeps consistent findings and flags divergences. It also addresses Deming’s concern about measurement system bias: a calibrated instrument that is never independently checked may drift without detection.

A reconciliation step classifies each blind finding: new (not in the calibrated report or resolved history — a genuine blind spot), confirmed (found independently by both — higher confidence), regression (matches a resolved finding from a prior round — was fixed, now back), or noise (rediscovered something already caught or correctly excluded). Only new findings and regressions merge into the final report. The practitioner sees one clean report, not two overlapping ones.

The dual review metadata — how many findings came from each source across rounds — reveals whether calibration is developing systematic biases. If the blind reviewer consistently catches findings the calibrated reviewer misses in a particular dimension, that dimension’s calibration may need resetting. The correction mechanism has its own feedback loop.

4. Change Management — The Pipeline Handles the Second Pass

Software changes. Requirements shift. Stakeholders revise priorities. A tech lead reverses an architecture decision. Every real project goes through this. The question is whether the methodology has a mechanism for it or whether the developer has to improvise.

Most AI methodologies are strongest on the first pass. BMAD’s course-correction handles mid-sprint changes; SDD tools like OpenSpec support delta-based updates. GVM goes further, with three distinct change modes that cover the full range of how software actually evolves:

  1. Append — the requirements phase is still in progress. New requirements are added to the existing document. The ID sequence continues. Low ceremony.
  2. New round — the pipeline is complete and new scope arrives. New numbered documents (requirements-002.md, test-cases-002.md) with distinct ID prefixes. Previous round’s artefacts are immutable. High ceremony.
  3. In-place update — a decision reversal or priority change within an existing round. The artefact is edited with a changelog entry recording what changed, when, and why. Medium ceremony.
Ceremony is proportional to risk. Appending a requirement during active elicitation is low risk. Starting a new round after the pipeline is complete is high risk — every downstream artefact may need regeneration.

The pipeline detects staleness automatically. If requirements.md has been modified after test-cases.md was generated, the status phase flags it: “requirements.md changed RE-3 priority on 2026-03-29 — test-cases.md and specs may need update.” Changelog-aware staleness names the specific IDs that changed, not just the file timestamps.

Architecture Decision Records (ADRs) use a supersession protocol. When a decision is reversed, the new ADR records the change and references the old one. The old ADR’s status is updated to “Superseded by ADR-012” but its content is immutable — it remains as a historical record of what was decided and why.

None of this exists in SDD, BMAD, or vibe coding. When requirements change in those methodologies, the developer starts over or improvises. GVM has a protocol.

5. Session Resilience — Work Survives Context Exhaustion

AI coding agents have context windows. When the context fills up, the agent loses track of what it was doing. This is not a theoretical concern — it is the primary failure mode of any AI-assisted build that takes more than one session.

BMAD acknowledges this problem. Its solution is hyper-detailed development stories that carry enough context to restart. GVM takes a different approach: every build chunk produces two persistence artefacts.

  • Pre-build prompt (build/prompts/P{X}-C{XX}.md) — everything a fresh agent needs to execute the chunk: spec excerpts, dependency handovers, cross-cutting conventions, deliverables list, TDD instructions.
  • Post-build handover (build/handovers/P{X}-C{XX}.md) — what was actually built: files created, tests passing, deviations from spec, surfaced requirements, upstream fixes, lessons learned.

When a new session starts, the build phase reads the handover directory, cross-references against the implementation guide’s dependency matrix, and knows exactly which chunks are done, which are in progress, and which are blocked. No context from prior sessions is needed. The handovers are the context.

This also enables parallel dispatch. When the dependency matrix shows chunks with no mutual dependencies, they can be built simultaneously in isolated git worktrees, each with its own prompt and handover. The parallelism is safe because each chunk’s inputs and outputs are fully specified in its artefacts.

6. Boundaries Before Features — The Walking Skeleton

The walking-skeleton pattern is not new — Alistair Cockburn introduced it in Crystal Clear, and Freeman & Pryce made it the spine of Growing Object-Oriented Software, Guided by Tests. ATDD-style frameworks have converged on similar discipline: build the skeleton, then write the first acceptance test against it, then start the TDD cycle. The pattern addresses a discipline that AI-assisted development still routinely ignores: build features against mocks, discover at integration time that the real boundaries don’t behave like the mocks said, and unwind weeks of feature work that was correct against a fiction.

GVM’s contribution is not the skeleton; it is the registry-backed gate. /gvm-walking-skeleton wires a thin end-to-end execution thread through every external integration boundary — HTTP APIs, databases, cloud SDKs, file systems, subprocess calls, email — and produces boundaries.md: a registry that names every boundary, its real-vs-sandbox status, and any divergence between the sandbox and production. Downstream phases gate against this registry mechanically. /gvm-build’s WS-5 rule refuses chunks if the skeleton is red in CI — the rule reasons that you cannot reliably build features on a foundation that doesn’t connect. /gvm-test’s VV-4(d) audits the sandbox-divergence notes when verifying the build, so a feature that worked in sandbox but failed in production has a recorded reason rather than a mystery. The skeleton is shared ground; the gates that consume its output are the structural addition.

The skeleton is not a prototype to be discarded. It is the spine the features attach to. Every feature chunk imports the skeleton’s real boundary clients and exercises them through the same path the production code will. The mocks that remain in the test suite are budgeted — see §7 below.

7. Outside-In Build Discipline — Tests, Mocks, Fixtures, and the Smoke Gate

A chunk can ship with passing unit tests and broken end-to-end behaviour. This was the most common failure mode observed across the early production builds — multiple chunks shipped green at the unit level while their integration into the running product was silently broken. Unit tests verify each layer in isolation; they do not verify the wiring between layers. Tracer-bullet chunks built in early phases looked correct against their own tests but were never imported by the later consumer chunk that was supposed to use them. By the time the verification phase ran, the wiring gap had compounded across multiple phases.

Outside-in test ordering is not GVM’s contribution. ATDD frameworks have long used the same ordering — outer loop is a failing acceptance test from the user’s perspective, inner loop is the TDD cycle that drives implementation (Adzic, Specification by Example, 2011; the broader BDD lineage from North onward). The structural addition GVM brings is four rules that sit beneath the outer-loop acceptance test, each closing a specific defect class observed when outside-in alone proved insufficient:

TDD-1 (outside-in ordering). Shared ground with the broader ATDD tradition: the acceptance test is the first deliverable for chunks with a user-facing surface, written before any unit test. The chunk transitions Red → Green on the acceptance test; both transitions are recorded in the handover. Pure-internal-helper chunks may exempt themselves with a documented rationale; an exemption with no rationale is treated as no exemption (Fagan: silent skips are not skips). Grounded in Freeman & Pryce’s outside-in discipline and Beck’s TDD cycle from Test-Driven Development: By Example (the Red→Green→Refactor slogan is the XP community formulation of Beck’s cycle).

TDD-2 (mock budget at the external boundary). Each test mocks at most one collaborator, and only when that collaborator sits at the network or process edge. Mocking an internal class is a defect: the seam the mock pretends to verify is the seam the test is silently bypassing (Metz, POODR Ch. 9 — prefer real objects for code you own; the protocol-drift defect class follows from Freeman & Pryce’s GOOS critique of internal mocking). The methodology names a defined set of external-boundary categories (HTTP clients, sockets, file systems, subprocess, OS environment and signals, paths, plus database and cloud SDKs) plus a third-party-SDK rule. Severity escalates from Important to Critical when the mock target appears in a project-level .cross-chunk-seams allowlist — these are the cross-chunk protocols whose drift would break production wiring. The rule is enforced at runtime in Python, TypeScript (vitest, jest, testdouble, sinon), and Go (testify, gomock, and *Mock|*Fake|*Stub|*Spy struct types) by the EBT contract lint that runs in /gvm-code-review Panel B.

TDD-3 (realistic-fixture catalogue). Synthetic happy-path fixtures encode the engineer’s mental model; realistic fixtures expose the gap between the model and the world (Bach & Bolton, Rapid Software Testing). Six known-edge-shape domains are catalogued (data-analysis, web/UI, API, parsing, security validation, concurrency) with starter fixture shapes per domain — short categorical codes, malformed inputs, percent-encoding edges, identical baseline-vs-actual datasets. /gvm-code-review Panel C emits an Important finding when a chunk in one of the three highest-ROI domains (data-analysis, parsing, security validation) ships with synthetic fixtures only; chunks can claim exemption with a documented rationale.

Hard Gate 8 (chunk-acceptance smoke). After the chunk’s tests pass and the wiring audits run, but before the handover is written, the chunk runs a smoke command that exercises its user-facing surface end-to-end and asserts structural contracts on the output (required HTML elements, required JSON keys, non-empty headlines, no placeholder substrings). Non-zero exit code blocks the handover — the chunk is incomplete until the smoke is green. Chunks authored before the rule’s release date are exempt as carry-over per NFR-1, so existing work continues to ship without retroactive enforcement.

Two further audits sit beside the four rules. The wiring matrix (a four-column table in the implementation guide: entry point, consumed modules, wiring chunk, demanded by) is the audit trail proving every built module has a path to the running product and a consumer that demanded it. The matrix refuses the implementation-guide write if any cell is empty. /gvm-build Hard Gate 7 mechanically greps every entry point for the modules the matrix says it consumes; the inverse audit catches the dual failure mode (a module built but never written into the matrix). The grounding is Feathers’ consumer-side seam discipline from Working Effectively with Legacy Code: a producer that no consumer demanded is dead code regardless of how green its tests are.

MVP-1 (Minimum Viable Product first ordering). Vertical slicing (Cohn, User Stories Applied; Patton, User Story Mapping) applies within chunks: TDD-1 makes the acceptance test the first deliverable. MVP-1 applies across chunks. The first user-facing chunk in the implementation guide must constitute the smallest end-to-end shippable slice, exercising one user-visible behaviour through every layer it touches against the walking skeleton’s wired boundaries. A guide whose first user-facing chunk is a layer fragment (data model only, API endpoint with no caller, UI shell with no backing data) is refused at impl-guide write time by /gvm-tech-spec Phase 5 and again at /gvm-build read time by Hard Gate 9. The two gates compose: the impl guide must satisfy both.

Four named project shapes claim exemption with the in-guide marker MVP-1 exempted: <category> — rationale: <why>: library (no user-facing surface), refactor (stable interface, implementation-only), performance-driven (the architectural constraint is the product; the first slice must exercise it), and fully-specified (regulatory rebuild where user feedback adds no information).

The defect class addressed is layer-first ordering that defers user-visible feedback to integration time. A build that ships green tests across every layer can still discover at integration that the layers do not compose into the product the user wanted. A non-gated companion paragraph in the skill prose directs practitioners to optimise chunk sequencing for the earliest runnable MVP. Position is not gated because that would force exemption inflation on legitimate infrastructure-first cases (data migration, auth bootstrapping, regulatory setup).

Horizontal architecture, vertical sequence. MVP-1 governs the delivery sequence; the cross-cutting spec and architecture overview from /gvm-tech-spec govern the patterns every slice is built against; code review enforces both. Slice 5 uses the same data-access conventions, error-handling approach, logging discipline, and naming rules as slice 1 — conceptual integrity (Brooks, The Mythical Man-Month, 1975). The first slice exercises one user behaviour through the architecture the spec already defined; subsequent slices inherit those patterns rather than reinvent them. A reader who interprets “vertical slice first” as licence to invent slice-local architecture is misreading the rule. If slice N’s data-access pattern diverges from slice 1’s, the divergence is a code-review finding regardless of whether each slice individually shipped a green smoke gate.

Each rule cites a defect class observed in production: TDD-1 closes unit-test-green / system-broken; TDD-2 closes internal-mock protocol drift; TDD-3 closes synthetic-fixture-only verification (a real defect: a privacy scan tripped on real-world short categorical codes that synthetic fixtures didn’t exercise); Hard Gate 8 closes the chunk-completion smoke gap; the wiring matrix closes the tracer-bullet engine boundary; MVP-1 closes layer-first phase ordering that defers user-visible feedback to integration time. Without the defect class, no rule. The audit trail is in methodology-changelog.md — see §12.

8. Most AI Methodologies Assume Greenfield — GVM Routes Existing Codebases

Most AI development methodologies assume greenfield. You write a spec. You generate code. There is no existing codebase to contend with.

Most software work is not greenfield. It is maintenance, extension, migration, and renovation. GVM’s site survey phase addresses this directly: before any pipeline work begins on an existing codebase, the AI explores the architecture, identifies patterns and conventions in use, diagnoses the structural state, and recommends where to enter the pipeline.

The site survey produces a health assessment across multiple dimensions (architecture, test coverage, documentation, dependency management) and routes the user to the appropriate starting point. A well-structured codebase with missing documentation enters at doc-review. A codebase with no tests enters at test-cases. A codebase that needs new features enters at requirements. The entry point is determined by diagnosis, not assumption.

The construction industry equivalent: you survey a site before renovation. You don’t start knocking down walls until you know what’s load-bearing.

This matters because the majority of developers adopting AI tools are working in existing codebases. A methodology that only handles greenfield is a methodology that only handles the minority case.

The comparison is not about which methodology is “better” in aggregate. SDD and BMAD are good at what they do — BMAD in particular has made significant progress on brownfield support, test-driven workflows, and change management. The comparison is about degree: where GVM’s pipeline takes a structural capability further (test-first at the requirements stage, calibrated review that improves across rounds, three distinct change modes with automated staleness detection) and where it addresses problems others have not yet attempted (review calibration, per-chunk session resilience with dependency-aware parallelism, design validation against requirements with explicit API boundary contract and performance checking before any build work begins).

The breadth of review coverage is worth noting explicitly. Across the full lifecycle, GVM reviews work from four distinct angles: document quality (requirements, test cases, specs, user documentation), design correctness (data model, UI/UX, API contracts, performance), code quality (references, contracts, logic, naming panels in parallel), and product verification (integration, smoke tests, stub audit). Each uses the same progressive calibration system. Most other methodologies have a single review mechanism — typically a linter or a PR comment. GVM has four, and they all get better over time.

The Cost

The pipeline looks expensive at first glance. Phases in sequence, paired artefacts, multi-panel review running in parallel and sometimes across multiple rounds with dual review, structural tests pinning the methodology’s own prose, an audit trail for every rule. For a script or a prototype, none of this is warranted — vibe coding ships a script faster than the pipeline can write the impact map.

For a serious or complex system, the calculus inverts. Defects caught in requirements cost less than defects caught in design, which cost less than defects caught in code, which cost less than defects caught in test, which cost less than defects caught in production (Boehm, Software Engineering Economics, 1981). The structural decisions described above push detection toward the cheap end of the curve. Test cases that surface untestable requirements cost less than the rounds of vibe-coded debugging they replace. Walking-skeleton boundary checks cost less than the integration failures they pre-empt. The chunk-acceptance smoke costs less than the post-merge bug hunt it replaces. The pipeline trades small, predictable upfront work for large, unpredictable downstream work.

9. The Pipeline Separates Construction from Review

The pipeline runs construction and review as distinct phases with their own prompts, their own grounding choices, and their own success criteria. The separation makes the right type of grounding available to the right phase, because construction and review respond to different types of knowledge.

Expert grounding focuses the AI and raises baseline quality. It does not guarantee a correct build. Grounding raises the baseline; structure catches what the baseline still misses.

The PRISM study (Hu, Rostami & Thomason, USC, March 2026) provides a possible mechanism from the research side — expert personas improve LLM alignment on generative tasks but reduce factual accuracy on knowledge-dependent tasks. The companion paper, The Grounded Vibe Methodology, examines the grounding mechanism in full.

Expert works are declarative — they teach what good code looks like. API documentation is procedural — it specifies exact method signatures and framework semantics. Construction needs procedural grounding (how to use this specific framework correctly) more than declarative grounding (whether the result meets quality criteria). Review needs the opposite. A methodology that blends writing and reviewing in a single pass cannot apply different grounding to each.

A related structural addition: recurring review findings feed back into construction as build checks. When the same issue is flagged in three or more consecutive review rounds, it is promoted to a concrete instruction loaded during build prompt generation. The construction phase learns from the review phase’s history. This feedback path exists because the pipeline separates the two phases with a structured interface between them: the calibration file (introduced in §3) accumulates recurring findings, and the build checks file carries them forward into construction prompts. An integrated write-and-review loop has no interface to carry this feedback.

10. Exploratory Testing Has a Phase

Scripted tests verify what someone thought to write down. Real software fails along paths nobody thought to write down. Exploratory testing is the discipline of looking for those paths — structured, time-boxed, practitioner-driven, and historically left out of every AI development methodology because its output is not a green check.

GVM’s /gvm-explore-test phase gives exploratory testing the same structural status as scripted verification. The practitioner writes a YAML charter (mission, tour, scope, duration) and the skill runs the timer. Defects are captured through structured prompts — severity in the practitioner’s words, Given/When/Then in the practitioner’s framing, reproduction steps the practitioner verifies. The skill does not auto-classify severity; that is a judgment the practitioner makes about the risk of the defect in their context. The output is a paired session report (test/explore-NNN.md + .html) that /gvm-test reads via VV-4(d) when computing the verification verdict.

The structural choice is to treat exploratory testing as a first-class pipeline phase rather than as “unstructured QA we should probably do.” The charter format is from Bach & Bolton’s Rapid Software Testing; the time-box discipline is from session-based test management. What GVM adds is the data flow: the charter feeds the session, the session produces a report, the report flows into the verification verdict, and the verdict gates the release. Exploratory testing without that data flow is advisory; with it, it is enforced.

Severity is the practitioner’s call. The methodology does not let the skill decide whether a defect is Critical or Minor — that classification depends on context the practitioner has and the model does not (regulatory exposure, customer impact, the cost of the workaround). The skill records what the practitioner said.

11. The Release Is a Quality Gate, Not a Calendar Event

Most AI development workflows end at “the code works.” Releasing is a manual step: bump a version, write some notes, push a tag. Whether the documentation matches the build, whether review findings were addressed, whether the tests actually passed — these are left to the developer to verify.

GVM’s release phase (/gvm-deploy) is a structured quality gate that aggregates the pipeline’s own output before allowing a release. It reads every review verdict produced during the pipeline run. If any review — code review, document review, design review — produced a “not ready” verdict, the release is blocked. Caveats are surfaced for the practitioner’s decision. No silent releases.

The release phase also checks documentation accuracy against the current build — not the planned build, not the build from two rounds ago, the current one. If the README describes a feature that was descoped, or if setup instructions reference a dependency that was removed, the release is blocked until the documentation is corrected. Stale documentation is treated as a release defect, not an acceptable gap.

This is a structural decision, not a quality preference. The pipeline produces verdicts; the release phase consumes them. The data flow is what prevents the bypass — a practitioner cannot tag a release without the pipeline having produced the evidence.

The skill does not deploy anything. Deployment mechanisms — CI/CD pipelines, container orchestration, cloud providers — are organisation-specific and outside the pipeline’s scope. The release phase produces a tagged, documented commit with accurate release notes and a changelog. What happens after the tag is pushed is not the methodology’s concern.

This matters because the final step of a process defines whether the preceding steps were enforced or advisory. If any developer can tag a release without checking verdicts, the review phases are advisory. If the release phase structurally requires verdict data, the review phases are enforced. The release gate closes the loop.

12. The Methodology Has an Audit Trail on Itself

A methodology that adds rules without recording why ends up bloated, ungrounded, or both. Practitioners cannot tell why a particular rule exists, whether it is still earning its keep, or whether the rule was added in response to evidence or in response to a single bad day. After enough rounds of growth, the rules become tradition rather than discipline.

GVM treats its own evolution as a defect class to address. Three structural pieces sit alongside each other:

NFR-3 — the methodology changelog. Every behaviour-changing rule lives in methodology-changelog.md with a mandatory rule reference, the defect class addressed, and the validation method (typically a structural test that pins the rule’s prose so it cannot be paraphrased away). The inclusion heuristic is concrete: “if a chunk that was acceptable yesterday would be refused today (or vice versa), it is a changelog entry.” Typo and formatting edits do not qualify; behaviour changes do. /gvm-status reads the most recent dated entry and surfaces it in a Methodology section so practitioners see, on every status check, what last changed and why.

NFR-2 — diff-budget tracking. Each chunk handover records a Diff budget delta integer. The build summary aggregates per-chunk values into a cumulative running total; when methodology additions cross 500 lines, the user is prompted to consider whether the rule set is bloating. This is surface-and-track, not a refusal gate — the bound is a conversation, not a wall — but it makes growth visible at the time it happens rather than discoverable by complaint after the fact.

NFR-1 — carry-over exemption. When a new gate is introduced, it applies to chunks authored on or after the gate’s release date and exempts everything authored earlier as “carry-over.” The release date is a literal constant in the relevant SKILL.md; the exemption is an mtime comparison on the chunk’s prompt file. This is the rule that lets the methodology evolve without retroactively breaking projects mid-flight. Distributed self-versioning — each major rule release adds its own _V{MAJOR}_{MINOR}_{PATCH}_RELEASE_DATE constant co-located with the gates that read it — means there is no central install-version file to fall behind.

A fourth structural piece (codified as shared rule 22) sits in the same family: cost-optimised model selection. Different pipeline steps need different models. Architectural reasoning, requirements elicitation, and document review need the deep reasoning of the primary model. Code generation, structured extraction, and panel-level finding emission are guided work that runs well on Sonnet. HTML rendering is template-filling that runs well on Haiku. The pipeline picks the right model for each step rather than running everything on the most expensive option. This is structural because it forms part of the contract between the skill and the runtime — the skill names the model, the runtime obliges — not a per-invocation guess.

The audit trail and the model-selection rule have the same root: the methodology is not a manifesto. It is a working system, and working systems need accountability for their growth, their cost, and their mistakes. The changelog records the why; the diff budget records the cost; the carry-over rule preserves continuity; the model selection records the trade-off. None of these are visible in the pipeline output, but all of them are visible to the practitioner who wants to ask, “why is this rule here?”

When the Pipeline Is Overkill

The pipeline has a domain of fit. Inside it — serious or complex systems with real failure cost — the discipline pays for itself many times over (see The Cost). Outside it, the overhead exceeds what the work warrants.

A throwaway script doesn’t earn an impact map. A 50-line spike to test a hypothesis doesn’t need a wiring matrix or a chunk-acceptance smoke. A narrow refactor with no user-facing change doesn’t need an outside-in acceptance test. The pipeline would work in all these cases — nothing about the discipline prevents it from building the application. But the trade it offers — small, predictable upfront cost against large, unpredictable downstream cost — only pays off when the downstream cost is actually large. For work where downstream cost is bounded, the trade is the wrong way around.

The pipeline executes discipline. It does not choose when to apply itself. The user judges where the line falls between work that warrants the structure and work that doesn’t; the pipeline runs whatever discipline that judgment selects. A practitioner who runs the full pipeline on every script wastes effort. A practitioner who skips it on a serious system takes on risk it was designed to reduce. Fit and judgment are upstream of every gate — structurally, the pipeline cannot correct for them.

Records for the Future

Where the pipeline runs, it produces software and a structured artefact trail as a side effect: requirements with priorities and rationales, test cases traceable to those requirements, technical specifications with named architecture decisions, calibrated review reports and the calibration history they feed, per-chunk handovers naming what was built and what deviated, exploratory testing session reports, and a methodology changelog explaining why each rule exists.

The most concrete use is maintenance and extension. An engineer adding a feature six months later wants to know why the existing data model has the shape it has, why the auth flow takes the path it does, why a particular library was chosen. A pipelined project has those answers in the requirements, the relevant ADR, and the handover that built the affected code. A vibe-coded project answers the same questions by guessing, asking, or reading the code and inferring intent. The pipeline’s artefacts reduce the cost of maintenance — and the longer the gap between original build and maintenance task, the more they pay back.

The same trail serves other readers when needed. An auditor reconstructing a trade-off finds it in an ADR with date and alternatives considered. A postmortem after an outage finds the calibration history of what was reviewed and what was missed. A successor maintainer inherits the project’s record of its own thinking, not just the code.

These artefacts cost less to produce as a pipeline side effect than to reconstruct after the fact. The trail is also the only structural answer to “did the discipline actually happen on this project?” — the artefacts are the evidence that the methodology ran.

Why Structure Matters as Much as Staffing

GVM’s other contribution — citation-based expert activation — makes the pipeline produce higher-quality output by staffing each phase with named authorities. That mechanism is real and experimentally supported for evaluative and structural tasks. But it is not what makes the pipeline structurally different. And as §9 describes, the relationship between grounding and pipeline phase is nuanced — grounding focuses the AI but does not guarantee correctness, and the pipeline’s phase separation is what makes it possible to apply the right type of grounding at the right time.

Strip out the expert grounding entirely. Replace every named expert with a generic instruction. The pipeline still catches early (impact mapping before requirements, test cases before specs, walking-skeleton boundaries before features), still calibrates across rounds and corrects for its own bias, still separates construction from review so the right type of grounding reaches the right phase, and still gates the release on the pipeline’s own verdict data. The body of this paper describes the full set; the point here is that none of it requires named experts to function.

Expert grounding makes the output better — demonstrably for review and design, and with undemonstrated effect for construction. The pipeline structure makes the process reliable and makes the grounding applicable where it works. Both matter. Together they are more than either alone — but the structure enables the grounding, not the other way around.

The claim

The GVM pipeline’s structural decisions — impact mapping before requirements, test-first ordering, calibrated review with user-owned triage, three-mode change management, per-chunk session resilience, the walking skeleton, outside-in build discipline (TDD-1/2/3, the chunk-acceptance smoke gate, the wiring-matrix audit, and MVP-first phase ordering), existing-codebase routing, the separation of construction from review, exploratory testing as a phase, a release quality gate that enforces the pipeline’s own verdicts, and an audit trail on the methodology itself — address failure modes that the spec, the tests, and the code cannot catch on their own. These decisions function independently of expert grounding and represent a distinct contribution alongside it. Well-engineered software needs an end-to-end pipeline.

Deployment Topology — Single User to Organisation

The pipeline’s data model — artefacts, calibration, build checks, expert activations — implies a deployment question that the methodology itself does not resolve: where does the data live, and who can see it?

Three tiers, each with different sharing boundaries:

Deployment tiers
TierScopeStorageShared ViaExample Data
ProjectOne codebase, one teamFiles in the repoGitCalibration scores, build checks, handovers, specs, review reports
OrganisationAll projects, all teamsCentral database or shared serviceAPI / shared filesystemCross-project build check patterns, org-wide expert activation data, validated stack/domain patterns
PluginAll users of the methodologyExpert reference files shipped with the pluginPlugin updatesCanonical expert scores, scoring methodology, shared rules

For a single developer, only the project tier matters. Calibration, build checks, and handovers live in the repo. Expert reference files live in the plugin install path. The local home directory (~/.claude/gvm/) holds activation logs and discovered experts. This is the current implementation.

For a team on one project, the project tier still works. Build checks in reviews/build-checks.md are committed to the repo and shared via git. Every team member’s GVM instance reads the same checks. Calibration accumulates across review rounds regardless of who runs the review. No infrastructure change is needed.

For an organisation with multiple teams and projects, the optional Application Brief becomes more important. AI makes it cheap to start projects — the barrier drops from months of planning to a weekend with an AI agent. The cost of creating is now lower than the cost of evaluating whether something should exist. The brief inverts this: cheap to write, cheap to review, but gates the expensive downstream pipeline work. Organisations that mandate the brief for all projects get a portfolio-level view of what is being proposed before resources are committed.

A second gap also emerges at the org level. Build checks are project-local. A pattern that surfaces in three projects — “async error boundaries missing in React/TypeScript codebases” — is discovered independently by each team, promoted independently in each project, and never shared across projects. The same defect class is found and fixed three times instead of once.

The organisation tier addresses this with a central store that aggregates build check patterns across projects. The architecture decisions an org deployer would face:

  • Promotion threshold: a pattern becomes org-level when it appears in 5+ projects with matching stack or domain tags. This is deliberately conservative — org-level patterns are loaded into every matching project’s build prompts, consuming context and adding review criteria. Too many patterns degrade the signal-to-noise ratio. Five independent occurrences across different projects and teams is a strong signal that the pattern is systemic, not coincidental.
  • Filtering by relevance: org-level patterns carry stack tags (React, Python, Django) and domain tags (fintech, healthcare). A project only loads patterns that match its own stack and domain. A financial services team is not forced to apply patterns from a mobile gaming project.
  • Team override: teams can reject org-level patterns that don’t apply to their specific project. The central store records the rejection with rationale — this feedback improves pattern quality over time.
  • Graduation to plugin: when a pattern has been validated across enough projects and teams, it becomes a candidate for inclusion in the expert reference files as a permanent key principle. This is a curation decision, not an automatic promotion.

The methodology already defines the data model that the org tier would aggregate: build checks with IDs, root cause patterns, expert attribution, stack/domain tags, promotion criteria, and retirement rules. The deployment question is infrastructure — a database, an API, access controls — not methodology. But the methodology’s data model was designed to support this aggregation, even though the current implementation does not require it.

The three-tier pattern mirrors how other engineering disciplines handle institutional knowledge: project-level lessons learned, organisation-level best practice libraries, and industry-level standards. The difference is that GVM’s data model is machine-readable from the start — build checks are structured artefacts, not prose documents. An org-level aggregator can query, filter, and propagate them programmatically.
