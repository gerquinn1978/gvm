Verifying AI-Generated Code

A layered verification architecture for code you didn't write

Conor Brennan & Gerard Quinn

April 2026

This paper in context. This paper covers the testing and verification architecture. For the full methodology, see The Grounded Vibe Methodology. For the pipeline structure that this verification layer operates within, see Well-Engineered Software Needs an End-to-End Pipeline. For a case study showing the verification approach applied to a real build (207 tests, 7 review rounds), see Building a Production Tool with GVM.

AI-generated code has failure modes that traditional quality gates were not designed to catch: hallucinated dependencies, embedded secrets, systematic vulnerability patterns, and logic that looks correct but violates unstated invariants. Effective verification requires layering detection mechanisms with distinct error models, so that what one layer is blind to, another catches.

This paper describes such an architecture across five mechanism families: tool-based layers, construction-time gates, expert-grounded code review, exploratory testing, and a self-correcting feedback loop. The tool-based layers run dependency verification, credential scanning, static analysis, property-based testing, and mutation testing. Construction-time gates refuse builds that fail their preconditions, alongside review-time rules that surface boundary discipline as findings. Expert-grounded code review uses parallel defect-class panels. Exploratory testing brings a human error model that no automated layer can replicate. The self-correcting loop combines progressive calibration, blind reviewer bias correction, build check promotion, a finding quality gate, and user-owned triage. A release quality gate aggregates every verdict and refuses the tag if any layer reports not-ready.

The architecture is implemented in the Grounded Vibe Methodology (GVM) and is portable across ten language stacks.

The Problem

AI-generated code has different failure modes than human-written code. Not uniformly worse — different. A human developer who writes an insecure SQL query at least used a real library to do it. An AI coding assistant might import a package that does not exist, embed a secret in a constant, or generate a function that looks plausible but violates an invariant no test was written to catch.

The evidence is accumulating across independent studies:

Vulnerability rates

AI-generated pull requests contain 1.75x more logic errors and 3x more readability issues than human-written PRs (CodeRabbit, 2025). 25.1% of AI code samples contain a confirmed vulnerability (AppSec Santa, 2026, 534 samples across 6 LLMs). Injection-class weaknesses — SSRF, command injection, NoSQL injection, path traversal — account for 33.1% of all confirmed findings.

Perry et al. (ACM CCS, 2023) found that developers with AI assistance wrote significantly less secure code than those without — and were more likely to believe they had written secure code.
Supply chain risk

Roughly 20% of AI-recommended packages do not exist. 43% of hallucinated package names are repeatable across runs of the same prompt, making them exploitable attack vectors (Spracklen et al., USENIX Security, 2025; 576,000 code samples). The rate varies by model — commercial models hallucinate at 5.2%, open-source at 21.7% — but the pattern is consistent.

Secret leakage

28.65 million hardcoded secrets were detected on public GitHub in 2025, a 34% year-over-year increase — the largest single-year jump recorded. AI-assisted commits show elevated secret leakage rates compared to the 1.5% baseline (GitGuardian, State of Secrets Sprawl, 2026).

These are not edge cases. They are systematic patterns that emerge from how large language models generate code: they optimise for plausibility, not correctness. A function that looks right, passes a superficial reading, and matches the statistical distribution of training data can still be wrong in ways that matter.

The response so far has been to add AI to existing quality gates — AI-powered code review, AI-generated tests, AI-assisted security scanning. This creates a pipeline where an LLM reviews the output of another LLM, using the same training data, the same architectural biases, and the same blind spots. The detector and the generator share an error model. This is not verification. It is corroboration.

Why Traditional Gates Fall Short

Software engineering has always used layered quality techniques. Code review catches design issues. Unit tests catch logic bugs. Integration tests catch wiring failures. The insight, documented by Capers Jones across decades of industry data, is that no single technique exceeds roughly 85% defect removal efficiency — but combining independent techniques compounds the coverage multiplicatively.

Capers Jones's data on defect removal efficiency spans thousands of projects across several decades. The consistent finding: no individual technique — inspection, testing, or static analysis — reaches comprehensive coverage on its own. Only stacking multiple independent techniques reaches 95%+ cumulative removal.

Traditional gates were designed for human failure modes: typos, off-by-one errors, forgotten null checks, misunderstood requirements. AI failure modes are structurally different:

  • Hallucinated dependencies — the code imports a package that does not exist in any registry. A human developer would not invent a library name; an LLM routinely does.
  • Fabricated API surfaces — the code calls a method that looks like it should exist on a real library but does not. The method name is plausible, the arguments are reasonable, and the return type makes sense. It is entirely fictional.
  • Systematic vulnerability patterns — AI-generated code exhibits specific vulnerability concentrations (33.1% injection-class) that differ from human vulnerability distributions.
  • Credential embedding — LLMs reproduce patterns from training data, including hardcoded secrets, API keys, and connection strings that should be environment variables.
  • Plausible but incorrect logic — code that reads correctly, passes a casual review, satisfies simple test cases, but violates an invariant that was never tested.

A code review conducted by a human who understands the domain catches many of these. But AI-assisted development is adopted precisely because the volume of code exceeds what humans can review line by line. The verification architecture must scale with the generation rate, not depend on human attention that is already the bottleneck.

Distinct Error Models

The principle that makes layered verification work is not just "run more checks." It is that each layer must use a fundamentally different detection mechanism — a different error model. Two detectors that share an error model will tend to miss the same defects, no matter how many times you run them. The corroboration trap described above — where the detector and the generator share training data and biases — is what happens when layers share an error model.

A static analysis tool is a different error model. It does not assess plausibility. It pattern-matches against known vulnerability signatures using deterministic rules. A credential scanner searches for entropy patterns and known secret formats, indifferent to whether the code "looks right." A property-based testing framework generates thousands of random inputs and checks invariants mechanically.

A caveat: these layers are not entirely free of LLM involvement. The LLM orchestrates their execution, and in some cases contributes to their setup — the LLM identifies which invariants to test, writes the property-based test code, and authors the tests that mutation testing runs against. The independence is in the detection mechanism, not in the end-to-end process. Bandit applies deterministic rules regardless of who wrote the code. Hypothesis generates random inputs regardless of who wrote the test. The LLM's blind spots in code generation do not predict the tools' blind spots in detection — and that partial independence is what makes the layering effective, even though it falls short of full separation.

The compounding principle

If each independent layer catches 50% of the defects the others miss, three layers reach 87.5% and five reach 96.9%. These figures assume full independence; partial correlation between layers reduces the compound rate. But the principle holds: the less correlated the detectors, the more their coverage compounds. Correlated detectors do not compound — they merely agree.

The verification architecture described here layers five detection mechanisms, each with a distinct error model, each targeting specific AI failure modes. A sixth layer — expert-grounded review — uses the same model family as the generator but applies structural diversity through named expert frameworks rather than undifferentiated prompting.

The Verification Architecture

Five layers, ordered from cheapest and fastest (run first, fail fast) to most expensive and most informative (run last, on critical paths only). Each layer targets a specific category of AI failure mode with a detection mechanism that does not share the LLM's error model — avoiding the corroboration trap described above.

Layer 1: Dependency Verification

Failure mode: Hallucinated packages — imports that reference libraries which do not exist in any registry.

Detection mechanism: Attempt to resolve every new dependency against the actual package registry. For interpreted languages (Python, Node), import the package and check the exit code. For compiled languages (Go, Rust, Java), the build step itself is the verification — a missing dependency fails compilation.

Error model independence: Registry lookup. No machine learning, no heuristics — the package either exists or it does not.

The 43% repeatability rate of hallucinated package names (USENIX Security, 2025) makes this an exploitable attack vector. An attacker can register the hallucinated name and serve malicious code to every project that trusts the AI's recommendation.

A hallucinated dependency poisons everything downstream — tests, static analysis, and review are all wasted effort if the code cannot even resolve its imports. The attack surface is real: researchers have coined the term "slopsquatting" (coined by security researcher Seth Larson, 2025) for the practice of registering hallucinated package names with malicious payloads, exploiting the predictability of AI package hallucinations.

Layer 2: Credential Scanning

Failure mode: Hardcoded secrets — API keys, connection strings, tokens, and passwords embedded in source code.

Detection mechanism: Entropy analysis and pattern matching against known secret formats. Tools like gitleaks and trufflehog scan for high-entropy strings, known provider patterns (AWS keys beginning with AKIA, GitHub tokens with ghp_), and common assignment patterns (password = "...").

Error model independence: Pattern matching against known formats and entropy thresholds. Shares nothing with the LLM's generative model.

GitGuardian (2026): 28.65 million hardcoded secrets on public GitHub in 2025, a 34% year-over-year increase. Once a secret is committed, it persists in git history even after deletion from the working tree.

Any detected secret is a hard block. The remediation is mechanical: replace the hardcoded value with an environment variable or configuration lookup. This layer runs before static analysis because a leaked secret is a more urgent problem than a code quality issue.

Layer 3: Static Analysis (SAST)

Failure mode: Systematic vulnerability patterns — injection, XSS, path traversal, insecure deserialisation, and other categories where AI-generated code shows elevated rates.

Detection mechanism: Dedicated SAST tools that apply deterministic rule sets and data-flow analysis. Each language has established tools: bandit (Python), gosec (Go), eslint-plugin-security (JavaScript/TypeScript), cargo-audit (Rust), SpotBugs with FindSecBugs (Java).

Error model independence: Rule-based pattern matching and data-flow analysis. The tools encode human-authored vulnerability knowledge, not learned distributions.

ISSTA 2024: a single SAST tool produces warnings on only 52% of vulnerability-contributing changes, and 22% of vulnerable changes remain undetected even with SAST tooling, due to limitations of SAST rules. This is why SAST is one layer among five, not the entire strategy.

HIGH and CRITICAL severity findings block progression. Complexity findings (cyclomatic complexity, cognitive complexity) are reported but do not block — they inform refactoring decisions rather than gate the build.

Layer 4: Property-Based Testing

Failure mode: Plausible but incorrect logic — code that passes specific example-based tests but violates general invariants.

Detection mechanism: Generate random inputs and verify that invariant properties hold across all of them. Where an example-based test checks one input-output pair, a property-based test checks thousands — probing edge cases, boundary values, and combinations that a human tester would not think to specify.

Error model independence: Randomised input generation. The test framework knows nothing about how the code was generated — it only knows the invariant and whether the code satisfies it.

Ravi & Coblenz (OOPSLA, 2025) found each property-based test caught approximately 50x as many mutations as the average unit test, with 76% of mutations detected within the first 20 random inputs. The study quantifies what practitioners have long observed: property tests are disproportionately valuable per test written.

Property-based testing is disproportionately effective against AI-generated code because AI optimises for the common case. A function that handles typical inputs correctly but fails on empty collections, negative numbers, or Unicode edge cases will pass example-based tests and fail property-based tests. The invariant patterns — reversibility, idempotency, commutativity, round-trip encoding, preservation of size — are precisely the properties that plausible-looking code is most likely to violate.

Layer 5: Mutation Testing

Failure mode: Inadequate test suites — tests that pass but would not detect injected faults, indicating they are not actually verifying the behaviour they claim to verify.

Detection mechanism: Systematically inject small code changes (mutants) — flip a comparison operator, change a constant, remove a conditional — and verify that at least one test fails for each mutation. Surviving mutants indicate test gaps.

Error model independence: Syntactic code transformation. The mutation engine modifies the code mechanically, with no awareness of what the code is supposed to do.

Mutation testing is the most expensive layer and runs last, targeting only critical-path code: MUST-priority requirements, security-sensitive logic, and business rules with high consequence of failure. It is optional — the first four layers provide substantial coverage without it — but for code where correctness is non-negotiable, mutation testing answers the question that the other layers cannot: "would our tests actually catch a bug if one were introduced?"

Construction-Time Verification

The five tool-based layers run after code exists. A complementary set of mechanisms runs during construction itself: hard gates that refuse builds failing their preconditions, and review-time rules that surface boundary discipline as findings. These catch defect classes that surface only at construction time — wiring that was never made, contracts that were never honoured, fixtures that never tested the model against the world, and slices that defer user-visible value to later phases. Tests-pass-but-product-broken failures happen because tests verify each layer in isolation while the wiring, the contracts, and the fixtures between layers go unverified. Construction-time verification closes that gap.

The walking skeleton and boundary registry

Before any feature work, a thin end-to-end execution thread is wired through every external integration boundary — HTTP APIs, databases, cloud SDKs, file systems, subprocess calls, email. The output is boundaries.md: a registry naming every boundary, its real-vs-sandbox status, and any divergence between the sandbox and production. Downstream phases gate against this registry mechanically. The build phase refuses chunks if the skeleton is red in CI — you cannot reliably build features on a foundation that does not connect. The test phase audits the sandbox-divergence notes, so a feature that worked in sandbox but failed in production has a recorded reason rather than a mystery.

Walking skeleton from Cockburn (Crystal Clear) and Freeman & Pryce (Growing Object-Oriented Software, Guided by Tests). Both establish the discipline that AI-assisted development still routinely ignores: build features against mocks, discover at integration time that the real boundaries do not behave like the mocks said, and unwind weeks of feature work that was correct against a fiction.

TDD-1, TDD-2, TDD-3

Three rules sit beneath the outside-in acceptance test, each closing a specific defect class.

TDD-1 (outside-in ordering). The acceptance test from the user’s perspective is the first deliverable for any chunk with a user-facing surface, written before any unit test. The chunk transitions Red → Green on the acceptance test; both transitions are recorded in the chunk handover. Pure-internal-helper chunks may exempt themselves with a documented rationale; an exemption with no rationale is treated as no exemption (Fagan: silent skips are not skips). Grounded in Freeman & Pryce’s outside-in discipline and Beck’s TDD cycle from Test-Driven Development: By Example (the Red→Green→Refactor slogan is the XP community formulation of Beck’s cycle).

TDD-2 (mock budget at the external boundary). Each test mocks at most one collaborator, and only when that collaborator sits at the network or process edge. Mocking an internal class is a code smell: the seam the mock pretends to verify is the seam the test is silently bypassing (Metz, POODR Ch. 9 — prefer real objects for code you own; the protocol-drift defect class follows from Freeman & Pryce’s GOOS critique of internal mocking). The methodology names a defined set of external-boundary categories — HTTP clients, sockets, file systems, subprocess, OS environment and signals, paths, plus database and cloud SDKs. Severity escalates from Important to Critical when the mock target appears in a project-level .cross-chunk-seams allowlist — these are the cross-chunk protocols whose drift would break production wiring. The rule is enforced at runtime in Python, TypeScript (vitest, jest, testdouble, sinon), and Go (testify, gomock, plus *Mock|*Fake|*Stub|*Spy struct types) by an EBT contract lint that runs in the code review phase.

TDD-3 (realistic-fixture catalogue). Synthetic happy-path fixtures encode the engineer’s mental model; realistic fixtures expose the gap between the model and the world (Bach & Bolton, Rapid Software Testing). Six known-edge-shape domains are catalogued — data analysis, web/UI, API, parsing, security validation, concurrency — with starter fixture shapes per domain (short categorical codes, malformed inputs, percent-encoding edges, identical baseline-vs-actual datasets). Code review emits an Important finding when a chunk in one of the three highest-ROI domains (data analysis, parsing, security validation) ships with synthetic fixtures only.

Chunk-acceptance smoke gate and the wiring matrix

After the chunk’s tests pass and the wiring audits run, but before the handover is written, the chunk runs a smoke command that exercises its user-facing surface end-to-end and asserts structural contracts on the output (required HTML elements, required JSON keys, non-empty headlines, no placeholder substrings). Non-zero exit code blocks the handover. The chunk is incomplete until the smoke is green.

Beside the chunk smoke sits the wiring matrix: a four-column table in the implementation guide naming entry point, consumed modules, wiring chunk, and the consumer that demanded each module. The matrix refuses the implementation-guide write if any cell is empty. The build phase mechanically greps every entry point for the modules the matrix says it consumes; the inverse audit catches the dual failure mode (a module built but never written into the matrix). The grounding combines Feathers’ seam concept (Working Effectively with Legacy Code) with YAGNI applied to producer modules: a producer that no consumer demanded is dead code regardless of how green its tests are.

MVP-first phase ordering

Vertical slicing applies within chunks (TDD-1 makes the acceptance test the first deliverable). MVP-1 (Minimum Viable Product first ordering) applies across chunks. The first user-facing chunk in the implementation guide must constitute the smallest end-to-end shippable slice exercising one user-visible behaviour through every layer it touches against the walking skeleton’s wired boundaries. A guide whose first user-facing chunk is a layer fragment (data model only, API endpoint with no caller, UI shell with no backing data) is refused at impl-guide write time and again at impl-guide read time. The two gates compose: tech-spec refuses the write, build refuses the read.

Four named project shapes claim exemption with the literal in-guide marker MVP-1 exempted: <category> — rationale: <why>: library (no user-facing surface), refactor (stable interface, implementation-only), performance-driven (the architectural constraint is the product; the first slice must exercise it rather than a token user-visible behaviour), and fully-specified (regulatory rebuild where user feedback adds no information). An exemption with no rationale is treated as no exemption (Fagan: silent skips are not skips). The defect class addressed is layer-first ordering that defers user-visible feedback to integration time: a build that ships green tests across every layer can still discover at integration that the layers do not compose into the product the user wanted.

Horizontal architecture, vertical sequence. MVP-1 governs the delivery sequence; the cross-cutting spec and the architecture overview from /gvm-tech-spec govern the patterns every slice is built against. Slice 5 uses the same data-access conventions, error-handling approach, logging discipline, and naming rules as slice 1 — Brooks’s conceptual integrity. The first slice exercises one user behaviour through the architecture the spec already defined; subsequent slices inherit those patterns rather than reinvent them. A reader who interprets “vertical slice first” as licence to invent slice-local architecture is misreading the rule: if slice N’s data-access pattern diverges from slice 1’s, the divergence is a code-review finding regardless of whether each slice individually shipped a green smoke gate.

Each rule cites a defect class observed in production. TDD-1 closes unit-test-green / system-broken. TDD-2 closes internal-mock protocol drift. TDD-3 closes synthetic-fixture-only verification. The smoke gate closes the chunk-completion gap. The wiring matrix closes the tracer-bullet engine boundary — modules built but never imported by the consumer that was supposed to use them. MVP-1 closes layer-first phase ordering that defers user-visible feedback to integration time. Without the defect class, no rule.

Expert-Grounded Review

The five tool-based layers use detection mechanisms that are mechanistically distinct from the LLM — but the LLM remains involved in orchestration and, for testing layers, in writing the tests themselves. LLM-based code review adds a different kind of value: it catches design issues, architectural violations, and requirement mismatches that deterministic tools cannot assess. The question is how to make that review as diverse as possible in what it detects.

The approach is structural diversity through defect-class partitioning. Instead of prompting a reviewer with “review this code,” the review dispatches parallel panels, each scanning for a specific class of defect. This orthogonal partitioning is grounded in Perspective-Based Reading research (Laitenberger, Atkinson, Schlich & El Emam, “An experimental comparison of reading techniques for defect detection in UML design documents,” Fraunhofer IESE, Journal of Systems and Software, 2000) — teams with non-overlapping scanning mandates detected 41% more unique defects than checklist-based teams at 58% lower cost per defect (the original study covered UML design-document inspection; the analogy to parallel code-review panels is structural, not a replicated finding). Drew, Võ & Wolfe’s satisfaction-of-search research shows why: finding one defect reduces the likelihood of finding others nearby, so a broad scan misses what a focused scan catches.

  • A references panel scans every import, path, and cross-reference — Hunt & Thomas’s DRY principle applied to dependency verification.
  • A contracts panel traces every interface — Keeling’s decision capture framework applied to producer/consumer alignment.
  • A logic panel traces every execution path — Fagan’s entry/exit criteria applied to branch completeness.
  • A naming panel verifies terminology — Brooks’s conceptual integrity applied to specification compliance.
Each panel receives specific files, specific expert principles, and the instruction to scan only its defect class. A references panel does not comment on logic errors. A contracts panel does not flag naming issues. The scanning mandates do not overlap.

Each panel applies a liberal detection threshold — Green & Swets’s Signal Detection Theory: lower the R1 criterion to maximise recall at the cost of false positives, which the synthesis phase filters. Each panel scans in two passes to counter satisfaction-of-search within its own mandate, and capture-recapture estimation (Wohlin) provides a statistical basis for estimating residual defects from the overlap between panels.

This is not the same as model diversity — the panels share an underlying model family. But the defect-class partitioning provides genuine diversity of scanning mandate. A references panel and a logic panel look at the same function and search for different things, apply different criteria, and catch different categories of issue. Combined with the five tool-based layers that share nothing with the model, the review panels add a sixth error model that, while not fully independent, is structurally diversified.

The approach follows established principles — the PBR finding that orthogonal perspectives outperform checklist-based review (Laitenberger et al., 2000), Fagan’s finding that role-specific checklists outperform general impressions, and the broader literature on structured expert judgement outperforming unstructured judgement.

Early observational data suggests the diversity is real. Panel findings cluster by defect class in ways consistent with the partitioning: references panels find missing imports and broken paths; contracts panels find interface mismatches and missing error contracts; logic panels find unreachable branches and missing edge cases; naming panels find specification drift and inconsistent terminology. The overlap between panels is predominantly at the instance level — two panels finding the same defect from different angles — rather than the category level. Some panels produce findings with essentially zero overlap with any other panel.

In one project's review cycle, the blind reviewer confirmed calibration bias: calibrated panels had developed tunnel vision on a recurring theme while the blind panel caught config validation, error handling, and untested paths that the calibrated panels overlooked.

The sample is still small, and a controlled study comparing finding distributions across expert-framed versus generic LLM review would strengthen the claim. What the data does establish is that the architecture produces the intended effect — whether it does so reliably and at scale remains open.

Spec-driven and ATDD-style toolchains have begun to incorporate similar patterns. The convergence reflects the same underlying research: parallel review with non-overlapping mandates is the documented structural choice for defect detection. What remains distinctive in the architecture described here is three additions to the panel pattern: capture-recapture estimation that converts “should we run another round?” into a calculation, calibration data that persists across rounds, and the user-owned triage rule described below.

A Self-Correcting Verification System

The five layers and the expert review panels catch defects. But a verification system that never learns from its own findings is static — it catches the same categories of issue at the same rate, indefinitely. Three mechanisms make the system adaptive.

Iterative Convergence

The review layer is not a single pass. Multiple review rounds are expected, and each round catches different things. The first round surfaces the obvious issues — the problems visible when attention is spread across the entire codebase. Fixing those issues changes the code, and the changed code exposes subtler problems that were masked by the noise of the first-round findings. The second round, operating on a cleaner artefact, catches what the first round could not see.

Fagan (IBM, 1976) observed this in manual inspections: removing major defects in the first pass revealed minor defects that were invisible while the major ones dominated reviewer attention. The same principle applies when the reviewer is an LLM.

This is a different kind of layering from the five tool-based layers. Those are parallel — different tools applied to the same code. The review loop is serial — the same reviewer applied to progressively evolving code. Each pass operates on a different artefact than the last, because the fixes from the previous round have changed what is there to find. The value is not in repeating the same check, but in the fact that each iteration presents a different surface to the reviewer.

Progressive Calibration with Bias Correction

Each review round produces findings. These findings accumulate into a calibration file that tracks: which issues were found, which were resolved, which recurred, and which areas scored well or poorly. On subsequent rounds, the reviewer loads this history and applies it — giving more scrutiny to areas that have been problematic and less to areas that have been clean.

This makes reviews more precise over time. It also introduces a systematic bias: areas that scored well in prior rounds receive less attention. Resolved findings are trusted without re-verification. The reviewer develops blind spots shaped by its own history.

The correction mechanism is a blind reviewer, dispatched automatically from the third round onwards. The blind reviewer receives the same code and the same expert references, but no calibration history, no resolved findings, no anchor examples. It reviews as if seeing the project for the first time. Its findings are then reconciled with the calibrated review:

  • New findings — caught by the blind reviewer but not the calibrated one. These are genuine blind spots. Merged into the final report.
  • Confirmed findings — caught by both reviewers independently. Higher confidence. Kept.
  • Regressions — match a resolved finding from a prior round. The fix did not stick. Flagged.
  • Noise — match a finding the calibrated reviewer correctly excluded. Discarded.
This applies Wang et al.'s self-consistency principle (ICLR, 2023) at the review level: two independent assessments with different information sets, reconciliation that keeps consistent findings and flags divergences.

The dual review runs automatically, not on request. A practitioner who has run five calibrated rounds and developed blind spots is the least likely to think they need a blind review. The cost — one additional parallel reviewer plus reconciliation — is proportional to the value of catching what calibration systematically misses.

Build Check Promotion

Review findings fix the output. Build checks fix the process.

Deming: recurring defects are common-cause variation — the process must change, not just the output. Fagan (IBM, 1976): checklists derived from past inspections are the most effective calibration tool for future inspections.

When a finding recurs in three or more consecutive review rounds, or when a resolved finding reappears (a regression), the finding is promoted to a build check — a concrete instruction that is injected into the code generation prompt for all subsequent builds. The AI that generates the code receives the pattern to avoid as part of its instructions, preventing the defect from being generated in the first place.

Process feedback, not just output correction

Traditional quality systems inspect the output and fix what they find. The build check promotion creates a feedback loop from inspection back to generation: recurring defects modify the generation instructions, not just the generated code. The system learns from its own review history.

Not all build checks are equal. A check that traces to published expert literature — McConnell's prohibition on magic numbers, Hunt & Thomas's fail-fast principle — is a permanent engineering principle that the project's review history surfaced but that exists independently of the project. These Tier 1 checks never retire. The code may be clean because the check is enforced. Removing it would reintroduce the class of defect the system learned to prevent.

Project-specific checks with no grounding in expert literature — "the search module keeps using raw SQL instead of the query builder" — are Tier 2. After three consecutive rounds without being triggered, the practitioner is asked whether to keep or retire them. The system flags; the user decides. A practitioner who knows the module is fragile will keep the check. One who knows the module was refactored away will retire it.

Finding Quality Gate

A verification system that never stops finding issues is not thorough — it is miscalibrated. Reviewers can generate findings indefinitely by lowering the threshold for what counts as a problem. Without a stop condition, each review round produces diminishing-value observations dressed up as findings, and the practitioner cannot distinguish signal from noise.

The quality gate is a three-part test that every finding must pass before it enters the report. First, consumer impact: state the consequence as a sentence — "a developer would build the wrong thing," "a reader would cite a wrong source." If you cannot finish that sentence with a concrete outcome, the finding is polish. Second, propagation: does the finding cause a downstream problem in another artefact? A wrong requirement ID propagates through tests, specs, and code. A left-aligned number in a table propagates nowhere. Third, diminishing returns: each review round holds new findings to a higher bar than the last. Round 1 catches structural problems. Round 3 should not be surfacing redundant sentences or minor formatting preferences.

The stop condition is explicit: when all remaining observations fail the consumer impact test, the artefact is done. A clean review — no actionable findings — is a valid outcome. It means the quality system worked.

The combination of iterative convergence, progressive calibration, blind reviewer bias correction, tiered build check promotion, and a finding quality gate creates a verification system that improves with each iteration: reviews get more precise, blind spots get caught, systemic patterns get prevented at the source, and the system knows when to stop.

User-Owned Finding Triage

Self-correction handles the system’s detection. Triage handles its disposition. They are different mechanisms, and conflating them creates a particular failure mode in review skills: after dispatching panels and surfacing findings, the skill is tempted to triage them itself. It silently filters borderlines, defers Important findings to a future hardening release, or bundles the verdict with the user’s first sight of the finding list. Each move is locally reasonable: the panel’s emit threshold becomes a post-synthesis filter, and the skill claims it is saving the user time. Globally, each move is a defect. The user never sees what was hidden, and the calibration record drifts away from what panels actually found.

The methodology treats this as a structural defect class. Every Critical or Important finding is presented to the practitioner before any disposition is recorded. The skill may recommend “fix / defer / dismiss” per finding but must not decide. Forbidden patterns (silent suppression, blanket deferral under cover of a stricter criterion, bundling the verdict with the finding list) are spelled out verbatim and pinned by structural tests so the gate cannot be paraphrased away. A “fix all” authorisation does not extend to new Critical findings discovered during the fix; those re-enter triage.

This is a small rule with a large effect on trust. The verification report a practitioner reads is the same one the panels emitted. Anything the skill might have hidden, it cannot hide. The practitioner’s authority over disposition is not advisory; it is the gate.

Exploratory Testing as Verification

The architecture so far covers automated verification: tools and reviewers operating without human intervention beyond orchestration. A complementary mechanism uses human attention deliberately.

The error model is genuinely distinct from every layer above. No rule set, no random input, no LLM scan — a human running the software and noticing what is wrong. Scripted tests verify what someone thought to write down; exploratory testing is the discipline of looking for the paths nobody thought to write down. The defect classes it reaches are precisely those automated verification cannot anticipate: a sequence of operations no one wrote a test for, a state combination that no fixture covered, a usability failure that compiles, lints, and tests clean.

The methodology gives exploratory testing first-class pipeline status. The practitioner writes a YAML charter (mission, tour, scope, duration) and the skill runs the timer. Defects are captured through structured prompts: severity in the practitioner’s words, Given/When/Then in the practitioner’s framing, reproduction steps the practitioner verifies. The skill does not auto-classify severity; that is a judgment the practitioner makes about the risk of the defect in their context. The output is a paired session report that the build verification phase reads when computing the final verdict.

Charter format from Bach & Bolton, Rapid Software Testing; the time-box discipline from session-based test management. Severity is the practitioner’s call — the classification depends on context the practitioner has and the model does not (regulatory exposure, customer impact, the cost of the workaround). The skill records what the practitioner said.

Making It Portable

A verification architecture that only works for Python is not an architecture — it is a script. Most teams use multiple languages, and AI coding assistants generate code in whatever language the project requires. The verification architecture must be language-agnostic in principle but stack-specific in execution.

Dependency verification for Python (python3 -c "import {package}") is a different command than for Go (go build ./...), but the principle — verify that every dependency resolves against a real registry — is the same.

A stack-aware tooling reference maps each verification layer to concrete commands for each supported technology. The reference covers dependency verification, linting, security SAST, complexity analysis, credential scanning, property-based testing libraries, and mutation testing tools across ten language stacks: Python, TypeScript/JavaScript, Go, Rust, Java, C#, Ruby, PHP, Swift, and Kotlin.

For technologies not in the reference, a discovery process activates: identify the language's standard toolchain, find the most widely adopted tools for each verification category (preferring built-in over third-party, widely adopted over niche, actively maintained over stale), test the commands, and add them to the reference. The verification architecture self-extends to new stacks rather than requiring manual configuration for each one.

Not every stack has mature tooling for every layer. Swift, for example, lacks a widely adopted mutation testing tool. The architecture accommodates gaps honestly — skip what does not exist rather than pretend it does.

The release gate is what makes every preceding layer enforced rather than advisory. The pipeline produces verdicts; the release phase consumes them. A practitioner cannot tag a release without the pipeline having produced the evidence. Stale documentation is treated as a release defect, not an acceptable gap.

Pipeline steps run on different models. Architectural reasoning, requirements elicitation, and document review run on the primary reasoning model. Code generation, panel-level finding emission, and structured extraction run on Sonnet. HTML rendering runs on Haiku. The pipeline picks the right model for each step rather than running everything on the most expensive option — a structural contract between the skill and the runtime, not a per-invocation guess.

The build is structured as dependency-ordered chunks, each with its own TDD cycle and quality gates. Verification runs per-chunk, not as a monolithic step at the end. A defect caught in chunk 3 is fixed in chunk 3, before chunks 4 through N are built on top of it.

The progressive calibration and build check promotion operate across review rounds. On the first code review, findings are recorded. On the second, calibration begins. From the third round onwards, the blind reviewer activates automatically. Recurring findings promote to build checks that modify subsequent generation prompts. The pipeline gets better at preventing defects with each iteration.

What Remains Open

This architecture addresses the AI code quality problems that have published evidence and established tooling. Several areas remain open:

  • Model diversity in review. The expert-grounded review panels use different frameworks but typically the same model family. The grounding effect has been validated across Claude, GPT, and Gemini (see Grounding Produces Measurably Better Output), but cross-model review — using one model to review another's output — has not yet been tested. Genuine model diversity in review would reduce correlated errors further and is the theoretically correct approach as tooling constraints ease.
  • Mutation testing maturity. Not all language ecosystems have mature mutation testing tools. Swift has no widely adopted option. Go's tooling is limited. As these ecosystems mature, the fifth verification layer becomes more universally applicable.
  • Emerging failure modes. As AI coding capabilities evolve, new failure patterns will emerge that current layers do not target. The architecture is designed to be extensible — new layers can be added without restructuring existing ones — but identifying what those layers should be requires ongoing observation.
  • Quantifying layer effectiveness. The individual evidence (ISSTA for SAST coverage, USENIX for dependency hallucination rates) is strong, but no study has measured the compound effectiveness of all layers running together on AI-generated code. The multiplicative independence assumption is theoretically sound but empirically untested at the system level. A controlled study comparing defect escape rates with and without layered verification on AI-generated codebases would validate or refine the architecture.
  • Expert framing effectiveness. Observational data shows expert-framed panels catching meaningfully different defect categories with low cross-panel overlap. A controlled study comparing expert-framed versus generic LLM review would establish whether the pattern holds broadly.

Conclusion

AI-generated code fails differently from human-written code. The verification architecture must match. Layered verification with distinct error models catches what no single technique can, because each layer is blind to different things than the others.

The architecture spans five mechanism families. Tool-based layers catch defects in code that exists. Construction-time gates close defect classes before code reaches review: walking-skeleton boundary checks, the chunk-acceptance smoke, the wiring matrix, and MVP-first phase ordering refuse work that fails its preconditions, while mock-budget and realistic-fixture rules surface boundary discipline as findings the practitioner triages. Expert-grounded code review adds structural diversity within the LLM itself, separated by defect class. Exploratory testing adds a human error model that no automated layer can replicate. The self-correcting feedback loop combines progressive calibration, blind reviewer bias correction, build check promotion, a finding quality gate, and user-owned triage. A release gate aggregates every verdict and blocks the tag if any layer reports not-ready.

The result is a verification pipeline that improves with each iteration. Not by running the same checks harder, but by compounding the coverage of genuinely different detection mechanisms.
