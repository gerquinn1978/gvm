Building a Production Tool with GVM

A case study: how the Grounded Vibe Methodology built Assay, a blind-review experiment runner for LLM research — from requirements to release, with every artefact traced.

April 2026

The problem

We needed to test whether expert grounding — citing specific authorities in prompts — measurably improves AI output quality. The experiment required generating hundreds of artefacts across multiple LLM providers, reviewing each one blind (the reviewer cannot know which prompt produced it), extracting scores into structured data for statistical analysis, and doing all of this without a single methodological shortcut that would invalidate the results.

This is not a coding problem. It is a logistics and integrity problem. The blind-review constraint alone introduces shuffle mappings, anonymised IDs, and a separation between generation and review that must survive process crashes, budget exhaustion, and rate limiting. Add three providers (Anthropic, OpenAI, Google), concurrent API dispatch, cost tracking, and an audit trail, and you have a system where a single implementation error can silently corrupt an entire experiment.

We built Assay using the Grounded Vibe Methodology — the same methodology the experiment was designed to validate. This is the case study of that build.

What was built

Assay is a command-line tool that runs controlled experiments on LLM output. A researcher defines an experiment in YAML — producer groups with different prompts, review configurations with scoring rubrics, budget limits, concurrency settings — and Assay handles the rest.

Three commands cover the full workflow:

  • assay generate — dispatches prompts to LLM providers, stores artefacts, tracks cost, and persists state atomically so a crashed or paused run resumes exactly where it stopped.
  • assay review — shuffles artefacts into anonymised order, sends each to reviewers with only the rubric and the content (no group label, no index, no filename), so a reviewer cannot see which group produced an artefact (blindness), enforced at the protocol level.
  • assay extract — parses review output into CSV scores with a separate group-mapping file for post-hoc unblinding.

The tool supports Anthropic Claude, OpenAI GPT, and Google Gemini. It handles concurrent dispatch with per-provider semaphores, exponential backoff with interruptible sleep, budget enforcement that can abort mid-stream, and an append-only JSONL audit trail that records every API call with prompt hashes and blindness verification flags.

Assay was the tool used to run the three-model experiment reported in the companion paper Grounding Produces Measurably Better Output: 180 dashboards, 1,080 blind reviews, three models.

The final release ships with 38 requirements implemented (all MUST priorities met), 207 passing tests, 97 acceptance test cases verified, and zero critical or important code review findings.

The pipeline in practice

GVM structures a build as a sequence of phases, each producing artefacts that the next phase consumes. Here is how those phases played out for Assay.

Requirements

The pipeline begins with /gvm-requirements, which produces a structured requirements document grounded in Wiegers' elicitation framework and Patton's story mapping. For Assay, this yielded 38 requirements across seven domains: experiment definition, experiment execution, cost management, data output, audit trail, security, and logging & progress.

Every requirement carries a MoSCoW priority and a unique ID that traces forward through every subsequent phase. When a test case references TC-EP-5-01, you can trace it back to requirement EP-5 (reviewer blindness) without ambiguity. Here is what a GVM requirement looks like:

EP-5 Must

The review phase must not include any group identifier, group label, producer prompt, or other metadata that would reveal which group an artefact belongs to in the reviewer's prompt context. Reviewers see only the artefact content and the review rubric.

The language is specific enough to test directly: either the reviewer prompt contains group identifiers or it does not. This requirement produced three test cases (blindness verification, shuffle mapping isolation, and unblinding separation) and drove the architectural decision to anonymise artefacts at the protocol level rather than relying on prompt engineering.

Test cases before architecture

The test cases phase runs before the tech spec — not after. This is deliberate. If a requirement cannot be expressed as a concrete test with given/when/then conditions and property-based invariants, the requirement is the problem, and the cheapest time to discover that is before any architecture exists.

For Assay, /gvm-test-cases produced 97 acceptance test cases. These became the acceptance criteria for the build: every chunk of implementation had to pass its traced test cases before proceeding.

Tech spec and architecture decisions

The tech spec phase produced two spec documents and nine Architecture Decision Records (ADRs). The ADRs record not just what was decided but why, and what alternatives were rejected. When the code review later questioned the state persistence approach, the answer was already documented in ADR-004 (JSON with atomic temp-write-rename, chosen over SQLite for portability and crash safety).

ADR traceability in practice

Docstrings in the production code reference their ADRs directly: the state.py module cites ADR-004 and ADR-006. A reviewer reading the code can trace any design decision to its rationale without leaving the codebase.

Build: five phases, fourteen planned chunks plus a hardening tail

The build phase decomposes the spec into ordered chunks — each small enough to implement in a single session, each producing a testable increment. Assay's build ran across five phases. The first four planned fourteen chunks; the fifth phase added a hardening tail of three or more chunks that surfaced as the system stabilised:

Table 1 — Build phases and their deliverables

Each chunk follows a strict loop: generate the implementation prompt from the spec, write the code, run the tests, verify the chunk's acceptance criteria pass, and produce a handover document that gives the next session everything it needs to continue — even if the context window is exhausted.

Expert grounding across phases

GVM activates named experts at every phase. For Assay, the expert roster shifted as the pipeline progressed — the requirements phase needed different authorities than the build phase, and the review phase different authorities again.

Table 2 — Expert panels by pipeline phase
PhaseKey expertsWhat they governed
RequirementsWiegers, Patton, Cooper, RobertsonElicitation methodology, persona-grounded requirements, MoSCoW prioritisation
Test casesCopeland, Beizer, North, KanerTechnique selection (BVA, equivalence partitioning), BDD structure, exploratory scenarios

The experts shaped the code through the spec and review phases, not through construction. The patterns in Assay trace to experts who defined the architecture (tech spec) or enforced quality standards (code review). Construction is grounded in framework documentation — procedural knowledge about how to use the APIs correctly — not expert principles. This separation is a finding from the methodology itself (see the blog post, What we got wrong).

Specific patterns trace to specific phases:

  • Protocol-based provider abstraction (spec: Ramalho, Fluent Python; Fowler, Enterprise Patterns) — instead of an inheritance hierarchy, each provider satisfies a Python Protocol. Adding a new provider means implementing two methods. The spec phase defined this pattern from Ramalho's structural typing guidance and Fowler's plugin architecture principles.
  • Atomic writes everywhere (spec: Keeling, Design It!) — state files, artefacts, and reviews all use write-to-temp-then-rename. The pattern was specified in ADR-004 because the spec phase experts made crash safety a first-class architectural concern.
  • Frozen dataclasses with MappingProxyType (spec: Ramalho, Fluent Python) — all configuration models are immutable value objects. Concurrency-shared dictionaries are wrapped in MappingProxyType to prevent accidental mutation. Specified during the tech spec phase as a concurrency safety measure.
  • Shared dispatcher extraction (review: Hunt & Thomas, DRY principle) — the generator and reviewer initially had separate dispatch implementations. The code review flagged this as recurring duplication, and the build responded by extracting a shared Dispatcher class. This pattern was driven by the review phase, not the original spec.
  • Registry-based factory (spec: Fowler, Enterprise Patterns) — providers register in a dictionary rather than a conditional chain. Defined in the spec as the provider discovery mechanism.
The grounding thesis, demonstrated

These are not abstract principles applied after the fact. Every pattern listed above is visible in the production code, and each traces to a named expert whose work was cited during the phase that produced it. Protocol-based abstraction exists because Ramalho was active during the spec phase. Atomic writes exist because Keeling was active during architecture decisions. The code is the evidence. If you read the Assay source and then read the expert panel table, the connection is not inferred — it is structural. This is what expert grounding produces: not vague "best practices," but specific, identifiable patterns that would not be there without the citation.

Documentation, review, and release

After the build and code review, three final phases completed the pipeline:

/gvm-doc-write produced an operations guide covering experiment workflow, configuration patterns, troubleshooting, and output interpretation. /gvm-doc-review scored it 6.0/10 and found three accuracy issues — including a wrong claim about how the Google provider handles system prompts, and a code recipe that would raise a FileNotFoundError because pd.read_csv() does not support glob patterns. The review caught factual errors in the documentation that would have sent users down wrong paths.

/gvm-deploy aggregated all review verdicts (code review: Ship; doc review: 6.0/10 with important findings), verified that the README matched the implemented features, proposed version v0.1.0 based on semver analysis, generated release notes and a changelog, and created an annotated git tag. The release gate enforced everything upstream: if any review had blocked, the release would have blocked.

What the reviews caught

Assay went through seven rounds of code review. The review history is the strongest evidence that the pipeline's structure adds value — because the reviews caught failures that no other part of the process would have.

Second, total findings did not converge smoothly to zero. They spiked in round 5 because the streaming implementation introduced new surface area. The review system adapted to new code rather than assuming old verdicts still held.

The hallucination the review caught

In round 1, the Anthropic provider adapter used await client.messages.create() with streaming enabled. Correct syntax. Plausible parameters. The code compiled. Tests passed with a mocked client. But the correct call is a context manager (async with client.messages.stream()), not an await. The AI hallucinated the API, and it would have failed the first time it ran against the real SDK.

This specific example appears in the blog post AI Can Build Fully Engineered Software because it illustrates the core problem: plausible, compilable, test-passing code that is wrong.

The code review phase caught it. TDD did not — because the mocked client accepted the hallucinated call signature. The unit tests were correct tests of wrong code. Only a reviewer grounded in the Anthropic SDK's actual interface could see that the call pattern was fabricated.

The blind reviewer's catch

From round 3 onwards, GVM runs dual review: a calibrated reviewer (with full project history, resolved findings, anchor examples) and an uncalibrated blind reviewer (same criteria, same expert references, no history). Round 3 is the threshold because round 1 establishes the first scoring anchor, round 2 establishes a pattern of expectations, and by round 3 anchoring bias has had enough history to create systematic blind spots — the point at which rater drift becomes detectable in assessment research. The blind reviewer exists to catch what calibration bias causes the calibrated reviewer to miss.

In round 7, the calibrated panel was ready to ship. The blind reviewer found one critical issue: an await on an async generator that should have been iterated directly. The calibrated panel had seen similar patterns resolve correctly in prior rounds and did not flag it. The blind reviewer, seeing the codebase fresh, caught it.

Why this matters

Without dual review, round 7 would have shipped with a critical bug. The calibrated reviewer's familiarity with the codebase — normally an advantage — became a blind spot. The mechanism that corrected for this bias was structural, not a matter of reviewer skill. It was built into the process.

Build check promotion

Three findings recurred across five or more consecutive rounds, indicating systemic patterns rather than one-off mistakes:

  • BC-001: Provider names defined in two places (registry and validation list). Resolved in round 7 by exporting a single VALID_PROVIDERS frozenset from the factory module.
  • BC-002: Token summation duplicated across phases. Resolved by unifying via a CostTracker.total_tokens property.
  • BC-003: Resource-owning classes not using context managers. AuditLogger added __enter__/__exit__, but callers continued using bare instantiation. Still monitored.

When a finding recurs in three or more consecutive rounds, GVM promotes it to a build check — a concrete instruction that gets loaded into subsequent build prompts under "Known Patterns to Avoid." The build system learns from its own review history. This is the feedback loop from review to construction that most AI development processes lack.

Build checks are tiered at promotion time. All three of Assay's checks are tier 1 — general engineering practice (DRY, resource lifecycle, single source of truth) that applies to any project. Tier 1 checks are permanent. Tier 2 checks — tied to a specific library or domain convention — are flagged for retirement after three quiet rounds, but the user decides whether to remove them.

What surprised us

Review findings do not converge linearly

The expectation was a steady decline: each round fixes findings, the next round finds fewer. Instead, findings spiked in round 5 (from 19 to 30) because the streaming implementation introduced new code paths that the prior reviews had never seen. The review system correctly treated new code as new surface area requiring fresh scrutiny.

This matters for teams adopting GVM. If your review counts spike after a feature addition, the process is working — it means the reviews are actually re-evaluating the codebase rather than rubber-stamping prior verdicts.

The build improved its own architecture

The generator and reviewer phases shared identical dispatch logic: TaskGroup + Semaphore + pause/stop/budget/retry. The code review flagged the duplication as a recurring finding. By the time it was promoted to build check BC-002, the fix was clear: extract a shared Dispatcher class that both phases use with different callbacks. This eliminated roughly 150 lines of duplicated code and created a single place to maintain the concurrency and retry logic.

The review system drove this architectural improvement, not upfront design. The original spec had separate implementations for each phase — a reasonable choice at the time. The reviews identified the emerging duplication pattern and the build responded. This is what iterative convergence looks like in practice.

Mocked tests can pass with hallucinated code

The Anthropic streaming hallucination passed all unit tests because the mocked client accepted whatever call signature it was given. The tests were structurally correct — they verified the right behaviour — but they tested against a mock that didn't enforce the real SDK's interface.

The lesson: mocked tests verify your logic, not your integration. For API boundaries where the AI is likely to hallucinate the call pattern, you need either integration tests against the real SDK or a review phase that checks the actual API surface. Assay uses both: provider tests verify against current SDK interfaces, and the code review explicitly checks API usage patterns.

The result

Assay shipped as v0.1.0 on 4 April 2026. It has since been used to run the three-model experiment that produced the results reported in Grounding Produces Measurably Better Output: 180 dashboards generated across Claude, GPT, and Gemini, with 1,080 blind reviews and statistical analysis showing significant quality improvements from expert grounding (p < 0.001, all pairwise comparisons).

The tool that was built to test the methodology was itself built by the methodology. A process that produces traceable, reviewed, tested artefacts should be able to build any software system — including the tool that validates its own claims.

What we learned from building Assay with GVM

Let the test cases phase challenge your requirements. The test cases phase is not just about generating tests — it is a quality gate on the requirements themselves. If you cannot express a requirement as a concrete test, the requirement is ambiguous. We discovered this for two SHOULD requirements that were initially marked PARTIAL because their acceptance criteria were imprecise. The test cases phase would have caught this earlier if we had been more rigorous about flagging untestable language.

Expect review counts to fluctuate, not decline. New code introduces new findings. A spike after a feature addition is the system working, not failing. The trajectory that matters is critical findings — those should drop to zero and stay there.

ADRs pay off during review, not during design. Writing ADRs feels like overhead during the spec phase. Their value becomes clear during code review, when a reviewer questions a design choice and the answer is already documented with alternatives considered and rationale given. For Assay, ADR-004 (state persistence strategy) and ADR-007 (signal handling approach) were referenced multiple times during review rounds.

The build check mechanism needs time to mature. Build checks only trigger after three consecutive rounds of the same finding. For Assay, this meant the first two rounds could not benefit from build checks at all. On larger projects with more review rounds, the feedback loop from review to construction becomes more valuable over time. The methodology gets better the longer you use it.

Dual review earns its cost from round 3. The blind reviewer adds one parallel review per round. For rounds 1 and 2, this overhead is not justified — there is not enough calibration history to create meaningful bias. From round 3 onwards, the blind reviewer caught findings in every round that the calibrated panel missed, including the only critical finding in round 7. The cost is one extra review; the return is catching what your experienced reviewers have been trained to overlook.
